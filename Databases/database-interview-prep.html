🧱 Database Fundamentals
The building blocks every database interview starts with — storage, indexes, relational vs non-relational.
How Databases Store Data
At the lowest level, databases store data in pages (typically 8KB or 16KB). Pages are organized into tablespaces on disk. When you query, the DB reads pages into a buffer pool (in-memory cache). The goal: minimize disk I/O by keeping hot pages in memory.

B-Tree — The Foundation of Every Index
Almost all database indexes use a B-Tree (Balanced Tree). It keeps data sorted and enables: exact lookups O(log N), range queries, ORDER BY without sorting. Each node holds multiple keys, keeping the tree shallow even for billions of rows.
💡 ANALOGY
A B-Tree is like a phone book with table of contents. The table of contents (root node) tells you which section to open. Each section header (internal nodes) narrows it down further. The actual entries (leaf nodes) have the data. You never read the whole book — you jump directly to the right page in O(log N) steps.
Index Type Structure Best for NOT good for
B-Tree Balanced tree, sorted Equality, ranges, ORDER BY, LIKE 'abc%' LIKE '%abc' (leading wildcard)
Hash Hash table Exact equality only (=) Range queries, ORDER BY
Full-text Inverted index Text search (MATCH AGAINST) Numeric ranges
Spatial (R-Tree) Rectangle tree Geospatial queries (ST_Within) Non-spatial data
Bitmap Bit arrays Low cardinality (status, gender) High cardinality
🎯 INTERVIEW QUESTIONS
Q: What is a database index and what are the tradeoffs?
An index is a separate data structure (usually B-Tree) that maps column values to row locations, enabling fast lookups without scanning all rows. Tradeoffs: READS get faster (O(log N) vs O(N) full scan). WRITES get slower — every INSERT/UPDATE/DELETE must also update all relevant indexes. Storage increases — indexes consume disk space (sometimes as much as the table). Too many indexes can hurt write-heavy tables more than the read benefit is worth. Rule: index columns used in WHERE, JOIN ON, ORDER BY, GROUP BY — but profile first.
Q: What is the difference between clustered and non-clustered indexes?
Clustered index: the table rows are physically stored in the index order. There can be only ONE clustered index per table (it IS the table). In MySQL InnoDB, the primary key is always the clustered index. Lookup by PK is extremely fast — no extra step needed. Non-clustered index: a separate structure pointing to rows. After finding the key in the index, there's an extra "lookup" to fetch the actual row (unless it's a covering index). A table can have many non-clustered indexes.

⚖️ ACID & Transactions
The most important database interview topic. Isolation levels, deadlocks, and what every developer must know about transactions.
ACID Properties
Property Meaning What guarantees it
Atomicity All operations in a transaction succeed, or all are rolled back — never partial. Transaction log (WAL / undo log)
Consistency DB moves from one valid state to another. All constraints satisfied. Constraints, triggers, app logic
Isolation Concurrent transactions don't interfere with each other. Locks, MVCC
Durability Committed transactions survive crashes — written to persistent storage. Write-Ahead Logging (WAL)
💡 BANK TRANSFER ANALOGY
Atomicity: Transfer $100 from A to B — both debit AND credit happen, or neither.
Consistency: Total money before transfer = total money after (conservation).
Isolation: Two simultaneous transfers from A don't see each other's intermediate states.
Durability: Once "Transfer confirmed" — even a power cut can't lose it.
Isolation Levels — Critical for Interviews
Level Dirty Read Non-repeatable Read Phantom Read Performance
READ UNCOMMITTED ❌ Possible ❌ Possible ❌ Possible Fastest
READ COMMITTED ✅ Prevented ❌ Possible ❌ Possible Fast (default in PG)
REPEATABLE READ ✅ Prevented ✅ Prevented ❌ Possible* Medium (default MySQL)
SERIALIZABLE ✅ Prevented ✅ Prevented ✅ Prevented Slowest
SQL — TRANSACTION EXAMPLE 
-- Set isolation level
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

START TRANSACTION;

-- Check balance
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE; -- lock row!

-- Deduct from sender
UPDATE accounts SET balance = balance - 100 WHERE id = 1;

-- Add to receiver
UPDATE accounts SET balance = balance + 100 WHERE id = 2;

COMMIT; -- both changes permanent
-- or ROLLBACK; -- undo everything
Read Anomalies Explained
Dirty Read: Transaction A reads uncommitted changes from Transaction B. If B rolls back, A has read data that never existed.

Non-repeatable Read: Transaction A reads a row twice, gets different values — Transaction B updated it between reads.

Phantom Read: Transaction A queries rows matching a condition twice, gets different rows — Transaction B inserted/deleted matching rows between the two queries.
Deadlocks
Deadlock: Transaction A locks row 1, waits for row 2. Transaction B locks row 2, waits for row 1. Both wait forever. DBs detect this and kill one transaction (the "victim").

Prevention: Always acquire locks in the same ORDER across all transactions. Keep transactions short. Use SELECT FOR UPDATE carefully.
🎯 INTERVIEW QUESTIONS
Q: What are the four ACID properties?
Atomicity: all-or-nothing — a transaction's operations all succeed or all fail together. Consistency: the database moves from one valid state to another, maintaining all defined rules and constraints. Isolation: concurrent transactions execute as if they were serial — they don't see each other's intermediate states (level of isolation is configurable). Durability: once committed, data survives system failures — written to non-volatile storage via WAL.
Q: What is the difference between optimistic and pessimistic locking?
Pessimistic locking: lock the resource before reading — assumes conflicts will happen. Use SELECT FOR UPDATE to lock rows. Transaction holds the lock until commit. Good for: high-contention write scenarios. Downside: reduced concurrency, deadlock risk. Optimistic locking: read without locking, check at write time if data changed (using a version column). If changed, retry. Good for: read-heavy, low-contention scenarios. Downside: retry logic complexity. Most web apps use optimistic locking for better scalability.
Q: What is MVCC?
Multi-Version Concurrency Control keeps multiple versions of each row. When a transaction modifies a row, the DB creates a new version instead of overwriting. Readers see the version that was current when their transaction started. This means readers don't block writers and writers don't block readers — dramatically improving concurrency. PostgreSQL and InnoDB (MySQL) both use MVCC. The tradeoff: old versions accumulate and must be cleaned up (VACUUM in PostgreSQL, purge in InnoDB).

🗂️ Indexing Deep Dive
The single most impactful performance topic. Composite, covering, partial indexes — and when NOT to index.
Index Design Rules
SQL — INDEX PATTERNS 
-- Single column index
CREATE INDEX idx_email ON users (email);
-- Good for: WHERE email = 'x'  ORDER BY email

-- Composite index — COLUMN ORDER MATTERS!
CREATE INDEX idx_user_status_date ON orders (user_id, status, created_at);
-- Supports: WHERE user_id = 1
--           WHERE user_id = 1 AND status = 'active'
--           WHERE user_id = 1 AND status = 'active' AND created_at > '2024-01-01'
-- Does NOT help: WHERE status = 'active'  ← skips leading column!
-- Does NOT help: WHERE created_at > '...' ← skips two columns!

-- Covering index — includes all columns query needs (no table lookup)
CREATE INDEX idx_covering ON orders (user_id, status)
  INCLUDE (total_amount, created_at);   -- PG syntax
-- SELECT total_amount, created_at WHERE user_id=1 AND status='paid'
-- → satisfied entirely from index, NO table read!

-- Partial index — index only a subset of rows
CREATE INDEX idx_pending ON orders (created_at)
  WHERE status = 'pending';  -- only indexes pending orders (smaller, faster)

-- Functional index
CREATE INDEX idx_lower_email ON users (LOWER(email));
-- WHERE LOWER(email) = 'hany@example.com'  ← now uses index
The Left-Prefix Rule
Critical Rule: For a composite index on (A, B, C), the DB can use it for queries involving A, A+B, or A+B+C. It CANNOT skip leading columns. If you query only on B or C, the index is NOT used. This is the most common indexing mistake in interviews.
When Indexes Hurt
Don't over-index! Each index means: INSERT creates N+1 writes (one per index). UPDATE may update multiple indexes. DELETE cleans up all indexes. For write-heavy tables (logs, events), too many indexes can make writes 2-5x slower. Also: indexes on low-cardinality columns (status with 3 values on a large table) may not help — the DB might prefer a full scan anyway. Always measure with EXPLAIN.
Situation Index helps? Why
WHERE status='active' (99% of rows) ❌ No Full scan is faster for high % of rows
WHERE email = 'x@x.com' ✅ Yes High cardinality, unique value
LIKE '%searchterm%' ❌ No Leading wildcard prevents B-tree use
LIKE 'search%' ✅ Yes Trailing wildcard works with B-tree
WHERE YEAR(created_at) = 2024 ❌ No Function on column prevents index use
WHERE created_at >= '2024-01-01' ✅ Yes Direct column comparison
🎯 INTERVIEW QUESTIONS
Q: What is a covering index?
A covering index includes all columns a query needs, so the query can be satisfied entirely from the index without reading the actual table (no "table lookup" or "heap fetch"). This is extremely fast because the index is smaller and often cached. Example: index on (user_id, status) with INCLUDE (total_amount) — a query SELECT total_amount WHERE user_id=1 AND status='paid' reads only the index. Create covering indexes for your most frequent, performance-critical queries.
Q: Why does a function call on an indexed column prevent index usage?
WHERE YEAR(created_at) = 2024 cannot use an index on created_at because the DB would need to apply YEAR() to every row's value to find matches — the index is sorted by the raw value, not the computed value. Fix: rewrite to a range — WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'. Or create a functional/expression index: CREATE INDEX ON orders (YEAR(created_at)). This is a very common performance anti-pattern.

📐 Normalization
Organizing data to reduce redundancy and improve integrity — and when to deliberately break the rules.
Normal Forms
Form Rule Eliminates
1NF All column values are atomic (no repeating groups, no arrays in cells) Repeating groups
2NF 1NF + every non-key column depends on the WHOLE primary key (no partial dependencies) Partial dependencies
3NF 2NF + no transitive dependencies (non-key columns don't depend on other non-key columns) Transitive dependencies
BCNF Every determinant is a candidate key Anomalies in 3NF tables with overlapping keys
Denormalization — When to Break the Rules
Normalization is great for write integrity but can hurt read performance — lots of JOINs for common queries. Denormalization deliberately adds redundancy to improve read speed.
Good reasons to denormalize: A query JOINing 5 tables runs 10K/sec — flatten into one table. Reporting/analytics tables (OLAP) benefit from denormalized star schemas. Read-heavy data that changes rarely (user's country stored in orders table). Caching computed values (order_total stored instead of recalculated).
Denormalization risks: Update anomalies — if user changes email, must update all denormalized copies. Data inconsistency if updates miss some copies. Increased storage. Make sure you have the READS to justify it.
🎯 INTERVIEW QUESTIONS
Q: Explain 3NF with an example.
3NF: no transitive dependencies. Example: Order table has (order_id, customer_id, customer_city, customer_country). customer_city → customer_country is a transitive dependency (city determines country, not order_id). This is a 3NF violation — if a customer moves cities, we must update all their orders. Fix: move city/country to a Customers table. Orders only stores customer_id (FK). Now city and country depend directly on customer_id, not transitively through city.
Q: When would you choose denormalization over strict normalization?
Denormalize when: (1) A normalized query requires many JOINs and runs too slowly under load, (2) Read-to-write ratio is very high (reads >> writes), (3) The duplicated data changes rarely (e.g., product category name in an orders table), (4) Analytics/reporting databases (OLAP) — star/snowflake schema with fact and dimension tables. Always normalize first, then denormalize specific queries that prove to be bottlenecks. Don't prematurely denormalize.

SQL — MYSQL TABLE DESIGN 
CREATE TABLE users (
  id         BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  email      VARCHAR(255) NOT NULL UNIQUE,
  name       VARCHAR(100) NOT NULL,
  status     ENUM('active', 'inactive', 'banned') DEFAULT 'active',
  metadata   JSON,                        -- MySQL 5.7+ native JSON
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- utf8mb4: full Unicode including emoji (utf8 in MySQL is only 3-byte!)

CREATE TABLE orders (
  id         BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  user_id    BIGINT UNSIGNED NOT NULL,
  total      DECIMAL(10,2) NOT NULL,    -- use DECIMAL not FLOAT for money!
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
🎯 INTERVIEW QUESTIONS
Q: Why use BIGINT for IDs instead of INT?
INT UNSIGNED max value is ~4.29 billion. Sounds like a lot, but high-traffic apps can hit this in years. Twitter famously ran into this problem. BIGINT UNSIGNED supports ~18.4 quintillion — effectively unlimited. The performance difference is negligible. Always use BIGINT for IDs in production systems that might grow. Also consider using UUID or Snowflake IDs for distributed systems to avoid ID collision across multiple database instances.
Q: Why never use FLOAT for monetary values?
FLOAT and DOUBLE use binary floating-point representation, which cannot exactly represent many decimal fractions. 0.1 + 0.2 = 0.30000000000000004 in floating-point. For money, even a fraction of a cent error is unacceptable — accumulated over millions of transactions it becomes significant. Use DECIMAL(10,2) — it stores exact decimal values. Or store amounts as integers (cents instead of dollars): 1099 cents instead of $10.99 — this is what Stripe and most payment systems do.

🔗 JOINs & Advanced Queries
Every JOIN type, CTEs, window functions, subqueries — the SQL you'll be asked to write in interviews.
All JOIN Types
SQL — JOIN TYPES 
-- INNER JOIN — only matching rows in BOTH tables
SELECT u.name, o.total
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- LEFT JOIN — all users, with order data if it exists (NULL if not)
SELECT u.name, o.total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;
-- Users with NO orders: name=Hany, total=NULL

-- Find users with NO orders (anti-join pattern)
SELECT u.name FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;   -- NULL means no matching order!

-- SELF JOIN — join table to itself
SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;
CTEs & Window Functions
SQL — CTE + WINDOW FUNCTIONS 
-- CTE (Common Table Expression) — readable temp result
WITH monthly_revenue AS (
  SELECT
    DATE_FORMAT(created_at, '%Y-%m') AS month,
    SUM(total) AS revenue
  FROM orders
  WHERE status = 'completed'
  GROUP BY month
),
top_months AS (                         -- chain CTEs
  SELECT * FROM monthly_revenue
  WHERE revenue > 100000
)
SELECT * FROM top_months ORDER BY revenue DESC;

-- Window functions — calculations across related rows
SELECT
  name,
  salary,
  RANK() OVER (ORDER BY salary DESC) AS salary_rank,
  DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rank,
  ROW_NUMBER() OVER (ORDER BY salary DESC) AS row_num,
  SUM(salary) OVER (PARTITION BY dept_id) AS dept_total,
  LAG(salary, 1) OVER (ORDER BY salary) AS prev_salary
FROM employees;

-- Running total
SELECT date, revenue,
  SUM(revenue) OVER (ORDER BY date ROWS UNBOUNDED PRECEDING) AS running_total
FROM daily_sales;
🎯 INTERVIEW QUESTIONS
Q: What is the difference between RANK(), DENSE_RANK(), and ROW_NUMBER()?
All three number rows within a window partition. ROW_NUMBER(): always unique sequential numbers 1,2,3,4 — no ties. RANK(): tied rows get the same rank, then skips — 1,2,2,4 (no rank 3 after two rank-2s). DENSE_RANK(): tied rows get same rank, no skipping — 1,2,2,3. Use ROW_NUMBER for pagination (always unique), RANK/DENSE_RANK for leaderboards and competitions. Interview trick: "find top 3 earners per department" — use DENSE_RANK with PARTITION BY dept.
Q: What's the difference between WHERE and HAVING?
WHERE filters BEFORE aggregation — it filters individual rows. You cannot use aggregate functions in WHERE. HAVING filters AFTER aggregation — it filters groups. You can use aggregate functions in HAVING. Execution order: FROM → WHERE (filter rows) → GROUP BY → HAVING (filter groups) → SELECT → ORDER BY. Example: WHERE salary > 50000 filters employees before grouping. HAVING COUNT(*) > 5 filters departments that have more than 5 employees after grouping.

⚡ MySQL Performance & EXPLAIN
Reading query execution plans, identifying bottlenecks, and applying the right fixes.
EXPLAIN — Reading Query Plans
SQL — EXPLAIN OUTPUT 
EXPLAIN SELECT * FROM orders WHERE user_id = 123 AND status = 'active';

-- Key columns to check:
-- type: access method (best → worst)
--   system → const → eq_ref → ref → range → index → ALL
-- ALL = FULL TABLE SCAN = bad! Fix with an index.
-- ref/range = index usage = good

-- rows: estimated rows examined (lower is better)
-- key: which index was used (NULL = no index!)
-- Extra:
--   "Using index"  = covering index ✅ (fast)
--   "Using where"  = filter applied after index
--   "Using filesort" = extra sort step ⚠️ (add index on ORDER BY col)
--   "Using temporary" = temp table ⚠️ (avoid GROUP BY without index)

-- Full format with cost estimates
EXPLAIN FORMAT=JSON SELECT ...;
EXPLAIN ANALYZE SELECT ...;  -- MySQL 8.0+ actually runs it
Common Performance Fixes
SQL — QUERY OPTIMIZATION 
-- ❌ SLOW: SELECT * fetches unnecessary columns
SELECT * FROM users WHERE email = 'x@x.com';

-- ✅ FAST: select only what you need
SELECT id, name FROM users WHERE email = 'x@x.com';

-- ❌ SLOW: N+1 problem (loop query in app code)
-- for each user: SELECT * FROM orders WHERE user_id = ?

-- ✅ FAST: single JOIN
SELECT u.id, u.name, COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id;

-- ❌ SLOW: function on indexed column kills index
SELECT * FROM orders WHERE YEAR(created_at) = 2024;

-- ✅ FAST: range on the column itself
SELECT * FROM orders
WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';

-- ❌ SLOW: OR on different columns
SELECT * FROM users WHERE email = 'x' OR phone = 'y';

-- ✅ FAST: UNION (uses each index separately)
SELECT * FROM users WHERE email = 'x'
UNION
SELECT * FROM users WHERE phone = 'y';
🎯 INTERVIEW QUESTIONS
Q: What does "Using filesort" in EXPLAIN mean and how do you fix it?
"Using filesort" means MySQL needs to sort the result set using an extra sorting operation (not necessarily file-based, despite the name). This is slow for large datasets. Fix: add an index on the ORDER BY column(s). If your query is WHERE status='active' ORDER BY created_at, create a composite index on (status, created_at). The query can use the index both for filtering and for sorted order, eliminating the filesort. For GROUP BY, same principle — index on the GROUP BY column(s).
Q: What is the slow query log and how do you use it?
The slow query log records queries that exceed a time threshold (long_query_time, default 10s — set to 0.5s or 1s in practice). Enable with: SET GLOBAL slow_query_log = 'ON'; SET GLOBAL long_query_time = 1; Analyze with pt-query-digest (Percona Toolkit) to find the top time-consuming query patterns. This is the primary tool for production performance investigation — shows real-world slow queries, not theoretical ones.

Table Partitioning
SQL — PARTITIONING 
-- RANGE partitioning — great for time-series data
CREATE TABLE events (
  id         BIGINT NOT NULL,
  event_date DATE NOT NULL,
  data       JSON
)
PARTITION BY RANGE (YEAR(event_date)) (
  PARTITION p2022 VALUES LESS THAN (2023),
  PARTITION p2023 VALUES LESS THAN (2024),
  PARTITION p2024 VALUES LESS THAN (2025),
  PARTITION pmax  VALUES LESS THAN MAXVALUE
);

-- Querying only 2024 partition (partition pruning!)
SELECT * FROM events
WHERE event_date BETWEEN '2024-01-01' AND '2024-12-31';
-- Only scans p2024 partition — ignores all others!

-- Drop old partition (instant operation!)
ALTER TABLE events DROP PARTITION p2022;
-- vs DELETE FROM events WHERE YEAR(event_date) = 2022; ← slow!
🎯 INTERVIEW QUESTIONS
Q: What is replication lag and how do you mitigate it?
Replication lag is the delay between a write on the primary and it appearing on replicas. Causes: heavy write load on primary, slow network, replica hardware slower than primary. Mitigation: (1) Read-your-own-writes: route reads to primary for 1-2s after a write, (2) Synchronous replication (semi-sync): primary waits for at least one replica to acknowledge before committing — reduces lag but slows writes, (3) Scale replicas: more replicas share the replication work, (4) Parallel replication: MySQL 5.7+ applies relay log in parallel by schema or transaction.

📄 MongoDB Core
Documents, BSON, CRUD operations, and the fundamental MongoDB concepts every interview will test.
Documents & Collections
MongoDB stores data as BSON documents (Binary JSON) in collections. Unlike SQL tables, collections don't enforce a schema — each document can have different fields.
MONGODB — CRUD 
// INSERT
db.users.insertOne({
  name: "Hany",
  email: "hany@example.com",
  age: 30,
  tags: ["developer", "backend"],  // arrays native
  address: {                            // nested document
    city: "Cairo",
    country: "Egypt"
  }
});

// FIND — query operators
db.users.find({ age: { $gte: 25, $lt: 35 } });     // range
db.users.find({ tags: "developer" });              // match array element
db.users.find({ "address.city": "Cairo" });        // nested field
db.users.find({ $or: [{ age: 30 }, { age: 25 }] }); // OR
db.users.find({ email: { $regex: /gmail\.com$/ } }); // regex

// Projection — select fields
db.users.find({}, { name: 1, email: 1, _id: 0 }); // 1=include, 0=exclude

// UPDATE
db.users.updateOne(
  { _id: ObjectId("...") },
  { $set: { name: "Hany Updated" }, $inc: { age: 1 } }
);
db.users.updateMany({ status: "inactive" }, { $set: { archived: true } });

// DELETE
db.users.deleteOne({ _id: ObjectId("...") });
db.users.deleteMany({ createdAt: { $lt: new Date('2020-01-01') } });
Common Update Operators
Operator What it does Example

$unset Remove a field {$unset: {oldField: ""}}
$inc Increment a number {$inc: {views: 1}}
$push Append to array {$push: {tags: "new"}}
$addToSet Add to array if not exists (unique) {$addToSet: {tags: "dev"}}
$pull Remove from array by value {$pull: {tags: "old"}}
$rename Rename a field {$rename: {old: "new"}}
🎯 INTERVIEW QUESTIONS
Q: What is BSON and how does it differ from JSON?
BSON (Binary JSON) is MongoDB's internal storage format. Differences from JSON: (1) Binary encoded — more efficient for storage and traversal, (2) Supports additional types not in JSON: Date, Binary, ObjectId, Decimal128, Int32, Int64 — JSON only has Number (double), (3) Includes length prefixes — can skip fields without parsing entire document, (4) Not human-readable (unlike JSON). When you use MongoDB drivers, they automatically serialize/deserialize between your language objects and BSON.
Q: What is an ObjectId?
ObjectId is MongoDB's default _id type — a 12-byte BSON type: 4 bytes timestamp (seconds since epoch), 5 bytes random value (machine + process), 3 bytes incrementing counter. This makes ObjectIds: globally unique without coordination between nodes, sortable by creation time (ObjectId embeds timestamp), lightweight to generate (no DB roundtrip needed). You can extract creation time: new ObjectId().getTimestamp(). Used in sharded clusters to avoid ID conflicts across shards.

📊 MongoDB Schema Design
The most important MongoDB decision: embed vs reference. Real patterns used by production systems.
Embed vs Reference
MONGODB — EMBED VS REFERENCE 
// ✅ EMBED — when data belongs to parent, accessed together
{
  _id: ObjectId("..."),
  name: "Hany",
  address: {              // embedded — 1 read to get user + address
    street: "123 Main",
    city: "Cairo"
  },
  phones: [               // embed arrays too
    { type: "mobile", number: "01011..." }
  ]
}
// Pros: 1 read, atomicity, no $lookup needed
// Cons: document grows large, can't query embedded alone

// ✅ REFERENCE — when data is independent, accessed separately
// users collection:
{ _id: ObjectId("user1"), name: "Hany" }

// orders collection (references user):
{
  _id: ObjectId("order1"),
  userId: ObjectId("user1"),  // reference
  items: [...],
  total: 150.00
}
// Pros: independent queries, smaller docs, no data duplication
// Cons: requires $lookup (JOIN equivalent), 2 reads
Decision Framework
Embed when Reference when
Data belongs to parent (address → user) Data has independent lifecycle (orders exist after user deletion)
Always accessed together Accessed independently
One-to-one relationship Many-to-many relationship
One-to-few (user has 1-5 addresses) One-to-many unbounded (user has millions of events)
Child doesn't need querying alone You query the child collection directly
⚠️ Document size limit: 16MB. Never embed unbounded arrays — a user's entire order history, all comments on a post, etc. These will hit the 16MB limit. Reference instead, or use the Bucket Pattern (group related data: {month: "2024-01", events: [...50 events]}).
🎯 INTERVIEW QUESTIONS
Q: How do you model a many-to-many relationship in MongoDB?
Option 1: Store arrays of references in both documents. User: {following: [userId1, userId2]}, Post: {likes: [userId1, userId3]}. Simple but arrays can grow large. Option 2: Intermediate collection (like SQL junction table). {userId: X, postId: Y, likedAt: Date} — this works well for large-scale many-to-many with metadata. Option 3: Denormalize (embed summaries). Post: {likes_count: 1203, recent_likers: [...last 5 users]}. Choice depends on query patterns and relationship scale.
Q: How does MongoDB handle transactions?
Since MongoDB 4.0 (replica sets) and 4.2 (sharded clusters), MongoDB supports multi-document ACID transactions similar to SQL. Use for: operations that must be atomic across multiple documents/collections (transfer between accounts). Single-document operations are ALWAYS atomic — an update to one document is atomic even without a transaction. Transactions in MongoDB have overhead — design schemas to avoid needing them by embedding related data. When you do need them: session.startTransaction(), try operations, session.commitTransaction().

🔄 Aggregation Pipeline
MongoDB's powerful data transformation pipeline — the equivalent of SQL's GROUP BY, JOIN, and computed columns combined.
Pipeline Stages
MONGODB — AGGREGATION 
db.orders.aggregate([
  // 1. $match — filter documents (like WHERE) — put FIRST for performance
  { $match: { status: "completed", createdAt: { $gte: new Date("2024-01-01") } } },

  // 2. $lookup — LEFT JOIN another collection
  { $lookup: {
    from: "users",
    localField: "userId",
    foreignField: "_id",
    as: "user"         // result is an array!
  }},

  // 3. $unwind — flatten array from $lookup
  { $unwind: "$user" },

  // 4. $group — GROUP BY equivalent
  { $group: {
    _id: "$user._id",         // group by user
    userName: { $first: "$user.name" },
    totalSpent: { $sum: "$total" },
    orderCount: { $sum: 1 },
    avgOrder: { $avg: "$total" },
    maxOrder: { $max: "$total" }
  }},

  // 5. $project — shape output (like SELECT)
  { $project: {
    _id: 0,
    userId: "$_id",
    userName: 1,
    totalSpent: { $round: ["$totalSpent", 2] },
    orderCount: 1
  }},

  // 6. $sort — ORDER BY
  { $sort: { totalSpent: -1 } },

  // 7. $limit / $skip — pagination
  { $limit: 10 }
]);
Key Aggregation Operators
Stage SQL Equivalent Key use
$match WHERE Filter — always put first!
$group GROUP BY Aggregate ($sum, $avg, $count, $push)
$project SELECT (fields + computed) Reshape, compute new fields
$sort ORDER BY Sort results
$lookup LEFT JOIN Join another collection
$unwind Flatten (no SQL equiv) Expand array to multiple docs
$limit / $skip LIMIT / OFFSET Pagination
$facet Multiple GROUP BY (no SQL equiv) Multi-dimensional aggregation
🎯 INTERVIEW QUESTIONS
Q: Why should $match be placed first in the pipeline?
$match early reduces the number of documents flowing through subsequent stages. If you have 10M documents and filter to 1,000 with $match, subsequent $lookup, $group, $sort operate on 1,000 docs instead of 10M — orders of magnitude faster. Also, $match at the start can use indexes. If $match comes after $group or $project, the query has already abandoned index usage. Golden rule: filter early, project late.

Index types, explain(), and the patterns that make MongoDB fast at scale.
MongoDB Index Types
MONGODB — INDEXES 
// Single field
db.users.createIndex({ email: 1 }, { unique: true });
// 1 = ascending, -1 = descending

// Compound index — order matters (same left-prefix rule as SQL)
db.orders.createIndex({ userId: 1, status: 1, createdAt: -1 });

// Multikey index — automatically created when indexing array field
db.posts.createIndex({ tags: 1 });
// Supports: find({ tags: "mongodb" }) ← finds docs where array contains value

// Text index — full-text search
db.articles.createIndex({ title: "text", body: "text" });
db.articles.find({ $text: { $search: "mongodb performance" } });

// TTL index — auto-delete documents after expiry
db.sessions.createIndex({ createdAt: 1 }, { expireAfterSeconds: 3600 });
// Sessions automatically deleted 1 hour after createdAt

// Partial index — only index documents matching a filter
db.orders.createIndex(
  { userId: 1, createdAt: -1 },
  { partialFilterExpression: { status: "pending" } }
);

// Analyze query performance
db.orders.find({ userId: 123 }).explain("executionStats");
// Look for: COLLSCAN (bad) vs IXSCAN (good)
// totalDocsExamined should ≈ nReturned
🎯 INTERVIEW QUESTIONS
Q: What is a TTL index in MongoDB?
A TTL (Time-To-Live) index automatically deletes documents after a specified time. Create with: createIndex({createdAt: 1}, {expireAfterSeconds: 86400}). MongoDB runs a background thread every 60 seconds that removes expired documents. Uses: session storage, cache collections, temporary data, logs with retention policies, OTP codes. The field must be a Date type. Note: TTL indexes only work on single-field indexes, not compound indexes.
Q: What is the WiredTiger storage engine?
WiredTiger is MongoDB's default storage engine since MongoDB 3.2. Key features: document-level concurrency control (not collection-level like MMAPv1), compression (snappy by default — reduces disk by 80%+), MVCC for concurrent reads and writes, checkpoint-based durability with journaling. In practice: WiredTiger's document-level locking means multiple operations on different documents in the same collection don't block each other — critical for high-concurrency workloads.

MVCC & VACUUM
PostgreSQL's MVCC never overwrites or deletes rows in place. Instead, it creates new versions (tuple versions). Old versions ("dead tuples") accumulate and must be cleaned by VACUUM.
SQL — POSTGRESQL ESSENTIALS 
-- Sequences (PostgreSQL's auto-increment)
CREATE TABLE users (
  id    BIGSERIAL PRIMARY KEY,   -- BIGSERIAL = BIGINT + sequence
  email TEXT NOT NULL UNIQUE,
  data  JSONB                     -- JSONB: binary JSON, indexable
);

-- ENUM type
CREATE TYPE order_status AS ENUM ('pending', 'paid', 'shipped');

-- Schemas (namespaces — like databases within a database)
CREATE SCHEMA analytics;
CREATE TABLE analytics.events (...);

-- VACUUM — reclaim space from dead tuples
VACUUM users;           -- reclaim space, non-blocking
VACUUM FULL users;      -- compact table, LOCKS TABLE (use carefully!)
VACUUM ANALYZE users;   -- vacuum + update statistics for planner

-- Check table bloat
SELECT relname, n_dead_tup, n_live_tup,
  round(n_dead_tup::numeric / nullif(n_live_tup,0) * 100, 2) AS dead_pct
FROM pg_stat_user_tables ORDER BY n_dead_tup DESC;
🎯 INTERVIEW QUESTIONS
Q: What is VACUUM in PostgreSQL and why is it needed?
PostgreSQL's MVCC never deletes rows in place — it marks old versions as "dead tuples." These accumulate and waste disk space. VACUUM reclaims this space and updates statistics used by the query planner. autovacuum runs automatically in the background. VACUUM FULL compacts the table to minimum size but requires an exclusive lock. Without regular VACUUM on write-heavy tables: (1) table bloat — 10GB table grows to 100GB, (2) index bloat, (3) transaction ID wraparound (critical bug preventing all writes). Monitor with pg_stat_user_tables.n_dead_tup.
Q: What is the difference between TEXT and VARCHAR in PostgreSQL?
In PostgreSQL, TEXT and VARCHAR(n) are stored identically — there's no performance difference. VARCHAR(n) adds a length constraint. TEXT has no length limit. The PostgreSQL documentation actually recommends TEXT or VARCHAR without a length limit over VARCHAR(n) unless you have a specific reason for the constraint. This is different from MySQL where VARCHAR has an internal representation that differs from TEXT types.

🚀 PostgreSQL Advanced Features
JSONB, recursive CTEs, partitioning, full-text search, and the features that make PostgreSQL uniquely powerful.
JSONB — Best of Both Worlds
SQL — JSONB 
-- Store dynamic data in JSONB column
CREATE TABLE products (
  id       BIGSERIAL PRIMARY KEY,
  name     TEXT NOT NULL,
  metadata JSONB              -- flexible attributes
);

INSERT INTO products (name, metadata) VALUES
('Laptop', '{"brand":"Dell","ram":16,"tags":["electronics","computing"]}');

-- Query JSON fields
SELECT metadata->>>'brand' AS brand  -- ->> returns text
FROM products
WHERE metadata->>>'brand' = 'Dell'
  AND (metadata->>>'ram')::int >= 16;

-- Array containment
SELECT * FROM products
WHERE metadata @> '{"tags": ["electronics"]}';  -- @> = contains

-- GIN index on JSONB (makes all queries above fast)
CREATE INDEX idx_metadata ON products USING GIN (metadata);

-- Recursive CTE — hierarchical data (categories, org charts)
WITH RECURSIVE category_tree AS (
  -- Base case: root categories
  SELECT id, name, parent_id, 0 AS depth
  FROM categories WHERE parent_id IS NULL

  UNION ALL

  -- Recursive case: children
  SELECT c.id, c.name, c.parent_id, ct.depth + 1
  FROM categories c
  JOIN category_tree ct ON c.parent_id = ct.id
)
SELECT REPEAT('  ', depth) || name AS tree
FROM category_tree;
Table Partitioning
SQL — DECLARATIVE PARTITIONING 
-- Declarative partitioning (PostgreSQL 10+)
CREATE TABLE measurements (
  id          BIGSERIAL,
  sensor_id   INT NOT NULL,
  recorded_at TIMESTAMPTZ NOT NULL,
  value       NUMERIC
) PARTITION BY RANGE (recorded_at);

-- Create monthly partitions
CREATE TABLE measurements_2024_01
  PARTITION OF measurements
  FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE measurements_2024_02
  PARTITION OF measurements
  FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

-- Queries auto-use partition pruning:
SELECT * FROM measurements
WHERE recorded_at >= '2024-01-01' AND recorded_at < '2024-02-01';
-- Only scans measurements_2024_01!
🎯 INTERVIEW QUESTIONS
Q: What's the difference between JSON and JSONB in PostgreSQL?
JSON stores the exact input text, preserving whitespace, key order, and duplicate keys. It must be re-parsed on every operation. JSONB stores data in a decomposed binary format — parsed on write, but operations are much faster. JSONB: supports indexes (GIN indexes for containment @>, key existence ?), faster reads, smaller storage (deduplicates keys). JSON: faster writes (no parsing), preserves exact input. In practice: always use JSONB unless you specifically need to preserve the exact input format or key ordering.

-- "Seq Scan" = full table scan (maybe needs index)
-- "Index Scan" = uses index ✅
-- "Index Only Scan" = covering index ✅✅ (fastest)
-- "Hash Join" vs "Nested Loop" — planner chooses based on row estimates

Index Use case

Connection Pooling
PostgreSQL creates a new process per connection! Each connection uses ~5-10MB RAM. 1000 connections = 5-10GB just for connections. Use a connection pooler:

PgBouncer — most popular. Modes: session (one DB conn per client session), transaction (conn returned after each transaction — most efficient), statement. For most apps: transaction pooling.

Pgpool-II — adds load balancing and replication routing.

Typical setup: App → PgBouncer (10K client connections) → PostgreSQL (50-100 DB connections).
🎯 INTERVIEW QUESTIONS
Q: What configuration parameters most impact PostgreSQL performance?
Most impactful: (1) shared_buffers — PostgreSQL's buffer pool. Set to 25% of RAM. (2) work_mem — memory per sort/hash operation (watch out: can multiply by connections × operations). (3) effective_cache_size — hint to planner about total cacheable memory (OS + PG). Set to 75% of RAM. (4) max_connections — keep low, use PgBouncer. (5) checkpoint_completion_target=0.9 — spread checkpoint writes. (6) wal_buffers — WAL write buffer. Use pgtune.leopard.in to generate baseline config.
Q: What is ANALYZE and when should you run it?
ANALYZE collects statistics about table content (column cardinality, value distributions, correlation with physical order) and stores them in pg_statistic. The query planner uses these statistics to estimate row counts and choose optimal join strategies and index usage. autovacuum runs ANALYZE automatically. Run manually after: bulk data loads (millions of rows), major data distribution changes, after restoring from backup. Without good statistics, the planner makes poor decisions — "rows=1 estimated vs 100,000 actual" causes sequential scans where indexes should be used.

Decision Framework
Choose PostgreSQL when: Complex SQL queries with many JOINs, need full ACID transactions, advanced features (JSONB, partitioning, full-text, geospatial via PostGIS, time-series via TimescaleDB), analytics/reporting, data integrity is critical. PostgreSQL is the "default best choice" for most new projects.
Choose MySQL when: Team is highly experienced with MySQL, running on WordPress/Drupal (MySQL ecosystem), need simpler setup, existing infrastructure is MySQL-based. MySQL 8.0 has closed much of the gap with PostgreSQL.
Choose MongoDB when: Data is truly document-shaped with varying attributes (product catalogs, user profiles, CMS), schema changes frequently during development, need native horizontal sharding, team is more comfortable with JSON than SQL, hierarchical data with variable depth.

🎯 INTERVIEW QUESTIONS
Q: MySQL vs PostgreSQL — which would you choose and why?
For new projects, I'd default to PostgreSQL: it's more standards-compliant, has more powerful features (JSONB, window functions, recursive CTEs, better partitioning, PostGIS), and better ACID semantics. MySQL has a larger ecosystem for web frameworks and is slightly simpler to get started. If the team has deep MySQL expertise or the stack heavily uses MySQL tooling (Vitess, PlanetScale), MySQL makes sense. MySQL 8.0 has closed many gaps but PostgreSQL still leads in features.
Q: When would MongoDB be better than PostgreSQL?
MongoDB excels when: (1) Data is document-centric with highly variable structure — product catalogs where different product types have completely different attributes, (2) Development speed is priority and schema evolves rapidly, (3) Need native horizontal sharding across many nodes (sharded MongoDB clusters scale more easily than PostgreSQL), (4) Workload is simple reads/writes by document ID with occasional aggregations — no complex multi-table JOINs. For most apps with relational data, PostgreSQL with JSONB columns gives you the flexibility without sacrificing SQL power.
