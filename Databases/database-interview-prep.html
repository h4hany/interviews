🔒 ACID & Indexes
Transactions, isolation levels, and index types.
ACID: Atomicity (all or nothing), Consistency (constraints hold), Isolation (concurrent transactions don’t interfere), Durability (committed data persists). Isolation levels: Read uncommitted, Read committed, Repeatable read, Serializable. Indexes: B-tree (default), hash, composite (order matters), covering index.
-- Composite index: (a, b) helps WHERE a = ? AND b = ?
CREATE INDEX idx_ab ON t(a, b);
Q: When would you use a composite index?
When queries filter or sort by multiple columns. The column order matters: put equality filters first, then range/sort. (a, b) can be used for “WHERE a = ?” or “WHERE a = ? AND b = ?”, but not for “WHERE b = ?” alone.
Q: What is a covering index?
An index that contains all columns required by a query (including in the SELECT list). The database can satisfy the query from the index alone without reading the table (index-only scan), which is faster and reduces I/O.

🔗 SQL & Joins
INNER, LEFT, subqueries, window functions.
Join types
INNER JOIN: Only rows that match in both tables. LEFT (OUTER) JOIN: All rows from the left table; matching rows from the right, or NULL where there is no match. Use when you want to keep all left rows (e.g. all users and their orders, or NULL if no orders). RIGHT JOIN is the same but from the right. FULL OUTER: all from both, NULL where no match.
SELECT u.name, o.id
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;
Subqueries and window functions
Subqueries: Use for EXISTS, IN, or a scalar (single value) in SELECT/WHERE. Correlated subquery references outer query. Window functions: ROW_NUMBER(), RANK(), DENSE_RANK(), SUM() OVER (PARTITION BY ... ORDER BY ...) compute over a partition without collapsing rows. Useful for top-N per group, running totals, and rankings.
Q: INNER JOIN vs LEFT JOIN — when to use which?
Use INNER when you only want rows that have a match in both tables. Use LEFT when you want all rows from the left table and optional data from the right (e.g. list all users and their last order, or NULL if they have none).
Q: What is the difference between ROW_NUMBER, RANK, and DENSE_RANK?
ROW_NUMBER assigns a unique number per row (no ties). RANK leaves gaps after ties (1,2,2,4). DENSE_RANK does not leave gaps (1,2,2,3). Use ROW_NUMBER for a unique ordering; RANK/DENSE_RANK when you care about ranking with ties.

📈 Scaling & Replication
Read replicas, sharding, CAP.
Primary accepts writes and replicates to replicas (eventual consistency for reads). Sharding: partition data by shard key (e.g. user_id) across nodes; choose key to avoid hot spots. CAP: in a partition you cannot have all three of Consistency, Availability, Partition tolerance; choose CP or AP. CAP: you can’t have Consistency, Availability, and Partition tolerance all at once; choose CP or AP.

📄 NoSQL
Document, key-value, when to use.
Document stores (e.g. MongoDB)
Documents (e.g. JSON/BSON) with flexible schema. Good for nested data, catalogs, content, and when schema evolves often. Query by fields and use indexes. Avoid deep nesting if you need to query inside it. Replication and sharding are common. Weaker consistency and no joins; model data so related info is in one document or accept application-level joins.
Key-value (e.g. Redis)
Simple get/set by key. Use for cache, sessions, rate limiting, pub/sub, and simple lookups. Very fast, in-memory option. Persistence (RDB, AOF) available. Not for complex queries or relationships. Good as a cache layer in front of SQL or as a session store.
Q: When would you choose NoSQL over SQL?
When schema is fluid or you need horizontal scale and can accept eventual consistency. Document DBs suit nested, variable-shaped data. Key-value suits cache and simple lookups. Choose SQL when you need ACID, complex queries, joins, and strong consistency.
Q: What are the downsides of document stores?
No joins (denormalize or join in app). Weaker consistency guarantees. Duplication if same data lives in many documents. Schema flexibility can lead to inconsistency. Complex queries and transactions are harder than in SQL. Good fit for read-heavy, flexible-schema workloads.

📋 Database cheatsheet
Full quick reference for interviews.
ACID & isolation
Atomicity, Consistency, Isolation, Durability. Isolation levels: Read uncommitted → Read committed → Repeatable read → Serializable. Know dirty read, non-repeatable read, phantom read.
Indexes
B-tree (default), hash, composite (left-prefix rule), covering index. Indexes speed reads, slow writes, use storage.
SQL & joins
INNER vs LEFT. Subqueries: EXISTS, IN, scalar. Window functions: ROW_NUMBER, RANK, DENSE_RANK, SUM OVER.
Scaling & CAP
Read replicas (eventual consistency). Sharding (partition by key). CAP: in a partition, choose CP or AP.
NoSQL
Document: flexible schema, nested data. Key-value: cache, sessions. Use when schema is fluid or you need horizontal scale and can relax strong consistency.
