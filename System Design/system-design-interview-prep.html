🗺️ The Interview Framework
Most candidates freeze at "Design Twitter." This 4-step framework gives you a structured approach that interviewers love.
The 4-Step Process
1
Clarify Requirements (5 min)
Ask about scope, scale, and constraints. Never assume. Ask: Who are the users? What features are in scope? How many users (DAU)? Read-heavy or write-heavy? Latency requirements? Consistency vs availability preference?
2
Estimate Scale — Back-of-Envelope (5-10 min)
Calculate: QPS (queries per second), storage needed, bandwidth. Rough estimates show you think at scale. Example: 100M DAU × 10 requests/day = 1000M requests/day = ~12,000 QPS peak (×2-3 for peaks).
3
High-Level Design (10-15 min)
Draw the big picture: clients → DNS → CDN → load balancers → servers → databases. Cover the happy path end-to-end. Get buy-in from the interviewer before going deep.
4
Deep Dive — Design Details (15-20 min)
Pick 2-3 components to go deep on. Handle bottlenecks, failure cases, scaling. Discuss tradeoffs explicitly: "I chose X over Y because... the tradeoff is..."
Back-of-Envelope Estimation Template
ESTIMATION TEMPLATE
// Scale assumptions
DAU = 100M users
Reads per user per day = 10
Writes per user per day = 2

// QPS
Read QPS  = 100M × 10 / 86400  ≈ 11,600 (~12K QPS)
Write QPS = 100M × 2  / 86400  ≈ 2,300  (~2.3K QPS)
Peak      = avg × 2-3             ≈ 35K read, 7K write

// Storage (5 year)
Tweets: 2.3K writes/s × 200 bytes × 5yr × 3.15e7s/yr ≈ 72TB
Media:  10% have images × 200KB avg                      ≈ 7.2PB

// Bandwidth
Incoming: Write QPS × avg_size = 2.3K × 200B = 460KB/s
Outgoing: Read QPS × avg_size  = 12K  × 1KB  = 12MB/s
Things Interviewers Want to See
✅ Green flags: Clarifying before designing. Stating assumptions. Explaining tradeoffs ("X is better for our use case because..."). Addressing failure modes. Knowing when to use which technology and WHY. Showing awareness of consistency vs availability tradeoffs.
🚩 Red flags: Starting to code immediately. Using a database without justification. Not asking about scale. Only happy-path design with no failure handling. Not knowing numbers (can't estimate anything). Saying "it depends" without following up with what it depends on.
🎯 INTERVIEW QUESTIONS
Q: How do you approach a system design interview?
Four steps: (1) Clarify requirements and constraints — ask about scale, features in scope, latency/consistency needs. Never assume. (2) Estimate scale — calculate QPS, storage, bandwidth. Shows you think at scale. (3) High-level design — draw the overall architecture, cover the happy path. (4) Deep dive — pick 2-3 bottlenecks to discuss deeply, cover tradeoffs explicitly. Communicate your thinking throughout.
Q: How do you estimate QPS from DAU?
DAU × actions_per_day / seconds_per_day. 1 day = 86,400 seconds (round to 100K). For peak, multiply by 2-3x. Example: 10M DAU, 5 reads/day = 50M reads/day / 100K = ~500 QPS average, ~1500 QPS peak. For write-heavy systems, separate read and write QPS as they scale differently.

🎯 INTERVIEW QUESTIONS
Q: What does it mean for a server to be stateless?
A stateless server doesn't store any client session data in local memory. Each request is self-contained with all needed information. This enables horizontal scaling — any server can handle any request since there's no session affinity needed. State is stored in shared infrastructure: databases, Redis, etc. Stateful servers require sticky sessions (same client always goes to same server), which complicates load balancing and prevents easy scaling.
Q: When would you choose vertical scaling over horizontal?
Vertical scaling is simpler and appropriate for: (1) databases, which are harder to shard/distribute — scale up first, (2) quick short-term fixes when horizontal scaling requires app changes, (3) workloads that don't parallelize well. However, vertical has a hard ceiling and creates a SPOF. Most production systems start vertical and migrate horizontal as scale demands. Always have a horizontal plan as the long-term strategy.

⚖️ CAP Theorem
The most fundamental theorem in distributed systems. You WILL be asked about this. Understanding it explains every database choice you'll ever make.
The Three Properties
C — Consistency
Every read gets the most recent write or an error. All nodes see same data at same time.
A — Availability
Every request gets a non-error response. System is always up and responding.
P — Partition Tolerance
System works even when network partitions (node communication failures) occur.
⚠️ In a distributed system, network partitions WILL happen. P is non-negotiable. You choose between C and A.
Real-World Database Choices
TypeChoosesExamplesUse when
CP SystemsConsistency over AvailabilityHBase, MongoDB, Redis, ZooKeeperFinancial data, inventory, config
AP SystemsAvailability over ConsistencyCassandra, CouchDB, DynamoDBSocial feeds, shopping carts, DNS
CA SystemsConsistency + AvailabilityTraditional RDBMS (PostgreSQL, MySQL)Single-node or network is reliable
💡 REAL EXAMPLE — AMAZON SHOPPING CART
Amazon chose AP (availability over consistency) for shopping carts. If a network partition occurs, they'd rather show you a slightly stale cart than show an error. The cart might briefly show an item you already removed — that's okay. If they chose CP, your cart could be unavailable during a network partition — unacceptable for an e-commerce site.
PACELC — The Extended Model
CAP only addresses partitions. PACELC extends it: even when there's no partition (P), a system must trade off latency (L) vs consistency (C).
PACELC: During Partition → choose A or C. Else (no partition) → choose L (latency) or C (consistency). DynamoDB: PA/EL (high availability, low latency). BigTable: PC/EC (consistent, accepts latency).
🎯 INTERVIEW QUESTIONS
Q: Explain CAP theorem.
CAP theorem states that a distributed system can only guarantee two of three properties: Consistency (all nodes return the same data), Availability (every request gets a response), and Partition Tolerance (system works despite network failures). In practice, network partitions always happen, so P is mandatory — you're really choosing between C and A when a partition occurs. Banks need CP; social media can accept AP.
Q: Is a relational database CA? Is that possible?
A single-node RDBMS is CA in the sense that it provides consistency and availability, but it avoids the partition problem by not being distributed. Once you try to distribute a relational database (master-slave, clustering), you face the same C vs A tradeoff during partitions. Modern systems often say "CA is not a valid choice in distributed systems" — you must choose CP or AP when partitions occur.
Q: Would you choose consistency or availability for a banking system?
Consistency. A bank cannot show stale balance data — if you have $100 and make two withdrawals from different devices simultaneously, the system must prevent overdrawing. It's better to show an error ("service temporarily unavailable") than to process a transaction based on stale data. Banks use CP systems like relational databases with ACID transactions. Brief unavailability is acceptable; financial inconsistency is not.

Consistent Hashing
When you add/remove a server from a standard hash ring, all keys must be remapped. With consistent hashing, only keys from the affected range need remapping — minimizing cache invalidation and data movement.
💡 ANALOGY
Imagine a circular clock face. Servers are placed at random positions on the clock. Each key is hashed to a position, and assigned to the first server clockwise from it. If Server B at 3 o'clock is removed, only keys between Server A (12 o'clock) and Server B (3 o'clock) move to Server C — not all keys. Virtual nodes (each server appears multiple times) improve distribution.
🎯 INTERVIEW QUESTIONS
Q: What is consistent hashing and why is it important?
Consistent hashing places both servers and keys on a hash ring. Each key is assigned to the nearest server clockwise. When a server is added/removed, only 1/N keys need remapping (N = servers), vs N-1/N with simple modular hashing. This is critical for distributed caches — when a cache node fails, you don't want to invalidate every key, just those that belonged to that node. Used in Amazon DynamoDB, Apache Cassandra, Redis Cluster.
Q: When would you use an L7 load balancer instead of L4?
Use L7 when you need: URL-based routing (route /api/ to backend servers, /static/ to CDN), SSL/TLS termination (decrypt once at LB, plain HTTP internally), content-based routing (route by HTTP headers), A/B testing, rate limiting per user/IP, WebSocket support with sticky connections. L4 is better for raw TCP performance, non-HTTP protocols (databases, game servers), or when you can't/don't want to inspect application data.

⚡ Caching Strategies
The most impactful performance optimization. Cache-aside, write-through, write-back, TTL, and eviction. Almost every system design uses caching.
Cache Write Strategies
CACHE-ASIDE (LAZY LOADING)
1. App checks cache → miss
2. App reads from DB
3. App writes to cache
4. Returns to client
✅ Cache only what's needed
⚠️ Cache miss penalty (3 trips)
WRITE-THROUGH
1. App writes to cache
2. Cache writes to DB
3. Both always in sync
✅ No stale data
⚠️ Writes are slow (2 hops)
StrategyWrite pathProsCons
Cache-asideApp → DB, then app populates cacheOnly cache hot dataCold start, cache miss penalty
Write-throughApp → Cache → DB (sync)Cache always freshSlow writes, cache filled with unused data
Write-backApp → Cache (async flush to DB)Fast writesRisk of data loss if cache crashes
Write-aroundApp → DB directly, skip cacheAvoid filling cache with write-once dataRead-back causes cache miss
Cache Eviction Policies
PolicyEvictsBest for
LRULeast Recently UsedGeneral purpose (most common)
LFULeast Frequently UsedLong-term popularity patterns
TTLAfter fixed time-to-liveTime-sensitive data (weather, stock)
FIFOFirst In First OutSimple, fair eviction

Cache Problems & Solutions
Cache Stampede (Thundering Herd): A cached item expires → thousands of requests hit the DB simultaneously. Solution: (1) Probabilistic early expiry — proactively refresh before expiry, (2) Mutex lock — first request fetches, others wait, (3) Stale-while-revalidate — serve stale data while refreshing in background.
Cache Penetration: Queries for non-existent keys bypass cache (nothing to cache) and hammer DB. Solution: Cache null results with short TTL, or use a Bloom filter to check if key could exist before querying DB.
Cache Avalanche: Many cache keys expire at the same time → DB overloaded. Solution: Add random jitter to TTL (expire at TTL ± random seconds), stagger cache warming.
🎯 INTERVIEW QUESTIONS
Q: What is cache-aside and when do you use it?
Cache-aside (lazy loading): app checks cache first. On miss, app reads from DB, writes result to cache, returns to client. On hit, return from cache. Best for read-heavy workloads where you only cache what's actually accessed. Downside: first access (cold cache) has 3 trips. Stale data possible if DB updated directly. Most Redis usage follows this pattern.
Q: What is cache invalidation and why is it hard?
Cache invalidation means removing or updating cached data when the underlying data changes. It's hard because: (1) you need to know all cache keys affected by a DB change — complex for denormalized data, (2) race conditions — invalidate then update, or update then invalidate? What if invalidation fails? (3) distributed systems — which nodes need invalidation? Solutions: TTL-based expiry (simple, some staleness), event-driven invalidation (complex, accurate), write-through (always fresh, slower writes).

🗄️ Database Design
Indexes, ACID, transactions, replication — the database knowledge every senior engineer needs.
Indexes
Indexes speed up reads at the cost of writes and storage. Without an index, every query requires a full table scan O(N). With a B-tree index, lookup is O(log N).
SQL — INDEX DESIGN
-- Primary key (automatically indexed)
CREATE TABLE users (
  id BIGINT PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,  -- unique index
  name VARCHAR(100),
  created_at TIMESTAMP
);

-- Composite index — order matters! (email, created_at)
-- Supports queries on: email alone, email+created_at
-- Does NOT help: created_at alone
CREATE INDEX idx_user_email_date ON users (email, created_at);

-- Covering index — includes all columns query needs
-- No table lookup needed → faster
CREATE INDEX idx_covering ON orders (user_id) INCLUDE (total, status);

🎯 INTERVIEW QUESTIONS
Q: What is replication lag and how do you handle it?
Replication lag is the delay between a write to the master and it appearing on replicas. During this window, reads from replicas return stale data. Mitigations: (1) Read-your-writes consistency — route a user's own reads to master for a short window after writing, (2) Sticky routing — route specific users/sessions to specific replicas, (3) Synchronous replication — no lag but slower writes, (4) Read from master for sensitive data, replicas for eventually-consistent data.
Q: When should you use a composite index vs multiple single indexes?
Use composite index when queries filter by multiple columns together (WHERE user_id = X AND status = Y). The leading column must match the query. Use multiple single indexes when queries filter on each column independently. Composite indexes can serve as covering indexes (avoid table lookup). Rule: index columns in order of selectivity (most selective first) and based on actual query patterns.

💡 REAL WORLD EXAMPLE — INSTAGRAM
Instagram uses PostgreSQL for user accounts, follows, and comments (relational data needing consistency). They use Cassandra for the activity feed and notifications (write-heavy, high volume, eventual consistency fine). They use Redis for caching and rate limiting. Most large systems use multiple databases for different use cases.
🎯 INTERVIEW QUESTIONS
Q: When would you choose MongoDB over PostgreSQL?
MongoDB when: data is document-like with nested structures (product catalogs with varying attributes), schema changes frequently during development, don't need multi-document ACID transactions, need horizontal scaling. PostgreSQL when: need ACID transactions, complex queries with joins, data integrity constraints, strong consistency. Note: PostgreSQL now supports JSON columns, blurring the line — often PostgreSQL with JSON is preferred over MongoDB for its ACID guarantees.
Q: What is BASE and how does it differ from ACID?
BASE: Basically Available (system stays available), Soft state (state may change over time without input), Eventual consistency (system will eventually reach consistent state). ACID is strong consistency, BASE accepts temporary inconsistency for availability and performance. NoSQL databases often use BASE to achieve scalability. Example: Cassandra writes succeed even if some nodes are down — those nodes get updates when they recover (eventual consistency).

🔁 Replication & Sharding
How to scale databases beyond a single machine — the two fundamental strategies and their tradeoffs.
Sharding (Horizontal Partitioning)
Sharding splits a large dataset across multiple database nodes. Each shard holds a subset of the data. Unlike replication (copies of same data), sharding splits the data itself.
Sharding StrategyHow it worksProsCons
Range-baseduser_id 1-1M → shard 1, 1M-2M → shard 2Simple, range queries easyHotspot if data not evenly distributed
Hash-basedshard = hash(user_id) % num_shardsEven distributionRange queries impossible, resharding is hard
Directory-basedLookup table: entity → shardFlexible, can rebalanceLookup table is SPOF and bottleneck
GeographicEuropean users → EU shard, US users → US shardData sovereignty, low latencyUneven distribution by geography
⚠️ Resharding Problem: When data grows and you need to add shards, you must move large amounts of data. This is why consistent hashing is often used — adding a shard only moves ~1/N of data instead of remapping everything.
Hotspot Problem
A hotspot (celebrity problem) occurs when one shard gets disproportionately more traffic. Example: a viral tweet by a celebrity generates millions of reads to the shard containing that tweet.
Solutions: Add a random suffix to hot keys to spread across multiple shards (e.g., celebrity_user → celebrity_user_1, celebrity_user_2). Use caching in front of the shard. Identify and pre-warm hot keys. Special handling for "VIP" entities.
🎯 INTERVIEW QUESTIONS
Q: What is database sharding and what are its challenges?
Sharding splits data across multiple database nodes, each holding a subset. Challenges: (1) Cross-shard queries — JOINs across shards require application-side aggregation, (2) Resharding — adding shards requires moving data, (3) Hotspots — uneven data distribution overloads specific shards, (4) Transaction complexity — no ACID across shards, (5) Operational complexity — more nodes to manage. Most companies delay sharding as long as possible using vertical scaling, read replicas, and caching first.
Q: What's the difference between replication and sharding?
Replication creates copies of the SAME data on multiple nodes — improves read throughput, provides redundancy/failover. Every node has all the data. Sharding splits DIFFERENT data across nodes — improves both read AND write throughput for large datasets. Each node has only a subset of data. Often combined: each shard has 1 master + 2 replicas for both scale and redundancy.

Delivery Guarantees

🎯 INTERVIEW QUESTIONS
Q: When would you use a message queue in your design?
Use a message queue when: (1) decoupling services — producer shouldn't wait for consumer, (2) traffic smoothing — buffer writes during traffic spikes (queue absorbs burst), (3) async processing — tasks that don't need immediate response (email sending, video encoding), (4) fan-out — one event needs to trigger multiple services, (5) retry/reliability — automatically retry failed operations. Red flag: using queues when synchronous response is needed or when adding unnecessary complexity to simple flows.
Q: What's the difference between Kafka and RabbitMQ?
Kafka: distributed commit log, messages retained for configurable period (replay possible), pull-based consumers, consumer groups, partitioned for parallelism, very high throughput (millions/sec). Best for: event streaming, event sourcing, audit logs, big data pipelines. RabbitMQ: traditional message broker, messages deleted after consumed, push-based, routing/exchange patterns (direct, topic, fanout), lower throughput but more features. Best for: task queues, complex routing, when you need exactly-once semantics, microservice communication.

💡 TOKEN BUCKET ANALOGY
Imagine a toll booth with tokens. Your bucket holds 10 tokens. Each API call costs 1 token. Tokens refill at 2/second. You can burst through 10 requests instantly, but sustained rate is limited to 2/second. This matches real API usage: occasional bursts are fine, sustained abuse is blocked.
Distributed Rate Limiting
In a distributed system, requests hit different servers. Each server can't just use local memory — you need a shared store (Redis) to track counts across all nodes.
REDIS — SLIDING WINDOW
-- Redis sliding window rate limiter (Lua script — atomic)
local key = "rate:user:" .. user_id
local now = redis.call('TIME')[1] * 1000
local window = 60000  -- 60 second window in ms
local limit = 100

-- Remove old entries outside window
redis.call('ZREMRANGEBYSCORE', key, 0, now - window)

-- Count remaining in window
local count = redis.call('ZCARD', key)

if count < limit then
  redis.call('ZADD', key, now, now)  -- add current request
  redis.call('EXPIRE', key, 60)
  return 1  -- allowed
else
  return 0  -- rejected
end
🎯 INTERVIEW QUESTIONS
Q: Explain the token bucket algorithm.
Token bucket: a bucket holds up to N tokens. Each request consumes 1 token. Tokens are added at a fixed rate (e.g., 10/second). If bucket is empty, request is rejected. If bucket is full, new tokens are discarded. This allows burst traffic up to bucket size while enforcing average rate. Implementation: store {token_count, last_refill_time} per user in Redis. On each request, calculate tokens added since last refill, check if enough tokens, decrement atomically.
Q: Why is the fixed window algorithm problematic?
Fixed window has a boundary problem: if your limit is 100 req/min, a user can send 100 requests at 11:59:50 and 100 more at 12:00:10 — 200 requests in 20 seconds while technically "respecting" the limit. The sliding window solves this by considering a rolling 60-second window, not a fixed clock-based window. For most APIs, sliding window counter is the best balance of accuracy and memory efficiency.

Idempotency
An idempotent operation produces the same result whether called once or many times. Critical for retry logic — if a payment request times out, did it succeed? Retrying non-idempotent requests causes double charging.

🎯 INTERVIEW QUESTIONS
Q: When would you use gRPC over REST?
gRPC is ideal for: internal microservice communication (where you control both sides), high-performance binary protocol (protobuf is 3-10x smaller than JSON), streaming (server/client/bidirectional streaming), polyglot environments (generates client code in any language from proto). Use REST for: public APIs (wider tooling support, browser-friendly), when JSON is required, simpler CRUD services where proto overhead isn't justified.
Q: Why is cursor-based pagination better than offset pagination?
Offset pagination has two problems: (1) Performance — OFFSET N requires scanning and discarding N rows. For page 1000 with 20 items, the DB processes 20,000+ rows. Gets slower as offset grows. (2) Data consistency — if new items are inserted while paginating, items shift and users may see duplicates or skip items. Cursor-based uses WHERE id > last_seen_id which uses the index directly (O(log N)), and new inserts don't affect the cursor position.

Quorum
Quorum defines how many nodes must acknowledge a read/write for it to succeed. With N replicas, write quorum W, read quorum R:
Strong consistency: W + R > N. Example: N=3, W=2, R=2 → 2+2=4 > 3. At least one node in every read overlaps with every write.

Common: N=3, W=2, R=2 (majority quorum) — tolerate 1 node failure. Or W=3, R=1 for fast reads.
🎯 INTERVIEW QUESTIONS
Q: How does eventual consistency work in practice?
In eventual consistency, writes propagate asynchronously to replicas. During the propagation window, different nodes may return different values. Eventually (milliseconds to seconds), all nodes converge to the same value. Handle stale reads: show "last updated" timestamps, use version numbers, allow users to refresh, or accept minor inconsistency (like counts). Systems often offer tunable consistency — you can request stronger guarantees at the cost of latency.
Q: What is quorum and how does it ensure consistency?
Quorum requires a majority of nodes to agree on a read/write. With W + R > N, every read must overlap with every write by at least one node. That overlapping node has the latest data. Example: 5 nodes, W=3, R=3 → W+R=6 > 5. Any 3-node read set intersects with any 3-node write set (pigeonhole). This provides strong consistency while tolerating (N-W) write failures and (N-R) read failures.

🎯 INTERVIEW QUESTIONS
Q: 301 vs 302 redirect — which to use for URL shortener?
301 Permanent Redirect: browser caches the redirect and never hits your server again for that URL. Saves server load but you can't track clicks or update the destination. 302 Temporary Redirect: browser always asks your server, even if seen before. You can track every click, update destinations, and show analytics. Choice depends on requirements: if you need analytics → 302. If it's a pure shortener wanting minimal server load → 301.
Q: How would you scale this to billions of URLs?
Multiple strategies: (1) Shard the database by short_code hash — spread across multiple DB nodes. (2) Cache aggressively in Redis — most URLs follow power law distribution (20% of URLs get 80% of traffic). (3) Use a CDN for the redirect layer — cached 301s don't even hit your servers. (4) Use separate read replicas for redirect lookups (read-heavy). (5) Archive old/expired URLs to cold storage, keep only active ones in hot DB.

📰 Design: News Feed
Twitter/Instagram/Facebook feed. Fan-out strategies, ranking, pagination, and the celebrity problem.
The Core Problem: Fan-Out
When user A (10M followers) posts, how do you deliver that post to 10M feeds? Two approaches:
FAN-OUT ON WRITE (Push)
Post → push to all followers' feed cache immediately
Reads are fast (pre-computed)
Good for small follower counts
WRITE DISADVANTAGE
Celebrity with 10M followers = 10M writes per post
Wastes storage for inactive users
Write amplification problem
FAN-OUT ON READ (Pull)
Feed assembled at read time — fetch from all followees
No write amplification
Always fresh data
READ DISADVANTAGE
Slow reads — merge N posts from N followees
N queries per feed load
Can't pre-rank or pre-cache
Hybrid approach (Twitter/Instagram): Fan-out on write for regular users (< 1K followers). Fan-out on read for celebrities (> 1M followers). Merge at read time: pre-built feed + real-time celebrity posts. This is the real-world solution to the celebrity problem.
High-Level Architecture
POST /post → Post Service → Posts DB + Message Queue
↓ Fan-Out Service consumes queue
→ For each follower: push post_id to Redis feed cache (sorted set by time)
→ Celebrities: skip fan-out, inject at read time
GET /feed → Feed Service
→ Get user's feed from Redis (post IDs)
→ Fetch post details from Posts cache/DB
→ Merge celebrity posts + rank by relevance
→ Return hydrated feed
🎯 INTERVIEW QUESTIONS
Q: How would you handle a celebrity with 100M followers posting?
Use hybrid fan-out: don't fan-out for users above a follower threshold (e.g., 1M followers). Instead, at feed read time: fetch the pre-built feed from cache (from regular user follows), then fetch the last N posts from celebrities the user follows (a separate query), merge and rank them. This avoids the write amplification of pushing to 100M feeds. The tradeoff: slightly more complex read path, but writes stay manageable.
Q: How do you implement infinite scroll / feed pagination?
Use cursor-based pagination. Store each user's feed as a Redis sorted set with timestamp as score. For the first load: ZREVRANGE feed:user:123 0 19 (first 20 items). For next page: client sends cursor (timestamp of last item), server does ZREVRANGEBYSCORE feed:user:123 cursor -inf LIMIT 0 20. This is stable (new posts added to front don't affect past cursor positions) and fast (Redis sorted set is O(log N + items)).

Online Presence
Heartbeat approach: Client sends heartbeat every 5 seconds. Server stores {user_id: last_heartbeat} in Redis with 10s TTL. User is "online" if key exists. On disconnect, key expires in 10s. Presence service aggregates and publishes to friends. For 50M users, this is 10M heartbeats/sec — batch updates, don't broadcast to all friends immediately.
🎯 INTERVIEW QUESTIONS
Q: How do you route a message when sender and receiver are on different servers?
Use Redis Pub/Sub or a message broker. Each user has a pub/sub channel (user:{id}). The chat server the receiver is connected to subscribes to that channel. When A sends a message, A's server publishes to B's channel. The server B is connected to receives it and forwards via B's WebSocket. If B is offline, fall back to push notifications (APNs/FCM). This is how Slack and Discord route messages across their server clusters.
Q: Why Cassandra for message storage instead of PostgreSQL?
Chat messages are: write-heavy (23K/sec), time-ordered (natural for sorted storage), partition-friendly by conversation_id, rarely updated (mostly inserts), and require horizontal scale. Cassandra excels at: high write throughput, linear horizontal scaling, efficient time-series storage with composite keys, tunable consistency. PostgreSQL struggles with: 23K inserts/sec on a single node, sharding complexity, large table performance. Alternative: MySQL with horizontal sharding (Facebook Messenger's actual approach — HBase).
