📋 Requirements & Scope
Clarify functional and non-functional requirements before designing.
Functional requirements
Core features the system must support: e.g. user can post, follow others, view a timeline, like/comment. Be specific enough to drive design (read/write ratio, who can do what). Ask: single user or multi-tenant? Real-time or batch?
Non-functional requirements
Scale: DAU, QPS (reads vs writes). Latency: p99 or average (e.g. < 200ms). Availability: 99.9% (three nines) or 99.99%. Consistency: strong (read-after-write) vs eventual. State assumptions explicitly (e.g. single region first, 10M DAU).
Q: Why clarify requirements before drawing the system?
Requirements drive scale (QPS, storage), consistency choices, and which components you need (cache, queue, DB). Without them you might over- or under-design. Stating scope (e.g. read-heavy, write path simple) keeps the interview focused.

📐 Capacity Estimation
QPS, storage, and bandwidth back-of-envelope.
QPS (queries per second)
Start from DAU. Assume reads per user per day (e.g. 10) and writes (e.g. 2). Total read QPS ≈ DAU × reads_per_user / 86400; peak is 2–3× average. Use round numbers: 1M DAU, 10 reads/user/day → ~100–120 read QPS; 2 writes → ~25 write QPS.
Storage and bandwidth
Storage: records × size per record × retention (e.g. 5 years). Add for indexes and replication. Bandwidth: QPS × request/response size. E.g. 100K QPS × 10 KB ≈ 1 GB/s. State assumptions (e.g. 1 KB per request).
Q: How do you estimate write QPS from DAU?
DAU × (writes per user per day) / 86400. Example: 10M DAU, 5 writes/user/day → 10M × 5 / 86400 ≈ 580 write QPS. Use peak multiplier (e.g. 2–3×) if asked for peak capacity.

⚖️ Load Balancing
Algorithms, health checks, L4 vs L7.
Algorithms
Round-robin: Rotate across backends; simple, even if some are slower. Least connections: Send to the server with fewest active connections; good for long-lived or variable-length requests. Weighted: Assign more traffic to more capable nodes. IP hash: Same client often to same server (sticky by IP).
Health checks and layers
Health checks (HTTP or TCP) remove unhealthy backends from the pool. Sticky sessions (cookies or server-assigned) when state is local to a server. L4 (TCP/UDP): no awareness of HTTP. L7 (HTTP): can route by path, header, cookie; TLS termination; more flexible.
Q: When would you use L7 load balancing instead of L4?
When you need HTTP-aware routing: by URL path, host header, or cookie (e.g. canary, A/B, sticky session). L7 also allows TLS termination at the LB. Use L4 when you only need to distribute TCP/UDP and want lower latency and simpler config.

💾 Caching Strategies
Cache-aside, write-through, TTL, invalidation.
Cache-aside (lazy loading)
App reads from cache first. On miss: read from DB, then write to cache. Writes go to DB; cache can be updated or invalidated on write. Simple and robust; cache can hold a subset of data. Risk: thundering herd if many requests miss the same key — use locking or single-flight.
Write-through and TTL
Write-through: Every write goes to cache and DB so cache stays in sync; read miss is rare but writes are slower. TTL: Expire entries after a time to avoid stale data; balance freshness vs load. Invalidation: On write, delete or update the cached entry so next read gets fresh data from DB.
Q: What is cache stampede and how do you mitigate it?
When a popular key expires, many requests hit the DB at once (stampede). Mitigate: (1) probabilistic early expiry (e.g. refresh a bit before TTL), (2) single-flight (one request fills cache, others wait), (3) lock per key so only one recomputes.

🔀 Sharding & Partitioning
Horizontal partitioning and consistent hashing.
Shard key and partitioning
Partition data by a shard key (e.g. user_id, tenant_id) so each shard holds a subset. Key choice matters: avoid hot shards (one key with most traffic). Range partitioning: by key range; can create hotspots. Hash partitioning: key → hash → shard; even distribution but range queries need to hit all shards.
Consistent hashing
Hash keys and nodes onto a ring. Key goes to the next node clockwise. Adding or removing a node only moves keys that now map to a different node; minimal rebalancing. Use virtual nodes (replicas of each node on the ring) for more even distribution. Hot shards: monitor, split the shard, or use a different key.
Q: Why use consistent hashing for sharding?
When nodes are added or removed, only a fraction of keys need to move (those adjacent to the change on the ring). With simple modulo hashing, adding a node would reassign almost all keys. Consistent hashing reduces rebalancing cost and failure blast radius.

📑 Replication
Leader-follower, sync vs async, consistency.
Leader-follower (primary-replica)
One leader accepts writes; followers replicate the write log (WAL or binlog). Sync replication: Leader waits for N replicas to ack before confirming write; strong consistency, higher latency. Async replication: Leader confirms immediately; replicas lag; eventual consistency. Reads can go to followers to scale read throughput.
Trade-offs
Strong consistency: read from leader or sync replicas; higher latency, fewer read replicas. Eventual consistency: read from any replica; lower latency, scale reads, but may see stale data. Use leader for read-after-write; use replicas for bulk or timeline reads that tolerate lag.
Q: When would you read from a replica instead of the leader?
When the use case tolerates staleness: e.g. timeline, feed, analytics. Reading from replicas spreads load and scales read QPS. Always read from leader (or sync replica) when the user must see their own just-written data (read-after-write consistency).

🔍 Search & Ranking
Inverted index, retrieval, re-ranking.
Inverted index
Maps each term to a list of document IDs (posting list) that contain the term. Enables fast lookup: for a query, intersect or merge posting lists. Store doc frequency, positions for phrase queries. Index is built at ingest; search is over the index, not raw docs. Shard index by doc_id; query all shards and merge top-K.
Retrieval and re-ranking
Retrieval: Get candidates (e.g. BM25 for keyword, vector search for semantic). Retrieve more than K (e.g. 1000). Re-ranking: Apply a heavier model or signals (ML ranker, diversity, recency) to the candidates and return top-K. Two-stage keeps latency low while improving relevance.
Q: Why use a two-stage retrieve-then-rerank pipeline?
First stage (e.g. BM25, vector search) is fast and reduces millions of docs to hundreds of candidates. Second stage (ML model, complex features) runs on a small set and improves relevance. Running the heavy ranker on the full corpus would be too slow for low latency.

📋 System design cheatsheet
Full quick reference for interviews.
Flow
Requirements (functional, non-functional) → Capacity (QPS, storage, bandwidth) → High-level diagram → Load balancer → App servers → Cache → DB (sharding, replication) → Special (search, queue, CDN).
Key concepts
Load balancing: round-robin, least connections, L4 vs L7. Caching: cache-aside, write-through, TTL, stampede mitigation. Sharding: shard key, consistent hashing. Replication: leader-follower, sync vs async, read from replica. Search: inverted index, retrieve then re-rank.
