🎯 Step 1 — Clarify Requirements First
Always ask these before drawing anything. This alone impresses 80% of interviewers.
  • What scale? Yassir operates in 45 cities (Algeria, Morocco, Tunisia, France). ~2M active users, ~100K active drivers. Not Uber-global but still needs serious scale.
  • Core features only? Agree to focus on: ride request → driver match → real-time tracking → trip completion → payment. Not recommendations, ads, or analytics.
  • Latency requirements? Driver matching must happen in under 3 seconds. Location updates every 4–5 seconds. Payment within 10 seconds of trip end.
  • Consistency vs Availability? Ride matching must be strongly consistent (no double-booking). Location tracking can be eventually consistent.
  • What's the traffic pattern? Peaks at morning/evening rush hours and weekends. Surge pricing implies demand spikes. System must auto-scale.
🏗️ High-Level Architecture
Start with this on the whiteboard — boxes first, details later.
[Rider App] ──→ [API Gateway] [Driver App] ──→ [API Gateway] │ ┌───────────┼───────────┐ ↓ ↓ ↓ [Auth Service] [Location [Ride Service] Service] │ │ [Redis GEO] [PostgreSQL (hot cache) + MongoDB] │ │ [Kafka [Dispatch Stream] Service] │ │ [Driver [Notif. Location DB] Service] (WebSocket + Push)
⚙️ Core Microservices
Each owns its data. Communication via REST + Kafka events.
🚗 Location Service
Receives driver GPS every 4–5s via WebSocket. Converts lat/lng to Geohash. Stores in Redis GEO (for fast proximity queries) + writes to Cassandra for history.
→ Redis GEOADD/GEORADIUS for O(log N) nearest-driver lookup
🎯 Dispatch Service
Receives ride request → queries Location Service for nearby drivers → scores by ETA + rating → sends push notification to best driver → handles accept/decline with timeout retry.
→ Uses distributed lock (Redis SETNX) to prevent double-assignment
🛣️ Ride Service
Manages the ride state machine: REQUESTED → ACCEPTED → ARRIVING → IN_PROGRESS → COMPLETED. Each state change is atomic in PostgreSQL. Publishes Kafka events per state change.
→ Strong consistency needed here — no eventual consistency tolerated
💳 Payment Service
Triggered by COMPLETED event from Kafka. Calculates fare (distance × rate + surge). Charges stored payment method. Handles retries, refunds, receipts asynchronously.
→ Idempotent — uses trip_id as idempotency key to prevent double charges
📍 The Hard Part — Real-Time Location
This is where most candidates lose points. Know both approaches.
GEOHASH (What Yassir/Uber actually use)
Convert GPS coordinates into a short string. 37.7564, -122.4016 → 9q8yy.

Nearby locations share the same prefix. Drivers in the same geohash cell are stored together in Redis.

When rider requests ride: Hash their location → query Redis for all active drivers in that cell + 8 neighboring cells → rank by distance → dispatch.
🌍 Real Numbers
6-char geohash ≈ 1.2km × 0.6km cell. Uber processes 100M+ location updates daily. Redis GEOADD is O(log N) per write — handles this at 200K driver updates/minute easily.
✅ GEOHASH PROS O(1) insert speed
Works with any DB
Easy to cache
Simple to understand
❌ GEOHASH CONS Edge case: drivers near cell boundaries may be missed
Fix: always query 9 cells (target + 8 neighbors)
🔒 Driver Matching — The Race Condition Problem
Two riders requesting at the same time can both be assigned the same driver. Here's how to prevent it.
  • Redis Distributed Lock: When dispatch picks driver X, immediately SET driver:X:lock <rideId> NX EX 30. NX = only if not exists. If lock fails, driver is busy → try next candidate.
  • Driver accepts within 15s: If no response, release lock, notify next driver. Prevents a driver from blocking rides while phone is off.
  • Exactly-once guarantee: Ride state transitions use database transactions. UPDATE rides SET driver_id=X WHERE id=Y AND driver_id IS NULL — atomic check-and-set.
  • Kafka for async work: After ACCEPTED state, Kafka event triggers notifications to rider, map updates, billing prep. The ride service itself stays fast.
💡 Interview Gold
Mentioning the distributed lock + idempotent payment key puts you in the top 10% of candidates. Most people forget these and get called out when the interviewer asks "what if two riders request at the same time?"
🗄️ Database Strategy
Different data has different requirements. Don't use one DB for everything.
Redis (In-Memory)
Active driver locations (Geohash + GEORADIUS), distributed locks, session tokens, surge pricing cache. Evict after driver goes offline.
→ Sub-millisecond reads. Perfect for hot, fast-changing data.
PostgreSQL (Relational)
Users, drivers, rides, payments. Strongly consistent. ACID compliant. Use read replicas for analytics. Shard by city/region at scale.
→ Financial data (fares, payments) must be strongly consistent.
Cassandra (Wide-Column)
Driver location history. Append-only, time-series data. Partition key = driver_id, clustering key = timestamp. Write-heavy workload.
→ Handles millions of writes/sec. Eventual consistency is fine for history.
Kafka (Event Stream)
Decouples services. Ride state changes → Kafka topic → consumed by Notification, Payment, Analytics services independently. Replay on failure.
→ Prevents cascading failures. Each service processes at its own pace.

🛡️ Resiliency & Failure Handling
Interviewers WILL ask "what if X fails?" Be ready.
  • Redis goes down: Fallback to PostgreSQL GIS queries (slower but works). Redis is rebuilt from driver heartbeats in seconds once back up.
  • Dispatch Service crashes: Kafka consumer group rebalances. Another instance picks up. Ride remains in REQUESTED state — not lost.
  • Driver app loses internet: Ride stays ACCEPTED. Rider sees "driver is connecting..." for up to 60s. After timeout, reassign to next driver.
  • Payment service fails: Kafka retries. Idempotent design means retry = same result. Charge is queued and processed when service recovers.
  • Entire datacenter down: Geo-routing shifts to secondary region (e.g., Casablanca → Paris backup). Data is replicated via Kafka cross-region replication.

2. Draw boxes, not details
High-level first: API Gateway → Services → Databases. Fill details as time allows.
3. Name the hard problems

4. State your trade-offs

6. Do the math
Estimate: writes/sec, storage/day, latency target. Shows engineering thinking beyond just architecture diagrams.
