// PHASE 1 — IDEA
💡 Idea Capture & Validation
Before writing a single line of code — or even a PRD — use AI to pressure-test your idea. Most failed projects fail at this step: the scope wasn't clear, the user wasn't defined, or the MVP wasn't minimal enough.
Rule: Don't touch a keyboard for code until you can answer: Who is the user? What exact pain does this solve? What is the smallest version that proves the idea works?
Idea Validation & Scoping
PROMPT 01
Idea Stress Test & Problem Fit
FIRST STEPCLAUDE / GPT
When to use: You have a vague idea and want to pressure-test it before spending time planning it.
I have an app idea and I want you to help me validate and sharpen it before I start building. MY IDEA: [Describe your idea in 2-5 sentences. Be rough — this is brainstorm stage.] HELP ME ANSWER THESE QUESTIONS: 1. The Problem: What specific pain point does this solve? Who feels this pain the most? How do they solve it today (manually, competitor, don't solve it)? 2. The User: Who is the primary user? Describe them precisely (role, context, tech literacy, budget). 3. The Value: What does the user gain? Time saved? Money saved? Status? Fun? 4. The Risk: What's the #1 reason this could fail? What has to be true for this to work? 5. The Scope Trap: What am I probably trying to build that's NOT actually the core idea? What should I cut for v1? 6. The MVP Test: What is the SMALLEST possible version that proves this is worth building? THEN: Give me a sharpened 3-sentence description of the idea that I can use as the foundation for a PRD. Be direct and critical — I want honest pushback, not encouragement.
PROMPT 02
Competitive Landscape — What Already Exists
RESEARCH
I'm building [YOUR SHARPENED IDEA — use output from Prompt 01]. Before I write a PRD, help me understand what already exists. GIVE ME: 1. Existing solutions: List 3-5 tools/apps that already solve this (or adjacent problems). For each: what they do well, what they do poorly, what their users complain about. 2. My differentiation: Based on the gaps, what angle could I take that none of them own? 3. Build vs buy: Are there APIs, SDKs, or no-code tools I should use instead of building from scratch? (e.g. Stripe for payments, Auth0 for auth, Algolia for search) 4. Tech risk: Is there any part of this that's technically hard or uncertain? Flag it. 5. Time estimate: Rough estimate — how long would this take a solo developer working with AI assistance? Be honest about whether this is worth building vs buying/adapting.
PROMPT 03
Generate Idea Brief — The Foundation Doc
OUTPUT: IDEA.mdFEEDS INTO PRD
Output: Save this as IDEA.md. It's the input to all subsequent prompts.
Based on everything we've discussed, write a clean Idea Brief document I'll save as IDEA.md. IF STARTING FRESH, include answers to: - App name (working title): [NAME] - The problem: [One precise sentence] - The user: [Specific person, context, pain] - The solution: [What the app does in one sentence] - The MVP scope: [Exactly what's in v1 — bullet list] - NOT in v1: [What you explicitly cut] - Tech stack preference: [e.g. Next.js + Supabase / FastAPI + React / Node + Postgres / whatever fits] - Solo or team: [Who is building this] FORMAT AS: A clean markdown document with these exact sections: # [App Name] — Idea Brief ## The Problem ## The User ## The Solution ## MVP Scope ## Out of Scope (v1) ## Tech Stack ## Success Metric (how do I know v1 worked?)
// PHASE 1 — DOCUMENT
📄 PRD — Product Requirements Document
The PRD is what your AI agent reads to understand the entire product. A great PRD means the agent makes fewer wrong assumptions, asks fewer clarifying questions, and writes better code. This is the most important document you'll create.
Why PRD first? When you give Cursor or Claude a well-structured PRD, it can: understand all features holistically, make consistent naming decisions, avoid building things that conflict with each other, and generate the right database schema from day one. Without a PRD, agents build features in isolation and create a mess.
PRD Generation Prompts
PROMPT 01
Generate Full PRD from Idea Brief
OUTPUT: PRD.mdMOST IMPORTANT PROMPT
Input: Paste your IDEA.md content after the prompt. Output: Save as PRD.md — this feeds everything else.
You are a senior product manager and technical architect. Read the Idea Brief below and generate a complete Product Requirements Document (PRD.md) I will use to instruct an AI code agent to build this product. IDEA BRIEF: [Paste your IDEA.md content here] GENERATE A PRD WITH THESE EXACT SECTIONS: ## 1. Overview Product name, one-paragraph description, primary user persona, core value proposition. ## 2. User Stories All user stories in format: "As a [user], I want to [action] so that [benefit]." Group by feature area. Include happy path AND error cases. ## 3. Features — MVP Every feature in the MVP. For each: - Feature name - Description (what it does, not how) - Acceptance criteria (bullet list — how do I know it's done?) - Priority: P0 (must ship) / P1 (should ship) / P2 (nice to have) ## 4. Data Models Every entity in the system. For each: - Entity name - Fields with types (id, name: string, email: string, createdAt: timestamp, etc.) - Relationships (User has many Posts, etc.) ## 5. API Surface Every API endpoint needed. Format: METHOD /path — Description Request body (if any) Response shape ## 6. Tech Stack Decision Recommended stack with reasoning. Include: - Frontend framework + why - Backend framework + why - Database + why - Auth solution + why - Deployment target + why - Any key third-party services (payments, email, storage, etc.) ## 7. Non-Functional Requirements Performance, security, scalability expectations for MVP. ## 8. Out of Scope Explicitly list what is NOT in v1. ## 9. Open Questions Any decisions that need to be made before building starts. IMPORTANT: Be specific and complete. Vague PRDs produce vague code. Every feature should be precise enough that a developer could build it without asking questions.
PROMPT 02
PRD Review — Find Gaps Before Building
QUALITY CHECK
Review this PRD as a senior engineer about to build it. Find every gap, ambiguity, or missing decision that would cause problems during development. PRD: [Paste your PRD.md] CHECK FOR: 1. Missing user flows — what happens when X fails? What's the empty state? What's the error state? 2. Undefined data — are there fields referenced in the UI that aren't in the data model? 3. Auth gaps — which endpoints are public vs authenticated? What permissions exist? 4. Edge cases — what happens with empty lists, duplicate entries, concurrent edits? 5. Missing endpoints — does every user story have the API call it needs? 6. Tech conflicts — does the chosen stack actually work well together for this use case? 7. Scope creep signals — anything in the PRD that's clearly out of MVP scope but snuck in? Output: numbered list of issues with suggested resolutions. Then rate the PRD 1-10 for buildability.

// PHASE 1 — PLANNING
🗂️ Phase Planning — Split PRD into Milestones
The PRD tells the agent WHAT to build. Phases tell it WHEN and IN WHAT ORDER. Each phase is a .md file you feed to the agent when you start that phase. This prevents context overflow and keeps each session focused.
Why phases matter: AI agents have context limits. If you feed them the entire PRD and say "build everything", they lose track. Phases solve this: each session = one phase file + specific instructions. The agent finishes phase 1 completely before you start phase 2.
Phase Generation Prompts
PROMPT 01
Generate All Phase Files from PRD
OUTPUT: PHASE-1.md → PHASE-N.mdKEY PROMPT
Output: Multiple .md files — one per phase. Each is a self-contained instruction set for the AI agent.
Read this PRD and split it into execution phases. Each phase should be a self-contained, shippable increment that I can give to an AI code agent as a focused task. PRD: [Paste your PRD.md] RULES FOR PHASES: - Each phase must be independently testable (I can run the app and verify it works) - Backend comes before frontend for each feature area - Infrastructure (auth, DB, config) is always Phase 1 - No phase should take more than 2-3 days of AI-assisted coding - Each phase builds on the previous — no orphaned features FOR EACH PHASE, OUTPUT A MARKDOWN FILE: # PHASE-[N]: [Name] ## Goal One sentence: what works at the end of this phase. ## What to Build Numbered list of specific tasks the agent must complete. ## Files to Create List every file that should exist after this phase. ## API Endpoints (if backend work) METHOD /path — description for each endpoint in this phase. ## Database Changes (if any) Tables/columns to create or modify. ## Success Criteria How I verify this phase is complete (checklist). ## Dependencies Which previous phases must be done first. Generate all phases now. Aim for 4-6 phases for an MVP.

For every file you create, add a brief comment at the top explaining its purpose 5. After completing each task, confirm it before moving to the next 6. If you find a bug or conflict with the PRD, flag it before proceeding 7. When the phase is complete, list all files created/modified CURRENT PROJECT STATE: [Describe what's already built: "Phase 1 and 2 are complete. The auth system and user model are working. The frontend Next.js app is set up with the design system."] Begin with Task 1. Confirm before starting each subsequent task.

// PHASE 2 — BUILD
🖥️ Backend Build Prompts
Build the backend first. All endpoints, authentication, database operations, and business logic — before touching the frontend. This gives you a stable API contract the frontend can rely on.
Order matters: Backend first → test with Postman/REST client → then build frontend against working endpoints. Never build frontend and backend simultaneously — you'll spend all your time debugging which side has the bug.

Cursor Model Settings
Recommended Cursor setup: Default model = claude-3-5-sonnet-20241022 for daily coding. Switch to claude-3-7-sonnet for complex architectural work. Use cursor-fast (Claude Haiku) for autocomplete to save credits.
When to switch models mid-project+
  • Stuck on a bug for 30+ min: switch to o1 or Claude 3.7 with extended thinking. The extra cost is worth the breakthrough.
  • Adding a big new feature: start a new Cursor chat with fresh context. Feed it the PRD + current CONTEXT.md + the phase file.
  • Agent keeps breaking things: switch to a more conservative model (Claude 3.5 Sonnet is safer than Claude 3.7 Sonnet for not over-engineering).
  • Running low on credits: switch to Gemini 2.0 Flash or DeepSeek V3 via Cursor's "Use API key" for cheap continuation.
Context window matters — what this means in practice+
  • Claude 3.5/3.7: 200K tokens (~150K words). Can read your entire medium-sized codebase in one context window.
  • GPT-4o: 128K tokens. Smaller context = more summarization needed for large projects.
  • Gemini 1.5/2.0: 1-2M tokens. Best choice when you need to give the model the entire codebase at once.
  • Practical rule: Use Cursor's file attachment selectively. Only include files directly relevant to the current task. Too much context confuses the model — more is not always better.

Golden Rules of Vibe Coding
1. Always have a PRD before touching code+
The agent needs a source of truth. Without a PRD, it makes assumptions that compound into architectural problems. 30 minutes writing a PRD saves 3 hours of debugging bad code. When in doubt: stop coding, update the PRD, then continue.
2. One phase at a time — verify before moving forward+
Never ask the agent to do Phase 2 work while Phase 1 is incomplete. Each phase should be verifiable: you can run the app and see Phase 1 working. Fix bugs in Phase 1 before starting Phase 2. Technical debt from unverified phases multiplies.
3. Backend first, always+
Build and test the API before the frontend. When you build them simultaneously, you can't tell which side has the bug. A working backend with curl tests gives you confidence. Then the frontend just connects to something real.
4. Fresh context for each session+
Start each Cursor session with a fresh chat. Give it: PRD.md + current CONTEXT.md + the specific phase file. Don't rely on the agent "remembering" previous sessions — it doesn't. Fresh context prevents drift and hallucination from stale information.
5. Never let the agent choose the scope+
Agents WILL add features you didn't ask for. They'll add extra validation, create new tables, add helper utilities. This seems helpful but creates unreviewed code. In your prompts: explicitly say "only implement what's in this phase file, do not add anything extra". Review every file the agent creates.
6. Commit often — give yourself recovery points+
Commit after every phase completion, and after every significant working state. When the agent breaks something, git reset is your best friend. Vibe coding produces a lot of "it was working 20 minutes ago" moments — commits are your undo button.
