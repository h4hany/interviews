Coding Interview Prompts
Full simulation prompts for technical coding rounds. Replace the orange fields with your specifics, then paste into Claude, ChatGPT, or Gemini.
How to use: Copy the full prompt → paste into a new AI chat → respond as you would in a real interview. Don't ask for hints. Time yourself. When done, the AI will give detailed feedback.
Full Coding Round Simulation
Complete 45-minute mock interview. One problem, interviewer behavior, timed feedback.
PROMPT 01
Standard Coding Interview — Any Level
ALL LEVELS45 MINCLAUDE / GPT
When to use: Standard weekly coding practice. Works for junior through senior levels. Change the difficulty and language.
You are a senior software engineer at [COMPANY NAME e.g. Google, Meta, a fintech startup] conducting a live coding interview. I am interviewing for a [LEVEL e.g. Senior / Mid / Junior] Software Engineer position. ═══ YOUR BEHAVIOR AS INTERVIEWER ═══ - Give me exactly ONE coding problem appropriate for [LEVEL] at [COMPANY NAME] - Do NOT give hints unless I say "I am completely stuck and need a hint" - Respond ONLY with brief, realistic interviewer-style replies - If I ask clarifying questions, answer them as a real interviewer would (sometimes say "that's a good question", sometimes redirect with "what do you think?") - Keep ALL responses short (2-4 lines max) until I say I'm done ═══ MY CONSTRAINTS ═══ - Language: [Python / JavaScript / TypeScript / Java / Go] - Target difficulty: [Easy / Medium / Hard] - Focus area (optional): [arrays / trees / dynamic programming / graphs / any] ═══ WHEN I SAY "DONE" ═══ Evaluate my solution on: 1. Correctness — does it handle all cases? 2. Time complexity — is it optimal? What is it? 3. Space complexity — can it be improved? 4. Edge cases I missed 5. Code quality (naming, structure, readability) 6. What a stronger answer would look like 7. One follow-up question you'd ask in a real interview 8. Overall signal: Strong Hire / Hire / No Hire / Strong No Hire ═══ START ═══ Introduce yourself as the interviewer in ONE sentence, then give me the problem. Do NOT explain it — state it the way a real interviewer would (concisely).
PROMPT 02
Live Debugging Session
MID-SENIOR30 MIN
When to use: Some companies (Stripe, Shopify) give you broken code to debug. This simulates that format.
You are a senior engineer at [COMPANY] running a debugging interview. I am a [LEVEL] candidate. ═══ FORMAT ═══ 1. Give me a realistic code snippet in [LANGUAGE] that has 2-4 bugs 2. Tell me what the code is SUPPOSED to do — but don't reveal the bugs 3. I will diagnose and fix the issues 4. Ask me questions as I debug: "what made you look there?", "how did you confirm that?" 5. When I'm done: reveal all bugs, score my debugging approach, and give one improvement tip ═══ BUG TYPES ═══ Include bugs from: [logic error / off-by-one / race condition / null handling / any] ═══ CONTEXT ═══ This is production code. The stakes are realistic. Start now.
PROMPT 03
Solution Review & Optimization
ALL LEVELSREVIEW
When to use: You already solved a problem and want deep feedback on your solution.
I just solved a coding problem and want expert-level review as if I submitted this in a real interview. ═══ THE PROBLEM ═══ [Paste the problem statement here] ═══ MY SOLUTION ═══ [Paste your code here] ═══ EVALUATE ON ═══ 1. Correctness — test it against: empty input, single element, duplicates, negative numbers, very large input 2. Time complexity — what is it and can it be improved? 3. Space complexity — can it be reduced? 4. Edge cases my solution misses 5. Code style — naming, structure, comments 6. What the optimal solution looks like (show code if different from mine) 7. Interview signal: Strong Hire / Hire / No Hire 8. The one thing I should change first Be direct and specific. Line-number references preferred.

Full System Design Round
SENIOR+60 MINSTAFF LEVEL
When to use: Weekly system design practice. Most realistic simulation of a real system design round.
You are a staff engineer at [COMPANY e.g. Stripe, Netflix, Uber] running a system design interview. I am a [Senior / Staff / Principal] candidate. ═══ YOUR BEHAVIOR ═══ - Give me ONE system design problem relevant to [COMPANY]'s domain - Be a REAL interviewer: terse, probing, skeptical where appropriate - Ask "why did you choose X over Y?" at least twice - Ask about: scale numbers, failure modes, bottlenecks, trade-offs - Do NOT volunteer information — make me think for it - If I go down a wrong path, ask "is that really the right approach here?" (don't correct outright) - Keep responses to 2-3 sentences ═══ MY BACKGROUND ═══ [2-3 sentences about your experience level and what you've built] ═══ AFTER I SAY "DONE" ═══ Score me on: 1. Requirements gathering — did I ask the right questions? 2. High-level design — was the architecture sound? 3. Deep dives — did I go deep enough on the right components? 4. Scale — did I think about bottlenecks and how to address them? 5. Trade-offs — did I articulate WHY I made choices? 6. What I missed (top 3 things) 7. What a strong answer added that I didn't 8. Verdict: Strong Hire / Hire / No Hire ═══ START ═══ State your name and the question in one sentence. No preamble.

// PROMPT ENGINEERING
Prompt Fundamentals
The principles that work across every AI model. Master these and you'll get better results from Claude, GPT-4, Gemini, and any future model.
The core insight: AI models are prediction engines. They predict the most likely continuation of your input. Better prompts create better context, which leads to better predictions. These fundamentals help you give better context.
Core Techniques
These 6 techniques improve results across every use case.
TECHNIQUE 01
Role + Context + Constraint Formula
UNIVERSALALWAYS USE
Why it works: AI performs dramatically better when given an explicit role and specific constraints. "Role priming" shifts the model's vocabulary, style, and depth of response toward the right persona.
THE FORMULA: You are [SPECIFIC ROLE with seniority/expertise level]. [CONTEXT: relevant background about the task or situation] [YOUR TASK: what you want, stated precisely] [CONSTRAINTS: format, length, tone, what to avoid] [OUTPUT FORMAT: how you want the answer structured] EXAMPLE — WEAK PROMPT: "Explain Kubernetes" EXAMPLE — STRONG PROMPT: You are a senior DevOps engineer with 8 years of Kubernetes production experience. I am a backend developer who understands Docker but has never used Kubernetes. Explain Kubernetes to me using analogies to Docker concepts I already know. Keep it under 400 words. Use bullet points for the key concepts. End with: the one thing I need to understand before anything else. KEY RULES: → Be specific about the role's expertise level ("senior" > "expert" > nothing) → Give context about the AUDIENCE (who is reading this answer) → State what to AVOID, not just what to do → Specify output structure when it matters → One clear ask is better than three
TECHNIQUE 02
Chain of Thought — Force Visible Reasoning
LOGIC TASKSANALYSIS
WHEN TO USE: Complex reasoning, math, code debugging, decision-making, multi-step problems. TECHNIQUE: Add one of these phrases to force visible reasoning: → "Think step by step before answering." → "Before giving your final answer, work through the problem out loud." → "Reason through this first, then conclude." → "Show your work." → "What are the key considerations? Then answer." EXAMPLE TEMPLATE: [Your question or problem] Before answering, think through: 1. What are the key factors in this decision? 2. What would each option mean in practice? 3. What am I possibly missing? Then give your recommendation. WHY IT WORKS: Models that "think out loud" make fewer logical errors. The reasoning tokens serve as working memory. For code: ask the model to explain the approach before writing it — the code quality improves significantly.
TECHNIQUE 03
Few-Shot Examples — Show Don't Tell
FORMAT CONTROLSTYLE MATCHING
WHEN TO USE: When you want a specific format, tone, or style that's hard to describe in words. TECHNIQUE: Provide 1-3 examples of the output you want, then ask for more. TEMPLATE: I want you to write [CONTENT TYPE] in this exact style and format. Here are examples of what I want: Example 1: [Paste an example of the style/format you want] Example 2: [Another example if needed] Now write [YOUR SPECIFIC REQUEST] using this same style, format, and tone. Do not deviate from the pattern set by the examples. WORKS GREAT FOR: → LinkedIn posts in your voice → Code in your team's style → Emails that match your tone → Any content where "like this" is clearer than describing it
TECHNIQUE 04
Output Format Control
STRUCTUREJSON / MARKDOWN
CONTROL OUTPUT FORMAT WITH EXPLICIT INSTRUCTIONS: For structured data (JSON): Respond ONLY with valid JSON. No preamble, no markdown code blocks, no explanation. Format: { "field1": "...", "field2": ["...", "..."], "field3": { "nested": "..." } } For markdown documents: Format your response as a markdown document with: - H2 headers for main sections - Bullet points for lists - Bold for key terms - A TL;DR at the top For tables: Present this as a markdown table with columns: [COL1] | [COL2] | [COL3] Sort rows by [CRITERION]. For concise answers: Answer in exactly [N] sentences. Answer in under [N] words. Use only bullet points — no prose. Give me ONLY the code, nothing else. KEY INSIGHT: If you say "answer in 3 bullet points" at the END of a long prompt, models often ignore it. Put FORMAT INSTRUCTIONS near the start, not the end.
TECHNIQUE 05
Negative Instructions — What NOT to Do
QUALITY CONTROL
MOST UNDER-USED TECHNIQUE: Telling the AI what to AVOID is often more effective than telling it what to do. COMMON NEGATIVES TO ADD: → "Do not use filler phrases like 'Great question!' or 'Certainly!'" → "Do not add a disclaimer or caveat at the end." → "Do not summarize what you just said." → "Do not hedge every statement with 'it depends' — give me your actual recommendation." → "Do not use passive voice." → "Do not explain what you're about to do — just do it." → "Do not repeat my question back to me." → "Do not be diplomatic. Be direct." TEMPLATE ADDITION: Add a "Do NOT" section to any prompt: ...your main prompt... DO NOT: - Start with "Certainly!" or any affirmation - Add a disclaimer about limitations - Use more than 3 sentences for the introduction - Use the word "utilize" (use "use" instead) - End with "I hope this helps!"
TECHNIQUE 06
Iterative Refinement — The Power of Follow-Ups
WORKFLOWADVANCED
PRINCIPLE: The best output rarely comes from one prompt. Build iteratively. THE ITERATIVE WORKFLOW: Step 1 — Draft: Generate a [CONTENT TYPE] about [TOPIC]. Keep it rough. Step 2 — Critique: "What are the 3 weakest parts of what you just wrote?" Step 3 — Improve: "Rewrite focusing on fixing those 3 weaknesses." Step 4 — Constrain: "Now cut it to 50% of the length without losing the key points." Step 5 — Tone: "Make it [more direct / warmer / more technical / simpler]." USEFUL FOLLOW-UP PROMPTS FOR ANY RESPONSE: → "Now make it 50% shorter." → "Now make it more direct and less hedged." → "Give me 3 alternative versions." → "What did you leave out that might be important?" → "Now write this from the opposite perspective." → "Critique your own answer — what's weak about it?" → "Now rewrite the introduction to be more compelling." → "What would a skeptic say about this?"
// PROMPT ENGINEERING — IMAGE
Image Generation Prompts
Master formulas for Midjourney, DALL-E 3, Stable Diffusion, and Flux. Every image prompt has a structure — learn it once and apply to any subject.
The Image Prompt Formula: [Subject] + [Setting/Environment] + [Style/Medium] + [Lighting] + [Mood/Atmosphere] + [Technical parameters] — in roughly that order. The more specific each element, the more predictable and controllable the result.

// PROMPT ENGINEERING — VIDEO
Video Generation Prompts
Sora, Runway Gen-3, Kling, Pika, and Hailuo. Video prompts need camera direction, motion specifics, and temporal flow — different from image prompts.
Video Prompt Formula: [Camera move] + [Subject doing action] + [Setting] + [Lighting/time] + [Mood] + [Speed/style]. The key difference from image prompts: you must describe MOTION explicitly. What is moving? How fast? What direction?

The 6 Universal Prompt Rules
1. Give the AI a SPECIFIC role with seniority level+
"You are a senior X" outperforms "you are an expert X" which outperforms nothing. The more specific the role, the more the model draws on relevant knowledge patterns. Add the context of WHO is asking (audience) not just WHO is answering.
2. State CONSTRAINTS, not just goals+
Tell the AI what NOT to do. "Do not add disclaimers", "Do not summarize what you just wrote", "Do not hedge every statement". Negative constraints are often more powerful than positive instructions because they prevent the AI's default filler behavior.
3. Specify the OUTPUT FORMAT explicitly+
Without format instructions, AI defaults to markdown with headers and bullets. Specify: plain prose / JSON / numbered list / table / code only / one-sentence answer. Put format instructions NEAR THE TOP of the prompt, not buried at the end where they get ignored.
4. Use "Think step by step" for complex reasoning+
Chain-of-thought prompting (asking the AI to reason visibly before answering) significantly reduces logical errors. For code: "explain your approach before writing it". For decisions: "list key considerations, then recommend". The reasoning tokens serve as working memory.
5. Show examples (few-shot) when words aren't enough+
When you want a specific style, tone, or format that's hard to describe — show 1-3 examples first. "Write more like this: [example]" is more powerful than 5 sentences trying to describe the style. This is especially true for creative content and specific writing voices.
6. Iterate — one prompt is rarely your best result+
The best output comes from iteration. Draft → critique ("what are the 3 weakest parts?") → improve → shorten → tone-adjust. Build a workflow, not a single prompt. Save prompts that work well. Build a library (this one) rather than starting from scratch each time.
