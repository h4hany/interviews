#### ❓ Explain the Node.js Event Loop in depth.

**Answer:**

Node.js runs on a **single-threaded event loop** powered by **libuv**, which delegates I/O operations to the OS kernel or a thread pool. The event loop has **six phases**, executed in order:

```
   ┌─────────────────────────────────┐
   │           timers                │  ← setTimeout, setInterval callbacks
   │─────────────────────────────────│
   │      pending callbacks          │  ← I/O errors deferred from prev cycle
   │─────────────────────────────────│
   │        idle, prepare            │  ← internal use
   │─────────────────────────────────│
   │            poll                 │  ← retrieve new I/O events; execute callbacks
   │─────────────────────────────────│
   │            check                │  ← setImmediate callbacks
   │─────────────────────────────────│
   │       close callbacks           │  ← socket.on('close', ...)
   └─────────────────────────────────┘
```

**Microtask queue** (process.nextTick, Promise callbacks) runs **between each phase**, not during phases. `process.nextTick` runs before Promises in the microtask queue.

**Staff-level insight:** Understanding this matters at Yassir because a ride dispatch service processing thousands of concurrent WebSocket connections must avoid blocking the event loop. Long-running synchronous CPU work (e.g., route calculation) must be offloaded to **Worker Threads** or a separate compute service to prevent head-of-line blocking for all other requests.

```javascript
// BAD: Blocking the event loop
app.get('/route', (req, res) => {
  const result = computeHeavyRoute(req.params); // blocks everything
  res.json(result);
});

// GOOD: Offload to Worker Thread
const { Worker } = require('worker_threads');
app.get('/route', (req, res) => {
  const worker = new Worker('./routeWorker.js', { workerData: req.params });
  worker.on('message', (result) => res.json(result));
  worker.on('error', (err) => res.status(500).json({ error: err.message }));
});
```

#### ❓ Worker Threads vs. Cluster Module — when do you use each?

**Answer:**

| | **Worker Threads** | **Cluster** |
|---|---|---|
| **Isolation** | Shared memory (SharedArrayBuffer) possible | Separate OS processes, no shared memory |
| **Use case** | CPU-intensive tasks within a single service (e.g., image processing, ML inference, crypto) | Scale HTTP server across CPU cores |
| **Memory** | Lower overhead (threads) | Higher overhead (forked processes) |
| **Failure isolation** | Worker crash doesn't kill main thread | Worker crash doesn't kill master |
| **Communication** | `postMessage` / SharedArrayBuffer | IPC messages / load balancer |

**At Yassir scale:** Use **Cluster** to saturate all CPU cores for the ride-hailing API Gateway. Use **Worker Threads** inside a pricing service to compute surge pricing algorithms without blocking the main request thread.

```javascript
// Cluster for HTTP scaling (rides API)
const cluster = require('cluster');
const numCPUs = require('os').cpus().length;

if (cluster.isPrimary) {
  for (let i = 0; i < numCPUs; i++) cluster.fork();
  cluster.on('exit', (worker) => {
    console.log(`Worker ${worker.process.pid} died, restarting...`);
    cluster.fork(); // auto-restart
  });
} else {
  require('./server'); // each worker runs the full HTTP server
}
```

#### ❓ How do you handle memory leaks in a long-running Node.js service?

**Answer:**

Common sources in production Node.js services:
1. **Event listener accumulation** — `emitter.on()` in loops without `.removeListener()`
2. **Closure capturing large objects** — callbacks holding references preventing GC
3. **Unbounded in-memory caches** — Maps/Sets that grow without eviction
4. **Uncleaned timers/intervals** — `setInterval` without `clearInterval` on shutdown
5. **Stream backpressure ignored** — writing faster than consumer can handle

**Diagnosis tools:**
- `--inspect` flag + Chrome DevTools heap snapshots
- `heapdump` npm package for production snapshots
- `process.memoryUsage()` logged as a metric to Cloud Monitoring
- `clinic.js` (clinic heap) for profiling in staging

```javascript
// Memory-safe cache with TTL eviction (production pattern)
class BoundedCache {
  constructor(maxSize = 1000, ttlMs = 60_000) {
    this.cache = new Map();
    this.maxSize = maxSize;
    this.ttlMs = ttlMs;
  }

  set(key, value) {
    if (this.cache.size >= this.maxSize) {
      // Evict oldest entry (LRU-lite)
      const firstKey = this.cache.keys().next().value;
      this.cache.delete(firstKey);
    }
    this.cache.set(key, { value, expiresAt: Date.now() + this.ttlMs });
  }

  get(key) {
    const entry = this.cache.get(key);
    if (!entry) return null;
    if (Date.now() > entry.expiresAt) {
      this.cache.delete(key);
      return null;
    }
    return entry.value;
  }
}
```

#### ❓ Explain Node.js Streams and when you'd use them at Yassir's scale.

**Answer:**

Streams process data **chunk by chunk** without loading the entire payload into memory. The four types: Readable, Writable, Duplex, Transform.

**Key concept — backpressure:** The consumer must signal to the producer when it's ready for more data. Ignoring backpressure fills Node's internal buffer and causes memory bloat.

**Yassir use cases:**
- **Log aggregation**: Stream driver/courier activity logs from services → transform → Pub/Sub → BigQuery without loading millions of records in memory
- **CSV export of order history**: Pipe DB query results through a transform stream into an HTTP response
- **Real-time location updates**: Readable stream of GPS events piped through a transform that applies Kalman filtering, then to WebSocket clients

```javascript
// Streaming large order history export — production pattern
const { Transform, pipeline } = require('stream');
const { promisify } = require('util');
const pipelineAsync = promisify(pipeline);

async function exportOrderHistory(res, userId) {
  const dbStream = db.query(`SELECT * FROM orders WHERE user_id = $1`, [userId])
    .stream(); // pg cursor stream

  const jsonTransform = new Transform({
    objectMode: true,
    transform(row, _enc, cb) {
      cb(null, JSON.stringify(row) + '\n'); // NDJSON
    }
  });

  res.setHeader('Content-Type', 'application/x-ndjson');
  await pipelineAsync(dbStream, jsonTransform, res);
  // pipeline handles backpressure and cleanup automatically
}
```

#### ❓ Explain the CAP Theorem and how it applies to Yassir's fintech layer.

**Answer:**

**CAP Theorem** states that a distributed system can guarantee at most **two of three**:
- **C**onsistency — every read receives the most recent write
- **A**vailability — every request receives a (non-error) response
- **P**artition Tolerance — system continues despite network partitions

Since **Partition Tolerance is non-negotiable** in any distributed system, real choices are:
- **CP** (Consistency + Partition Tolerance): sacrifice availability during partitions
- **AP** (Availability + Partition Tolerance): sacrifice strong consistency; accept eventual consistency

**For Yassir's fintech (payments/wallet):**
- **CP is required** for financial transactions — a user's wallet balance must never show an incorrect amount that results in double-spending
- Use **PostgreSQL with serializable isolation** for wallet mutations
- Use **distributed locks (Redis SETNX)** for critical sections like payment debit
- Apply **optimistic locking** (version columns) for high-contention updates

**For Yassir's ride-hailing (driver locations):**
- **AP is acceptable** — a driver's location being 2 seconds stale is fine; showing an error to users is not
- Use **Redis** (in-memory, AP by default) for driver location cache with short TTL

#### ❓ Implement a Circuit Breaker pattern in Node.js.

```javascript
// Production-grade Circuit Breaker implementation
class CircuitBreaker {
  constructor(fn, options = {}) {
    this.fn = fn;
    this.failureThreshold = options.failureThreshold || 5;
    this.resetTimeout = options.resetTimeout || 30_000; // 30s
    this.successThreshold = options.successThreshold || 2;

    this.state = 'CLOSED'; // CLOSED | OPEN | HALF_OPEN
    this.failureCount = 0;
    this.successCount = 0;
    this.nextAttempt = Date.now();
  }

  async call(...args) {
    if (this.state === 'OPEN') {
      if (Date.now() < this.nextAttempt) {
        throw new Error(`Circuit is OPEN. Service unavailable.`);
      }
      this.state = 'HALF_OPEN';
    }

    try {
      const result = await this.fn(...args);
      this.onSuccess();
      return result;
    } catch (err) {
      this.onFailure(err);
      throw err;
    }
  }

  onSuccess() {
    this.failureCount = 0;
    if (this.state === 'HALF_OPEN') {
      this.successCount++;
      if (this.successCount >= this.successThreshold) {
        this.state = 'CLOSED';
        this.successCount = 0;
      }
    }
  }

  onFailure(err) {
    this.failureCount++;
    if (this.state === 'HALF_OPEN') {
      this.state = 'OPEN';
      this.nextAttempt = Date.now() + this.resetTimeout;
      return;
    }
    if (this.failureCount >= this.failureThreshold) {
      this.state = 'OPEN';
      this.nextAttempt = Date.now() + this.resetTimeout;
    }
  }
}

// Usage: wrapping external payment gateway call
const paymentBreaker = new CircuitBreaker(
  (payload) => paymentGateway.charge(payload),
  { failureThreshold: 3, resetTimeout: 15_000 }
);

async function processPayment(payload) {
  try {
    return await paymentBreaker.call(payload);
  } catch (err) {
    if (err.message.includes('Circuit is OPEN')) {
      // Fallback: queue payment for async retry via Kafka
      await kafkaProducer.send('payments.retry', payload);
      return { status: 'queued', message: 'Payment queued for retry' };
    }
    throw err;
  }
}
```

#### ❓ How do you implement rate limiting at Yassir scale?

**Answer:**

At Yassir's scale, rate limiting must be:
- **Distributed** — multiple API Gateway instances share state via Redis
- **Sliding window** — more accurate than fixed window (no burst at window reset)
- **Tiered** — different limits for anonymous vs authenticated vs partner APIs

**Algorithm: Sliding Window Log with Redis**

```javascript
// NestJS Guard — Distributed Sliding Window Rate Limiter
import { Injectable, CanActivate, ExecutionContext, HttpException, HttpStatus } from '@nestjs/common';
import { InjectRedis } from '@liaoliaots/nestjs-redis';
import Redis from 'ioredis';

@Injectable()
export class RateLimiterGuard implements CanActivate {
  constructor(@InjectRedis() private readonly redis: Redis) {}

  async canActivate(context: ExecutionContext): Promise<boolean> {
    const request = context.switchToHttp().getRequest();
    const userId = request.user?.id || request.ip;
    const key = `rate:${userId}`;
    const windowMs = 60_000; // 1 minute
    const maxRequests = 100;
    const now = Date.now();
    const windowStart = now - windowMs;

    const pipeline = this.redis.pipeline();
    pipeline.zremrangebyscore(key, '-inf', windowStart);   // remove old entries
    pipeline.zadd(key, now, `${now}-${Math.random()}`);    // add current request
    pipeline.zcard(key);                                    // count in window
    pipeline.expire(key, 60);                              // TTL cleanup

    const results = await pipeline.exec();
    const requestCount = results[2][1] as number;

    if (requestCount > maxRequests) {
      throw new HttpException(
        { error: 'Too Many Requests', retryAfter: 60 },
        HttpStatus.TOO_MANY_REQUESTS
      );
    }
    return true;
  }
}
```

#### ❓ How do you design a multi-tier caching strategy for a delivery platform?

**Answer:**

```
Request → [CDN Cache] → [API Gateway Cache] → [Redis L1] → [Redis L2 / Cluster] → [PostgreSQL]
```

**Layer breakdown for Yassir food delivery:**

| Cache Layer | Tool | TTL | Use Case |
|---|---|---|---|
| **CDN** | Cloudflare / Cloud CDN | 5–60 min | Static assets, public restaurant menus |
| **API Gateway** | Apigee response cache | 30s–5min | Popular restaurant listings by city |
| **Application (L1)** | In-memory (Node.js Map) | 10–30s | Hot restaurant data per pod |
| **Distributed (L2)** | Redis | 1–15 min | User cart, driver location, session |
| **Database** | PostgreSQL query cache | — | Complex aggregated queries |

**Cache invalidation strategy:**
- **Write-through**: Update cache on every DB write (consistent, higher write latency)
- **Cache-aside**: Application checks cache, fetches from DB on miss, populates cache
- **Event-driven invalidation**: Pub/Sub event `restaurant.menu.updated` → invalidate all cache keys matching `menu:{restaurantId}:*`

```javascript
// Cache-aside with automatic invalidation via Pub/Sub (NestJS pattern)
@Injectable()
export class RestaurantService {
  constructor(
    @InjectRedis() private redis: Redis,
    private db: DatabaseService,
    private pubsub: PubSubService
  ) {}

  async getRestaurantMenu(restaurantId: string) {
    const cacheKey = `menu:${restaurantId}`;
    const cached = await this.redis.get(cacheKey);
    if (cached) return JSON.parse(cached);

    const menu = await this.db.query(
      'SELECT * FROM menu_items WHERE restaurant_id = $1 AND active = true',
      [restaurantId]
    );

    await this.redis.setex(cacheKey, 900, JSON.stringify(menu)); // 15 min TTL
    return menu;
  }

  async updateMenu(restaurantId: string, updates: MenuUpdate[]) {
    await this.db.transaction(async (trx) => {
      await trx.updateMenuItems(restaurantId, updates);
      // Publish invalidation event inside transaction
      await this.pubsub.publish('menu.invalidate', { restaurantId });
    });
  }
}

// Subscriber on all API Gateway pods
pubsub.subscribe('menu.invalidate', async ({ restaurantId }) => {
  await redis.del(`menu:${restaurantId}`);
});
```

#### ❓ How do you handle database scaling for Yassir's order volume?

**Answer:**

**Phase 1: Vertical + Read Replicas** (current scale)
- Promote primary PostgreSQL, add 2–3 read replicas
- Route read-heavy queries (order history, restaurant search) to replicas
- Connection pooling via **PgBouncer** (transaction mode) to handle thousands of Node.js connections efficiently

### Challenge 1: Distributed Rate Limiter (Token Bucket)

**Problem:** Implement a token bucket rate limiter that works across multiple API Gateway instances, limiting each user to 100 requests/minute, with burst capacity.

**Constraints:** O(1) per check, Redis-backed, atomic operations only.

```javascript
// Token Bucket Rate Limiter — Production Node.js
const Redis = require('ioredis');
const redis = new Redis(process.env.REDIS_URL);

/**
 * Token Bucket Rate Limiter
 * Capacity: maxTokens, Refill: maxTokens per windowMs
 * Uses Redis Lua script for atomicity
 */
const tokenBucketScript = `
local key = KEYS[1]
local capacity = tonumber(ARGV[1])
local refillRate = tonumber(ARGV[2])  -- tokens per second
local now = tonumber(ARGV[3])
local requested = tonumber(ARGV[4])

local bucket = redis.call('HMGET', key, 'tokens', 'lastRefill')
local tokens = tonumber(bucket[1]) or capacity
local lastRefill = tonumber(bucket[2]) or now

-- Calculate tokens to add since last refill
local elapsed = math.max(0, now - lastRefill)
local newTokens = math.min(capacity, tokens + (elapsed * refillRate))

if newTokens >= requested then
  -- Grant request
  redis.call('HMSET', key, 'tokens', newTokens - requested, 'lastRefill', now)
  redis.call('EXPIRE', key, 120)
  return {1, math.floor(newTokens - requested)}
else
  -- Deny request
  redis.call('HMSET', key, 'tokens', newTokens, 'lastRefill', now)
  redis.call('EXPIRE', key, 120)
  return {0, math.floor(newTokens)}
end
`;

class TokenBucketRateLimiter {
  constructor({ capacity = 100, refillRatePerSecond = 1.67 } = {}) {
    this.capacity = capacity;
    this.refillRate = refillRatePerSecond; // 100 tokens/min ≈ 1.67/s
  }

  async checkAndConsume(userId, tokensRequested = 1) {
    const key = `ratelimit:tokenbucket:${userId}`;
    const now = Date.now() / 1000; // seconds

    const [allowed, remainingTokens] = await redis.eval(
      tokenBucketScript,
      1,
      key,
      this.capacity,
      this.refillRate,
      now,
      tokensRequested
    );

    return {
      allowed: allowed === 1,
      remainingTokens: parseInt(remainingTokens),
      retryAfterSeconds: allowed ? 0 : Math.ceil((tokensRequested - remainingTokens) / this.refillRate)
    };
  }
}

// Express middleware integration
const limiter = new TokenBucketRateLimiter({ capacity: 100, refillRatePerSecond: 1.67 });

async function rateLimitMiddleware(req, res, next) {
  const userId = req.user?.id || req.ip;
  const { allowed, remainingTokens, retryAfterSeconds } = await limiter.checkAndConsume(userId);

  res.set('X-RateLimit-Remaining', remainingTokens);

  if (!allowed) {
    return res.status(429).json({
      error: 'Too Many Requests',
      retryAfter: retryAfterSeconds
    });
  }
  next();
}
```

**Complexity:** O(1) time, O(1) space per user. Redis Lua script ensures atomicity without distributed locks.

### Challenge 2: LRU Cache with O(1) Operations

**Problem:** Implement an LRU Cache with `get` and `put` in O(1) time. Used for in-memory caching of driver locations at the pod level.

```javascript
class LRUCache {
  constructor(capacity) {
    this.capacity = capacity;
    this.map = new Map(); // key → Node
    // Doubly linked list: head (most recent) ←→ ... ←→ tail (least recent)
    this.head = { key: null, value: null, prev: null, next: null };
    this.tail = { key: null, value: null, prev: null, next: null };
    this.head.next = this.tail;
    this.tail.prev = this.head;
  }

  get(key) {
    if (!this.map.has(key)) return -1;
    const node = this.map.get(key);
    this._remove(node);
    this._addToFront(node);
    return node.value;
  }

  put(key, value) {
    if (this.map.has(key)) {
      const node = this.map.get(key);
      node.value = value;
      this._remove(node);
      this._addToFront(node);
    } else {
      if (this.map.size >= this.capacity) {
        // Evict LRU (tail's prev)
        const lru = this.tail.prev;
        this._remove(lru);
        this.map.delete(lru.key);
      }
      const newNode = { key, value, prev: null, next: null };
      this._addToFront(newNode);
      this.map.set(key, newNode);
    }
  }

  _remove(node) {
    node.prev.next = node.next;
    node.next.prev = node.prev;
  }

  _addToFront(node) {
    node.next = this.head.next;
    node.prev = this.head;
    this.head.next.prev = node;
    this.head.next = node;
  }
}

// Usage: cache last 10,000 driver locations in memory
const driverCache = new LRUCache(10_000);
driverCache.put('driver:abc123', { lat: 36.7538, lng: 3.0588, updatedAt: Date.now() });
const loc = driverCache.get('driver:abc123');
```

**Complexity:** O(1) get and put. O(n) space.

### Challenge 5: Async Task Processor with Concurrency Control

**Problem:** Process a large batch of geocoding tasks asynchronously with a maximum concurrency of N, respecting external API rate limits.

```javascript
/**
 * Async task processor with controlled concurrency
 * Used for: batch geocoding courier pickup points, bulk price calculations
 */
class ConcurrentTaskProcessor {
  constructor(concurrency = 10, rateLimit = 50) {
    this.concurrency = concurrency;
    this.rateLimit = rateLimit; // requests per second
    this.intervalMs = 1000 / rateLimit;
    this.queue = [];
    this.running = 0;
    this.lastRequestTime = 0;
  }

  async process(tasks, taskFn) {
    const results = new Array(tasks.length);
    let taskIndex = 0;

    await new Promise((resolve, reject) => {
      const runNext = async () => {
        if (taskIndex >= tasks.length && this.running === 0) {
          return resolve();
        }
        if (this.running >= this.concurrency || taskIndex >= tasks.length) return;

        // Rate limiting: enforce minimum interval between requests
        const now = Date.now();
        const timeSinceLastRequest = now - this.lastRequestTime;
        if (timeSinceLastRequest < this.intervalMs) {
          await new Promise(r => setTimeout(r, this.intervalMs - timeSinceLastRequest));
        }

        const currentIndex = taskIndex++;
        const task = tasks[currentIndex];
        this.running++;
        this.lastRequestTime = Date.now();

        taskFn(task)
          .then(result => { results[currentIndex] = { success: true, result }; })
          .catch(error => { results[currentIndex] = { success: false, error: error.message }; })
          .finally(() => {
            this.running--;
            runNext(); // Fill concurrency slot
          });

        // Fill up to concurrency limit
        if (this.running < this.concurrency && taskIndex < tasks.length) {
          runNext();
        }
      };

      runNext();
    });

    return results;
  }
}

// Usage: batch geocode 10,000 restaurant addresses
const processor = new ConcurrentTaskProcessor(10, 50); // 10 concurrent, 50 req/s

const addresses = await db.query('SELECT id, address FROM restaurants WHERE geocoded = false LIMIT 10000');
const results = await processor.process(addresses.rows, async (row) => {
  const coords = await geocodingService.geocode(row.address);
  await db.query('UPDATE restaurants SET lat=$1, lng=$2, geocoded=true WHERE id=$3',
    [coords.lat, coords.lng, row.id]);
  return coords;
});

console.log(`Processed: ${results.filter(r => r.success).length} / ${results.length}`);
```

**Complexity:** O(n) where n = tasks. Memory: O(concurrency) active tasks at any time.

### ❓ Tell me about a time you drove a major architectural decision across multiple teams.

**STAR Answer:**

**Situation:** At Kinship, our platform had grown from a monolith into a loosely coupled set of services, but teams were duplicating infrastructure concerns: each service had its own auth middleware, custom error handling, and inconsistent logging formats. This was causing debugging nightmares across 6 services and a maintenance burden.

**Task:** As the Staff Engineer, I needed to define and drive adoption of a shared backend platform — without being a bottleneck or creating an ivory tower architecture.

**Action:**
1. **Discovery phase (2 weeks)**: Audited all 6 services, documented pain points, interviewed team leads. Created a matrix of commonalities vs. divergences.
2. **RFC process**: Wrote a Request for Comments document proposing a shared NestJS library (`@kinship/backend-core`) covering auth, logging, error handling, rate limiting, and health checks. Published it to Notion and gave teams 1 week to comment.
3. **Prototype first**: Built a proof of concept in the lowest-risk service. Demonstrated a 30% reduction in boilerplate and a new service standing up in <1 hour instead of 2 days.
4. **Incremental adoption**: Didn't mandate a big-bang migration. Created a migration guide and paired with each team during their next sprint cycle to adopt the shared library.
5. **Owned the library**: Became the primary maintainer, set up automated publishing via CI/CD, versioned with semver, maintained a changelog.

**Result:** All 6 services migrated over 8 weeks. New service onboarding time dropped from ~2 days to ~4 hours. Observability improved dramatically — all services now emit consistent structured logs consumed by the same Datadog dashboard. One P2 incident was caught in 8 minutes instead of the previous average of 45 minutes.

### ❓ Describe a time you had a technical conflict with a peer or lead.

**STAR Answer:**

**Situation:** At Escape Ventures, I recommended migrating from a polling-based notification system to an event-driven architecture using Kafka. A senior engineer strongly opposed, citing operational complexity and the team's lack of Kafka experience.

**Task:** Navigate the disagreement constructively and arrive at the best technical decision for the team, not just win the argument.

**Action:**
1. **Acknowledged their concern first**: They were right that Kafka has operational overhead. I didn't dismiss that.
2. **Made the trade-offs explicit**: Created a one-pager comparing: polling vs WebSockets vs Kafka on dimensions of scalability, operational complexity, team learning curve, and cost.
3. **Proposed a middle ground**: Started with Redis Pub/Sub for notifications (simpler, team already knew Redis) with a documented migration path to Kafka once team confidence and load warranted it.
4. **Time-boxed the decision**: Proposed a 3-month review point to re-evaluate based on actual load data.
5. **Socialized with the team**: Brought the question to a team design review so the decision had broader input, not just our two opinions.

**Result:** We shipped Redis Pub/Sub in 2 weeks. At the 3-month review, load data confirmed we needed Kafka for message replay and consumer group support. The gradual transition was much smoother because the team had grown their event-driven intuition with Redis first. The senior engineer became a champion of the Kafka migration.

### ❓ How do you mentor engineers and grow a team's technical maturity?

**STAR Answer:**

**Situation:** Joined Kinship where the backend team had 4 mid-level engineers who wrote working code but struggled with system design thinking and production readiness.

**Task:** Elevate team maturity without creating dependency on me as a single source of truth.

**Action:**
1. **1-on-1 technical pairing**: Weekly 45-min sessions per engineer focused on a specific growth area (one on database query optimization, one on testing strategies, one on API design).
2. **Design review culture**: Established a lightweight RFC process for any feature touching >2 services. Engineers had to write the RFC themselves; I gave structured feedback. Over time they internalized the thinking.
3. **Blameless post-mortems**: After every incident, ran a structured post-mortem. Focus: what did the system fail to make obvious, not who made the mistake. Published findings to the team wiki.
4. **Stretch assignments**: Deliberately assigned a mid-level engineer to own a cross-service migration end-to-end, with me as the advisor, not the driver. Watched them grow into a senior engineer over 6 months.
5. **Tech talks**: Monthly 30-minute internal talks. Engineers took turns. Topics ranged from "What I learned debugging a memory leak" to "Introduction to distributed tracing."

**Result:** All 4 engineers grew measurably. One was promoted to Senior within 9 months. The team started catching design issues in review that previously made it to production. Incident MTTR dropped by 60%.

### ❓ How do you influence product decisions as a Staff Engineer?

**Answer (Behavioral + Framework):**

At Staff level, you're not just executing product decisions — you're helping shape them. My approach:

1. **Bring data, not opinions**: When advocating for a technical investment, I translate it into business impact. "Reducing our p99 API latency from 800ms to 200ms increases our delivery completion rate by X% based on our A/B test data" — that's a product argument, not just an engineering one.

2. **Join product planning early**: I ask to attend product discovery sessions, not just sprint planning. Understanding user problems early lets me flag technical risks or opportunities before they become expensive.

3. **Write clearly**: I've learned that a well-written 1-pager in product terms (user impact, risk, effort) is more persuasive than a technically perfect architecture diagram that product managers can't read.

4. **Build trust through delivery**: Influence is earned by consistently delivering. When you have a track record of shipping reliable systems on time, product and business leaders trust your technical judgment.

5. **Frame technical debt as product risk**: Instead of "we need to refactor the order service," say "our current order service architecture limits us to 2x traffic before we need a rewrite mid-launch. If we invest 3 sprints now, we get 20x headroom for the next 18 months."

## 12. Quick Revision Cheat Sheet

### 🔑 Node.js Key Concepts

| Concept | Key Point |
|---|---|
| **Event Loop Phases** | timers → pending callbacks → idle/prepare → poll → check → close callbacks |
| **Microtask Queue** | Runs between phases. `process.nextTick` before Promises |
| **libuv Thread Pool** | Default 4 threads. Handles fs, crypto, DNS. Set `UV_THREADPOOL_SIZE` |
| **Worker Threads** | Shared memory possible. For CPU-bound. `worker_threads` module |
| **Cluster** | Multiple processes. OS-level load balancing. No shared memory |
| **Backpressure** | Readable.pipe handles it automatically. Manual: check `writable.write()` return |
| **Memory Heap** | V8 manages. Old space (long-lived), New space (short-lived, GC frequently) |
| **Event Emitter leak** | Default max 10 listeners. `setMaxListeners(0)` to disable warning |

### 📐 Important Formulas & Thresholds

| Metric | Rule of Thumb |
|---|---|
| **Redis memory** | 64 bytes per key overhead. Plan for 2x actual data size |
| **PostgreSQL connections** | Max = RAM(GB) × 100 / 4 (rough). Use PgBouncer |
| **Node.js heap** | Default 1.4GB (64-bit). Override with `--max-old-space-size=4096` |
| **Kafka partition count** | Partitions = desired throughput / throughput per partition |
| **Rate limit** | Token bucket: capacity = burst size, refill rate = sustained rate |
| **SLO targets** | p50 <50ms, p95 <200ms, p99 <500ms for Yassir APIs |
| **Cache hit ratio target** | >90% for hot data. <80% indicates cache design issue |

### 🏛 Architecture Patterns Quick Reference

| Pattern | When to Use | Yassir Example |
|---|---|---|
| **Saga (Choreography)** | Distributed transactions, loose coupling | Order flow: inventory → payment → delivery |
| **Saga (Orchestration)** | Complex workflows needing visibility | Driver onboarding process |
| **Outbox Pattern** | Guarantee event publishing alongside DB write | Order confirmed → publish event |
| **CQRS** | Read-heavy with complex queries | Order history (read) vs. order placement (write) |
| **Circuit Breaker** | Protect against failing external services | Payment gateway, SMS provider |
| **Bulkhead** | Isolate failure domains | Payments failures don't degrade ride booking |
| **Strangler Fig** | Incremental monolith migration | Migrating Yassir Express from monolith to microservices |
| **API Gateway** | Single entry point, cross-cutting concerns | Apigee: auth, rate limiting, routing |
| **BFF** | Tailor API to client needs | Mobile BFF vs. web ops dashboard BFF |
| **Event Sourcing** | Full audit trail, replayability | Financial transactions (wallet debits/credits) |

// 1. Global Exception Filter
@Catch()
export class GlobalExceptionFilter implements ExceptionFilter {
  catch(exception: unknown, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse();
    const request = ctx.getRequest();

    const status = exception instanceof HttpException
      ? exception.getStatus()
      : HttpStatus.INTERNAL_SERVER_ERROR;

    response.status(status).json({
      statusCode: status,
      timestamp: new Date().toISOString(),
      path: request.url,
      correlationId: request.headers['x-correlation-id'],
    });
  }
}

// 2. Async Config (12-factor app compliance)
@Injectable()
export class DatabaseConfigService implements TypeOrmOptionsFactory {
  constructor(private configService: ConfigService) {}
  createTypeOrmOptions(): TypeOrmModuleOptions {
    return {
      type: 'postgres',
      host: this.configService.get('DB_HOST'),
      ssl: this.configService.get('NODE_ENV') === 'production',
      entities: [__dirname + '/**/*.entity{.ts,.js}'],
    };
  }
}

// 3. Interceptor for logging + correlation ID
@Injectable()
export class LoggingInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    const start = Date.now();
    const request = context.switchToHttp().getRequest();
    return next.handle().pipe(
      tap(() => logger.info({
        path: request.url,
        method: request.method,
        durationMs: Date.now() - start,
        correlationId: request.headers['x-correlation-id'],
      }))
    );
  }
}
