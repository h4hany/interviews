💎 Ruby Basics
Symbols, hashes, modules, and common interview points.
Symbols vs Strings
Symbols are immutable, interned identifiers: the same symbol literal always refers to the same object in memory. Use them for hash keys, method names, and when identity matters. Strings are mutable and better for user-facing text. Symbols save memory when reused across the app.
:user.object_id == :user.object_id   # true (same object)
"user".object_id != "user".object_id # false (new string each time)

h = { name: "Alice", age: 30 }  # symbol keys
Modules: include vs extend
include adds the module as instance methods (mixed into the class). extend adds the module as class methods. extend on an instance gives that instance the module's methods as singleton methods. Use modules for namespacing and shared behavior (mixins).
module Helper
  def greet; "Hi"; end
end
class Foo
  include Helper   # instance methods
end
class Bar
  extend Helper    # class methods: Bar.greet
end
Metaprogramming
Ruby lets you define and invoke methods dynamically. define_method creates methods at runtime. send or public_send invokes a method by name. attr_accessor, has_many in Rails are built with metaprogramming. Be aware of performance and readability when using it.
Q: When would you use a symbol instead of a string?
Use symbols for hash keys, identifiers, and when you need identity comparison or memory efficiency (e.g. many repeated keys). Use strings for user input, output, and when you need to modify the value.
Q: What is the difference between proc and lambda?
Lambda checks argument count (arity); proc does not. return in a lambda exits only the lambda; in a proc it returns from the enclosing method. Use lambdas when you want strict, method-like behavior.

🔒 Blocks & Procs
Blocks, procs, lambdas, and yield.
Blocks and yield
A block is a chunk of code passed to a method (in curly braces or do...end). The method invokes it with yield. Blocks are not objects by default; they are part of the method call. You can pass arguments to yield and use block_given? to check if a block was passed.
def twice
  yield 1
  yield 2
end
twice { |n| puts n }  # 1, 2
Procs and lambdas
Proc is an object that holds a block; create with Proc.new { } or proc { }, call with .call. Lambda is a proc with strict arity (wrong number of args = error) and return only exits the lambda. Create with lambda { } or ->() { }. Storing a block as a proc lets you pass it around or call it later.
p = proc { |a, b| [a, b] }
p.call(1)      # [1, nil] — proc does not care about arity
l = lambda { |a, b| [a, b] }
l.call(1)      # ArgumentError — lambda enforces arity
Q: How do you pass a block to a method and call it?
Either use yield inside the method (and call the method with method { }), or capture the block with &block in the method signature and call block.call. The & converts a block to a Proc and vice versa.
Q: Proc vs lambda — return behavior?
In a lambda, return returns from the lambda only. In a proc, return returns from the enclosing method (can cause LocalJumpError if the proc was called from a context that does not allow return). So lambdas behave more like methods.

🏗️ MVC & ActiveRecord
Models, validations, callbacks, and query interface.
MVC in Rails
Model (ActiveRecord): business logic, validations, DB access. View: templates (ERB, etc.) that render HTML or JSON. Controller: receives requests, talks to the model, and renders a view or redirects. Keep controllers thin; put logic in models or service objects.
Validations and callbacks
Validations run before save (e.g. validates :name, presence: true). Failed validations prevent save and add errors to model.errors. Callbacks run at lifecycle points: before_save, after_commit, etc. Use after_commit for side effects (e.g. sending email) so they run only after the transaction commits.
class User < ApplicationRecord
  validates :email, presence: true, uniqueness: true
  before_save :normalize_email
  after_commit :send_welcome_email, on: :create
end
Query interface
where, joins, includes (eager load to avoid N+1), pluck (returns array of values without loading AR objects), select, order, limit. Scopes are reusable query fragments. Chain them for readable queries.
Q: When do you use after_commit instead of after_save?
Use after_commit when the side effect (e.g. sending email, enqueueing a job) should happen only after the transaction successfully commits. after_save runs inside the transaction; if the transaction rolls back, the side effect already ran.
Q: What is the difference between pluck and select?
pluck(:col) runs a SQL query that returns only that column as an array of values; no ActiveRecord objects are built. select(:col) still returns AR objects but only with that attribute loaded. Use pluck when you only need raw values for performance.

🔗 Associations
has_many, belongs_to, through, N+1.
Association types
belongs_to: the model has a foreign key (e.g. Post belongs_to :user). has_many: one-to-many (User has_many :posts). has_many :through: many-to-many via a join model (User has_many :roles, through: :assignments). has_and_belongs_to_many: many-to-many with a join table that has no extra columns. Add dependent: :destroy or :nullify to control what happens when the parent is deleted.
class User < ApplicationRecord
  has_many :posts, dependent: :destroy
  has_many :comments, through: :posts
end
class Post < ApplicationRecord
  belongs_to :user
  has_many :comments
end
N+1 and eager loading
N+1: loading a list of records and then accessing an association on each causes one query per record. Fix with includes (eager load), eager_load (always LEFT OUTER JOIN), or preload (separate queries). includes is usually enough; Rails picks the strategy. Use references when you need to filter by an included association.
# Bad: N+1
Post.all.each { |p| p.user.name }

# Good
Post.includes(:user).each { |p| p.user.name }
Q: has_many :through vs has_and_belongs_to_many?
Use has_many :through when the join model has its own attributes or you need to work with it (e.g. Assignment with a role_name or assigned_at). Use has_and_belongs_to_many when the join table is only two foreign keys with no extra columns.
Q: How do you find and fix N+1 queries?
Use the Bullet gem in development, or watch logs for many similar queries. Fix by eager loading: Model.includes(:association) (or eager_load / preload). For filtering by association use includes(...).where(associations: { ... }) and references(:association) if needed.

🛤️ Routing & Controllers
REST, resources, nested routes, strong params.
Resources and REST
resources :posts generates the seven RESTful routes: index, show, new, create, edit, update, destroy. Use only: or except: to limit. resource (singular) for a single resource (e.g. profile). member adds a route on a specific id (e.g. post/1/publish); collection adds a route on the collection (e.g. posts/search).
resources :posts do
  member   { get :publish }
  collection { get :search }
end
# Nested: resources :comments (under posts)
Strong parameters
Never pass params directly into Model.new or update (mass assignment vulnerability). Use strong parameters: params.require(:post).permit(:title, :body, :user_id). This whitelists which keys can be set from the request. Define in a private method in the controller.
def create
  @post = Post.new(post_params)
  # ...
end
private
def post_params
  params.require(:post).permit(:title, :body, :published)
end
Q: When do you use nested routes?
When a resource logically belongs to another (e.g. comments under a post). Nested routes give URLs like /posts/5/comments. Do not nest more than one level deep; keep resources shallow where possible. Use shallow: true so show/edit/update/destroy use the resource id only.
Q: What are strong parameters and why use them?
Strong parameters whitelist which request parameters can be used for mass assignment. Without them, an attacker could pass user[admin]=true and escalate privileges. Always use require and permit in the controller for create/update.
