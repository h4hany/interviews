💎 Ruby Basics
The foundational Ruby concepts that separate developers who use Ruby from those who understand it.
Everything is an Object
In Ruby, everything is an object — including integers, nil, true, and false. Every object has methods. This is the foundation of Ruby's design.
RUBY 
# Everything is an object — even primitives
42.class          # => Integer
3.14.class        # => Float
"hello".class    # => String
nil.class         # => NilClass
true.class        # => TrueClass
42.respond_to?(:+) # => true

# Symbols — immutable, unique identifiers (more efficient than strings)
:name.class       # => Symbol
:name.object_id == :name.object_id # => true (always same object!)
"name".object_id == "name".object_id # => false (new string each time)

# Truthiness: ONLY nil and false are falsy!
# 0 is TRUTHY in Ruby (unlike JS/C)
if 0; puts "0 is truthy!"; end # => "0 is truthy!"
String vs Symbol — Key Difference
String Symbol
Mutable by default Always immutable
New object each time Same object (unique per name)
Use for content/data Use for identifiers, keys
"name".frozen? => false :name.frozen? => true
RUBY 
# Frozen strings — immutable (like symbols for strings)
str = "hello".freeze
str << " world" # => FrozenError!

# frozen_string_literal magic comment (Rails default in newer versions)
# frozen_string_literal: true

# nil safety
user = nil
user&.name   # => nil (safe navigation operator &. — like ?. in JS)
user.name    # => NoMethodError!

# Multiple assignment
a, b, *rest = [1, 2, 3, 4]
# a=1, b=2, rest=[3,4]

# String interpolation (only double quotes!)
name = "Hany"
puts "Hello, #{name}!"  # => "Hello, Hany!"
puts 'Hello, #{name}!'  # => "Hello, #{name}!" — no interpolation
🎯 INTERVIEW QUESTIONS
Q: What's the difference between nil and false in Ruby?
Both are falsy, but they're different objects. nil is NilClass and represents "nothing/absence of value." false is FalseClass and is a boolean. All other values — including 0, empty string "", and empty array [] — are truthy. nil.nil? returns true; false.nil? returns false.
Q: Why use symbols over strings as hash keys?
Symbols are immutable and unique — :name always refers to the same object in memory. Strings create a new object each time they're used. Hash lookup with symbol keys is faster because there's no need to hash the string each time. In practice: use symbols for identifiers/keys ({name: "Hany"}), strings for actual text content.
Q: What is the safe navigation operator &. ?
The &. operator (introduced Ruby 2.3) calls a method only if the receiver is not nil, otherwise returns nil. It's equivalent to `obj && obj.method`. Example: `user&.address&.city` returns nil if user or address is nil, without raising NoMethodError. Equivalent to Rails' .try() method but faster and native.

🧬 OOP & Classes
Ruby is a pure OOP language. Understanding Ruby's class system is essential for any Rails work.
Class Fundamentals
RUBY 
class Animal
  # Class variable (shared across all instances)
  @@count = 0

  # attr_accessor = getter + setter
  # attr_reader  = getter only
  # attr_writer  = setter only
  attr_accessor :name, :age
  attr_reader :species

  def initialize(name, age)
    @name = name      # instance variable
    @age = age
    @@count += 1
  end

  # Class method (called on class itself)
  def self.count
    @@count
  end

  # Instance method
  def speak
    "..."
  end

  # Method visibility
  protected
  def internal_id; @id; end  # accessible to subclasses

  private
  def secret; "hidden"; end  # only within the class
end

class Dog < Animal  # single inheritance
  def speak
    "Woof! I'm #{@name}"   # override + super access
  end
end
Comparable — Elegant Sorting
RUBY 
class Box
  include Comparable  # gives you >, <, >=, <=, between?, clamp
  attr_accessor :volume

  def initialize(v); @volume = v; end

  # Only need to define <=> and Comparable gives you the rest!
  def <=>(other)
    @volume <=> other.volume  # spaceship operator
  end
end

boxes = [Box.new(3), Box.new(1), Box.new(2)]
boxes.sort.map(&:volume)  # => [1, 2, 3]
boxes.max.volume           # => 3
🎯 INTERVIEW QUESTIONS
Q: What's the difference between class methods and instance methods?
Instance methods operate on a specific object instance (accessed via instance variables @var). They're called on an object: dog.speak. Class methods operate on the class itself, defined with def self.method_name. They're called on the class: Animal.count. Class methods are often used for factory methods, statistics, or operations that don't need an instance.
Q: What is the spaceship operator <=> ?
The spaceship operator returns -1, 0, or 1 comparing two objects. -1 if left < right, 0 if equal, 1 if left > right. It's used by sort, min, max, and the Comparable module. By defining <=> in your class and including Comparable, you get all comparison operators for free.
Q: Difference between protected and private in Ruby?
private methods can only be called within the class itself, without an explicit receiver (you can't call self.private_method). protected methods can be called by instances of the same class or subclasses — useful for comparisons between instances. Neither is accessible from outside the class.

🧩 Modules & Mixins
Ruby's answer to multiple inheritance. The most powerful code reuse mechanism in the language.
include vs extend vs prepend
RUBY 
module Greetable
  def greet; "Hello, I'm #{name}"; end
end

# include — adds module methods as INSTANCE methods
class Person
  include Greetable
  attr_reader :name
  def initialize(n); @name = n; end
end
Person.new("Hany").greet   # => "Hello, I'm Hany"

# extend — adds module methods as CLASS methods
class Config
  extend Greetable
end
# Config.greet  (class-level method)

# prepend — inserts BEFORE the class in the method lookup chain
# Used for wrapping/decorating methods (like before_action)
module Logging
  def save
    puts "Saving..."
    result = super   # calls the original save
    puts "Saved!"
    result
  end
end

class Post
  prepend Logging
  def save; puts "DB write"; end
end
# Post.new.save prints: "Saving..." → "DB write" → "Saved!"
Method Lookup Chain
💡 METHOD LOOKUP ORDER
When you call a method, Ruby searches: prepended modules → class itself → included modules (in reverse order) → superclass → superclass's modules → Object → BasicObject. First match wins. This is why prepend can wrap methods and include adds methods that the class can override.
RUBY — ANCESTORS 
module M1; end
module M2; end
class A; include M1; end
class B < A; include M2; end

B.ancestors
# => [B, M2, A, M1, Object, Kernel, BasicObject]
🎯 INTERVIEW QUESTIONS
Q: What's the difference between include and extend?
include adds module methods as instance methods — instances of the class can call them. extend adds module methods as class methods — the class itself can call them. A common pattern is to use both via a self.included hook: when a module is included, it also extends the class with class-level methods.
Q: When would you use prepend?
prepend is used when you want to wrap or decorate existing methods. Unlike include, prepended modules sit BEFORE the class in the lookup chain, so their methods execute first and can call super to invoke the original. Rails' ActiveSupport::Concern uses this pattern. Real use cases: logging, authentication checks, instrumentation around existing methods.

🔷 Blocks, Procs & Lambdas
The most uniquely Ruby concept. Powers iterators, DSLs, and Rails' expressiveness.
Blocks
A block is a chunk of code passed to a method. It's not an object — it's syntax. Methods receive blocks via yield or by accepting a &block parameter.
RUBY 
# yield calls the block
def repeat(n)
  n.times { yield } if block_given?
end

repeat(3) { puts "Hello!" }  # prints "Hello!" 3 times

# yield with arguments
def transform(x)
  yield(x) * 2
end
transform(5) { |n| n + 1 }  # => 12  (5+1=6, 6*2=12)

# Explicit block capture with &
def run_later(&block)
  block.call    # store and call later
end
Proc vs Lambda
Feature Proc Lambda
Return behavior return exits ENCLOSING method return exits lambda only
Argument checking Lenient (ignores extras) Strict (raises ArgumentError)
Is a lambda? proc.lambda? => false lam.lambda? => true
Created with Proc.new { } or proc { } lambda { } or ->( ) { }
RUBY 
# Lambda — strict, predictable
square = ->(x) { x * x }
square.(5)   # => 25
square.call(5) # => 25

# Proc — lenient
greet = proc { |name| "Hello, #{name || 'stranger'}" }
greet.call        # => "Hello, stranger" (nil arg ok)

# Symbol to proc — powerful shorthand
[1,2,3].map(&:to_s)      # => ["1", "2", "3"]
["a","b"].map(&:upcase)  # => ["A", "B"]
# &:method converts symbol to proc via Symbol#to_proc
🎯 INTERVIEW QUESTIONS
Q: What's the main difference between a Proc and a Lambda?
Two key differences: (1) Return behavior — a return inside a Proc returns from the enclosing method, not just the Proc. A return inside a Lambda only exits the Lambda. This can cause LocalJumpError if the Proc outlives its enclosing method. (2) Argument checking — Lambdas strictly check argument count and raise ArgumentError; Procs ignore extra arguments and set missing ones to nil.
Q: What does & do when used with a method argument?
When & is used in a method definition (&block), it captures the block as a Proc object. When used in a method call ([1,2,3].map(&:to_s)), it converts the following object to a Proc via the to_proc method. Symbol#to_proc creates a proc that calls that method on its argument — :to_s.to_proc == proc { |obj| obj.to_s }.

📚 Enumerable & Collections
Ruby's Enumerable module is one of the most powerful in the language. Master these and coding interviews become easy.
Essential Enumerable Methods
RUBY 
users = [{name: "Hany", age: 30}, {name: "Sara", age: 25}, {name: "Ali", age: 30}]

# map — transform each element (returns new array)
users.map { |u| u[:name] }          # => ["Hany", "Sara", "Ali"]
users.map(&:name)                   # same (if using struct/model)

# select / filter — keep matching elements
users.select { |u| u[:age] >= 30 }  # => [{Hany,30}, {Ali,30}]

# reject — opposite of select
users.reject { |u| u[:age] >= 30 }  # => [{Sara,25}]

# reduce / inject — fold into single value
[1,2,3,4].reduce(0) { |sum, n| sum + n }  # => 10
[1,2,3,4].reduce(:+)                      # => 10 (shorthand!)

# group_by — very common in Rails
users.group_by { |u| u[:age] }
# => {30=>[{Hany}, {Ali}], 25=>[{Sara}]}

# each_with_object — build a new collection
users.each_with_object({}) { |u, h| h[u[:name]] = u[:age] }
# => {"Hany"=>30, "Sara"=>25, "Ali"=>30}

# flat_map — map + flatten
[[1,2],[3,4]].flat_map { |a| a.map { |x| x * 2 } }
# => [2, 4, 6, 8]

# any? / all? / none? / count
users.any? { |u| u[:age] > 28 }   # => true
users.all? { |u| u[:age] > 20 }   # => true
users.count { |u| u[:age] == 30 }  # => 2

# min_by / max_by / sort_by
users.min_by { |u| u[:age] }   # => {Sara, 25}
users.sort_by { |u| u[:age] }  # sorted array
🎯 INTERVIEW QUESTIONS
Q: What's the difference between map and each?
each iterates and returns the ORIGINAL array unchanged — it's for side effects. map iterates and returns a NEW array with transformed values. Rule: if you need the return values of the block, use map. If you're just doing side effects (printing, saving), use each.
Q: When would you use reduce/inject?
reduce accumulates a single value from a collection by applying a block that takes an accumulator and each element. Use it for: summing/multiplying values, building a hash from an array, flattening nested structures. It's the most powerful and general Enumerable method — technically you could implement all others with reduce.

🔮 Metaprogramming
Code that writes code. The magic behind Rails' attr_accessor, has_many, and validates.
method_missing & respond_to_missing?
RUBY 
class DynamicFinder
  # Called when a method isn't found
  def method_missing(name, *args)
    if name.to_s.start_with?("find_by_")
      field = name.to_s.sub("find_by_", "")
      puts "SELECT * WHERE #{field} = #{args.first}"
    else
      super  # IMPORTANT: delegate to parent if unhandled
    end
  end

  # ALWAYS implement respond_to_missing? alongside method_missing!
  def respond_to_missing?(name, include_private = false)
    name.to_s.start_with?("find_by_") || super
  end
end

finder = DynamicFinder.new
finder.find_by_email("hany@test.com")
# => "SELECT * WHERE email = hany@test.com"
define_method & Open Classes
RUBY 
# define_method — create methods programmatically
class Status
  STATES = [:pending, :active, :cancelled]

  STATES.each do |state|
    define_method("#{state}?") { @status == state }
    define_method("#{state}!") { @status = state }
  end
end
# Creates: pending?, active?, cancelled?, pending!, active!, cancelled!

# Open classes — add methods to existing classes
class String
  def palindrome?
    self == self.reverse
  end
end
"racecar".palindrome?  # => true

# send — call any method by name (even private!)
obj.send(:private_method)
obj.public_send(:method)  # safer — respects visibility
🎯 INTERVIEW QUESTIONS
Q: What is method_missing and when should you use it?
method_missing is called by Ruby when a method can't be found in the normal lookup chain. Override it to implement dynamic dispatch — like Rails' find_by_name magic. Always: (1) call super for unhandled methods to avoid swallowing errors, (2) implement respond_to_missing? so respond_to? works correctly. Avoid overusing it — it's slow and makes code hard to debug.
Q: What are open classes and what's the danger?
Open classes (monkey patching) let you add/modify methods on any existing class, including Ruby's own classes. Useful for adding convenience methods (e.g., 5.minutes in Rails). The danger: name collisions can silently break behavior, gems can conflict with each other, and debugging becomes hard. Best practice: use refinements (using/refine) for safer, scoped monkey patching.

🏗️ Rails MVC Architecture
Convention over Configuration. The request-response lifecycle and how Rails organizes everything.
Request-Response Lifecycle
📋 RAILS REQUEST FLOW
1. Browser sends HTTP request
2. Router (routes.rb) matches URL → controller#action
3. Middleware stack processes request (auth, sessions, logging)
4. Controller receives params, calls models, assigns @variables
5. Model queries/updates database via ActiveRecord
6. View renders ERB template with @variables
7. Response sent back to browser
RUBY — CONTROLLER 
class PostsController < ApplicationController
  before_action :authenticate_user!
  before_action :set_post, only: [:show, :edit, :update, :destroy]

  def index
    @posts = Post.published.page(params[:page])
    respond_to do |format|
      format.html
      format.json { render json: @posts }
    end
  end

  def create
    @post = current_user.posts.build(post_params)
    if @post.save
      redirect_to @post, notice: "Created!"
    else
      render :new, status: :unprocessable_entity
    end
  end

  private

  def set_post
    @post = Post.find(params[:id])
  end

  def post_params   # Strong parameters — security!
    params.require(:post).permit(:title, :body, :published)
  end
end
🎯 INTERVIEW QUESTIONS
Q: What is "Convention over Configuration"?
Rails makes decisions based on naming conventions so you don't have to write configuration. A class named Post maps to a posts table. A controller named PostsController maps to app/controllers/posts_controller.rb. Views are in app/views/posts/. Follow the conventions and Rails wires everything up automatically. Deviate and you need explicit configuration.
Q: What are strong parameters and why are they needed?
Strong parameters (params.require().permit()) protect against mass assignment attacks. Without them, a malicious user could pass is_admin: true in a POST request and your model would save it if you use User.new(params[:user]). Strong parameters whitelist exactly which fields are allowed, preventing attackers from modifying fields they shouldn't access.
Q: What is the Rails middleware stack?
Rails middleware is a stack of Rack components that each request passes through before reaching your controllers. Each middleware can process the request, modify it, short-circuit it, or pass it along. Common middleware: session management, CSRF protection, static file serving, logging, exception handling. You can see the stack with `rails middleware` and add custom middleware in config/application.rb.

🗄️ ActiveRecord
Rails' ORM. The most interview-tested Rails topic — N+1, associations, and query optimization.
Associations
RUBY 
class User < ApplicationRecord
  has_many :posts, dependent: :destroy
  has_many :comments, through: :posts
  has_one :profile
  belongs_to :company, optional: true

  # Polymorphic association
  has_many :notifications, as: :notifiable
end

class Post < ApplicationRecord
  belongs_to :user
  has_many :comments
  has_many_attached :images  # ActiveStorage

  # Validations
  validates :title, presence: true, length: { minimum: 3, maximum: 100 }
  validates :slug, uniqueness: true
  validate :published_at_in_future

  # Scopes
  scope :published, -> { where(published: true) }
  scope :recent, -> { order(created_at: :desc).limit(10) }
  scope :by_author, ->(user) { where(user: user) }

  # Callbacks
  before_save :generate_slug
  after_create :notify_subscribers
end
The N+1 Problem — Most Asked Interview Topic
⚠️ N+1 Problem: Loading a list then querying for each item individually. 100 users → 1 query for users + 100 queries for their posts = 101 queries. This is the #1 Rails performance issue.
RUBY — N+1 FIX 
# ❌ N+1 — 1 + N queries
Post.all.each { |post| puts post.user.name }

# ✅ includes — eager loading (2 queries total)
Post.includes(:user).each { |post| puts post.user.name }

# ✅ joins — for filtering only (1 query, no extra model loading)
Post.joins(:user).where(users: { active: true })

# ✅ preload — always separate queries (forces 2-query approach)
Post.preload(:user, :comments)

# ✅ eager_load — single LEFT JOIN (good for complex filtering)
Post.eager_load(:user).where(users: { role: "admin" })

# Detect N+1 in development
# gem 'bullet' — alerts you to N+1 problems
🎯 INTERVIEW QUESTIONS
Q: What is the N+1 problem and how do you fix it?
N+1 is when you load a collection (1 query) then access an association on each record in a loop (N more queries). Fix with eager loading: includes(:association) tells Rails to load the association for all records in 2 queries (or a JOIN). Use the Bullet gem in development to detect N+1 automatically. Also consider select and pluck to fetch only needed columns.
Q: When would you use joins vs includes?
Use joins when you need to FILTER by association attributes — it does a SQL JOIN but doesn't load the associated records into memory. Use includes when you need to ACCESS the associated records — it loads them into memory to prevent N+1. If you filter AND access, use eager_load which does a single LEFT OUTER JOIN and loads both.
Q: What are callbacks and when should you avoid them?
Callbacks (before_save, after_create, etc.) run automatically during model lifecycle. Avoid them for: business logic that should be explicit (use service objects instead), sending emails/notifications (side effects are surprising), complex cross-model operations. Good uses: data normalization (before_save: downcase_email), generating derived fields (slug generation).

🛣️ Rails Routing
How URLs map to controller actions. RESTful routes, nested resources, and custom routes.
RESTful Routes
RUBY — routes.rb 
Rails.application.routes.draw do
  # resources generates 7 RESTful routes:
  # GET    /posts        → index
  # GET    /posts/new    → new
  # POST   /posts        → create
  # GET    /posts/:id    → show
  # GET    /posts/:id/edit → edit
  # PATCH  /posts/:id    → update
  # DELETE /posts/:id    → destroy
  resources :posts do
    resources :comments       # nested: /posts/:post_id/comments
    member do
      post :publish           # POST /posts/:id/publish
    end
    collection do
      get :search             # GET /posts/search
    end
  end

  namespace :api do
    namespace :v1 do
      resources :users, only: [:index, :show, :create]
    end
  end

  root "home#index"
end
🎯 INTERVIEW QUESTIONS
Q: What's the difference between member and collection routes?
member routes operate on a SPECIFIC resource and include :id in the URL (e.g., POST /posts/:id/publish — publish a specific post). collection routes operate on the ENTIRE collection and don't include :id (e.g., GET /posts/search — search across all posts). Use member when the action is about one record, collection when it's about the set.
Q: What's the difference between namespace and scope?
namespace adds both URL prefix AND module prefix: `namespace :api` gives URL /api/users and controller Api::UsersController. scope only adds URL prefix without module: `scope :api` gives URL /api/users but still UsersController. Use namespace for versioned APIs where controllers are organized in subdirectories.

📋 Service Objects & Concerns
Keeping Rails apps maintainable. Fat models, skinny controllers, service objects — the right patterns for the right problems.
Service Objects Pattern
RUBY — SERVICE OBJECT 
# app/services/user_registration.rb
class UserRegistration
  def initialize(params)
    @params = params
  end

  def call
    user = User.new(@params)
    if user.save
      WelcomeMailer.with(user: user).welcome_email.deliver_later
      Analytics.track("user_registered", user_id: user.id)
      Result.success(user)
    else
      Result.failure(user.errors)
    end
  end
end

# In controller — thin and clean
def create
  result = UserRegistration.new(user_params).call
  if result.success?
    redirect_to dashboard_path
  else
    @errors = result.errors
    render :new
  end
end
🎯 INTERVIEW QUESTIONS
Q: When should you use a service object?
Service objects when: (1) action involves multiple models, (2) action has side effects (email, analytics, external APIs), (3) action has complex business logic, (4) action needs to be testable in isolation. Don't use for simple CRUD — a fat model or thin controller is fine. Rule: if your controller action is more than 10 lines or touches 3+ models, extract to a service.
Q: Concerns vs Service Objects — when to use each?
Concerns (ActiveSupport::Concern) are for sharing MODEL behavior across models — like Publishable concern with publish!/draft! methods. They're mixed into models. Service Objects are for BUSINESS LOGIC that doesn't belong in a model or controller — orchestrating multiple steps, external calls, complex workflows. Concerns = shared model capabilities; Services = complex operations.

🎯 INTERVIEW QUESTIONS
Q: What is Russian Doll caching?
Russian Doll caching nests cache blocks inside each other. Outer cache wraps a post, inner cache wraps the post's author. When only the author changes, only the inner cache is invalidated — the outer cache hits. This is efficient because Rails generates cache keys from cache_key (includes updated_at), so changing an association's updated_at automatically busts the relevant fragment while leaving unrelated fragments intact.

⚙️ Background Jobs
Sidekiq, ActiveJob, and the patterns for reliable async processing in Rails.
Sidekiq + ActiveJob
RUBY 
# app/jobs/send_welcome_email_job.rb
class SendWelcomeEmailJob < ApplicationJob
  queue_as :mailers
  sidekiq_options retry: 5, backtrace: true

  def perform(user_id)
    user = User.find_by(id: user_id)
    return unless user  # idempotent — handle deleted records

    WelcomeMailer.with(user: user).welcome_email.deliver_now
  end
end

# Enqueue jobs
SendWelcomeEmailJob.perform_later(user.id)    # async
SendWelcomeEmailJob.perform_now(user.id)      # sync
SendWelcomeEmailJob.set(wait: 5.minutes).perform_later(user.id)
🎯 INTERVIEW QUESTIONS
Q: What is idempotency and why is it important for background jobs?
Idempotency means running a job multiple times has the same effect as running it once. Jobs can be retried on failure, so you must design them to be safe if run again. Best practices: pass IDs not objects (objects may change), check if work is already done before doing it, use database unique constraints, use upsert/find_or_create instead of create. Example: always do User.find_by(id:) instead of User.find to handle deleted records.
Q: How does Sidekiq work?
Sidekiq uses Redis as a queue. When you call perform_later, the job is serialized to JSON and pushed to a Redis list. Sidekiq workers (multiple threads per process) poll Redis, pop jobs, and execute them. Sidekiq has a retry mechanism with exponential backoff for failed jobs and a dead queue for jobs that exceeded retries. It's thread-safe and much more efficient than Delayed::Job (DB-backed).
