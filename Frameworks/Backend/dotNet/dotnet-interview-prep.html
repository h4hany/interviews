⏳ Async/Await
Task, async/await, ConfigureAwait, and cancellation.
Key concepts
Async methods return Task or Task<T>. await unwraps the result and frees the thread. Use ConfigureAwait(false) in library code to avoid capturing the sync context. Use CancellationToken for cancellation.
public async Task<User> GetUserAsync(int id, CancellationToken ct = default)
{
  await _httpClient.GetAsync(url, ct);
  return await _db.Users.FindAsync(id, ct);
}
Q: What is the difference between Task.Run and async/await?
Task.Run offloads CPU-bound work to the thread pool. async/await is for I/O-bound work (no new thread; the method yields until I/O completes). Use Task.Run only when you have blocking or CPU-heavy code.

🔍 LINQ
Deferred execution, IEnumerable, IQueryable, and query syntax.
Deferred vs immediate execution
Deferred: The query is not executed until you iterate (foreach, ToList(), ToArray(), etc.). So you can build a query step by step and run it once. Immediate: Methods like Count(), First(), Single(), ToList() execute the query right away. Use deferred when you want to compose or delay; be aware that iterating twice runs the query twice unless you materialize with ToList().
var q = items.Where(x => x.Active);  // no query yet
var list = q.ToList();               // query runs here
IEnumerable vs IQueryable
IEnumerable: in-memory; operations are executed in .NET. IQueryable: expression tree; the provider (e.g. EF Core) translates to SQL and runs on the server. So filtering on IQueryable pushes the filter to the DB; filtering on IEnumerable runs in memory. Use IQueryable for DB-backed data to avoid loading everything.
Q: When does a LINQ query actually execute?
When you enumerate the result: foreach, ToList(), ToArray(), Count(), First(), etc. Until then it is deferred. With IQueryable, execution happens when the query is translated and sent to the provider (e.g. database).
Q: Why use IQueryable instead of IEnumerable for database queries?
IQueryable builds an expression tree that the provider (EF Core, etc.) converts to SQL. Filtering and projection happen in the database, so you only load what you need. With IEnumerable you would load full entities and filter in memory, which is inefficient for large data.

🪣 DI & Middleware
Service lifetime (Singleton, Scoped, Transient) and request pipeline.
Service lifetimes
Singleton: One instance for the entire application lifetime. Use for stateless services, caches, config. Do not hold request-specific or scoped data. Scoped: One instance per request (or per scope). Use for DbContext, request-scoped state. Transient: New instance every time it is resolved. Use for lightweight, stateless services. Avoid injecting Scoped into Singleton (captured dependency).
services.AddSingleton();
services.AddScoped();
services.AddTransient();
Middleware pipeline
Middleware runs in the order it is registered. Each middleware can call next() to pass to the next, or short-circuit (e.g. return 401). Order matters: e.g. authentication before authorization, exception handling first, CORS before endpoints. In minimal APIs and MVC, middleware wraps the entire request.
Q: Why not inject Scoped into Singleton?
Singleton lives for the app lifetime; Scoped is created per request. If a Singleton held a Scoped dependency, it would capture one scope (e.g. first request) and reuse it for all later requests, leading to wrong or stale data. Resolve Scoped from within a scope (e.g. in the controller or middleware).
Q: What is the difference between Use and Run in middleware?
Use can call next to pass to the next middleware. Run is terminal: it does not have a next delegate and typically ends the pipeline. Use Use when you want to add logic before and optionally after the rest of the pipeline.

🗄️ EF Core
DbContext, migrations, tracking, and eager loading.
DbContext and unit of work
DbContext represents a session with the database and a unit of work: track entities, cache them, and flush changes with SaveChanges. Register as Scoped so each request gets its own context. Use one DbContext per request; do not share across requests or use long-lived contexts for web apps.
Tracking and read-only queries
By default EF tracks entities so it can detect changes and persist them. For read-only queries use AsNoTracking() to avoid tracking overhead and prevent accidental updates. Use AsNoTrackingWithIdentityResolution() when you still want identity resolution but no change tracking.
var users = await _db.Users.AsNoTracking().Where(u => u.Active).ToListAsync();
Eager loading and N+1
Use Include and ThenInclude to load related data in one query and avoid N+1. For read-only projections, prefer Select to load only the columns you need. Avoid loading large graphs if you only need a few fields; project to DTOs or anonymous types.
Q: When do you use AsNoTracking?
For read-only queries: when you only read data and do not need to update or delete those entities. It improves performance and prevents the context from tracking entities you do not intend to change. Use it in APIs that return DTOs or in reporting queries.
Q: How do you avoid N+1 in EF Core?
Eager load with Include/ThenInclude when you need related entities. For simple data, use Select to project only the columns you need, possibly with a join, so EF generates a single efficient query instead of many.
