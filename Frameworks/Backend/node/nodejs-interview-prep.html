🔄 The Event Loop
The single most important concept in Node.js. Understanding this will help you answer 30% of all Node.js interview questions.
What is the Event Loop?
JavaScript is single-threaded — it can only do one thing at a time. But Node.js can handle thousands of concurrent connections without freezing. How? The Event Loop.

The Event Loop is a mechanism that continuously checks: "Is the call stack empty? If yes, grab the next task from the queue and push it to the stack."

Think of it as a traffic controller that decides which code runs when.
💡 ANALOGY
Imagine a restaurant. The waiter (call stack) can only serve one table at a time. When an order is placed (async task), instead of waiting at the kitchen, the waiter goes to serve other tables. When the food is ready (I/O complete), it goes to the pickup queue. The event loop is the manager who tells the waiter: "Hey, food is ready at table 3, go pick it up."
The Components
📚 Call Stack
main()
console.log("a")
🔬 Microtask Queue
Promise.then()
queueMicrotask()
📋 Macrotask Queue
setTimeout()
setInterval()
I/O callbacks
🔄 Event Loop: Stack empty? → drain Microtasks → pick 1 Macrotask → repeat
1. Call Stack — Where your synchronous code runs. LIFO (Last In, First Out). Functions are pushed when called, popped when they return.

2. Microtask Queue — Higher priority than macrotasks. Processed completely after each task before the next macrotask runs. Includes: Promise.then(), queueMicrotask(), MutationObserver.

3. Macrotask Queue (Task Queue) — Lower priority. One macrotask is processed, then microtasks are drained again. Includes: setTimeout, setInterval, setImmediate, I/O callbacks.
Critical Order of Execution
JAVASCRIPT 
// What order does this print?
console.log('1 - synchronous');

setTimeout(() => console.log('2 - setTimeout'), 0);

new Promise(resolve => {
  console.log('3 - Promise executor (sync!)');
  resolve();
}).then(() => console.log('4 - Promise.then (microtask)'));

console.log('5 - synchronous');

// OUTPUT ORDER: 1, 3, 5, 4, 2
// Why? Sync first → Promise executor is sync
// → microtasks (then) → macrotasks (setTimeout)
⚠️ Common Trap: setTimeout(fn, 0) does NOT mean "run immediately." It means "put this in the macrotask queue." A Promise.then() will always run before it, even if the setTimeout has 0ms delay.
Node.js Event Loop Phases
Node.js has 6 specific phases in its event loop (libuv). Each phase has its own FIFO queue:
Phase What runs here Key API
timers Expired setTimeout/setInterval callbacks setTimeout, setInterval
pending callbacks I/O callbacks deferred to next loop TCP errors
idle/prepare Internal Node.js use —
poll Retrieve new I/O events, execute their callbacks fs.read, net
check setImmediate callbacks setImmediate
close callbacks Close events (socket.on('close')) socket.close
📌 setImmediate vs setTimeout(0)
setImmediate(fn) runs in the check phase — always after the poll phase. setTimeout(fn, 0) runs in the timers phase. When called from the main module (outside I/O), the order is non-deterministic. When called inside an I/O callback, setImmediate always runs first.
🎯 TOP INTERVIEW QUESTIONS
Q: What is the event loop and how does it work?
The event loop is a mechanism that allows Node.js to perform non-blocking I/O despite JavaScript being single-threaded. It continuously monitors the call stack and task queues. When the stack is empty, it first drains all microtasks (Promise callbacks), then picks one macrotask (setTimeout, I/O) and executes it, then repeats.
Q: What's the difference between microtasks and macrotasks?
Microtasks (Promise.then, queueMicrotask, MutationObserver) have higher priority and are completely drained after each task before any macrotask runs. Macrotasks (setTimeout, setInterval, setImmediate, I/O callbacks) run one at a time, with all microtasks processed between each macrotask.
Q: What happens if you have an infinite loop in a Promise?
It will starve macrotasks. Microtasks are drained completely before any macrotask can run. An infinite microtask loop (like a Promise that keeps resolving and queuing .then()) will block the entire event loop, preventing any setTimeout, I/O, or other operations from ever running — essentially freezing Node.js.
Q: What does `setTimeout(fn, 0)` actually mean?
It means "schedule fn to run after a minimum of 0ms AND after the current synchronous code and all microtasks have completed." It does NOT run immediately. It goes into the macrotask queue and will run after all pending synchronous code and Promise callbacks have finished.

⚡ Synchronous vs Asynchronous
Blocking vs non-blocking — the foundation of why Node.js can handle massive I/O efficiently.
Synchronous (Blocking)
Synchronous code executes line by line. Each line must finish before the next one starts. If a line takes 5 seconds, everything stops for 5 seconds.
JAVASCRIPT — SYNC 
const data = fs.readFileSync('file.txt'); // BLOCKS here
console.log(data); // only runs AFTER file is fully read
processOtherRequests(); // stuck waiting...
Problem: In a web server, if one request does a sync file read, ALL other requests are frozen until that file is read. With 1000 users, this is catastrophic.
Asynchronous (Non-Blocking)
Async code starts an operation and immediately moves on. When the operation completes, a callback/promise handles the result — without blocking anything else.
JAVASCRIPT — ASYNC 
// Node.js hands the file read to the OS via libuv
// and continues executing immediately
fs.readFile('file.txt', (err, data) => {
  console.log(data); // runs LATER when file is ready
});

processOtherRequests(); // runs IMMEDIATELY, not waiting
console.log('This prints BEFORE the file is read!');
How Node.js Does Async I/O
💡 ANALOGY
Think of a McDonald's (Node.js) vs a sit-down restaurant (blocking server). At McDonald's, you order and step aside. The cashier (Node.js) immediately takes the next order. When your food is ready, they call your number (callback/event). At the blocking sit-down restaurant, you order, the waiter stands at your table waiting until the kitchen finishes your food — no other tables get served meanwhile.
Node.js achieves this through libuv, a C library that manages a thread pool for I/O operations. When you call fs.readFile():

1️⃣ Node passes the work to libuv's thread pool (4 threads by default)
2️⃣ The main JS thread is free immediately
3️⃣ When the OS finishes reading, libuv pushes the callback to the event queue
4️⃣ The event loop picks it up when the stack is empty
Blocking vs Non-Blocking — Key Differences
Aspect Synchronous Asynchronous
Execution Waits for operation to complete Continues while operation runs
Thread usage Blocks the thread Frees the thread
Performance Poor under load Excellent scalability
Code complexity Simple, linear More complex (callbacks/promises)
Use case Scripts, startup config Web servers, APIs
Example fs.readFileSync() fs.readFile()
🎯 TOP INTERVIEW QUESTIONS
Q: Is Node.js truly single-threaded?
Your JavaScript code runs on a single thread. But Node.js itself is NOT single-threaded — libuv uses a thread pool (default 4 threads) for heavy I/O operations. Network I/O (TCP/HTTP) uses OS async primitives (epoll/kqueue) and doesn't use the thread pool at all. So: JS = single thread, I/O = multi-threaded via libuv.
Q: What is blocking I/O and why is it bad in Node.js?
Blocking I/O means the execution thread waits until the operation completes. In Node.js, this freezes the event loop — no other code can execute, no other requests can be handled. This is why you should never use readFileSync, execSync, or any other synchronous operations in a production Node.js server handling multiple requests.
Q: When is it okay to use synchronous operations in Node.js?
It's acceptable to use synchronous operations: (1) During application startup/initialization before the server starts accepting requests, (2) In command-line scripts that don't need concurrency, (3) In build tools or one-off scripts. Never use sync operations inside request handlers or any code that serves concurrent users.

🤝 Promises
The modern way to handle async operations. Replaces callback hell with composable, readable code.
What is a Promise?
A Promise is an object representing the eventual completion or failure of an async operation. It's a placeholder for a value that's not yet available.
💡 ANALOGY
When you order food at a restaurant, the waiter gives you a receipt. That receipt is a "promise" — it represents the food that will come. You can do other things (talk, drink water) while waiting. When the food is ready (resolved), you eat. If the kitchen burns down (rejected), you get your money back.
Promise States
⏳ PENDING
initial state
→
✅ FULFILLED
resolve() called
or
❌ REJECTED
reject() called
Once settled (fulfilled or rejected), a Promise is IMMUTABLE — it cannot change state
Creating & Using Promises
JAVASCRIPT 
// Creating a Promise
const myPromise = new Promise((resolve, reject) => {
  // async work here
  const success = true;
  if (success) {
    resolve('Data loaded!');  // fulfills with value
  } else {
    reject(new Error('Failed!')); // rejects with error
  }
});

// Consuming a Promise
myPromise
  .then(value => console.log(value))   // 'Data loaded!'
  .catch(err => console.error(err))  // handles rejection
  .finally(() => console.log('Done')); // always runs
Promise Combinators — The Big 4
JAVASCRIPT — COMBINATORS 
const p1 = Promise.resolve(1);
const p2 = Promise.resolve(2);
const p3 = Promise.reject(new Error('fail'));

// ALL — waits for all, fails fast on any rejection
Promise.all([p1, p2]).then(([a, b]) => console.log(a, b)); // 1, 2
Promise.all([p1, p3]).catch(e => console.log(e)); // 'fail' — stops at p3

// ALL SETTLED — waits for all, never fails fast
Promise.allSettled([p1, p3]).then(results => {
  // [{status:'fulfilled',value:1}, {status:'rejected',reason:Error}]
  results.forEach(r => console.log(r.status));
});

// RACE — resolves/rejects with the FIRST settled promise
Promise.race([p1, p2]).then(v => console.log(v)); // 1 (whichever is first)

// ANY — resolves with the FIRST fulfilled, rejects if ALL reject
Promise.any([p3, p1]).then(v => console.log(v)); // 1 (ignores rejection)
Method Resolves when Rejects when
Promise.all ALL fulfilled ANY rejects (fast-fail)
Promise.allSettled ALL settled (any state) Never rejects
Promise.race FIRST settles (any state) FIRST rejects
Promise.any FIRST fulfills ALL reject
🎯 TOP INTERVIEW QUESTIONS
Q: What's the difference between Promise.all and Promise.allSettled?
Promise.all fails fast — if any single promise rejects, it immediately rejects with that error and you lose all other results. Promise.allSettled always waits for every promise to settle (fulfilled or rejected) and returns an array of result objects with a status field. Use allSettled when you need to know the outcome of ALL operations regardless of failures.
Q: Can a Promise be cancelled?
Natively, no. Once created, a Promise cannot be cancelled. Common workarounds: (1) AbortController for fetch requests, (2) A wrapper that ignores the resolve/reject if a "cancelled" flag is set, (3) External libraries like RxJS Observables which support cancellation. This is a known limitation of the Promises spec.
Q: What is Promise chaining and how does it work?
Each .then() returns a NEW promise. If you return a value from .then(), the next .then() receives it. If you return a promise, the chain waits for it to resolve. This allows sequential async operations to be written as a flat chain instead of nested callbacks. Errors propagate down the chain to the nearest .catch().

⏳ Async / Await
Syntactic sugar over Promises. Write async code that looks synchronous. The modern standard for async JS.
The Basics
JAVASCRIPT 
// async function ALWAYS returns a Promise
async function fetchData() {
  const response = await fetch('https://api.example.com/data');
  // await PAUSES execution here (but doesn't block the thread!)
  // other code in the event loop continues running
  const data = await response.json();
  return data; // wraps in Promise.resolve(data)
}

// Error handling with try/catch
async function safeLoad() {
  try {
    const data = await fetchData();
    console.log(data);
  } catch (err) {
    console.error('Failed:', err.message);
  }
}
Sequential vs Parallel — Critical Performance Difference
JAVASCRIPT — PERFORMANCE TRAP 
// ❌ SEQUENTIAL — takes 3 seconds total (1+1+1)
async function slow() {
  const a = await apiCall1(); // waits 1s
  const b = await apiCall2(); // then waits 1s more
  const c = await apiCall3(); // then waits 1s more
}

// ✅ PARALLEL — takes ~1 second total
async function fast() {
  const [a, b, c] = await Promise.all([
    apiCall1(), // all 3 start at the same time!
    apiCall2(),
    apiCall3()
  ]);
}

// ✅ PARALLEL with error resilience
async function resilient() {
  const results = await Promise.allSettled([
    apiCall1(), apiCall2(), apiCall3()
  ]);
}
⚠️ Most Common Interview Trap: Putting await inside a forEach loop does NOT work as expected — forEach doesn't wait for async callbacks. Use for...of for sequential async iteration, or Promise.all(arr.map(fn)) for parallel.
The await in forEach Problem
JAVASCRIPT — GOTCHA 
const ids = [1, 2, 3];

// ❌ WRONG — forEach ignores the returned Promise
ids.forEach(async (id) => {
  await processItem(id); // doesn't actually await!
});

// ✅ CORRECT — sequential
for (const id of ids) {
  await processItem(id); // processes 1, then 2, then 3
}

// ✅ CORRECT — parallel
await Promise.all(ids.map(id => processItem(id)));
🎯 TOP INTERVIEW QUESTIONS
Q: Does await block the event loop?
No! This is a very common misconception. await "pauses" the current async function, but it does NOT block the JavaScript thread. While waiting, the event loop is free to process other callbacks, handle other requests, run timers, etc. It's essentially syntactic sugar for .then() chaining — the engine suspends the function and returns control to the caller.
Q: What does an async function return?
An async function ALWAYS returns a Promise. If you return a plain value, it's wrapped in Promise.resolve(value). If you throw an error, it returns Promise.reject(error). This means you can always .then() or await an async function's result, even if it returns a non-Promise value.
Q: Async/Await vs Promises — when to use which?
Use async/await for most sequential async code — it's more readable. Use Promise combinators (Promise.all, race, etc.) when you need to run operations in parallel or need combinator logic. They're not mutually exclusive — you often use both: async/await for structure and Promise.all inside for parallel execution.

🔒 Closures
One of the most-asked JS interview topics. Every JS developer must understand closures deeply.
What is a Closure?
A closure is a function that remembers the variables from its outer scope even after the outer function has returned. The inner function "closes over" the variables.
💡 ANALOGY
Think of a backpack. When a function is created inside another function, it gets a backpack full of all the variables from its surrounding scope. Even when the outer function is done and gone, the inner function still carries that backpack and can access those variables anytime.
JAVASCRIPT 
function makeCounter() {
  let count = 0; // this is in the closure's "backpack"

  return function () {
    count++; // still accesses count even after makeCounter() returns
    return count;
  };
}

const counter = makeCounter(); // makeCounter is DONE, but...
console.log(counter()); // 1 — count is still alive!
console.log(counter()); // 2
console.log(counter()); // 3

const counter2 = makeCounter(); // fresh closure, independent count
console.log(counter2()); // 1 — separate from counter
Classic Interview Trap — var in a Loop
JAVASCRIPT — FAMOUS GOTCHA 
// ❌ BROKEN — all callbacks share the SAME i (var is function-scoped)
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// Prints: 3, 3, 3  (NOT 0, 1, 2)

// ✅ FIX 1 — use let (block-scoped, new i per iteration)
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// Prints: 0, 1, 2 ✓

// ✅ FIX 2 — IIFE to capture i (old school pre-let)
for (var i = 0; i < 3; i++) {
  ((j) => setTimeout(() => console.log(j), 100))(i);
}
// Prints: 0, 1, 2 ✓
Practical Use Cases
JAVASCRIPT — REAL USES 
// 1. Data privacy / encapsulation
function createBankAccount(initialBalance) {
  let balance = initialBalance; // PRIVATE — can't access directly
  return {
    deposit(n) { balance += n; },
    withdraw(n) { balance -= n; },
    getBalance() { return balance; }
  };
}

// 2. Memoization
function memoize(fn) {
  const cache = {}; // captured in closure
  return function(...args) {
    const key = JSON.stringify(args);
    if (cache[key]) return cache[key];
    return (cache[key] = fn(...args));
  };
}

// 3. Partial application
function multiply(x) {
  return (y) => x * y; // x is closed over
}
const double = multiply(2);
console.log(double(5)); // 10
🎯 TOP INTERVIEW QUESTIONS
Q: What is a closure? Give a real-world example.
A closure is a function that retains access to its lexical scope (outer variables) even when executed outside that scope. Real example: a counter factory function returns an inner function that keeps incrementing a private `count` variable. The count is inaccessible from outside but persists across calls — this is encapsulation via closures.
Q: Can closures cause memory leaks?
Yes. If a closure holds a reference to a large object (like a DOM node or big array), and the closure itself is held in memory (e.g., stored in a variable or event listener), the garbage collector cannot free that object. Common fix: set the reference to null when done, or use WeakRef/WeakMap for large objects that can be garbage collected.
Q: What's the difference between a closure and a regular function?
Every function in JavaScript is technically a closure — all functions close over their defining scope. The practical distinction is when a function accesses variables from an outer function's scope that has already returned. At that point, the closure keeps those variables alive in memory even after the outer function's stack frame is gone.

🚀 Concurrency in Node.js
Node.js is single-threaded but highly concurrent. Here's exactly how it achieves that — and where the limits are.
Concurrency vs Parallelism
Concept Meaning Node.js?
Concurrency Handling multiple tasks by switching between them (interleaving) ✅ Yes — via event loop
Parallelism Executing multiple tasks truly simultaneously on multiple cores ⚠️ Limited — via cluster/workers
💡 ANALOGY
Concurrency: A chef preparing 5 dishes by working on each a bit at a time — chopping one, then stirring another, then checking the oven.

Parallelism: 5 chefs each making one dish at exactly the same time on their own stoves.

Node.js is the one chef who's extremely good at switching between tasks without wasting time waiting.
Why Node Scales for I/O
Traditional servers (like Apache) create a new thread per request. Each thread blocks while waiting for database queries or file reads. With 10,000 requests, you need 10,000 threads — massive memory overhead.

Node.js uses a single thread + event loop. While waiting for I/O, the thread is free to handle other requests. 10,000 concurrent connections require just one thread.
JAVASCRIPT — SIMULATING CONCURRENCY 
// These 3 requests run "concurrently" — all start immediately
// The event loop handles responses as they arrive

const server = http.createServer(async (req, res) => {
  // Each request handler runs independently
  const user = await db.getUser(req.params.id); // non-blocking
  res.json(user);
});

// Request 1 is waiting for DB → event loop handles Request 2
// Request 2 is waiting for DB → event loop handles Request 3
// DB responds to Request 1 → event loop resumes it
// All this happens on ONE thread!
CPU-Bound vs I/O-Bound
Node.js Weakness — CPU-Bound work: If you do heavy computation (image processing, crypto, JSON parsing a 50MB file), it blocks the event loop and all other requests freeze. Node is great for I/O-bound work (database queries, file reads, HTTP calls) but not for CPU-bound work.

Solution: Worker Threads, child_process, or offloading to a dedicated service.
Work Type Examples Node handles? Solution
I/O-bound DB queries, HTTP, file I/O ✅ Excellent Use async/await
CPU-bound Image resize, encryption, sorting 10M items ⚠️ Blocks loop Worker threads
Mixed Real-time apps, APIs ✅ Good with care Don't block main thread
🎯 TOP INTERVIEW QUESTIONS
Q: How does Node.js handle 10,000 concurrent connections?
Through the event loop + non-blocking I/O. When a request comes in and needs to wait for something (DB, file, network), Node delegates that work to the OS (via libuv) and immediately moves on to handle the next request. When the OS signals the I/O is complete, the callback is queued and processed when the event loop picks it up. No threads are created per connection — all sharing one event loop thread.
Q: What is "callback hell" and how do you solve it?
Callback hell (also called "pyramid of doom") is deeply nested callbacks that become unreadable and hard to maintain. Solutions: (1) Named functions instead of anonymous callbacks, (2) Promises with .then() chaining, (3) async/await for synchronous-looking code, (4) Libraries like async.js for complex flows.
Q: When would you use Node.js and when wouldn't you?
USE Node.js for: REST APIs, real-time apps (chat, gaming), microservices, streaming data, proxy servers, SSR. AVOID Node.js for: CPU-intensive tasks like video encoding, scientific computing, ML inference (use Python/Go/Rust instead), or applications requiring heavy parallelism. The key rule: if it's I/O-heavy, Node.js shines; if it's CPU-heavy, it struggles.

🏗️ Node.js Architecture
V8 + libuv + Node.js core. Understanding the internals makes you a far better Node.js developer.
The Architecture Stack
Your JavaScript Code
express, koa, your app
Node.js Core (JavaScript)
fs, http, path, events, streams
Node.js Bindings (C++)
bridges JS to C++ world
V8 Engine
Executes JS, JIT compiles
libuv
Event loop + thread pool
Operating System
Linux/macOS/Windows — kernel, sockets, files
V8 — The JS Engine
V8 is Google's open-source JavaScript engine, written in C++. It's what Chrome uses — Node.js just runs it without a browser.

V8 does: parsing JS → AST → bytecode (Ignition interpreter) → optimized machine code (TurboFan JIT compiler). It also manages the heap and runs the garbage collector.
libuv — The Async Powerhouse
libuv is a C library that provides Node.js with:

🔁 The Event Loop — the main loop mechanism
🧵 Thread Pool (4 threads default, max 1024) — for blocking I/O: file system, DNS, crypto, zlib
📡 Async I/O — network I/O using OS primitives (epoll on Linux, kqueue on macOS, IOCP on Windows) — NOT the thread pool
⏰ Timers — setTimeout, setInterval management
📌 KEY INSIGHT
Network I/O (HTTP requests, TCP sockets) does NOT use libuv's thread pool — it uses the OS's async I/O directly. The thread pool is only for things the OS doesn't have native async support for (like file system and DNS on some systems).
🎯 TOP INTERVIEW QUESTIONS
Q: What is V8 and what does it do in Node.js?
V8 is Google's high-performance JavaScript engine written in C++. In Node.js, V8: (1) parses and executes JavaScript, (2) JIT-compiles frequently-executed code to machine code for performance, (3) manages the memory heap for JS objects, (4) runs the garbage collector to free unused memory. It's essentially what allows JavaScript to run outside the browser.
Q: What is libuv and why does Node.js need it?
libuv is a multi-platform C library that provides Node.js with: the event loop implementation, a thread pool for blocking I/O operations, async file system and DNS operations, and network I/O using OS-native async mechanisms. Without libuv, Node.js would either be blocking (waiting for each I/O operation) or would need to implement its own platform-specific async I/O for Linux/macOS/Windows.
Q: How can you increase libuv's thread pool size?
Set the UV_THREADPOOL_SIZE environment variable: `UV_THREADPOOL_SIZE=8 node app.js` or in code: `process.env.UV_THREADPOOL_SIZE = '8'` (must be set before any I/O). Default is 4, max is 1024. Increase it when you have many concurrent CPU/disk-intensive operations like heavy crypto or file processing.

Hoisting
Hoisting is JS moving declarations to the top of their scope during compilation (before execution). Only the declaration is hoisted — not the initialization.
JAVASCRIPT — HOISTING 
// What you write:
console.log(x); // undefined (not ReferenceError!)
var x = 5;
console.log(x); // 5

// What JS actually sees (after hoisting):
var x; // declaration hoisted to top, initialized to undefined
console.log(x); // undefined
x = 5; // assignment stays in place
console.log(x); // 5

Temporal Dead Zone (TDZ)
JAVASCRIPT — TDZ 
// let and const ARE hoisted, but in TDZ until declaration
// Accessing them before declaration = ReferenceError

console.log(y); // ❌ ReferenceError: Cannot access 'y' before initialization
let y = 10;    // TDZ ends here

// The TDZ exists from the start of the block until the let/const line
{
  // TDZ for 'z' starts here
  // console.log(z); would throw
  let z = 5; // TDZ ends, z is accessible
  console.log(z); // ✅ 5
}
🎯 TOP INTERVIEW QUESTIONS
Q: What is hoisting?
Hoisting is JavaScript's behavior of moving variable and function declarations to the top of their scope during the compilation phase, before code execution. var declarations are hoisted and initialized to undefined. Function declarations are fully hoisted (both declaration and body). let and const are hoisted but remain in the Temporal Dead Zone until their declaration line.
Q: What is the Temporal Dead Zone?
The TDZ is the period between the start of a block and the point where a let or const variable is declared. During this zone, the variable exists in memory but cannot be accessed — any attempt throws a ReferenceError. This prevents using variables before they're properly initialized, unlike var which silently returns undefined.
Q: Why should you prefer const and let over var?
var has two major problems: (1) function-scoped not block-scoped, so it "leaks" out of if/for blocks unexpectedly, (2) can be re-declared without error, causing silent bugs. const and let are block-scoped (predictable), cannot be re-declared, and const prevents reassignment. The TDZ with let/const also catches use-before-declaration bugs instead of silently returning undefined.

👆 The `this` Keyword
The most confusing concept in JavaScript. Mastering `this` requires understanding 4 binding rules.
The 4 Rules of `this`
JAVASCRIPT — 4 BINDING RULES 
// RULE 1: Default binding — standalone call → global (or undefined in strict)
function alone() { console.log(this); } // window / global / undefined(strict)
alone();

// RULE 2: Implicit binding — method call → the object before the dot
const obj = {
  name: 'Hany',
  greet() { console.log(this.name); }
};
obj.greet(); // 'Hany' — this = obj

// RULE 3: Explicit binding — call/apply/bind
function say() { console.log(this.name); }
say.call({ name: 'Ahmed' });   // 'Ahmed'
say.apply({ name: 'Sara' });  // 'Sara'
const sayHany = say.bind({ name: 'Hany' });
sayHany(); // 'Hany' — permanently bound

// RULE 4: new binding — constructor call → the new object
function Person(name) {
  this.name = name; // this = newly created object
}
const p = new Person('Hany');
console.log(p.name); // 'Hany'
Arrow Functions — Lexical `this`
JAVASCRIPT — ARROW FUNCTIONS 
// Arrow functions DON'T have their own `this`
// They inherit `this` from the SURROUNDING scope (lexical)

const timer = {
  name: 'Timer',
  start() {
    // Regular function loses `this` inside callback
    setTimeout(function() {
      console.log(this.name); // ❌ undefined — `this` lost!
    }, 100);

    // Arrow function inherits `this` from start()'s context
    setTimeout(() => {
      console.log(this.name); // ✅ 'Timer'
    }, 100);
  }
};

// Arrow functions cannot be used as constructors
// const a = new ArrowFn(); ❌ TypeError
// Arrow functions also cannot use call/apply/bind to change `this`
🎯 TOP INTERVIEW QUESTIONS
Q: What is `this` in JavaScript?
`this` is a runtime binding that refers to the context in which a function is called — not where it's defined. Its value depends on HOW the function is called: (1) standalone call → global/undefined, (2) method call → the object, (3) call/apply/bind → explicit object, (4) new → the new object. Arrow functions are the exception — they don't have their own `this` and use lexical (outer) `this`.
Q: What's the difference between call, apply, and bind?
All three explicitly set `this`. call(thisArg, arg1, arg2) — invokes immediately, passes args individually. apply(thisArg, [arg1, arg2]) — invokes immediately, passes args as an array. bind(thisArg) — returns a NEW function permanently bound to thisArg, does not invoke immediately. Use bind when you want to create a reusable function with a fixed context.

🧬 Prototypes & OOP
JavaScript's inheritance model. Classes are syntactic sugar — under the hood, it's all prototype chains.
The Prototype Chain
Every JS object has an internal [[Prototype]] link. When you access a property, JS looks it up on the object, then its prototype, then prototype's prototype... until it reaches null.
JAVASCRIPT 
const animal = {
  eat() { console.log('eating'); }
};

const dog = Object.create(animal); // dog's prototype = animal
dog.name = 'Rex';
dog.eat(); // ✅ found on prototype (animal)

// Prototype chain: dog → animal → Object.prototype → null
console.log(dog.hasOwnProperty('name')); // true (own property)
console.log(dog.hasOwnProperty('eat'));  // false (inherited)

// Modern class syntax — same thing under the hood
class Animal {
  eat() { console.log('eating'); }
}
class Dog extends Animal {
  constructor(name) {
    super(); // calls Animal constructor
    this.name = name;
  }
  bark() { console.log('woof'); }
}
🎯 TOP INTERVIEW QUESTIONS
Q: How does prototypal inheritance work in JavaScript?
Every object has a hidden [[Prototype]] link to another object. When you access a property that doesn't exist on the object itself, JavaScript walks up the prototype chain until it finds it or reaches null. This creates inheritance without classes — child objects delegate property lookups to parent objects through this chain.
Q: What's the difference between `__proto__` and `prototype`?
`__proto__` is an accessor property on ALL objects that points to the object's prototype (its [[Prototype]]). `prototype` is a property on FUNCTION objects — it becomes the [[Prototype]] of objects created with `new ThatFunction()`. So: `new Foo().__proto__ === Foo.prototype` — they're two sides of the same coin.
Q: What does `Object.create(null)` do?
It creates an object with NO prototype chain — pure key-value store with no inherited methods like toString, hasOwnProperty, etc. Useful for creating clean dictionaries/maps that won't be accidentally polluted by prototype methods. It's slightly faster for lookups since there's no prototype chain to walk, and safer against prototype pollution attacks.

📞 Callbacks
The original async pattern in Node.js. Understanding callbacks makes Promises and async/await click.
What is a Callback?
A callback is a function passed as an argument to another function, to be called later when an operation completes. The Node.js convention is error-first callbacks: the first argument is always an error (null if none), the second is the result.
JAVASCRIPT — NODE ERROR-FIRST PATTERN 
// Node.js error-first callback convention
fs.readFile('data.txt', 'utf8', (err, data) => {
  if (err) {
    console.error('Error:', err.message);
    return; // ← always return after handling error!
  }
  console.log(data); // only runs if no error
});

// Callback hell (pyramid of doom)
getUser(id, (err, user) => {
  getPosts(user.id, (err, posts) => {
    getComments(posts[0].id, (err, comments) => {
      // deeply nested, hard to read, error handling nightmare
    });
  });
});
🎯 INTERVIEW QUESTIONS
Q: What is the Node.js callback convention and why?
Node.js uses "error-first" callbacks: the first parameter is always an error object (null if success), followed by result parameters. This is a convention (not enforced by the language) that ensures errors are always handled explicitly. It prevents the common mistake of forgetting to check for errors in async operations.
Q: What is callback hell and how do you escape it?
Callback hell is when multiple nested async operations create deeply indented, hard-to-read code (the "pyramid of doom"). Escape routes: (1) Named functions to flatten the nesting, (2) Promises to chain operations linearly, (3) async/await for synchronous-looking code, (4) Modularization — break into smaller functions. Promises and async/await are the modern standard.

Q: What is uncaughtException and should you use it?

Q: What's the difference between operational and programmer errors?

🌊 Streams & Buffers
Process data piece by piece instead of loading it all into memory. Essential for large files, real-time data, and HTTP.
Why Streams?
💡 ANALOGY
Without streams: to watch a movie online, you'd have to download the entire 4GB file before watching it. With streams: you watch it as it downloads — a little chunk at a time. Node.js streams work the same way: process data chunks as they arrive, never holding the whole thing in memory.
Problem without streams: Reading a 10GB file with fs.readFile() loads the ENTIRE file into RAM. With a stream, you process it chunk by chunk using only ~64KB of memory at a time.
Stream Types
Type Direction Example
Readable Source of data fs.createReadStream(), HTTP request
Writable Destination fs.createWriteStream(), HTTP response
Duplex Both TCP socket, WebSocket
Transform Modify data zlib.createGzip(), crypto.createCipher
Using Streams
JAVASCRIPT 
// Pipe — the power of streams (most common pattern)
// Reads file, compresses it, writes to new file
// Memory used: ~64KB regardless of file size!
const readStream = fs.createReadStream('huge-file.txt');
const gzip = zlib.createGzip();
const writeStream = fs.createWriteStream('huge-file.txt.gz');

readStream.pipe(gzip).pipe(writeStream);

// Stream events
readStream.on('data', (chunk) => {
  console.log('Chunk size:', chunk.length);
});
readStream.on('end', () => console.log('Done reading'));
readStream.on('error', (err) => console.error(err));
🎯 TOP INTERVIEW QUESTIONS
Q: What is backpressure in streams?
Backpressure occurs when data is produced faster than it can be consumed. For example, reading from a fast SSD and writing to a slow network connection — the readable stream floods the writable stream's internal buffer. Node.js pipe() handles this automatically: when the writable buffer is full, it pauses the readable stream until the buffer drains. Without handling backpressure, you can run out of memory.
Q: What is a Buffer in Node.js?
A Buffer is a fixed-size chunk of memory outside the V8 heap, used to hold binary data. Since JavaScript historically had no byte manipulation, Node.js created Buffer to work with raw binary data from files, networks, and streams. Buffers are instances of Uint8Array. Common use: Buffer.from('hello', 'utf8'), buffer.toString(), reading binary file data.

Q: How does Node.js module caching work?
After the first require() call, Node.js caches the module in require.cache (an object keyed by filename). Q: Can you use require() in ES modules?
No — ESM files (.mjs or "type":"module") cannot use require() or module.exports. You can, however, use dynamic import() in CommonJS files (it returns a Promise). To use a CJS package from ESM: `import pkg from 'cjs-pkg'` (default import) or `const {fn} = await import('cjs-pkg')`. Going the other way (CJS importing ESM) requires dynamic import.

🎯 TOP INTERVIEW QUESTIONS
Q: How does garbage collection work in V8?
V8 uses a generational GC. New Space (short-lived objects) is collected frequently with a fast Scavenge (copy collector). Objects that survive 2+ collections are promoted to Old Space, collected less frequently with Mark-Sweep (marks live objects, sweeps dead ones) and Mark-Compact (also compacts memory to reduce fragmentation). GC pauses are minimized with incremental and parallel collection.
Q: How do you detect memory leaks in Node.js?
Tools and methods: (1) process.memoryUsage() to monitor heap growth over time, (2) Node.js --inspect flag + Chrome DevTools heap snapshots to compare before/after, (3) clinic.js or 0x for production profiling, (4) heapdump package to take heap snapshots programmatically, (5) Watch for monotonically increasing heap in your monitoring (Datadog, New Relic).

🌐 Cluster & Worker Threads
How to scale Node.js across multiple CPU cores — the two approaches and when to use each.
The Problem
Node.js is single-threaded. A machine with 8 CPU cores runs Node.js on only 1 core, wasting 7 cores. How do we use all cores?
Cluster Module — Multiple Processes
JAVASCRIPT — CLUSTER 
const cluster = require('cluster');
const os = require('os');

if (cluster.isPrimary) {
  // Master process — forks workers
  const numCPUs = os.cpus().length;
  console.log(`Master PID: ${process.pid}, forking ${numCPUs} workers`);

  for (let i = 0; i < numCPUs; i++) {
    cluster.fork(); // creates worker process
  }

  cluster.on('exit', (worker) => {
    console.log(`Worker ${worker.process.pid} died. Restarting...`);
    cluster.fork(); // auto-restart dead workers
  });
} else {
  // Worker process — runs your actual server
  const app = require('./app');
  app.listen(3000);
  console.log(`Worker ${process.pid} started`);
}
Worker Threads — True Parallelism for CPU Work
JAVASCRIPT — WORKER THREADS 
// main.js — run CPU-intensive work without blocking
const { Worker } = require('worker_threads');

function runWorker(data) {
  return new Promise((resolve, reject) => {
    const worker = new Worker('./cpu-task.js', { workerData: data });
    worker.on('message', resolve);
    worker.on('error', reject);
  });
}

// Main thread free while worker crunches numbers
const result = await runWorker({ n: 1000000 });
Cluster vs Worker Threads
Aspect Cluster Worker Threads
Use case Scale HTTP servers across cores CPU-intensive background tasks
Memory Separate process (no sharing) Shared memory (SharedArrayBuffer)
Communication IPC (slow) MessagePort (fast)
Isolation Full crash isolation Shared memory, less isolation
Overhead Higher (full process) Lower (thread)
🎯 TOP INTERVIEW QUESTIONS
Q: How does cluster load balancing work?
The master process listens on a port. When a connection comes in, the OS (on most platforms) or Node.js uses round-robin to distribute incoming connections to worker processes. Workers are fully independent Node.js processes — they each have their own event loop, V8 instance, and memory. Workers can communicate with the master via IPC (process.send/on('message')).
Q: When would you use worker_threads instead of cluster?
Use worker_threads for CPU-intensive work that needs to run in the background without blocking the main thread: image/video processing, cryptographic operations, large JSON parsing, ML inference, complex calculations. Use cluster for scaling an HTTP server across all CPU cores to handle more concurrent requests. Key difference: cluster = multiple server instances, worker_threads = background computation.

⚙️ libuv & Thread Pool
The C library that makes Node.js async I/O possible across all platforms.
What libuv Does
libuv implements the event loop and provides an abstraction layer over OS-specific async I/O mechanisms. It unifies epoll (Linux), kqueue (macOS/BSD), and IOCP (Windows) behind a single API that Node.js uses.

🎯 TOP INTERVIEW QUESTIONS
Q: Why does Node.js use a thread pool for file I/O but not for network I/O?
Modern operating systems have mature, highly-efficient async networking APIs: epoll on Linux, kqueue on macOS, IOCP on Windows. These allow thousands of network connections to be monitored with a single system call, with no threads needed. File system operations don't have equivalent universal async APIs across all OSes, so libuv uses a thread pool to simulate async behavior by running blocking fs calls in background threads.

var in loop + closure All iterations share same var — prints 3,3,3 not 0,1,2 Use let (block-scoped per iteration)
Sequential awaits await a; await b; runs them serially (slow) Promise.all([a(), b()]) for parallel
uncaughtException Using it to recover and continue is dangerous Log, notify, then process.exit(1)
readFileSync in server Blocks entire event loop for all users Always use async fs.readFile()
Promise not awaited Silent failure — error never caught Always await or .catch() promises
