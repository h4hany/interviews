// event loop — phase by phase
⚡ MICROTASK CHECK — runs between every single phase and between individual callbacks. Always before the next phase begins.
MICROTASK CHECK (nextTick → Promise)
01
TIMER QUEUE
setTimeout · setInterval
Executes callbacks whose delay has expired. A delay of 0ms is treated as ~1ms by the OS — not truly zero. Timers are checked against the current timestamp.
MICROTASK CHECK (A)
02
I/O QUEUE
fs.readFile · network · etc
Handles completed async I/O callbacks (file system, sockets). Includes an I/O polling phase — the loop may block here to wait for I/O if no other work is pending.
MICROTASK CHECK (B)
03
CHECK QUEUE
setImmediate
setImmediate() callbacks run here. Inside an I/O callback, setImmediate is guaranteed to fire before any setTimeout — because we're already past the Timer phase.
MICROTASK CHECK (C)
04
CLOSE QUEUE
socket.on('close') · cleanup
Cleanup callbacks, e.g. socket.on('close'). After this phase the loop checks active handles/requests and decides whether to do another iteration or exit.
MICROTASK CHECK (D)
↩ Loop repeats — or exits if no more work
Interview tip: "The event loop is not a single queue — it has distinct phases each with their own queue. Microtasks (nextTick + Promises) are checked between each phase, giving them higher priority than any loop phase."
// microtask queues — the highest priority

nextTick Queue PRIORITY 1

Callbacks passed to process.nextTick().

Not technically part of the event loop — it's a Node.js runtime construct. Runs before ANY phase begins. Used for: error handling, cleanup, deferred work that must happen before I/O.

Promise Queue PRIORITY 2

Resolved Promise callbacks: Promise.resolve().then(), async/await continuations.

Runs only after the entire nextTick queue is empty. Both queues are drained fully before the loop phase advances.

EXECUTION ORDER // Synchronous code runs first — always console.log('1 - sync'); process.nextTick(() => { console.log('3 - nextTick'); // nextTick queue }); Promise.resolve().then(() => { console.log('4 - Promise'); // Promise queue (after nextTick) }); setTimeout(() => { console.log('5 - timer'); // Timer Queue phase }, 0); console.log('2 - sync');
Output: 1 → 2 → 3 → 4 → 5
Sync completes → nextTick drains → Promise drains → Event loop phase (timer)
NESTED nextTick — DANGEROUS PATTERN process.nextTick(() => { console.log('nextTick 1'); process.nextTick(() => { console.log('nextTick 2 (nested)'); // still runs before Promise! }); }); Promise.resolve().then(() => console.log('Promise')); // Output: nextTick 1 → nextTick 2 → Promise // Nested nextTick callbacks join the SAME drain cycle!
⚠ Starvation risk: Recursively calling process.nextTick() can starve the event loop — it will keep processing the nextTick queue forever without advancing to I/O or timers.
// libuv — the async foundation
🔷

V8 Engine

Compiles and executes JavaScript. The JS call stack lives here. Single-threaded.

⚡

libuv

C library providing async I/O, event loop, and thread pool. OS abstraction layer.

🐧

OS Kernel

epoll (Linux), kqueue (macOS), IOCP (Windows). Handles network I/O natively.

⚙ libuv Thread Pool — UV_THREADPOOL_SIZE

Default: 4 threads. Max: 1024. Used for: File I/O, DNS lookups, crypto, zlib.

Network I/O

  • Handled by OS kernel directly
  • Does NOT use thread pool
  • Non-blocking via epoll/IOCP
  • Scales to 1000s of connections
  • No thread overhead

File I/O (fs module)

  • Uses libuv thread pool
  • OS lacks async primitives for FS
  • Limited by UV_THREADPOOL_SIZE
  • Can bottleneck under heavy load
  • Solution: increase pool size
Interview answer: "Node.js achieves non-blocking I/O via libuv, which interfaces with the OS kernel for async network operations, and uses a thread pool for file system operations where the OS lacks async primitives. This is why Node.js can handle thousands of concurrent HTTP connections with a single thread."

Scenario 1: Classic order quiz

SNIPPET setTimeout(() => console.log('A - timer'), 0); setImmediate(() => console.log('B - immediate')); process.nextTick(() => console.log('C - nextTick')); Promise.resolve().then(() => console.log('D - promise')); console.log('E - sync');
1 SYNC E - sync
2 NEXTTICK C - nextTick
3 PROMISE D - promise
4 TIMER A - timer   (may swap with B outside I/O context)
5 CHECK B - immediate

Scenario 2: Inside an I/O callback

SNIPPET const fs = require('fs'); fs.readFile(__filename, () => { // We are NOW inside the I/O phase setTimeout(() => console.log('A - setTimeout inside I/O'), 0); setImmediate(() => console.log('B - setImmediate inside I/O')); process.nextTick(() => console.log('C - nextTick inside I/O')); });
1 NEXTTICK C - nextTick inside I/O
2 CHECK B - setImmediate inside I/O  (GUARANTEED before timer here)
3 TIMER A - setTimeout inside I/O
Key insight: Outside an I/O callback, setTimeout(0) vs setImmediate is nondeterministic. Inside an I/O callback, setImmediate always wins because we've already passed the Timer phase in the current iteration.

// interview cheatsheet

Loop Phases (order)

  • Microtasks (nextTick → Promise)
  • Timer Queue (setTimeout/setInterval)
  • Microtasks again
  • I/O Queue (fs, net callbacks)
  • Microtasks again
  • Check Queue (setImmediate)
  • Microtasks again
  • Close Queue (cleanup)

Microtask Rules

  • Runs between EVERY phase
  • Also between individual callbacks
  • nextTick before Promise — always
  • Both queues fully drained first
  • Recursive nextTick = starvation risk
  • Not part of event loop itself

I/O Behavior

  • Network: kernel handles it (no thread)
  • File I/O: libuv thread pool
  • Default pool size: 4 threads
  • Env var: UV_THREADPOOL_SIZE
  • I/O polling may block loop
  • setTimeout vs I/O = nondeterministic

setImmediate vs setTimeout(0)

  • Top-level: nondeterministic order
  • Inside I/O: setImmediate wins always
  • setTimeout 0ms ≈ 1ms minimum
  • setImmediate = Check Queue phase
  • setTimeout = Timer Queue phase
  • In I/O, Timer phase already passed

process.nextTick

  • NOT part of the event loop
  • Queued in Node.js runtime side
  • Runs after current op completes
  • Before ANY phase including timers
  • Use for: error handling, cleanup
  • Avoid recursive calls (starvation)

Staff-level Gotchas

  • Blocking the loop = all I/O blocked
  • Heavy CPU → worker_threads
  • Many file ops → increase pool size
  • I/O polling minimizes CPU idle
  • Loop exits when no active handles
  • process.nextTick can starve I/O
One-liner for interviews: "Node.js event loop has 4 phases — Timer, I/O, Check, Close. Microtasks (nextTick then Promises) are drained between every phase. setImmediate is guaranteed before setTimeout inside I/O callbacks. File I/O uses a thread pool; network I/O goes through the OS kernel directly."
