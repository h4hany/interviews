📄 Semantic HTML
Tags that describe meaning and structure for browsers, developers, and assistive tech.
What are semantic tags?
Semantic tags are HTML elements that clearly describe their meaning and content both to the browser and to developers. They define the structure and purpose of the content inside them, making the code more readable and improving accessibility and search engine optimization (SEO).
Why do we need semantic tags?
1. Improved readability: Make the structure of the webpage more understandable to developers.

2. SEO (Search Engine Optimization): Search engines use the structure of the HTML page to understand its content. Semantic tags help search engines like Google understand what parts of the page are the most important, boosting search rankings for relevant content.

3. Accessibility: Screen readers and other assistive technologies rely on semantic tags to navigate and interpret the content more accurately. For example, a screen reader will treat a <nav> tag as navigation and allow users to quickly skip to the main content.

4. Better for browsers: Browsers also rely on semantic tags to correctly render and style the page. A <section> or <article> is treated differently than a simple <div>.

5. Maintainability: Using semantic tags provides a logical structure, making the code more maintainable and reducing technical debt.
Key semantic elements
Use these instead of generic <div> when they fit: header (intro/logo/nav), main (primary content, one per page), nav (navigation links), section (thematic grouping), article (self-contained content), aside (tangential content), footer.
<header><nav>...</nav></header>
<main>
  <article>...</article>
  <aside>...</aside>
</main>
<footer>...</footer>
Q: Why use semantic HTML?
Accessibility (screen readers use landmarks), SEO (crawlers understand structure), and maintainability. It also makes default styles and behavior more predictable. Avoid div soup when a semantic tag fits.
Q: How many <main> elements should a page have?
One. <main> represents the primary content of the document. Having more than one can confuse assistive tech and SEO. Use <section> or <article> to subdivide content inside main.

📌 Void elements & HTML entities
Elements without closing tags, whitespace behavior, and reserved characters.
Void elements
HTML elements that do not have closing tags or do not need to be closed are void elements. Examples: <br>, <br />, <img>, <hr>, <input>, <meta>, <link>. In HTML5 the trailing slash is optional.
Collapsing white space
In HTML, a blank sequence of whitespace characters (spaces, newlines, tabs) is treated as a single space character. This helps keep source readable without affecting how the page renders. Use &nbsp; or CSS (e.g. white-space: pre) when you need to preserve or force spaces.
HTML entities
Some characters are reserved in HTML (e.g. <, >, &, "). To display them in the page you use character entities: &lt;, &gt;, &amp;, &quot;. Named entities like &copy; (©) and &nbsp; (non-breaking space) are also common.
&lt;   &gt;   &amp;   &quot;   &copy;   &nbsp;
Q: What are void elements? Give examples.
Void elements have no content and no closing tag: <br>, <img>, <hr>, <input>, <meta>, <link>. They are self-closing by definition.
Q: Why use HTML entities?
To display reserved characters (e.g. <, >, &) that would otherwise be interpreted as markup, and to show symbols (©, ®, non-breaking space) in a way that works across encodings.

📐 Position & display (CSS)
Position and display properties commonly discussed in HTML/CSS interviews.
Position
Static: Default. Element follows the normal document flow.
Relative: Normal flow, but positioned relative to its original position (top/right/bottom/left).
Absolute: Positioned relative to the nearest positioned ancestor (or initial containing block). Removed from flow.
Fixed: Like absolute but relative to the viewport; doesn’t scroll with the page.
Sticky: Hybrid: normal until a threshold, then “sticks” like fixed within a container.
Initial / Inherit: Reset to default or inherit from parent.
Display
Inline: Element flows inline; width/height don’t apply. Examples: span, a.
Block: Element takes full width and starts on a new line. Examples: div, p.
Inline-block: Inline flow but you can set width and height.
Flex: Flex container; children follow flexbox rules.
Inline-flex: Flex container that behaves as an inline element.
Grid: Grid container for CSS Grid layout.
None: Element is not rendered and doesn’t take space.
Q: Difference between position: absolute and position: fixed?
Absolute is positioned relative to the nearest positioned ancestor (or the initial containing block). Fixed is positioned relative to the viewport and stays in place when the user scrolls.
Q: When would you use inline-block instead of inline or block?
When you need an element to flow inline (sit next to other content) but still apply width, height, and vertical padding/margin. Inline elements ignore those; block elements force a new line.

🖼️ SVG vs Canvas (HTML5)
Vector vs raster graphics; when to use each.
SVG (Scalable Vector Graphics)
Vector-based — composed of shapes (paths, circles, text).
Works better with larger surface and complex shapes.
Can be modified with CSS and scripts (DOM nodes, attributes).
Highly scalable — print at high quality at any resolution.
Good for icons, logos, charts, and accessible graphics (text and structure stay in DOM).
Canvas
Raster-based — composed of pixels.
Works better with smaller surface and pixel-level manipulation.
Can only be modified via scripts (no DOM for drawn content).
Less scalable — scaling up can look blurry.
Good for games, image processing, and dynamic pixel drawing.
Q: SVG vs Canvas — when do you choose which?
Use SVG when you need scalability, accessibility, or to style/script individual parts (icons, data viz, responsive graphics). Use Canvas when you need high-frequency pixel updates (games, filters) or don’t need DOM structure for the drawing.
Q: Can you style SVG with CSS?
Yes. SVG elements are part of the DOM and can be styled with CSS (fill, stroke, opacity, etc.) and animated. Canvas is a single bitmap; you can’t style individual drawn shapes with CSS.

♿ Forms & Accessibility
Labels, fieldset, ARIA, focus order, and contrast.
Labels and form structure
Always pair every input with a label using <label for="input-id"> so the label is associated with the control. Use <fieldset> and <legend> to group related inputs (e.g. radio groups). This gives screen readers clear structure and improves click targets.
<label for="email">Email</label>
<input id="email" type="email" name="email">

<fieldset>
  <legend>Choose one</legend>
  <input type="radio" id="a" name="opt"> <label for="a">A</label>
  <input type="radio" id="b" name="opt"> <label for="b">B</label>
</fieldset>
ARIA and focus
Use ARIA when native HTML is not enough: aria-label for unlabeled icons, aria-describedby to link hints or errors to inputs, role when you build custom widgets. Ensure logical focus order (tab order matches visual order) and visible focus styles. Do not rely on color alone; use sufficient color contrast (e.g. WCAG AA).
Q: Why is <label for="id"> important?
It programmatically associates the label with the input so screen readers announce the label when the input is focused, and clicking the label focuses the input. It also increases the clickable area and is required for accessibility.
Q: When do you use ARIA instead of native HTML?
When there is no suitable native element (e.g. custom dropdown, tab panel). Prefer native elements (button, input, select) first; use ARIA to expose state and semantics for custom components that cannot be built with natives alone.

🚀 SEO & Performance
Meta tags, structure, and loading best practices.
SEO basics
Use a meaningful <title> (unique per page, descriptive). Provide <meta name="description" content="..."> that summarizes the page; it often appears in search snippets. Use semantic structure and a clear heading hierarchy: one h1 per page, then h2, h3 in order. Avoid skipping levels.
Performance (images and loading)
Use loading="lazy" for images below the fold. Always set width and height to avoid layout shift (CLS). Prefer modern formats like WebP or AVIF with fallbacks (<picture> or source). Use decoding="async" for non-critical images.
<img src="photo.webp" alt="Description" width="800" height="600"
     loading="lazy" decoding="async">
Q: How does semantic HTML help SEO?
Search engines use structure to understand content. Semantic tags (main, article, nav, headings) signal what is important and how sections relate. A clear hierarchy and one h1 help crawlers and can improve how your page is ranked and displayed.
Q: Why set width and height on images?
To prevent layout shift (CLS): without dimensions the layout does not reserve space until the image loads, causing content to jump. Setting width/height reserves space and improves perceived performance and Core Web Vitals.

📋 HTML interview cheatsheet
Full quick reference for study and interviews.
Semantics
header, main (one per page), nav, section, article, aside, footer. Use them instead of div when they fit. Benefits: readability, SEO, accessibility, maintainability.
Void and entities
Void elements: br, img, hr, input, meta, link (no closing tag). Whitespace collapses to one space. Entities: &lt; &gt; &amp; &quot; &copy; &nbsp; for reserved or special characters.
Position and display
Position: static, relative, absolute, fixed, sticky. Display: inline, block, inline-block, flex, grid, none. Absolute vs fixed: absolute relative to ancestor, fixed relative to viewport.
SVG vs Canvas
SVG: vector, scalable, styleable with CSS, good for icons and charts. Canvas: raster, pixel-level, script-only, good for games and dynamic drawing.
Forms and A11y
label + for, fieldset + legend. ARIA when native is not enough. Focus order, visible focus, color contrast. Do not rely on color alone.
SEO and perf
Title, meta description, semantic structure, one h1, heading hierarchy. Lazy loading, width and height on images, WebP or AVIF with fallbacks.
