Virtual DOM & Reconciliation
💡 ANALOGY
The Virtual DOM is like a blueprint comparison. Instead of renovating your entire house every time, you compare the new blueprint to the old one, find only the changed rooms, and renovate only those. React compares old vDOM to new vDOM (diffing), finds the minimal set of changes (reconciliation), and applies only those to the real DOM (commit phase). This is why React is fast even with frequent state changes.

Keys — Why They Matter
TSX — KEYS 
// ❌ BAD — index as key (breaks on reorder/insert/delete)
items.map((item, index) => <Item key={index} {...item} />)
// If item at index 0 is removed, React thinks ALL items changed!
// Causes: wrong component state, input focus loss, animation bugs

// ✅ GOOD — stable unique ID as key
items.map((item) => <Item key={item.id} {...item} />)

// ✅ GOOD — stable hash for items without ID
items.map((item) => <Item key={`${item.name}-${item.type}`} {...item} />)

// Keys are also used to FORCE remount (reset state)
const [userId, setUserId] = useState(1);
<UserProfile key={userId} userId={userId} />
// Changing key forces full unmount + remount → fresh state!
🎯 INTERVIEW QUESTIONS
Q: What is the Virtual DOM and how does React use it?
The Virtual DOM is a lightweight JavaScript representation of the real DOM. When state or props change, React creates a new Virtual DOM tree and "diffs" it against the previous one using its reconciliation algorithm (Fiber). It finds the minimal set of changes needed and batches them into a single DOM update. This avoids expensive direct DOM manipulation — touching the real DOM is slow; comparing JavaScript objects is fast. React 18+ uses concurrent rendering with Fiber, allowing React to pause, resume, and prioritize rendering work.
Q: Why should you not use array index as a key?
React uses keys to identify which items changed, were added, or removed. Index keys break when: (1) Items are reordered — React sees different component at same index and destroys/recreates unnecessarily, (2) Items are inserted at the beginning — ALL subsequent items get new "identities" → performance hit and state bugs, (3) Items have internal state — a text input at index 1 that gets index 0 after deletion retains the wrong input value. Use stable, unique IDs. Only use index when the list is purely static and never reordered/filtered.
Q: What is React Fiber?
Fiber is React's reconciliation engine (rewritten in React 16). Key improvements over the old stack reconciler: (1) Incremental rendering — can split work into chunks and spread across frames, (2) Prioritization — urgent updates (user input) can interrupt and preempt lower-priority updates (data fetching), (3) Concurrency — the foundation for Concurrent Mode, Suspense, transitions, (4) Better error handling — error boundaries. Fiber represents each component as a "fiber" node in a linked list tree, enabling pausing and resuming traversal.

🎣 Hooks Deep Dive
The most important React topic. Every hook explained with gotchas, rules, and when to use each.
Essential Hooks
TSX — ALL CORE HOOKS 
import { useState, useEffect, useRef, useMemo, useCallback, useReducer } from 'react';

// useState — local state
const [count, setCount] = useState(0);
setCount(prev => prev + 1);  // functional update — safe with closures

// useEffect — side effects
useEffect(() => {
  const sub = api.subscribe(userId);  // setup
  return () => sub.unsubscribe();      // cleanup on unmount or dep change
}, [userId]);                            // deps: re-run if userId changes
// [] = run once on mount
// no array = run after every render (usually wrong!)

// useRef — mutable ref, no re-render on change
const inputRef = useRef<HTMLInputElement>(null);
const prevValue = useRef(value);  // track previous value
inputRef.current?.focus();

// useMemo — memoize expensive computation
const sortedItems = useMemo(
  () => [...items].sort((a, b) => a.price - b.price),
  [items]  // only re-sort when items changes
);

// useCallback — memoize function reference
const handleClick = useCallback(() => {
  setCount(prev => prev + 1);
}, []);  // stable reference — safe to pass as prop to memo() children

// useReducer — complex state logic
const [state, dispatch] = useReducer(reducer, initialState);
dispatch({ type: 'INCREMENT', payload: 5 });
Custom Hooks — Reusable Logic
TSX — CUSTOM HOOKS 
// Extract reusable stateful logic into custom hooks
function useLocalStorage<T>(key: string, initial: T) {
  const [value, setValue] = useState<T>(() => {
    try {
      const stored = localStorage.getItem(key);
      return stored ? JSON.parse(stored) : initial;
    } catch { return initial; }
  });

  const set = useCallback((val: T) => {
    setValue(val);
    localStorage.setItem(key, JSON.stringify(val));
  }, [key]);

  return [value, set] as const;
}

// useDebounce — delay value changes
function useDebounce<T>(value: T, delay = 500): T {
  const [debouncedValue, setDebounced] = useState(value);
  useEffect(() => {
    const timer = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(timer);
  }, [value, delay]);
  return debouncedValue;
}

// Usage — clean and reusable!
const [theme, setTheme] = useLocalStorage("theme", "dark");
const debouncedSearch = useDebounce(searchQuery, 300);
Rules of Hooks
⚠️ Rules of Hooks (enforced by ESLint plugin):

1. Only call hooks at the top level — never inside loops, conditions, or nested functions. React relies on the call order being the same every render.

2. Only call hooks from React function components or custom hooks — not regular JS functions.

Violation causes: "React Hook called conditionally" — React can't track which state belongs to which hook if order changes between renders.
🎯 INTERVIEW QUESTIONS
Q: What is the difference between useMemo and useCallback?
useMemo memoizes a computed VALUE — it caches the result of a function. useCallback memoizes a FUNCTION REFERENCE — it returns the same function instance across renders. useMemo: const sorted = useMemo(() => sort(items), [items]) — re-sorts only when items changes. useCallback: const handler = useCallback(() => doSomething(id), [id]) — returns same function reference unless id changes. Use useCallback when passing functions as props to memoized child components (otherwise memo() sees a new function every render and re-renders anyway).
Q: Explain the useEffect cleanup function.
The function returned from useEffect runs BEFORE the next effect and when the component unmounts. Use it to cancel subscriptions, clear timers, abort fetch requests, or remove event listeners — anything that would leak if not cleaned up. Without cleanup: subscribing in useEffect without unsubscribing creates memory leaks. When userId changes and effect re-runs: old subscription is cleaned up, new subscription is created. When component unmounts: subscription is cleaned up. Pattern: setup in effect body, teardown in return function.
Q: When would you use useReducer instead of useState?
useReducer is better when: (1) State has multiple sub-values that update together (a form with 10 fields), (2) Next state depends on previous state with complex logic, (3) Multiple actions can trigger the same state shape in different ways, (4) State transitions need to be testable in isolation (reducer is a pure function — easy to unit test). useState is simpler for single values or simple objects. Rule of thumb: if you have more than 3-4 related useState calls, consider useReducer.

When useState isn't enough. Redux Toolkit, Zustand, Jotai, and how to choose the right tool.
State Categories

Server state API responses, DB data, loading/error React Query / TanStack Query, SWR

⚠️ Most common mistake: Using Redux or Zustand for server state (API data). Server state has unique needs: caching, background refetching, stale data, deduplication. Use React Query / TanStack Query for server state and a lightweight store (Zustand) for client state.
Zustand — Modern Minimal State
TSX — ZUSTAND 
import { create } from 'zustand';
import { persist } from 'zustand/middleware';

interface CartStore {
  items: CartItem[];
  total: number;
  addItem(item: CartItem): void;
  removeItem(id: string): void;
  clear(): void;
}

export const useCartStore = create<CartStore>()(
  persist(
    (set, get) => ({
      items: [],
      total: 0,
      addItem(item) {
        set(state => ({
          items: [...state.items, item],
          total: state.total + item.price,
        }));
      },
      removeItem(id) {
        const item = get().items.find(i => i.id === id);
        set(state => ({
          items: state.items.filter(i => i.id !== id),
          total: state.total - (item?.price ?? 0),
        }));
      },
      clear: () => set({ items: [], total: 0 }),
    }),
    { name: 'cart-storage' }  // persists to localStorage
  )
);

// Usage — no providers needed!
const { items, addItem } = useCartStore();
// Only re-render on exact slice change:
const total = useCartStore(state => state.total);
Redux Toolkit — When You Need Full Power
TSX — REDUX TOOLKIT 
import { createSlice, createAsyncThunk } from '@reduxjs/toolkit';

// createAsyncThunk — handle async actions cleanly
export const fetchUser = createAsyncThunk(
  'user/fetch',
  async (userId: string, { rejectWithValue }) => {
    try {
      return await api.getUser(userId);
    } catch (err) {
      return rejectWithValue(err.message);
    }
  }
);

const userSlice = createSlice({
  name: 'user',
  initialState: { data: null, status: 'idle' },
  reducers: {
    logout: (state) => { state.data = null; }  // Immer = mutable writes!
  },
  extraReducers: (builder) => {
    builder
      .addCase(fetchUser.pending, (state) => { state.status = 'loading'; })
      .addCase(fetchUser.fulfilled, (state, action) => {
        state.status = 'success';
        state.data = action.payload;
      })
      .addCase(fetchUser.rejected, (state) => { state.status = 'error'; });
  }
});
🎯 INTERVIEW QUESTIONS
Q: When would you choose Zustand over Redux Toolkit?
Zustand is simpler, less boilerplate, no Provider needed, tiny bundle (~1KB). Use Zustand for: most medium apps, client state that doesn't need Redux's powerful tooling, teams new to state management. Use Redux Toolkit when: large teams need strong conventions, need Redux DevTools for time-travel debugging, complex derived state (RTK Query + selectors), existing Redux codebase, need robust middleware ecosystem. For new projects in 2024, Zustand + TanStack Query for server state is often the better choice over Redux.
Q: What is the difference between server state and client state?
Client state is owned by the frontend — it lives only in your app (theme, modal open/closed, selected tab). You control when it changes. Server state lives on the server — your app just has a cached copy (user data from API, product list). Challenges with server state: cache invalidation (when is your cached data stale?), background refetching, optimistic updates, deduplication of identical requests, loading/error states per query. React Query/TanStack Query solves all of these. Putting API responses in Redux is working against the grain — you end up reimplementing what React Query does better.

⚡ Performance Optimization
React renders fast by default. Know when and how to optimize — and crucially, when NOT to over-optimize.
Preventing Unnecessary Re-renders
TSX — MEMOIZATION 
import { memo, useMemo, useCallback } from 'react';

// React.memo — skip re-render if props unchanged
const ExpensiveList = memo(({ items, onSelect }: Props) => {
  return (
    <ul>{items.map(item => (
      <li key={item.id} onClick={() => onSelect(item)}>{item.name}</li>
    ))}</ul>
  );
});

// Parent — WRONG: new function every render, memo() pointless!
function Parent() {
  const [count, setCount] = useState(0);
  const items = useMemo(() => computeItems(), []);  // ✅ stable

  // ❌ New function ref every render → memo() re-renders anyway!
  const handleSelect = (item) => doSomething(item);

  // ✅ Stable function ref — memo() works!
  const handleSelect = useCallback((item) => doSomething(item), []);

  return <ExpensiveList items={items} onSelect={handleSelect} />;
}

// Code splitting — lazy load heavy components
const HeavyDashboard = lazy(() => import('./Dashboard'));

function App() {
  return (
    <Suspense fallback={<Spinner />}>
      <HeavyDashboard />
    </Suspense>
  );
}
When NOT to Optimize
Don't over-memoize! memo(), useMemo(), and useCallback all have overhead (memory + comparison cost). They're only worth it when:

✅ The component is genuinely expensive to render (100+ DOM nodes, heavy computation)
✅ memo() — only when re-renders with same props are actually happening
✅ useMemo — computation takes >1ms (sorting 1000s of items, complex calculations)
✅ useCallback — only when the function is a dependency in another hook or passed to memo() components

Premature optimization with memo/useMemo makes code harder to read with little benefit. Profile first with React DevTools Profiler!
React 18 Concurrent Features
TSX — CONCURRENT MODE 
import { useTransition, useDeferredValue } from 'react';

// useTransition — mark state update as non-urgent
const [isPending, startTransition] = useTransition();

const handleSearch = (query: string) => {
  setSearchQuery(query);  // urgent: update input immediately
  startTransition(() => {
    setFilteredItems(filterItems(allItems, query));  // non-urgent: can be interrupted
  });
};
// isPending: show loading indicator while transitioning

// useDeferredValue — defer non-urgent value update
const deferredQuery = useDeferredValue(searchQuery);
// deferredQuery lags behind searchQuery
// Heavy component re-renders with deferredQuery, not every keypress
const results = useMemo(
  () => heavySearch(items, deferredQuery),
  [items, deferredQuery]
);
🎯 INTERVIEW QUESTIONS
Q: What causes unnecessary re-renders in React?
Unnecessary re-renders happen when: (1) Parent re-renders causing child to re-render even though child's props didn't change — fix with React.memo(), (2) New object/array/function reference created inline every render ({} !== {} even with same content) — fix with useMemo/useCallback, (3) Context value changes causing all consumers to re-render even if they use only a small part — fix by splitting context or using state management library, (4) Missing or wrong dependency arrays in hooks causing stale closures or infinite loops. Use React DevTools Profiler to identify which components are expensive.
Q: What is the difference between useTransition and useDeferredValue?
Both help with non-urgent updates (React 18 concurrent features). useTransition: you CONTROL the state update — wrap a state setter in startTransition to mark it as low priority. The transition can be interrupted by urgent updates. Returns isPending to show loading UI. Use when: you have control over the state update. useDeferredValue: you RECEIVE a value and defer its downstream effects — React will use the old value for rendering while computing the new one, keeping the UI responsive. Use when: you receive a value from props or can't wrap the update in startTransition (e.g., in a third-party component).

Compound Components
TSX — COMPOUND COMPONENTS 
// Compound components — flexible composition with shared state
const TabsContext = createContext<TabsContextType>(null!);

function Tabs({ children, defaultTab }: Props) {
  const [active, setActive] = useState(defaultTab);
  return (
    <TabsContext.Provider value={{ active, setActive }}>
      <div className="tabs">{children}</div>
    </TabsContext.Provider>
  );
}

Tabs.Tab = function Tab({ id, children }: Props) {
  const { active, setActive } = useContext(TabsContext);
  return (
    <button
      className={active === id ? 'active' : ''}
      onClick={() => setActive(id)}
    >{children}</button>
  );
};

Tabs.Panel = function Panel({ id, children }: Props) {
  const { active } = useContext(TabsContext);
  return active === id ? <div>{children}</div> : null;
};

// Usage — flexible, readable, no prop drilling!
<Tabs defaultTab="overview">
  <Tabs.Tab id="overview">Overview</Tabs.Tab>
  <Tabs.Tab id="details">Details</Tabs.Tab>
  <Tabs.Panel id="overview"><Overview /></Tabs.Panel>
  <Tabs.Panel id="details"><Details /></Tabs.Panel>
</Tabs>
Controlled vs Uncontrolled Components
TSX — CONTROLLED vs UNCONTROLLED 
// Controlled — React owns the state
function ControlledInput() {
  const [value, setValue] = useState('');
  return <input value={value} onChange={e => setValue(e.target.value)} />;
  // ✅ Can validate on every keystroke
  // ✅ Can programmatically change value
  // ⚠️ Re-renders on every keystroke (use React Hook Form for perf)
}

// Uncontrolled — DOM owns the state
function UncontrolledInput() {
  const inputRef = useRef<HTMLInputElement>(null);
  const handleSubmit = () => console.log(inputRef.current?.value);
  return <input ref={inputRef} defaultValue="" />;
  // ✅ No re-renders while typing
  // ✅ Easy integration with non-React code
  // ⚠️ Can't easily synchronize with other state
}
🎯 INTERVIEW QUESTIONS
Q: What is the compound component pattern and when do you use it?
Compound components share implicit state through Context instead of prop drilling. The parent component manages state, sub-components access it via Context. Examples: Tabs/Tab/Panel, Select/Option, Accordion/Item. Benefits: flexible composition (users arrange sub-components however they want), no prop explosion (no isOpen/onChange/items passed to single component), intuitive API that mirrors HTML (like select/option). Use when: you have a component with multiple related sub-parts that share state, and you want to give users composition flexibility.
Q: What are Higher-Order Components (HOCs) and are they still used?
HOCs are functions that take a component and return an enhanced component: const Enhanced = withAuth(MyComponent). They add cross-cutting concerns (authentication, logging, theming) without modifying the wrapped component. Mostly replaced by hooks in modern React — a custom useAuth() hook is simpler and more readable than withAuth(). HOCs are still used in: class components (hooks don't work there), third-party libraries (React Redux's connect), when you need to wrap a component you don't own. Prefer hooks for new code.

🌐 Context API
Avoid prop drilling without a state library. Context pitfalls, performance patterns, and when to use Context vs Zustand.
Context Basics & Performance Trap
TSX — CONTEXT BEST PRACTICES 
// ❌ BAD — single context with many values → ALL consumers re-render
const AppContext = createContext({ user, theme, cart, settings });
// Change cart → user-consuming components re-render too!

// ✅ GOOD — split by concern (separate frequently-changing state)
const UserContext = createContext<User | null>(null);
const ThemeContext = createContext<string>('light');

// ✅ BEST — memoize context value to avoid unnecessary re-renders
function AuthProvider({ children }: { children: ReactNode }) {
  const [user, setUser] = useState<User | null>(null);

  const login = useCallback(async (credentials) => {
    const user = await api.login(credentials);
    setUser(user);
  }, []);

  const logout = useCallback(() => setUser(null), []);

  // Memoize the context value object
  const value = useMemo(() => ({ user, login, logout }), [user, login, logout]);

  return (
    <UserContext.Provider value={value}>
      {children}
    </UserContext.Provider>
  );
}

// Custom hook for safe consumption
function useAuth() {
  const context = useContext(UserContext);
  if (!context) throw new Error('useAuth must be inside AuthProvider');
  return context;
}
🎯 INTERVIEW QUESTIONS
Q: What is a common performance problem with Context and how do you fix it?
Problem: every consumer of a Context re-renders when ANY value in the context changes, even if they don't use the changed value. If you put {user, theme, notifications} in one context, changing notifications causes user-consuming components to re-render unnecessarily. Solutions: (1) Split context by concern — separate UserContext, ThemeContext, NotificationContext, (2) Memoize context value with useMemo so it doesn't change unless actual values change, (3) Use useCallback for functions in context value, (4) For high-frequency updates, use a state management library (Zustand, Jotai) which only re-renders components that subscribe to changed slices.
Q: When should you use Context vs Zustand/Redux?
Context: good for low-frequency updates (theme, auth user, language), data accessed by many components at different tree levels, when you want zero dependencies. Zustand/Redux: better for high-frequency updates (animations, real-time data), when you need to select specific slices (avoid unnecessary re-renders), complex update logic, multiple unrelated state slices, devtools. Context is not a state management solution — it's a dependency injection mechanism. For anything updated regularly or complex, use a proper store.

🔄 Data Fetching
React Query / TanStack Query — the modern standard for server state. Why it replaces useEffect for fetching.
Why React Query Beats useEffect for Fetching
TSX — REACT QUERY 
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';

// Basic query — handles loading, error, caching, refetching!
function UserProfile({ userId }: { userId: string }) {
  const { data: user, isLoading, error, isStale } = useQuery({
    queryKey: ['user', userId],   // cache key — auto-deduplicates
    queryFn: () => api.getUser(userId),
    staleTime: 5 * 60 * 1000,    // 5 min before considered stale
    gcTime: 10 * 60 * 1000,       // 10 min cache retention
    retry: 2,                       // retry failed requests twice
  });

  if (isLoading) return <Spinner />;
  if (error) return <ErrorMessage error={error} />;
  return <div>{user?.name}</div>;
}

// Mutation — create/update/delete with optimistic updates
function UpdateUser() {
  const queryClient = useQueryClient();

  const mutation = useMutation({
    mutationFn: (data: UpdateData) => api.updateUser(data),
    onMutate(newData) {              // optimistic update
      const prev = queryClient.getQueryData(['user', newData.id]);
      queryClient.setQueryData(['user', newData.id], newData);
      return { prev };              // return context for rollback
    },
    onError(err, _, context) {      // rollback on error
      queryClient.setQueryData(['user', _.id], context?.prev);
    },
    onSuccess() {                   // invalidate to refetch
      queryClient.invalidateQueries({ queryKey: ['user'] });
    },
  });
}
🎯 INTERVIEW QUESTIONS
Q: Why use React Query instead of useEffect for data fetching?
useEffect for fetching gives you a blank slate — you implement everything yourself: loading state, error handling, race conditions (two requests fired, second resolves first), cancellation, caching, deduplication, refetch on focus/online, stale data, background updates. React Query handles all of this. Additionally, useEffect has the "double invocation in StrictMode" problem where effects run twice in development. React Query gives you: automatic caching, background refetching, request deduplication, optimistic updates, infinite scroll, devtools. The setup cost is low; the benefits are massive.

const schema = z.object({
  email: z.string().email('Invalid email'),
  password: z.string().min(8, 'Min 8 characters'),
  age: z.number().min(18).max(120).optional(),
  role: z.enum(['admin', 'user']),
});

function LoginForm() {
  const { register, handleSubmit, formState: { errors, isSubmitting } } = useForm<FormData>({
    resolver: zodResolver(schema),
    defaultValues: { role: 'user' },
  });

  const onSubmit = async (data: FormData) => {
    await api.login(data);  // fully typed FormData
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input {...register('email')} type="email" />
      {errors.email && <span>{errors.email.message}</span>}

      <input {...register('password')} type="password" />
      {errors.password && <span>{errors.password.message}</span>}

      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? 'Logging in...' : 'Login'}
      </button>
    </form>
  );
}
🎯 INTERVIEW QUESTIONS
Q: Why is React Hook Form better than controlled forms for performance?
Controlled forms re-render the entire component on every keystroke — a form with 20 fields causes 20 re-renders per second while typing. React Hook Form uses uncontrolled inputs by default (refs, not state), so the component doesn't re-render while typing. Re-renders only happen for: initial render, form submission, specific field value subscriptions (watch()). With 100 users typing in complex forms, this is a significant performance difference. Additionally, React Hook Form integrates with Zod via zodResolver for type-safe validation derived from the same schema used for TypeScript types.

Q: What is the Testing Library philosophy and why is it better than Enzyme?
React Testing Library is guided by: "The more your tests resemble the way your software is used, the more confidence they can give you." This means: query elements by accessible roles/labels (as users find them), interact via userEvent (as users interact), test visible outcomes. Enzyme tested React internals (component state, instance methods, lifecycle) — tests broke on every refactor even if behavior stayed the same. RTL tests survive refactors because they test what users see and do, not how components are implemented. Tests should give you confidence that user-facing behavior works.

🎯 INTERVIEW QUESTIONS
Q: What is useOptimistic and how does it improve UX?
useOptimistic enables instant UI feedback before the server confirms an action. Instead of: click "Like" → wait 500ms → show liked, you get: click "Like" → immediately show liked → server confirms. If server fails, automatically rolls back. This is the "optimistic update" pattern. useOptimistic takes the real value and an update function, returns an optimistic value that is immediately updated when you call addOptimisticUpdate(), but reverts to the real value once the server action completes. Dramatically improves perceived performance for write operations.
Q: What are React Server Components?
Server Components (RSC) render on the server and send their HTML output to the client — no JavaScript bundle sent for them. Benefits: (1) Zero bundle size — import heavy libraries (markdown parsers, DB clients) without shipping to client, (2) Direct database access from components, (3) Faster initial page load (no hydration needed for server components), (4) Automatic code splitting. Limitations: no useState, useEffect, event handlers — they're not interactive. The mental model: Server Components for data-fetching/static content, Client Components ("use client") for interaction. Requires Next.js App Router or similar framework.

🎯 INTERVIEW QUESTIONS
Q: How does React Native work? How is it different from React?
React and React Native share the same component model and reconciliation algorithm, but React targets the browser DOM while React Native targets native mobile platform APIs. When you write <View> in RN, it becomes android.view.ViewGroup on Android and UIView on iOS — no WebView, no HTML. The JavaScript runs in a separate thread (Hermes JS engine), communicating with the native UI thread. React Native provides polyfills for common web APIs (Fetch, WebSockets, Timers) but doesn't have browser-specific APIs (window, document, localStorage). Shared logic (business logic, hooks, most components) can be shared between React and React Native.
Q: What is Hermes and why does React Native use it?
Hermes is a JavaScript engine built by Meta specifically for React Native. Advantages over V8/JSC: pre-compiled to bytecode at build time (faster startup — app doesn't need to parse JS on launch), lower memory usage, better garbage collection for mobile constraints. Default since RN 0.70. Performance impact: time-to-interactive reduced by 30-40% on Android for typical apps. Also enables: better debugging, faster development feedback, and is optimized for the mobile use case (not general-purpose web JS engine).

Q: What is the difference between ScrollView and FlatList?
ScrollView renders ALL its children immediately — great for small, fixed content (settings screen, form). If you put 1000 items in a ScrollView, all 1000 render at once: huge memory usage, slow initial render, janky scrolling. FlatList is virtualized — only renders items visible on screen plus a small buffer. Items outside the viewport are unmounted. For any list that could have more than ~20 items, always use FlatList. SectionList is FlatList with section headers. VirtualizedList is the base (use only for custom implementations).

import Animated, {
  useSharedValue, useAnimatedStyle, withSpring, withTiming, runOnJS
} from 'react-native-reanimated';

function AnimatedCard() {
  const translateX = useSharedValue(0);
  const opacity = useSharedValue(1);

  // Runs on UI thread — smooth even if JS is busy!
  const animatedStyle = useAnimatedStyle(() => ({
    transform: [{ translateX: translateX.value }],
    opacity: opacity.value,
  }));

  const handlePress = () => {
    translateX.value = withSpring(100);  // spring animation
    opacity.value = withTiming(0, { duration: 300 });
  };

  return (
    <Animated.View style={[styles.card, animatedStyle]}>
      <Pressable onPress={handlePress}>
        <Text>Swipe Me</Text>
      </Pressable>
    </Animated.View>
  );
}
🎯 INTERVIEW QUESTIONS
Q: Why use Reanimated 2 over Animated API?
React Native's built-in Animated API drives animations on the JS thread (unless using useNativeDriver: true, which is limited). If JS thread is busy, animations stutter. Reanimated 2 runs animation logic on the UI thread using worklets — compiled functions that run on the native side. This means animations stay smooth at 60/120fps even while JS is processing heavy updates. Additionally, Reanimated integrates with React Native Gesture Handler for fluid gesture-driven animations (swipe to delete, drag-and-drop) that feel native. For any production app with non-trivial animations, Reanimated 2 is the standard.

🎯 INTERVIEW QUESTIONS
Q: How do you securely store sensitive data in React Native?
AsyncStorage is NOT secure — it's plain text files on device, accessible with device access. For sensitive data: iOS — use Keychain Services (react-native-keychain wraps it). Android — use EncryptedSharedPreferences or Android Keystore (react-native-keychain also supports this). react-native-keychain provides a unified API: Keychain.setGenericPassword(username, password) and Keychain.getGenericPassword(). For authentication tokens, always use platform secure storage. Never store passwords or API keys in AsyncStorage, Redux state, or JavaScript variables that could be exposed.
Q: How do push notifications work in React Native?
Push notifications use platform services: APNs (Apple Push Notification service) for iOS, FCM (Firebase Cloud Messaging) for Android. Flow: (1) App registers for notifications, gets a device token, (2) App sends token to your backend, (3) Backend sends push via APNs/FCM with the device token, (4) Platform delivers to device even when app is closed. In React Native: use @notifee/react-native or @react-native-firebase/messaging. Always request permission before showing prompts (iOS requires explicit permission; Android 13+ also requires it). Handle: foreground notifications, background notifications, notification taps opening specific screens (deep linking).
