// TYPESCRIPT
📐 Types & Interfaces
The TypeScript foundation every Angular developer must master. type vs interface, union types, and narrowing.
type vs interface
TYPESCRIPT — TYPES & INTERFACES
// interface — open, mergeable, extends with extends keyword
interface User {
  id: number;
  name: string;
  email?: string;       // optional
  readonly createdAt: Date;  // immutable
}

interface User {        // declaration merging — adds to existing interface
  role: 'admin' | 'user';
}

interface AdminUser extends User {
  permissions: string[];
}

// type — more powerful, can't be merged, can use unions
type ID = number | string;           // union type
type Point = { x: number; y: number };
type 3DPoint = Point & { z: number };  // intersection type
type Status = 'active' | 'inactive' | 'banned';  // literal union
type Callback = (err: Error | null, data: unknown) => void;

// Type guards — narrow types at runtime
function isUser(obj: unknown): obj is User {
  return typeof obj === 'object' && obj !== null && 'id' in obj;
}

// Discriminated unions — exhaustive pattern matching
type Result<T> =
  | { status: 'success'; data: T }
  | { status: 'error';   error: string }
  | { status: 'loading' };

function render(result: Result<User>) {
  switch (result.status) {
    case 'success': return result.data.name;  // data typed as User
    case 'error':   return result.error;       // error typed as string
    case 'loading': return 'Loading...';
  }
}
🎯 INTERVIEW QUESTIONS
Q: What is the difference between type and interface in TypeScript?
interface is open — you can extend it with additional properties via declaration merging (important for augmenting third-party types). It supports extends and implements naturally. type is more powerful — can define unions (string | number), intersections (&), mapped types, conditional types, and aliases for primitives. You cannot merge type aliases. Rule of thumb in Angular projects: use interface for object shapes (User, Product), use type for unions, function types, and complex type transformations. Both compile away at runtime.
Q: What are discriminated unions and why are they useful?
A discriminated union is a union type where each member has a common literal property (the discriminant) that uniquely identifies it. TypeScript narrows the type within switch/if blocks based on this discriminant. Use case: representing different states (loading/success/error), API responses, action types in state management. With exhaustive checking (add a default: case that assigns to never), TypeScript catches at compile time if you add a new union member but forget to handle it — extremely valuable for maintainability.

🚀 Advanced Types
Mapped types, conditional types, and the utility types Angular devs use every day.
Utility Types
TYPESCRIPT — UTILITY TYPES
interface User { id: number; name: string; email: string; age: number }

// Built-in utility types
type PartialUser   = Partial<User>;         // all fields optional
type RequiredUser  = Required<User>;        // all fields required
type ReadonlyUser  = Readonly<User>;        // all fields readonly
type UserPreview   = Pick<User, 'id' | 'name'>;  // pick fields
type UserNoEmail   = Omit<User, 'email'>;   // exclude field
type StringMap     = Record<string, number>; // {[key: string]: number}
type NonNullUser   = NonNullable<User | null | undefined>; // removes null/undefined
type ReturnType    = ReturnType<typeof fetchUser>;   // infer return type
type Params        = Parameters<typeof fetchUser>; // infer param types

// Mapped type — transform every property
type Nullable<T> = {
  [K in keyof T]: T[K] | null;
};
type NullableUser = Nullable<User>; // { id: number|null; name: string|null; ... }

// Conditional type
type IsArray<T> = T extends any[] ? 'yes' : 'no';
type A = IsArray<string[]>;  // 'yes'
type B = IsArray<string>;    // 'no'

// Template literal types (TS 4.1+)
type EventName = `on${Capitalize<string>}`;
type CSSProperty = `--${string}`;  // CSS custom property

// Extract / Exclude
type Strings = Extract<string | number | boolean, string>;  // string
type NoStrings = Exclude<string | number | boolean, string>; // number | boolean
🎯 INTERVIEW QUESTIONS
Q: What is the difference between unknown and any?
both can hold any value, but any opts out of type checking entirely — you can do anything with an any value with no errors. unknown is the type-safe alternative — you can assign any value to unknown, but you can't use it until you narrow the type (with typeof, instanceof, or a type guard). Rule: never use any in production TypeScript. Use unknown for values whose type you genuinely don't know (API responses, JSON.parse results). This forces you to handle the type properly before using the value.
Q: How do Pick and Omit differ from Partial?
Partial<T> makes ALL properties of T optional — changes optionality but keeps all fields. Pick<T, K> creates a new type with only the specified keys K — useful for creating slimmer types for DTOs or view models. Omit<T, K> is the opposite — creates a type with all keys EXCEPT K. Common Angular pattern: Pick for response types (only expose needed fields to templates), Omit for create DTOs (exclude server-set fields like id and createdAt from POST body).

🔄 Generics
Write reusable, type-safe code. Generics are everywhere in Angular — HttpClient, Observables, signals.
Generics in Practice
TYPESCRIPT — GENERICS
// Generic function
function first<T>(arr: T[]): T | undefined {
  return arr[0];
}
const n = first([1, 2, 3]);  // T inferred as number
const s = first(['a', 'b']); // T inferred as string

// Generic constraint — T must have an id property
function findById<T extends { id: number }>(items: T[], id: number): T | undefined {
  return items.find(item => item.id === id);
}

// Generic class — type-safe HTTP service wrapper
class ApiService<T> {
  constructor(private http: HttpClient, private baseUrl: string) {}

  getAll(): Observable<T[]> {
    return this.http.get<T[]>(this.baseUrl);
  }

  getById(id: number): Observable<T> {
    return this.http.get<T>(`${this.baseUrl}/${id}`);
  }
}

// infer keyword — extract type from conditional type
type UnpackArray<T> = T extends (infer U)[] ? U : T;
type ItemType = UnpackArray<string[]>;  // string

type UnpackPromise<T> = T extends Promise<infer U> ? U : T;
type Resolved = UnpackPromise<Promise<User>>; // User

// Multiple type parameters
function zip<A, B>(a: A[], b: B[]): [A, B][] {
  return a.map((item, i) => [item, b[i]]);
}
🎯 INTERVIEW QUESTIONS
Q: What are generics and why are they useful in Angular?
Generics allow writing reusable code that works with multiple types while maintaining type safety. Without generics you'd need any (loses type safety) or overloads (verbose). In Angular, generics are everywhere: HttpClient.get<User[]>('/api/users') returns Observable<User[]> — TypeScript knows the exact type of the stream. signal<number>(0) creates a typed signal. EventEmitter<string> emits only strings. FormControl<string> in typed forms. The constraint syntax <T extends {...}> ensures T has minimum required shape.

🎁 TypeScript Decorators
Angular is built on decorators. Understanding them deeply explains how Angular's magic works.
Decorator Types
TYPESCRIPT — DECORATORS
// Class decorator — receives the constructor
function Sealed(constructor: Function) {
  Object.seal(constructor);
  Object.seal(constructor.prototype);
}

@Sealed
class BugReport {
  type = "report";
}

// Method decorator — logs execution time
function Log(target: any, key: string, descriptor: PropertyDescriptor) {
  const original = descriptor.value;
  descriptor.value = function(...args: any[]) {
    console.log(`Calling ${key}`);
    const result = original.apply(this, args);
    console.log(`Done ${key}`);
    return result;
  };
  return descriptor;
}

// Property decorator
function MinLength(min: number) {
  return function(target: any, propertyKey: string) {
    // attach metadata for validation runtime use
    Reflect.defineMetadata('minLength', min, target, propertyKey);
  };
}

// Angular uses decorators extensively:
@Component({              // class decorator
  selector: 'app-root',
  template: '...'
})
class AppComponent {
  @Input() title: string = '';   // property decorator
  @HostListener('click')          // method decorator
  onClick() {}
  constructor(@Inject(MY_TOKEN) val: string) {} // param decorator
}
🎯 INTERVIEW QUESTIONS
Q: How does @Component work under the hood?
@Component is a class decorator factory that takes a metadata object. Angular's compiler reads this metadata at build time using reflect-metadata. The decorator attaches the component metadata (selector, template, styles) to the class constructor. The Angular runtime (the compiler/JIT or Ivy AOT) uses this metadata to: create the component factory, compile the template into efficient JavaScript, register the component in the module's declarations. Ivy (Angular's current renderer) compiles decorators into static class fields (ɵcmp, ɵfac) during build, making them tree-shakeable.

🏗️ Architecture & Modules
NgModules, feature modules, lazy loading, and Angular's compilation model.
NgModule Architecture
ANGULAR — MODULE STRUCTURE
import { NgModule } from '@angular/core';
import { BrowserModule } from '@angular/platform-browser';
import { RouterModule } from '@angular/router';

@NgModule({
  declarations: [AppComponent, HeaderComponent],  // components, directives, pipes
  imports:      [BrowserModule, RouterModule],       // other modules
  exports:      [HeaderComponent],                   // make available to importers
  providers:    [UserService],                        // module-scoped providers
  bootstrap:    [AppComponent],                       // root component (AppModule only)
})
export class AppModule {}

// Feature module with lazy loading
const routes = [
  { path: 'admin', loadChildren: () =>
    import('./admin/admin.module').then(m => m.AdminModule) }
];

// SharedModule pattern — re-export common Angular modules
@NgModule({
  imports:  [CommonModule, FormsModule, ReactiveFormsModule],
  exports:  [CommonModule, FormsModule, ReactiveFormsModule,
             ButtonComponent, InputComponent],   // shared UI components
  declarations: [ButtonComponent, InputComponent],
})
export class SharedModule {}

// CoreModule pattern — singleton services + app-wide components
@NgModule({
  imports: [CommonModule],
  declarations: [NavbarComponent, FooterComponent],
  exports:  [NavbarComponent, FooterComponent],
  providers: [AuthService, LoggingService],  // app-singleton services
})
export class CoreModule {
  constructor(@Optional() @SkipSelf() parent: CoreModule) {
    if (parent) throw new Error('CoreModule already loaded!');
  }
}
🎯 INTERVIEW QUESTIONS
Q: What is the difference between declarations, imports, exports, and providers in NgModule?
declarations: components, directives, and pipes that belong to this module — they're "owned" by it. imports: other NgModules whose exported items this module needs. exports: subset of declarations (and imports) to make available to modules that import this one. providers: services available via DI in this module's component tree (in non-lazy modules, these become app-wide singletons unless scoped). A component can only be declared in ONE module. To use it elsewhere, the owning module must export it and the using module must import the owning module.
Q: How does lazy loading work in Angular?
Lazy loading defers loading a feature module's bundle until the route is navigated to. Router configuration: loadChildren: () => import('./feature/feature.module').then(m => m.FeatureModule). The router splits the module into a separate JS chunk at build time. When the user navigates to that route for the first time, Angular downloads the chunk, bootstraps the module, and renders the component. Benefits: smaller initial bundle, faster first load. Modern Angular (v17+) uses loadComponent with standalone components directly: loadComponent: () => import('./feature.component').then(c => c.FeatureComponent).

🧩 Components Deep Dive
Lifecycle hooks, @Input/@Output, ViewChild, component communication — everything about Angular components.
Lifecycle Hooks
ANGULAR — LIFECYCLE HOOKS
import { Component, OnInit, OnChanges, OnDestroy, AfterViewInit,
         Input, Output, EventEmitter, ViewChild, SimpleChanges,
         ElementRef } from '@angular/core';

@Component({ selector: 'app-user', template: `...` })
export class UserComponent implements OnInit, OnChanges, OnDestroy, AfterViewInit {

  @Input() userId!: number;             // ! = definite assignment
  @Input({ required: true }) name!: string;  // Angular 16+ required input
  @Output() userDeleted = new EventEmitter<number>();

  @ViewChild('myInput') inputEl!: ElementRef<HTMLInputElement>;

  // 1. constructor — DI only, no DOM access, no @Input values yet
  constructor(private userService: UserService) {}

  // 2. ngOnChanges — called when @Input values change (before ngOnInit)
  ngOnChanges(changes: SimpleChanges) {
    if (changes['userId'].currentValue !== changes['userId'].previousValue) {
      this.loadUser();
    }
  }

  // 3. ngOnInit — component initialized, @Input values available
  ngOnInit() { this.loadUser(); }

  // 4. ngAfterViewInit — DOM ready, ViewChild available
  ngAfterViewInit() { this.inputEl.nativeElement.focus(); }

  // 5. ngOnDestroy — cleanup subscriptions, timers, event listeners
  ngOnDestroy() { this.subscription?.unsubscribe(); }

  delete() { this.userDeleted.emit(this.userId); }
}

🎯 INTERVIEW QUESTIONS
Q: What is the difference between ngOnInit and constructor?
Constructor is a TypeScript/JavaScript concept — it runs when the class is instantiated. In Angular: use it ONLY for dependency injection. @Input values are NOT set yet, the template is not rendered, ViewChild is not available. ngOnInit is Angular's lifecycle hook — runs after the first change detection cycle. @Input values ARE set, child components are initialized. Use ngOnInit for: initialization logic, first data fetch, setting up subscriptions based on @Input values. Rule: constructor = inject services. ngOnInit = use them.
Q: When would you use ViewChild vs @Input/@Output?
@Input/@Output is declarative — data flows through the template. Preferred for most parent-child communication. ViewChild gives a direct reference to a child component/directive/element instance — use for: calling child component methods directly (child.focus(), child.reset()), accessing DOM elements (nativeElement.focus()), integrating with third-party libraries that need imperative APIs. Avoid overusing ViewChild — it creates tight coupling. If a parent needs to frequently call child methods, consider whether the logic should be in a shared service instead.

🎯 Directives & Pipes
Structural and attribute directives, custom directives, pure vs impure pipes, and the async pipe.
Custom Directive & Pipe
ANGULAR — DIRECTIVE + PIPE
import { Directive, HostListener, HostBinding, ElementRef, Input } from '@angular/core';
import { Pipe, PipeTransform } from '@angular/core';

// Attribute directive — modifies element behavior/appearance
@Directive({ selector: '[appHighlight]', standalone: true })
export class HighlightDirective {
  @Input('appHighlight') color: string = 'yellow';

  @HostBinding('style.backgroundColor') bgColor = '';

  @HostListener('mouseenter') onEnter() { this.bgColor = this.color; }
  @HostListener('mouseleave') onLeave() { this.bgColor = ''; }
}

// Usage: <p appHighlight="cyan">Hover me</p>

// Structural directive — creates/destroys DOM elements
@Directive({ selector: '[appUnless]', standalone: true })
export class UnlessDirective {
  constructor(private vcr: ViewContainerRef, private tmpl: TemplateRef<any>) {}

  @Input() set appUnless(condition: boolean) {
    if (!condition) { this.vcr.createEmbeddedView(this.tmpl); }
    else            { this.vcr.clear(); }
  }
}
// Usage: <div *appUnless="isAdmin">Not an admin</div>

// Pure pipe — called only when input reference changes (fast!)
@Pipe({ name: 'truncate', pure: true, standalone: true })
export class TruncatePipe implements PipeTransform {
  transform(value: string, limit = 100): string {
    return value.length > limit ? value.slice(0, limit) + '...' : value;
  }
}
// Usage: {{ description | truncate:50 }}
🎯 INTERVIEW QUESTIONS
Q: What is the difference between structural and attribute directives?
Structural directives change the DOM layout — they add, remove, or replace elements. Prefixed with * (syntactic sugar for ng-template). Examples: *ngIf, *ngFor, *ngSwitch. Attribute directives change the appearance or behavior of an existing element without touching the DOM structure. Examples: [ngClass], [ngStyle], custom highlight directive. One element can have multiple attribute directives but only one structural directive (because structural directives change the view container).
Q: What is the difference between pure and impure pipes?
Pure pipes (default) are called only when the input reference changes — like a memoized function. Angular won't re-execute a pure pipe if the same reference is passed (even if the object's contents changed). Very performant. Impure pipes (pure: false) are called on every change detection cycle regardless — expensive. Use impure pipes for: arrays/objects where contents change without reference change (async pipe is impure — it subscribes to Observable). Rule: always start with pure. If you need to detect internal mutations, use impure — but prefer immutable data patterns to avoid needing it.

💉 Dependency Injection
Angular's hierarchical injector system. One of the most tested topics in Angular interviews.
Injector Hierarchy & Providers
ANGULAR — DEPENDENCY INJECTION
import { Injectable, InjectionToken, Inject, Optional, SkipSelf } from '@angular/core';

// providedIn: 'root' — singleton, tree-shakeable (preferred)
@Injectable({ providedIn: 'root' })
export class AuthService {
  isLoggedIn$ = new BehaviorSubject<boolean>(false);
}

// InjectionToken — for non-class values
export const API_URL = new InjectionToken<string>('API_URL');

// Providing different types
const providers = [
  // useClass — swap implementation
  { provide: LoggerService, useClass: ConsoleLoggerService },
  // useValue — constant value
  { provide: API_URL, useValue: 'https://api.example.com' },
  // useFactory — dynamic creation
  { provide: HttpClient, useFactory: () => new HttpClient(...), deps: [HttpHandler] },
  // useExisting — alias one token to another
  { provide: NewLogger, useExisting: OldLogger },
  // multi provider — array of implementations
  { provide: HTTP_INTERCEPTORS, useClass: AuthInterceptor, multi: true },
  { provide: HTTP_INTERCEPTORS, useClass: LoggingInterceptor, multi: true },
];

// Inject in constructor (classic) or inject() function (Angular 14+)
@Injectable()
export class UserService {
  // Modern functional injection (usable anywhere)
  private http = inject(HttpClient);
  private apiUrl = inject(API_URL);

  // Or constructor injection
  constructor(
    @Inject(API_URL) private url: string,
    @Optional() private logger: LoggerService,
  ) {}
}

🎯 INTERVIEW QUESTIONS
Q: What is the difference between providedIn: 'root' and providing in a module?
providedIn: 'root' makes the service a singleton at the root injector level. The service is tree-shaken if never injected (not the case with NgModule providers). It's the modern preferred approach — clean, no circular dependency risk. Providing in an NgModule's providers array: registers the service in that module's injector. For lazy-loaded modules, this creates a child injector — the service is a new instance per lazy module, not a true app-wide singleton. Providing in a component's providers array: new instance per component instance — destroyed when the component is destroyed. Great for stateful services scoped to a component tree.
Q: What is the inject() function and when would you use it over constructor injection?
inject() is a functional API (Angular 14+) that retrieves a dependency from the current injector. Works in: constructors, field initializers, factory providers, and injection context functions. Benefits: usable in standalone functions (not just classes), cleaner with functional patterns, works in class fields (avoids constructor parameter boilerplate). Required for: composable functions (like Angular signals, guards as functions), injection outside constructors. Constructor injection remains valid — inject() is an alternative, not a replacement. Both work in components and services.

⚙️ Services & HttpClient
HTTP interceptors, error handling, retry strategies, and building robust data services.
HttpClient & Interceptors
ANGULAR — HTTP + INTERCEPTOR
import { Injectable, inject } from '@angular/core';
import { HttpClient, HttpInterceptorFn, HttpRequest, HttpHandlerFn } from '@angular/common/http';
import { Observable, throwError } from 'rxjs';
import { catchError, retry, timeout } from 'rxjs/operators';

// Functional interceptor (Angular 15+ — preferred)
export const authInterceptor: HttpInterceptorFn = (
  req: HttpRequest<unknown>,
  next: HttpHandlerFn
) => {
  const authService = inject(AuthService);
  const token = authService.getToken();

  if (!token) return next(req);

  const authReq = req.clone({
    headers: req.headers.set('Authorization', `Bearer ${token}`)
  });
  return next(authReq);
};

// Data service with error handling + retry
@Injectable({ providedIn: 'root' })
export class UserService {
  private http = inject(HttpClient);
  private apiUrl = 'https://api.example.com';

  getUsers(): Observable<User[]> {
    return this.http.get<User[]>(`${this.apiUrl}/users`).pipe(
      timeout(10000),          // error if no response in 10s
      retry({ count: 3, delay: 1000 }),  // retry 3x with 1s delay
      catchError(err => {
        // transform error for UI consumption
        return throwError(() => new Error(this.getErrorMessage(err)));
      })
    );
  }

  private getErrorMessage(err: any): string {
    if (err.status === 404) return 'Resource not found';
    if (err.status === 401) return 'Unauthorized';
    return err.message ?? 'Unknown error';
  }
}

// Register functional interceptors (standalone)
bootstrapApplication(AppComponent, {
  providers: [provideHttpClient(withInterceptors([authInterceptor]))]
});
🎯 INTERVIEW QUESTIONS
Q: What are HTTP interceptors and what are they used for?
Interceptors are middleware for HTTP requests and responses. They intercept every HttpClient request/response in the application. Common uses: (1) Auth token attachment — add Authorization header to all requests, (2) Error handling — globally catch 401/403 and redirect to login, (3) Loading indicator — show spinner on request, hide on response, (4) Request logging — log all API calls for debugging, (5) Caching — cache GET responses, (6) Response transformation — normalize API response format. Angular 15+ has functional interceptors (HttpInterceptorFn) which are simpler than the class-based approach and work with inject().

🛣️ Routing & Guards
Angular Router, lazy loading, route guards, resolve, and functional guard patterns.
Routes & Guards
ANGULAR — ROUTING + GUARDS
import { Routes } from '@angular/router';
import { inject } from '@angular/core';
import { AuthService } from './auth.service';

// Functional guard (Angular 14+ — preferred over class-based)
export const authGuard = () => {
  const auth = inject(AuthService);
  const router = inject(Router);
  return auth.isLoggedIn() || router.createUrlTree(['/login']);
  // return true = proceed, UrlTree = redirect, false = block
};

// Functional resolve guard
export const userResolver = (route: ActivatedRouteSnapshot) => {
  return inject(UserService).getUser(route.params['id']);
};

export const routes: Routes = [
  { path: '', redirectTo: '/home', pathMatch: 'full' },
  { path: 'home', component: HomeComponent },
  { path: 'login', component: LoginComponent },

  // Protected routes with canActivate
  {
    path: 'dashboard',
    canActivate: [authGuard],
    component: DashboardComponent,
  },

  // Lazy-loaded feature with resolve
  {
    path: 'users/:id',
    canActivate: [authGuard],
    resolve: { user: userResolver },  // data available before component loads
    loadComponent: () => import('./user-detail.component')
      .then(c => c.UserDetailComponent),
  },

  // Nested routes with children
  {
    path: 'admin',
    canActivate: [authGuard],
    loadChildren: () => import('./admin/admin.routes')

🎯 INTERVIEW QUESTIONS
Q: What are the different types of route guards in Angular?
canActivate: controls whether a route can be activated — returns true/false/UrlTree. canActivateChild: same but for child routes. canDeactivate: controls whether you can leave a route (warn about unsaved changes). resolve: pre-fetches data before the component loads — the component receives ready data via ActivatedRoute. canLoad/canMatch: controls whether a lazy-loaded module should be downloaded at all. Angular 14+ replaced class-based guards with functional guards (simpler, use inject(), more composable). canActivate + UrlTree is the modern way to redirect instead of calling router.navigate() in the guard.

Q: Why are reactive forms preferred over template-driven forms?
Reactive forms are defined in the component class — they're synchronous, explicit, and testable without a DOM. Validators are pure functions (easy to unit test). Essential RxJS Operators
ANGULAR — RXJS PATTERNS
import { Component, inject, DestroyRef } from '@angular/core';
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';
import { switchMap, mergeMap, concatMap, exhaustMap,
         combineLatest, debounceTime, distinctUntilChanged } from 'rxjs';

// switchMap — cancel previous, use latest (search, navigation)
searchTerm$.pipe(
  debounceTime(300),
  distinctUntilChanged(),
  switchMap(term => this.http.get(`/search?q=${term}`))
).subscribe(results => this.results = results);

// mergeMap — concurrent requests, all complete (parallel uploads)
files$.pipe(mergeMap(file => this.upload(file))).subscribe();

// concatMap — sequential, wait for each to complete (ordered actions)
actions$.pipe(concatMap(action => this.saveAction(action))).subscribe();

// exhaustMap — ignore new until current completes (form submit, prevent double)
submitBtn$.pipe(exhaustMap(() => this.form.submit())).subscribe();

// combineLatest — emit when ANY source changes (filters + data)
combineLatest([users$, filter$]).pipe(
  map(([users, filter]) => users.filter(u => u.role === filter))
).subscribe();

// Subject types
const subject    = new Subject<number>();            // no initial value, no replay
const behavior   = new BehaviorSubject<number>(0);    // initial value, replay last
const replay     = new ReplaySubject<number>(5);       // replay last 5 values
const asyncSubj  = new AsyncSubject<number>();          // emit only on complete

// Modern: takeUntilDestroyed — replaces manual unsubscribe!
@Component({...})
export class MyComponent {
  users$ = inject(UserService).getUsers().pipe(
    takeUntilDestroyed()   // auto-unsubscribes when component destroys
  );
}
🎯 INTERVIEW QUESTIONS
Q: What is the difference between switchMap, mergeMap, concatMap, and exhaustMap?
All four map each value to an inner Observable, but differ in how they handle concurrent inner observables. switchMap: cancels the previous inner observable when a new outer value arrives — use for search (cancel old request), navigation. mergeMap: subscribes to all inner observables concurrently — use for parallel operations (multiple uploads). concatMap: queues inner observables, waits for each to complete before starting the next — use for ordered sequential operations. exhaustMap: ignores new outer values while an inner observable is active — use for form submission (prevent double-submit). Memory rule: switch = search, merge = parallel, concat = sequential, exhaust = prevent duplicate.
Q: What is the difference between Subject, BehaviorSubject, and ReplaySubject?
Subject: plain multicast — subscribers only get values emitted AFTER they subscribe. No initial value. BehaviorSubject: requires an initial value. New subscribers immediately get the LAST emitted value (current state). Use for state (user, theme, auth status). ReplaySubject(n): replays the last N values to new subscribers. AsyncSubject: only emits the last value when the subject completes — rarely used. Rule: BehaviorSubject for state that has a current value (most Angular services). Subject for events/actions with no "current" concept. ReplaySubject when late subscribers need history (cache).

🔍 Change Detection
How Angular knows when to update the DOM. Zone.js, OnPush, and the key to Angular performance.
Default vs OnPush
💡 ANALOGY
Default CD is like a CCTV system that checks every room every second — thorough but expensive.

OnPush CD is like a motion sensor — it only checks a room when: a new object enters (new @Input reference), a button is pressed inside (DOM event), or someone explicitly rings the bell (markForCheck / async pipe). If nothing changed, the room isn't checked. Massive performance improvement for large trees.
ANGULAR — CHANGE DETECTION
import { Component, ChangeDetectionStrategy, ChangeDetectorRef,
         inject, Input } from '@angular/core';

// OnPush — only checks when:
//   1. @Input reference changes (not mutation!)
//   2. DOM event inside this component
//   3. async pipe emits
//   4. markForCheck() / detectChanges() called
@Component({
  selector: 'app-user-card',
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <!-- async pipe: subscribes, triggers CD on emission, unsubscribes -->
    <div *ngIf="user$ | async as user">{{ user.name }}</div>
  `
})
export class UserCardComponent {
  @Input() userId!: number;
  private cdr = inject(ChangeDetectorRef);

  user$ = inject(UserService).getUser(this.userId);

  // ❌ WRONG with OnPush — mutating input doesn't trigger CD
  badUpdate(newName: string) {
    this.user.name = newName;  // same reference — OnPush won't detect
  }

  // ✅ CORRECT — new reference triggers OnPush
  goodUpdate(newName: string) {
    this.user = { ...this.user, name: newName };  // new object
  }

  // Manual trigger when needed (e.g., from non-Angular callback)
  fromOutside() {
    this.cdr.markForCheck();   // schedule check in next cycle
    // or this.cdr.detectChanges(); // immediately check this subtree
  }
}
🎯 INTERVIEW QUESTIONS
Q: How does Angular's change detection work?
Zone.js patches browser async APIs (setTimeout, Promises, DOM events). When any async operation completes, Zone.js notifies Angular, which triggers a change detection cycle. Starting from the root, Angular walks down the component tree, checking each component's template bindings. With Default strategy: every component is checked every cycle. With OnPush: a component subtree is skipped unless its inputs changed by reference, it received a DOM event, or an async pipe emitted. Signals (Angular 17+) are even more granular — only the specific template expressions that depend on a changed signal are re-evaluated.
Q: What is the difference between markForCheck and detectChanges?
markForCheck(): marks this component and all its ancestors as dirty — they'll be checked in the next change detection cycle (scheduled). Doesn't run CD immediately. Use when updating from outside Angular (WebSocket callbacks, third-party lib). detectChanges(): immediately runs change detection on this component and its descendants synchronously. Use when you need the DOM updated right now. Also works outside of Zone.js. Rule: markForCheck for normal async updates (let Angular schedule it), detectChanges for imperative updates that need immediate DOM reflection.

ANGULAR — SIGNALS
import { Component, signal, computed, effect, inject } from '@angular/core';
import { toSignal, toObservable } from '@angular/core/rxjs-interop';

@Component({
  selector: 'app-counter',
  standalone: true,
  template: `
    <p>Count: {{ count() }}</p>          <!-- read signal -->
    <p>Double: {{ double() }}</p>        <!-- computed auto-updates -->
    <button (click)="increment()">+</button>
  `
})
export class CounterComponent {
  // signal() — writable reactive primitive
  count = signal(0);

  // computed() — derived signal, re-evaluates only when dependencies change
  double = computed(() => this.count() * 2);

  // effect() — side effects when signals change
  logEffect = effect(() => {
    console.log('Count changed:', this.count());
  });

  increment() {
    this.count.update(n => n + 1);  // update based on previous
    // or: this.count.set(5);         // set absolute value
  }

  // Bridge with RxJS
  users = toSignal(inject(UserService).getUsers(), { initialValue: [] });
  count$ = toObservable(this.count); // signal → Observable
}

// trackBy — prevent full list re-render
// <li *ngFor="let user of users; trackBy: trackById">
trackById(index: number, user: User) { return user.id; }
🎯 INTERVIEW QUESTIONS
Q: What are Angular Signals and how are they different from RxJS Observables?
Signals are synchronous reactive primitives — they hold a value and notify consumers when that value changes. Unlike Observables, they're always synchronous, always hold a current value (no need for initial values or subscriptions), and use fine-grained reactivity (only components that read a signal are re-rendered). Observables are lazy, async by default, handle streams of events over time, have powerful operators, and need explicit subscription management. Signals are better for component state; Observables for async streams, HTTP, and time-based operations. Angular 16+ allows interop: toSignal() converts Observable → Signal, toObservable() does the reverse.
Q: Why use trackBy with ngFor?
Without trackBy, when the array reference changes, Angular destroys and recreates ALL DOM elements even if most items are unchanged. This causes: unnecessary DOM operations (expensive), loss of component state, janky animations. With trackBy: Angular identifies each item by the returned key (usually id). Only items whose key is new get created; items that moved are reused (just repositioned); items no longer in the array are destroyed. For a list of 100 items where only one changed: trackBy = 1 DOM operation vs no trackBy = 100 DOM operations. Always use trackBy with ngFor for dynamic lists.

Q: What is fakeAsync and when do you use it?
fakeAsync() gives you synchronous control over asynchronous code in tests. 🎯 INTERVIEW QUESTIONS
Q: What are standalone components and why are they the new default?
Standalone components (Angular 14, default in 17) eliminate the need for NgModules. Each component declares its own imports directly. Benefits: less boilerplate (no module file), better tree-shaking (components only bundle what they import), simpler lazy loading (loadComponent directly), easier testing (import component directly without module setup), clearer dependency graph (you can see exactly what a component needs). New Angular projects should use standalone by default. NgModules still work and are needed for migrating large existing apps gradually.
Q: What are deferrable views and when should you use them?
@defer is an Angular 17 template syntax that splits heavy components into separate lazy-loaded chunks, loaded based on triggers: on viewport (loads when scrolled into view), on interaction (loads on user click/hover), on idle (loads when browser is idle), on timer(2000) (loads after 2 seconds), when (condition) (loads when expression is truthy). @placeholder shows while not yet loading. @loading shows during download. @error shows on failure. Use for: complex charts below the fold, rich text editors, third-party heavy widgets, any component not needed on initial render. No route change needed — it's inline lazy loading.
