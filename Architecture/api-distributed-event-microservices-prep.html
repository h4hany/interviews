🔗 REST Design Principles
The 6 REST constraints, correct HTTP method usage, status codes, and what makes an API truly RESTful vs just "HTTP-based".
The 6 REST Constraints
Constraint What it means Common violation
Stateless Each request must contain all info needed. Server stores no session. Storing user state in server-side session
Client-Server UI and data storage are separated. Evolve independently. Server-rendered HTML mixed with API
Cacheable Responses must declare if they're cacheable (Cache-Control, ETag). No cache headers on GET responses
Uniform Interface Resource identification, manipulation via representations, self-descriptive messages, HATEOAS. POST /getUser — verbs in URLs
Layered System Client can't tell if connected directly to server or via proxy/LB. Leaking internal topology to client
Code on Demand Optional: server can send executable code (e.g. JavaScript). Rarely used in practice
HTTP Methods — The Right Way
REST — HTTP METHODS & STATUS CODES 
// Resource: /users
// CRUD mapped to HTTP methods

GET    /users          → 200 OK  (list, paginated)
GET    /users/42       → 200 OK | 404 Not Found
POST   /users          → 201 Created  (body = new resource, Location header)
PUT    /users/42       → 200 OK | 204 No Content (full replace — idempotent)
PATCH  /users/42       → 200 OK  (partial update — NOT always idempotent)
DELETE /users/42       → 204 No Content | 404 Not Found (idempotent)

// Nested resources
GET    /users/42/posts        → posts belonging to user 42
POST   /users/42/posts        → create post for user 42
GET    /users/42/posts/7     → specific post — only nest 2 levels max!

// ❌ WRONG — verbs in URLs (RPC style, not REST)
POST /getUser/42
POST /deleteUser
GET  /fetchActiveUsers

// ✅ CORRECT — nouns, actions via HTTP method
GET    /users/42
DELETE /users/42
GET    /users?status=active

// Key status codes to know cold:
// 200 OK — general success
// 201 Created — resource was created (include Location header)
// 204 No Content — success, no body (DELETE, PUT with no return)
// 400 Bad Request — client sent invalid data
// 401 Unauthorized — not authenticated
// 403 Forbidden — authenticated but not authorized
// 404 Not Found — resource doesn't exist
// 409 Conflict — version conflict, duplicate resource
// 422 Unprocessable Entity — valid JSON but semantically wrong
// 429 Too Many Requests — rate limited
// 500 Internal Server Error — server crashed
// 503 Service Unavailable — overloaded or down
Idempotency — Critical Concept
💡 ANALOGY
Idempotency is like a light switch. If the light is already on and you press ON again, nothing changes — same result. DELETE /users/42 is idempotent: delete it 5 times, it's gone after the first. POST /orders is NOT idempotent: call it 5 times, you get 5 orders. This distinction matters enormously for retry logic in distributed systems — only idempotent operations are safe to retry automatically.
Method Safe? Idempotent? Safe to retry?
GET ✅ Yes ✅ Yes ✅ Yes
PUT ❌ No ✅ Yes ✅ Yes
DELETE ❌ No ✅ Yes ✅ Yes
POST ❌ No ❌ No ❌ Risky
PATCH ❌ No ⚠️ Depends ⚠️ Depends
🎯 INTERVIEW QUESTIONS
Q: What is the difference between PUT and PATCH?
PUT replaces the entire resource — you must send the complete representation. If you omit fields, they become null/default. PATCH applies a partial update — only the fields you send are modified. PUT is idempotent by definition (same full replacement = same result). PATCH is only idempotent if designed carefully (e.g. set field to absolute value vs increment). In practice: use PATCH for partial updates, PUT for full replacement. For user profile updates, PATCH is almost always correct — you don't want to wipe fields the client didn't include.
Q: What is HATEOAS and is it used in practice?
HATEOAS (Hypermedia As The Engine Of Application State) means responses include links to related actions/resources, so clients discover what they can do next without hardcoding URLs. Example: GET /orders/42 response includes links: {"cancel": "/orders/42/cancel", "items": "/orders/42/items"}. In theory: clients are fully discoverable. In practice: almost no public APIs implement true HATEOAS. It adds complexity without much benefit when clients are controlled by the same team. It's worth knowing for interviews but rarely implemented. The GitHub API and PayPal API are notable exceptions that do use it.
Q: When should you use 401 vs 403?
401 Unauthorized means the request lacks valid authentication credentials — the server doesn't know who you are. The client should authenticate (login, add a token). 403 Forbidden means the server knows who you are (authenticated) but you don't have permission to access this resource. Mnemonic: 401 = "Who are you?", 403 = "I know who you are and the answer is no." Common mistake: returning 401 when the user IS authenticated but lacks permission — that should be 403. Also: never return 403 for resources that don't exist for security reasons — use 404 to avoid leaking information about resource existence.
// QUIZ
A client calls DELETE /users/42 three times. The first call succeeds. What should the second and third calls return?

⚙️ REST Advanced Patterns
Pagination, filtering, caching with ETags, rate limiting, and bulk operations — the patterns that separate junior API design from production-grade APIs.
Pagination Strategies
REST — PAGINATION 
// 1. OFFSET pagination — simple but has problems at scale
GET /users?page=3&limit=20
GET /users?offset=40&limit=20
// Problem: if records inserted/deleted between pages → duplicates/skips

// 2. CURSOR pagination — stable, efficient (recommended for large sets)
GET /users?limit=20                    // first page
GET /users?cursor=eyJ1c2VyX2lkIjo&limit=20  // next page

// Response includes cursor for next page:
{
  "data": [...],
  "pagination": {
    "next_cursor": "eyJ1c2VyX2lkIjo0MH0=",  // base64({"user_id":40})
    "has_more": true,
    "total": 1240
  }
}
// Cursor = opaque token encoding the position (ID, timestamp, composite key)
// ✅ Stable under inserts/deletes  ✅ O(1) DB query with index

// 3. KEYSET (seek method) — fastest for ordered large sets
GET /posts?after_id=12345&limit=20
// SQL: WHERE id > 12345 ORDER BY id LIMIT 20  → uses index seek, not scan

// Filtering and sorting
GET /users?status=active&role=admin&sort=-created_at&fields=id,name,email
// - prefix = descending  |  fields = sparse fieldsets (avoid over-fetching)
ETag Caching & Rate Limiting
REST — CACHING & RATE LIMITING 
// ETag — conditional GET, avoid unnecessary data transfer
// Server: sends ETag (hash of resource content)
HTTP/1.1 200 OK
ETag: "a3f4b2c1d"
Cache-Control: max-age=60, must-revalidate

// Client: sends If-None-Match on next request
GET /users/42
If-None-Match: "a3f4b2c1d"

// Server: 304 if unchanged (no body!), 200 + new ETag if changed
HTTP/1.1 304 Not Modified   ← zero bandwidth, just a small header

// Optimistic concurrency control with ETag
PUT /users/42
If-Match: "a3f4b2c1d"      ← only update if still this version
// Server returns 412 Precondition Failed if version changed → conflict!

// Rate limiting response headers (RFC 6585 / GitHub style)
X-RateLimit-Limit: 1000      // max requests per window
X-RateLimit-Remaining: 42   // remaining in current window
X-RateLimit-Reset: 1714320000 // Unix timestamp when window resets
Retry-After: 60              // seconds to wait (on 429)

// Idempotency key — safe POST retries (Stripe pattern)
POST /payments
Idempotency-Key: "order-xyz-attempt-1"   // unique per attempt
// Server: store result keyed by idempotency-key
// Retry with same key → same result, not a duplicate charge!
🎯 INTERVIEW QUESTIONS
Q: What are the trade-offs between offset and cursor pagination?
Offset: simple to implement and understand. Allows jumping to any page. Works well for small datasets. Problems: inconsistent results when data changes between pages (new record on page 1 pushes everything — page 2 now has a duplicate); DB must scan all offset rows (OFFSET 10000 LIMIT 20 reads 10020 rows, discards 10000 = slow). Cursor: stable regardless of inserts/deletes between pages. Efficient — uses index seeks, not scans. Problems: can't jump to arbitrary page ("show page 47"), harder to implement, cursor must encode the sort field. Rule: use cursor pagination for large production datasets, infinite scroll, feeds. Use offset only for small admin UIs where jumping to specific pages matters.
Q: What is an idempotency key and when do you need it?
An idempotency key is a unique client-generated token sent with a POST request. The server stores the result keyed by this token. If the same request is retried (due to network timeout, client retry logic), the server detects the key was already processed and returns the stored result instead of executing the operation again. Critical for: payment processing (don't charge twice), order creation (don't duplicate orders), any state-changing operation that must not repeat. Implementation: store key + result in Redis with TTL. Check before processing. Return 200 (from cache) for duplicate keys. Stripe, Adyen, and most payment APIs require this.

🔺 GraphQL
Schema, resolvers, the N+1 problem, DataLoader, and when GraphQL is the right choice vs REST.
GraphQL Core Concepts
GRAPHQL — SCHEMA + RESOLVER 
# Schema definition language (SDL)
type User {
  id: ID!
  name: String!
  email: String!
  posts: [Post!]!         # relationship — loaded by resolver
}

type Post {
  id: ID!
  title: String!
  author: User!           # back-reference
}

type Query {
  user(id: ID!): User
  users(limit: Int = 20, cursor: String): UserConnection!
}

type Mutation {
  createUser(input: CreateUserInput!): User!
  updateUser(id: ID!, input: UpdateUserInput!): User!
}

type Subscription {
  messageAdded(roomId: ID!): Message!  # real-time via WebSocket
}

// ---- N+1 PROBLEM ----
// Query: { users { id name posts { title } } }
// Without DataLoader:
//   1 query for 100 users
//   100 queries for each user's posts = 101 queries! ← N+1

// ✅ FIX: DataLoader — batches + caches resolver calls
const postLoader = new DataLoader(async (userIds) => {
  // Called ONCE with all 100 userIds batched together
  const posts = await db.query(
    'SELECT * FROM posts WHERE user_id = ANY($1)', [userIds]
  );
  // Must return array in SAME ORDER as userIds
  return userIds.map(id => posts.filter(p => p.user_id === id));
});
// Result: 2 queries total regardless of N — 1 for users, 1 for all posts
REST vs GraphQL — When to Use What
Dimension REST GraphQL
Over/under-fetching Fixed response shape — often over-fetch Client specifies exact fields needed
Caching Excellent — HTTP caching (GET + URL) Harder — POST requests, need custom cache
Versioning URL versioning (/v2/) Evolve schema without versioning (deprecate fields)
Multiple clients Need separate endpoints or BFF layer One endpoint, each client queries what it needs
Complexity Simple, widely understood N+1, query depth attacks, schema complexity
Use when Public API, simple CRUD, caching critical Many clients with different needs, complex graphs
🎯 INTERVIEW QUESTIONS
Q: What is the N+1 problem in GraphQL and how do you solve it?
N+1 occurs when resolving a list of N objects that each trigger 1 additional query for their related data. Example: fetching 100 users then resolving each user's posts = 1 + 100 = 101 queries. GraphQL makes this easy to trigger accidentally because the nested resolution is invisible. Solution: DataLoader — batches multiple individual resolver calls into a single batch operation. DataLoader waits until the current event loop tick ends, collects all pending load calls, calls the batch function once with all keys, then resolves each individual promise. It also caches results within a request (same ID requested twice = only one DB call). Always implement DataLoader for any resolver that traverses a relationship.

⚡ gRPC & Protocol Buffers
Binary RPC protocol built on HTTP/2. Faster and more efficient than REST for internal service communication.
gRPC vs REST
PROTOBUF — SERVICE DEFINITION 
// .proto file — defines the service contract
syntax = "proto3";

message User {
  int32 id = 1;
  string name = 2;
  string email = 3;
  repeated string roles = 4;  // repeated = array
}

message GetUserRequest  { int32 id = 1; }
message CreateUserRequest { string name = 1; string email = 2; }

service UserService {
  // Unary RPC — classic request/response
  rpc GetUser (GetUserRequest) returns (User);

  // Server streaming — server sends stream of responses
  rpc ListUsers (ListRequest) returns (stream User);

  // Client streaming — client sends stream of requests
  rpc BulkCreate (stream CreateUserRequest) returns (BulkResult);

  // Bidirectional streaming — both sides stream
  rpc Chat (stream Message) returns (stream Message);
}

// WHY gRPC BEATS REST FOR INTERNAL SERVICES:
// 1. Binary Protobuf encoding: ~5-10x smaller payload than JSON
// 2. HTTP/2: multiplexing, header compression, persistent connection
// 3. Strongly typed: .proto = contract, auto-generated client code
// 4. Streaming: native bidirectional, no polling
// 5. Code generation: stubs in Go, Python, Java, TypeScript, etc.

// WHEN TO USE REST INSTEAD:
// - Public API (browser + curl friendly)
// - Caching required (HTTP cache headers)
// - Human-readable debugging matters
// - Simple CRUD with few clients
🎯 INTERVIEW QUESTIONS
Q: Why would you choose gRPC over REST for microservices communication?
gRPC advantages for internal service-to-service: (1) Performance — Protobuf binary encoding is 5–10x smaller than JSON, HTTP/2 multiplexes multiple RPCs over one TCP connection, no repeated header data. (2) Strong typing — the .proto file is the contract, client code is auto-generated, breaking changes are caught at compile time. (3) Streaming — native bidirectional streaming without polling or websocket complexity. (4) Deadline propagation — gRPC propagates timeouts across the call chain automatically. Trade-offs: not browser-friendly without gRPC-Web proxy, harder to debug (binary), less tooling than REST, learning curve. Use gRPC for: high-throughput internal services, streaming data, polyglot environments where strong typing across languages matters.

OAuth 2.0 Flows
OAUTH 2.0 — FLOWS 
// Authorization Code Flow (+ PKCE for SPAs/mobile)
// Most secure — use for apps that can store secrets or with PKCE

1. User → App: "Login with Google"
2. App → AuthServer: redirect with client_id, scope, code_challenge (PKCE)
3. User → AuthServer: login + consent
4. AuthServer → App: redirect with ?code=xyz
5. App → AuthServer: POST /token { code, code_verifier }  ← PKCE verify
6. AuthServer → App: { access_token, refresh_token, id_token }
7. App → Resource: GET /api with Authorization: Bearer {access_token}

// JWT Structure
// Header.Payload.Signature  (base64url encoded, dot-separated)

// Header:
{ "alg": "RS256", "typ": "JWT" }

// Payload (claims):
{
  "sub": "user-42",          // subject — who the token is about
  "iss": "auth.example.com",  // issuer
  "aud": "api.example.com",   // audience — intended recipient
  "exp": 1714320000,          // expiry (Unix timestamp)
  "iat": 1714316400,          // issued at
  "roles": ["admin", "user"]  // custom claims
}

// Signature: HMAC-SHA256(base64(header) + "." + base64(payload), secret)
// Use RS256 (asymmetric) in production — public key to verify, private to sign

// JWT pitfalls:
// ❌ Never put secrets (passwords, SSNs) in JWT payload — it's just base64!
// ❌ alg:none attack — always validate the algorithm header
// ❌ No revocation — use short expiry (15 min) + refresh tokens
// ✅ Keep access tokens short-lived (15 min)
// ✅ Store refresh tokens in httpOnly cookies (XSS safe)
OWASP API Top 10 (2023)
# Vulnerability Example / Fix
API1 Broken Object Level Auth /orders/42 — check it belongs to current user, not just that you're logged in
API2 Broken Auth Weak tokens, no rate limit on login, password reset flaws
API3 Broken Object Property Auth Returning/accepting admin fields users shouldn't see/set. Use response_model to filter.
API4 Unrestricted Resource Consumption No rate limit, no pagination limit. Single request returns 1M rows.
API5 Broken Function Level Auth Admin endpoints accessible to regular users. Check role on every route.
API8 Security Misconfiguration Debug endpoints in production, default credentials, overly permissive CORS (*)
🎯 INTERVIEW QUESTIONS
Q: What is the difference between authentication and authorization?
Authentication answers "who are you?" — verifying identity (login, token validation). Authorization answers "what can you do?" — verifying permissions. JWT authentication: server validates token signature and expiry (who are you). Authorization: server checks the user's role/scope against the required permission (are you allowed to do this?). Both are needed. A common mistake: checking only that a user is logged in (authenticated) without checking they own the resource they're accessing (authorized). This is OWASP API1 — Broken Object Level Authorization.
Q: Why use refresh tokens instead of long-lived access tokens?
Access tokens can't be revoked once issued (JWT is stateless — server doesn't track them). If a short-lived token (15 min) is stolen, it expires quickly. A long-lived token (30 days) stolen gives attacker 30 days of access. Solution: short-lived access token (15 min) + long-lived refresh token (days/weeks). Refresh token is stored server-side (can be revoked), access token is stateless (can't be revoked, but expires fast). When logout happens: delete the refresh token server-side. Attacker's stolen access token expires in ≤15 min and can't be renewed. Store refresh tokens in httpOnly, Secure cookies — not localStorage (XSS-vulnerable).

🔢 API Versioning & Evolution
Breaking vs non-breaking changes, versioning strategies, and how to evolve an API without breaking clients.
Versioning Strategies
REST — VERSIONING APPROACHES 
// 1. URI versioning (most common, most visible)
GET /v1/users
GET /v2/users
// ✅ Easy to test (curl/browser), explicit, cacheable
// ❌ "Unclean" URLs, version in URL violates REST theory

// 2. Header versioning
GET /users
Accept: application/vnd.myapi.v2+json
// or
GET /users
API-Version: 2024-01-15    ← Stripe/GitHub style (date-based)
// ✅ Clean URLs, header is metadata
// ❌ Not cacheable by default, less discoverable, harder to test

// 3. Query parameter versioning
GET /users?version=2
// ❌ Messy, version is not really a resource filter

// Breaking vs non-breaking changes

// ✅ NON-BREAKING (backward compatible):
//   - Adding new optional fields to response
//   - Adding new optional request parameters
//   - Adding new endpoints
//   - Adding new enum values (careful — clients may not handle unknown)

// ❌ BREAKING (requires version bump):
//   - Removing or renaming fields
//   - Changing field type (string → number)
//   - Changing HTTP method for existing endpoint
//   - Changing URL structure
//   - Making optional fields required
//   - Changing authentication mechanism

// Sunset deprecation headers (RFC 8594)
HTTP/1.1 200 OK
Deprecation: true
Sunset: Sat, 31 Dec 2024 23:59:59 GMT
Link: <https://api.example.com/v2/users>; rel="successor-version"
🎯 INTERVIEW QUESTIONS
Q: How would you deprecate a V1 API endpoint when V2 exists?
(1) Announce deprecation with timeline well in advance (3–6 months minimum). (2) Add Deprecation and Sunset response headers to all V1 responses. (3) Add logging/monitoring to track which clients are still hitting V1. (4) Reach out to teams/consumers who are still using V1 before the deadline. (5) On sunset date, return 410 Gone instead of removing silently — so clients get a clear error. (6) Keep the documentation up for 6–12 months after shutdown. Never just delete an endpoint without 410 — it looks like an outage. For internal APIs, use your internal communication channels. For public APIs, use a developer changelog and email.

📐 CAP & PACELC Theorems
The foundational trade-off in distributed systems. Every database choice reflects a CAP position.
CAP Theorem Explained
💡 ANALOGY
CAP is like a three-legged stool where you can only use two legs. In a distributed system with a network partition (which WILL happen), you must choose:

Consistency (C): Every read returns the most recent write — or an error. All nodes see the same data at the same time.
Availability (A): Every request gets a response — no errors or timeouts. Might not be the latest data.
Partition Tolerance (P): System continues to operate even when network drops messages between nodes.

Since partitions are a reality of networks, you always have P. So the real choice is: CP (consistent, may be unavailable during partition) or AP (available, may return stale data during partition).
System CAP Choice Behavior during partition
HBase, ZooKeeper, etcd CP Refuses requests to maintain consistency. Returns errors.
Cassandra, DynamoDB, CouchDB AP Stays available, may return stale data. Syncs later.
MySQL (single node) CA (no partition) N/A — single node, no network partition between nodes
MongoDB (default), Redis CP (configurable) Primary unavailable → rejects writes until new leader elected
PACELC — The Better Model
PACELC extends CAP by acknowledging that even without a partition, there's a latency/consistency trade-off. Full form: Partition: Availability vs Consistency; Else: Latency vs Consistency.

Cassandra is PA/EL — during partition stays available, normally trades consistency for low latency (eventual consistency).
MySQL (InnoDB) is PC/EC — during partition stays consistent, normally trades latency for consistency (synchronous writes).

This is why the question isn't just "CP or AP" — it's "how much latency are you willing to trade for consistency in the normal case?"
🎯 INTERVIEW QUESTIONS
Q: Can a system be both consistent and available?
CAP says you can't have all three — C, A, and P — simultaneously. Since network partitions are inevitable, CA systems don't really exist in distributed form. However: a single-node database (MySQL, PostgreSQL) has no partitions between nodes, so it can be both consistent and available. That's why traditional RDBMS is labeled CA in CAP diagrams — but it only works because it's not a distributed system. As soon as you add replicas across network boundaries, you re-enter the CAP trade-off. The real question is how you handle partitions when they occur — block (CP) or serve stale data (AP).

🔄 Consistency Models
Strong, eventual, causal, read-your-writes — the spectrum of consistency and when each model fits.
The Consistency Spectrum
Linearizability
(Strong consistency)
Every operation appears instantaneous. Read always returns latest write. Highest consistency, highest latency. Used by: etcd, ZooKeeper, Google Spanner.
Sequential Consistency
All operations appear in some sequential order, consistent across all nodes. Weaker than linearizability (no real-time guarantee). Theoretical model.
Causal Consistency
Causally related operations are seen in order. "If A caused B, all nodes see A before B." Comment on a post appears after the post. Used by: MongoDB causal sessions.
Read-Your-Writes
After you write, you always read your own write. Others may not see it yet. Common in social apps. Achieved by: sticky sessions, read-after-write routing.
Eventual Consistency
If no new writes, all replicas converge to same value eventually. No timing guarantee. Highest availability, lowest latency. Used by: Cassandra, DynamoDB, DNS.
↑ Stronger consistency, higher latency / Lower availability      Weaker consistency, lower latency / Higher availability ↓
Practical Consistency in Databases
CONSISTENCY — PRACTICAL PATTERNS 
// Cassandra: tunable consistency per operation
// W + R > N = strong consistency (quorum)
// N=3 replicas, W=2, R=2 → W+R=4 > 3 → always overlap → strong

session.execute(
  "SELECT * FROM users WHERE id=?",
  consistency=ConsistencyLevel.QUORUM     // read from majority

🎯 INTERVIEW QUESTIONS
Q: What is eventual consistency and when is it acceptable?
Eventual consistency means: if no new writes occur, all replicas will converge to the same value — eventually. No guarantee on when. Acceptable when: (1) Stale reads have no serious consequence — social media like counts, view counters, product recommendations. (2) The write latency benefit outweighs consistency cost — global apps where users are geographically far from each other. (3) The business logic tolerates conflicts — shopping cart (add items from two devices, merge them). NOT acceptable when: bank balances (incorrect balance could cause double-spending), inventory reservations (oversell), medical records (outdated diagnosis data). The key question: "What is the cost of a stale read?" If the answer involves money, health, or safety — use strong consistency.

🤝 Consensus — Raft & Paxos
How distributed systems agree on a single value. Leader election, log replication, and split-brain prevention.
Raft Consensus Algorithm
💡 ANALOGY
Consensus is like a company vote. Multiple managers (nodes) need to agree on one leader. Each manager has a vote. The leader sends updates to all managers (followers). If the leader disappears, a new election happens. As long as a majority (quorum) agrees, the decision stands. This is how etcd, ZooKeeper, and CockroachDB maintain consistency.
RAFT — KEY CONCEPTS 
// Raft roles
// LEADER: one per cluster. Handles all writes. Sends heartbeats.
// FOLLOWER: accepts logs from leader. Votes in elections.
// CANDIDATE: follower that started an election (timed out waiting for heartbeat)

// Write flow in Raft:
Client → Leader: "SET key=value"
Leader: appends to its own log (uncommitted)
Leader → Followers: AppendEntries RPC (log replication)
Majority ACK → Leader: marks entry as COMMITTED
Leader → Client: success
Leader → Followers: next heartbeat notifies commit

// Quorum rule: cluster of N nodes tolerates (N-1)/2 failures
// 3 nodes → quorum=2 → tolerates 1 failure
// 5 nodes → quorum=3 → tolerates 2 failures
// 7 nodes → quorum=4 → tolerates 3 failures

// Split-brain: network partition creates two groups, each thinks it's the leader
// Raft prevention: a leader can only commit if it hears from a majority
// Minority partition → leader steps down (can't commit) → safe

// Real systems using Raft:
// etcd (Kubernetes state store)
// CockroachDB (distributed SQL)
// TiKV (distributed key-value, TiDB)
// Consul (service discovery)

// Paxos vs Raft:
// Both solve the same problem
// Paxos: original algorithm, mathematically elegant, notoriously hard to implement
// Raft: designed for understandability, same guarantees, much easier to implement correctly
// Google Chubby uses Paxos. etcd uses Raft. New systems choose Raft.
🎯 INTERVIEW QUESTIONS
Q: What is split-brain and how does Raft prevent it?
Split-brain: a network partition separates nodes into two groups, and both groups elect a leader. Both leaders accept writes independently — leading to conflicting data with no way to resolve which is correct. Raft prevention: a leader can only commit a log entry after receiving acknowledgment from a MAJORITY (quorum) of nodes. If a network partition creates a minority group, its leader can't get majority ACKs → can't commit → becomes aware it lost leadership (election timeout) → steps down. The majority group's leader continues normally. This guarantees at most one leader can commit at any time. The minority group's data diverges but can't commit, so it's safe to discard when the partition heals.

💥 Failure Modes & Resilience
Circuit breaker, bulkhead, retry with exponential backoff, timeout, and fallback — the patterns that keep distributed systems alive.
Resilience Patterns
RESILIENCE PATTERNS 
// 1. CIRCUIT BREAKER — fail fast, don't hammer a sick service
// States: CLOSED (normal) → OPEN (failing) → HALF-OPEN (testing recovery)
class CircuitBreaker {
  state = 'CLOSED'
  failureCount = 0
  threshold = 5        // open after 5 failures
  timeout = 30000      // try again after 30s

  async call(fn) {
    if (this.state === 'OPEN') {
      if (Date.now() > this.nextAttempt)
        this.state = 'HALF-OPEN';
      else throw new Error('Circuit OPEN — fast fail');
    }
    try {
      const result = await fn();
      this.failureCount = 0;
      this.state = 'CLOSED';
      return result;
    } catch (e) {
      this.failureCount++;
      if (this.failureCount >= this.threshold) {
        this.state = 'OPEN';
        this.nextAttempt = Date.now() + this.timeout;
      }
      throw e;
    }
  }
}

// 2. RETRY with exponential backoff + jitter
async function retryWithBackoff(fn, maxAttempts = 3) {
  for (let i = 0; i < maxAttempts; i++) {
    try { return await fn(); }
    catch (e) {
      if (i === maxAttempts - 1) throw e;
      // exponential + jitter: 200ms, 400ms, 800ms +/- random
      const delay = 200 * 2**i + Math.random() * 100;
      await new Promise(r => setTimeout(r, delay));
    }
  }
}
// ⚠️ Only retry IDEMPOTENT operations! Don't retry POST payment!

// 3. BULKHEAD — isolate failures (like ship compartments)
// Separate thread pools / connection pools per downstream service
// If payment-service is slow, it uses ITS pool — won't exhaust user-service pool

// 4. TIMEOUT — never wait forever
// Rule: always set timeouts. 2 seconds for synchronous user-facing calls.
// cascade of no-timeouts = one slow service hangs entire system
🎯 INTERVIEW QUESTIONS
Q: What is a circuit breaker and why do you need it in microservices?
In microservices, if Service A calls Service B and B is slow/down, A's threads pile up waiting for B, exhausting A's connection pool, which then affects Service C that calls A. This cascade failure can take down the whole system. Circuit breaker solves this: when failure rate exceeds a threshold, the breaker OPENS — subsequent calls fail immediately (fast fail) without actually hitting the sick service. After a timeout, it enters HALF-OPEN: lets one test request through. If it succeeds, CLOSES (normal operation). If it fails, stays OPEN. Benefits: (1) Prevents cascade failures, (2) Gives the sick service time to recover without hammering it, (3) Fail fast = better UX (immediate error) than hang for 30 seconds then error. Hystrix (Netflix), Resilience4j, Polly (.NET) are popular implementations.
Q: Why add jitter to retry delays?
Without jitter: if 1000 clients all fail simultaneously (e.g. a server restart) and all retry at exactly 200ms, 400ms, 800ms — you get 1000 simultaneous requests at each retry interval — a thundering herd that overwhelms the recovering server and may cause it to fail again. With jitter (random +/- variation on the delay): retries are spread across time, reducing the peak load on the recovering service. The recovering server sees a gradual ramp-up instead of synchronized spikes. Full jitter (random within 0 to max_delay) is often the most effective. AWS recommends "decorrelated jitter" for their retry SDK.

📡 Events & Messaging Basics
Events vs commands vs queries, pub/sub vs message queues, and the delivery guarantees that matter in production.
Events vs Commands vs Queries
EVENT-DRIVEN — MESSAGES 
// COMMAND — imperative, one recipient, sender expects action
{ type: "CREATE_ORDER", payload: { userId: 42, items: [...] } }
// → Sent TO order-service. One consumer processes it. Can fail.

// EVENT — past tense, facts, sender doesn't care who consumes
{ type: "ORDER_CREATED", payload: { orderId: 99, userId: 42 }, ts: ... }
// → Published to event bus. Multiple consumers can react:
//   inventory-service: reserve items
//   notification-service: send confirmation email
//   analytics-service: track conversion
// Original publisher doesn't know who subscribes → loose coupling!

// QUERY — read-only, no side effects
GET /orders/99   ← REST query
// In CQRS: queries go to read model, not write model

// PUB/SUB vs MESSAGE QUEUE
// Message Queue (RabbitMQ, SQS): one consumer per message
// → Load balancing: multiple workers share the work
// → Message deleted after consumed

// Pub/Sub (Kafka, GCP Pub/Sub): many consumers per message
// → Each consumer group gets its own copy
// → Messages retained for replay (Kafka: days/weeks/forever)

// DELIVERY GUARANTEES
// At-most-once:  may lose messages, never duplicates (fire and forget)
// At-least-once: may duplicate, never loses (retry on failure) ← most common
// Exactly-once:  no loss, no duplicates ← very hard, expensive (Kafka transactions)

// ⚠️ At-least-once → consumers MUST be idempotent!
// Process same message twice should have same result as processing once
🎯 INTERVIEW QUESTIONS
Q: What is the difference between a message queue and a pub/sub system?
Message queue (RabbitMQ, SQS, ActiveMQ): each message is consumed by ONE consumer. Multiple consumers compete for messages — this is load balancing. Message is deleted once consumed. Use for: work queues, task distribution, background job processing. Pub/Sub (Kafka, GCP Pub/Sub, SNS): each message is delivered to ALL subscribers. Every consumer group gets its own independent copy. Messages are typically retained for replay. Use for: event broadcasting, when multiple services need to react to the same event, audit logging, event sourcing. Key insight: Kafka blurs the line — it's pub/sub with consumer groups (each group = competing queue). Multiple groups = pub/sub behavior. One group with many members = queue behavior.

  topic: 'orders',

  value: JSON.stringify(orderEvent)

// ✅ Guaranteed order for user-42's orders
// ❌ No order guarantee ACROSS different keys

// CONSUMER GROUP — each partition assigned to ONE consumer in group
// 3 partitions, 3 consumers → 1 partition each (parallel!)
// 3 partitions, 5 consumers → 2 consumers idle (can't have more consumers than partitions)
// 3 partitions, 2 consumers → one consumer gets 2 partitions

// OFFSET MANAGEMENT
// Consumer tracks its position (offset) per partition
// Stored in __consumer_offsets topic (or manually)
// Commit after processing (at-least-once) or before (at-most-once)

// RETENTION — Kafka stores messages for configurable duration
// Default: 7 days. Can be forever (log compaction for snapshots).
// Consumers can REPLAY from any offset → reprocess historical events

// REPLICATION
// Each partition has: 1 leader + N-1 replicas
// acks=all → wait for all in-sync replicas to confirm → durable
// acks=1  → leader only → faster but can lose messages on leader crash
// acks=0  → fire and forget → fastest, may lose
🎯 INTERVIEW QUESTIONS
Q: How does Kafka guarantee ordering?
Kafka guarantees ordering only within a partition, not across partitions. When you produce a message with a key, Kafka hashes the key to determine the partition. All messages with the same key always go to the same partition, so they maintain FIFO order. Strategy: use a business entity ID as the key (userId, orderId) to ensure all events for that entity are ordered. Across partitions (different keys), there's no ordering guarantee. This is a deliberate design — it allows horizontal scaling (multiple partitions processed in parallel) while preserving order for entities that need it. If you need global ordering for everything, use a single partition — but then you lose parallelism.
Q: How would you handle a Kafka consumer that is slow?
(1) Increase consumer parallelism — add more consumer instances to the consumer group (up to the number of partitions). (2) Increase partition count — allows more parallel consumers. (3) Optimize the consumer logic — batch process messages, async I/O, reduce per-message DB calls. (4) Increase max.poll.interval.ms — prevents session timeout while processing slow batches. (5) Reduce max.poll.records — fetch fewer messages per poll if processing is slow. (6) Scale the downstream dependency — if the consumer is slow because it writes to a DB, fix the DB. Monitor consumer lag (difference between latest offset and committed offset) — growing lag means consumers can't keep up.

🧩 Event Patterns & CQRS
CQRS, the Outbox pattern for guaranteed delivery, idempotent consumers, and event-carried state transfer.
CQRS — Command Query Responsibility Segregation
💡 ANALOGY
CQRS is like a restaurant with separate kitchen (writes) and dining room menu (reads). The kitchen is optimized for cooking (write operations — transactional, normalized). The dining room menu shows dishes (read operations — denormalized for fast display). They're different models. A change in the kitchen eventually updates the menu, but you don't cook at the table.
CQRS + OUTBOX PATTERN 
// CQRS: separate Read and Write models
// Write side: normalized DB, handles commands, enforces business rules
// Read side: denormalized, optimized for queries (materialized views, Elasticsearch)

// ─── THE OUTBOX PATTERN ───────────────────────────────────────────
// Problem: write to DB + publish to Kafka → two operations → can fail independently
// ❌ BAD: write order to DB, then publish OrderCreated to Kafka
//    If Kafka publish fails → DB has order, event never sent → inconsistent!

// ✅ GOOD: Outbox pattern — write event to DB in SAME transaction
BEGIN TRANSACTION;
  -- 1. Save business entity
  INSERT INTO orders (id, user_id, total) VALUES (99, 42, 150.00);
  -- 2. Save event to outbox table (same transaction!)
  INSERT INTO outbox (topic, key, payload, created_at)
  VALUES ('orders', '99', '{"type":"ORDER_CREATED",...}', NOW());
COMMIT;

-- Separate "relay" process reads outbox and publishes to Kafka
-- Debezium (CDC) reads DB transaction log → publishes to Kafka automatically
-- Guarantees: order is saved ↔ event is published (atomically)

// IDEMPOTENT CONSUMER — handle duplicate messages safely
async function processOrderCreated(event) {
  // Check if already processed (idempotency key = event ID)
  const processed = await redis.exists(`processed:${event.id}`);
  if (processed) return;  // already handled — skip!

  await reserveInventory(event.items);

  // Mark as processed AFTER successful processing
  await redis.set(`processed:${event.id}`, '1', 'EX', 86400);
}
🎯 INTERVIEW QUESTIONS
Q: What is the Outbox pattern and why do you need it?
Without the outbox pattern, publishing a database write AND a Kafka event are two separate operations — if Kafka is down or the process crashes between them, you get inconsistency: DB has the record, event was never published (other services never knew). The Outbox pattern solves this by writing the event to an outbox table in the SAME database transaction as the business entity. Now both succeed or both fail atomically. A separate relay process (or Debezium CDC connector) reads the outbox table and publishes to Kafka with at-least-once delivery. This guarantees exactly one successful write = exactly one event published eventually. The outbox also gives you a natural audit log.

Event Sourcing
EVENT SOURCING + SAGA 
// EVENT SOURCING: store events, not current state
// Instead of: UPDATE accounts SET balance=500 WHERE id=42
// Store: sequence of events

  { id: 1, agg: "account-42", type: "AccountOpened",    data: { balance: 0 } }

// rebuild read models (projections) from scratch
// Trade-offs: query complexity, event schema evolution is hard

// ─── SAGA PATTERN: distributed transactions without 2PC ────────────
// Problem: booking flight + hotel + car — all must succeed or all fail
// 2-Phase Commit (2PC) is too slow and locks resources

// SAGA: sequence of local transactions with compensating transactions

// CHOREOGRAPHY SAGA (event-driven, decentralized)
order-service:     OrderCreated event →
payment-service:   PaymentProcessed event →
inventory-service: ItemsReserved event →
shipping-service:  ShipmentCreated event  ← order fulfilled!

// If inventory fails: InventoryFailed event →
// payment-service listens: REVERSE the payment (compensating tx)
// order-service listens: mark order as FAILED

// ORCHESTRATION SAGA (central coordinator)
class OrderSaga {
  async execute(order) {
    try {
      await paymentService.charge(order);
      await inventoryService.reserve(order);
      await shippingService.schedule(order);
    } catch (e) {
      // Compensate completed steps in reverse order
      await inventoryService.release(order);
      await paymentService.refund(order);
    }
  }
}
🎯 INTERVIEW QUESTIONS
Q: What is the difference between Saga choreography and orchestration?
Choreography: each service publishes events and other services react to them independently. No central coordinator. Services are decoupled — they don't know about each other, only about events. Pros: simple, resilient (no single point of failure), true decoupling. Cons: hard to track the overall flow, debugging is difficult, cyclic dependencies can emerge. Orchestration: a central saga orchestrator explicitly calls each service in sequence. Orchestrator knows the full workflow. Pros: clear visibility of the transaction flow, easier to debug, easy to add timeout/compensation logic. Cons: orchestrator is a potential single point of failure and creates coupling to all services. Rule: use choreography for simple linear flows with few services. Use orchestration for complex multi-step workflows where you need visibility and explicit compensation logic.

Monolith vs Microservices
Microservices are not the default. Most systems should start as a modular monolith. Microservices add enormous operational complexity (distributed tracing, independent deployments, service discovery, distributed transactions) that isn't justified until you hit scaling or team organization problems that a monolith can't solve.

Failure isolation One bug can take down everything Failure isolated to one service (with circuit breakers)
Transactions ACID — easy Distributed — need Saga, eventual consistency
When to use Startup, small team, <20 engineers Large team, scaling bottlenecks, domain boundaries clear

🎯 INTERVIEW QUESTIONS
Q: How do you decide the size of a microservice?
Size by business capability and team ownership, not code line count. The "right size" is: (1) Owned by one team (2-pizza rule: ~6-10 engineers can understand and own it). (2) Has a clear bounded context — it models one domain concept completely. (3) Has independent deployment — can be deployed without coordinating with other teams. (4) Has a single database. Avoid: services so small they're just database wrappers with no logic (nano-services), services that require changes in multiple other services every time you modify them (wrong boundaries), and services that can't be deployed independently (tightly coupled). Start coarse-grained (fewer, larger services) and split when there's a clear reason — don't start with nano-services.

Distributed Monolith: Services deployed separately but tightly coupled. Change one service → must change 3 others. Deploy service A → must coordinate with teams B, C, D. Q: Do you always need a service mesh in microservices?
No. Service meshes add significant operational complexity — learning curve, debugging sidecar proxies, performance overhead from proxy hops. (2) You need zero-code infrastructure capabilities (mTLS for compliance, uniform observability). (3) You need sophisticated traffic management (canary deployments, fault injection for chaos testing). Not justified when: small number of services, team doesn't have Kubernetes/Istio expertise, operational complexity outweighs benefits. Start without: implement resilience patterns (circuit breaker, retry) in application code, use structured logging with correlation IDs, add a simple API gateway. Graduate to service mesh when those don't scale.

🗄️ Data in Microservices
Database per service, why shared databases are an anti-pattern, and how to handle distributed data consistency.
Database Per Service — The Golden Rule
Each service owns its own database. No other service can query its database directly. All data access goes through the service's API or events. This is the most important data rule in microservices — it enables independent deployment, independent scaling, and independent choice of database technology.
DISTRIBUTED DATA PATTERNS 
// ❌ ANTI-PATTERN: shared database
order-service  →  shared_db.orders

🎯 INTERVIEW QUESTIONS
Q: How do you handle data consistency across microservices without 2-Phase Commit?
2-Phase Commit (2PC) is too slow (synchronous, blocking), creates distributed locks, and fails badly if the coordinator crashes. Alternatives: (1) Saga pattern — sequence of local transactions with compensating transactions. If step 3 fails, compensate steps 1 and 2. Choreography (events) or orchestration (central coordinator). (2) Eventual consistency via events — accept that data will be consistent eventually. Use the Outbox pattern to guarantee event delivery. (3) Reservations/idempotency — instead of locking, create a "pending" state, then confirm or cancel. (4) Accept some inconsistency — "Shopping cart has items that are sold out" is sometimes acceptable if inventory was a soft reservation. The key question: what's the business cost of temporary inconsistency? Often it's acceptable and much cheaper than 2PC.

🌐 API Design
Concept Key Point
Idempotency GET, PUT, DELETE = idempotent. POST = not. Idempotency-Key header for safe POST retries.
401 vs 403 401 = not authenticated ("who are you?"). 403 = authenticated but no permission ("answer is no").
Cursor pagination Stable under inserts/deletes. O(1) via index seek. Use for large datasets and infinite scroll.
GraphQL N+1 Fix with DataLoader — batches all resolver calls in one tick into a single batch query.
gRPC vs REST gRPC: binary, HTTP/2, streaming, typed. Better for internal services. REST: human-readable, cacheable, public APIs.
JWT pitfalls Short-lived access (15min) + httpOnly refresh token. RS256 asymmetric in prod. Base64 ≠ encrypted.
🔮 Distributed Systems
Concept Key Point
CAP P is always required. Real choice: CP (reject on partition) or AP (stale reads). Banks = CP, social = AP.
Eventual consistency OK for: view counts, recommendations. NOT OK for: balances, inventory, medical.
Raft quorum N nodes, tolerates (N-1)/2 failures. Commit requires majority ACK. Prevents split-brain.
Circuit breaker CLOSED→OPEN on failures. Fast fail. HALF-OPEN tests recovery. Prevents cascade.
Observability Metrics = is something wrong? Logs = what happened? Traces = where is the bottleneck? All three needed.
⚡ Event-Driven
Concept Key Point
Kafka ordering Ordered within partition only. Use entity ID as key → all events for one entity ordered.
At-least-once Most common. May duplicate → consumers MUST be idempotent. Check event ID before processing.
Outbox pattern Write event to DB in same transaction. Relay publishes to Kafka. Guarantees event = DB write.
CQRS Separate read model (denormalized, fast) from write model (normalized, transactional).
Saga Choreography = event-driven, decentralized. Orchestration = central coordinator. Both use compensating transactions.
🧩 Microservices
Concept Key Point
DB per service Non-negotiable. No shared DB. Share data via API or events. Enables independent deploy + scale.
When to use Large teams, clear domain boundaries, scaling bottlenecks. NOT for small teams or startups.
Distributed monolith Deployed separately but change one → change all. Worst of both worlds. Fix: real bounded contexts.
API Gateway vs Mesh Gateway = north-south (external traffic). Mesh = east-west (service-to-service). Both can coexist.
Strangler Fig Migrate monolith incrementally. Never big-bang rewrite. Route new traffic to new service, strangle old.
