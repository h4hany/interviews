📦 Docker Basics
Containers vs VMs, the Docker architecture, and why Docker changed software delivery forever.

💡 ANALOGY
VMs are like apartments in separate buildings — each has its own plumbing, electricity, and structure. Containers are apartments in the same building — they share the foundation (kernel) and utilities, but have their own locked doors (namespaces) and allocated resources (cgroups). Containers are cheaper, faster to set up, but share more infrastructure.

🎯 INTERVIEW QUESTIONS
Q: What is Docker and what problem does it solve?
Docker packages an application and all its dependencies (libraries, config, runtime) into a container that runs consistently anywhere. It solves "works on my machine" — the container includes everything the app needs, so it runs identically on dev, staging, and production. Key benefits: reproducible environments, fast deployment, efficient resource usage vs VMs, easy horizontal scaling, and immutable deployable artifacts.
Q: What are Linux namespaces and cgroups? How do containers use them?
Namespaces provide isolation — each container gets its own view of: PID (process IDs), NET (network stack), MNT (filesystem mounts), UTS (hostname), IPC (inter-process communication), USER (user IDs). Cgroups (control groups) limit and account for resource usage — how much CPU, memory, disk I/O a container can use. Together, namespaces provide isolation and cgroups provide resource control, which is what makes containers feel like isolated environments while sharing the host kernel.
Q: What's the difference between docker stop and docker kill?
docker stop sends SIGTERM (graceful shutdown signal), waits 10 seconds for the process to clean up and exit, then sends SIGKILL. This allows the app to finish in-flight requests, close connections, flush buffers. docker kill immediately sends SIGKILL — instant termination, no cleanup. Always prefer docker stop. Use docker kill only when a container is unresponsive. You can customize the signal with --signal flag.

📄 Dockerfile Deep Dive
Every instruction explained, multi-stage builds that shrink images 10x, and cache optimization tricks.
All Dockerfile Instructions
DOCKERFILE — COMPLETE REFERENCE
# FROM — base image (always first)
FROM node:20-alpine AS base

# ARG — build-time variable (not in final image)
ARG NODE_ENV=production

# ENV — runtime environment variable (persists in image)
ENV PORT=3000 NODE_ENV=$NODE_ENV

# WORKDIR — set working directory (creates it if not exists)
WORKDIR /app

# COPY — copy files from build context
# COPY src dest  (src relative to build context)
COPY package*.json ./          # copy package files FIRST (cache trick!)

# RUN — execute command during BUILD (creates new layer)
RUN npm ci --only=production && \
    npm cache clean --force       # chain && minimize layers

COPY . .                        # copy rest of source AFTER npm install

# USER — switch to non-root user (security!)
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# EXPOSE — document which port (does NOT publish!)
EXPOSE 3000

# HEALTHCHECK — Docker health monitoring
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -qO- http://localhost:3000/health || exit 1

# CMD — default command (can be overridden at runtime)
CMD ["node", "server.js"]

# ENTRYPOINT — fixed command (CMD becomes its arguments)
# ENTRYPOINT ["node"]  →  docker run myimage server.js
Multi-Stage Builds — The Game Changer
Multi-stage builds let you use one stage to build and another to run, discarding build tools, compilers, and test dependencies from the final image. Result: images 5-10x smaller.
DOCKERFILE — MULTI-STAGE BUILD
# ---- Stage 1: Build ----
FROM node:20 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci                  # includes devDependencies
COPY . .
RUN npm run build           # compile TypeScript, bundle, etc.
RUN npm test                # run tests during build

# ---- Stage 2: Production (tiny!) ----
FROM node:20-alpine AS production
WORKDIR /app

# ONLY copy what's needed from builder stage
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/package*.json ./
RUN npm ci --only=production  # no devDependencies!

USER node
CMD ["node", "dist/server.js"]

# Result: builder=1.2GB  production=180MB  (7x smaller!)
# No TypeScript compiler, no jest, no source files in prod image
CMD vs ENTRYPOINT
AspectCMDENTRYPOINT
PurposeDefault arguments / commandFixed executable
Override at runtime✅ Yes — docker run img new-cmd⚠️ Needs --entrypoint flag

Shell form vs Exec form: CMD node app.js (shell form) runs via /bin/sh -c, so PID 1 is sh, not your app — signals like SIGTERM won't reach your process! Always use exec form: CMD ["node", "app.js"] — your process becomes PID 1 and receives signals directly.
🎯 INTERVIEW QUESTIONS
Q: What is a multi-stage build and why use it?
Multi-stage builds use multiple FROM instructions in one Dockerfile. Each stage can use a different base image. You COPY --from=stage_name only the artifacts you need. Benefits: (1) Final image contains only runtime necessities — no compilers, build tools, test frameworks, (2) Smaller attack surface (fewer packages = fewer vulnerabilities), (3) Smaller images = faster pulls and deploys, (4) Build secrets don't leak into final image. Typical result: Java Spring app from ~800MB to ~150MB.
Q: How does Docker layer caching work and how do you optimize for it?
Each Dockerfile instruction creates a layer. Docker caches layers and reuses them if the instruction and its inputs haven't changed. Cache is invalidated for a layer and all subsequent layers when: the instruction changes, or the files it COPYs change. Optimization: put things that change least frequently FIRST. Copy package.json first, run npm install (slow, cached as long as package.json unchanged), THEN copy source code (changes often). This way, source code changes don't invalidate the npm install layer.
Q: What's the difference between RUN, CMD, and ENTRYPOINT?
RUN executes during the BUILD phase, creates a new image layer, used for installing packages, compiling code. CMD defines the DEFAULT command/args when the container starts — can be overridden at runtime. ENTRYPOINT defines the main executable — makes the container act like a command. When both ENTRYPOINT and CMD are set, ENTRYPOINT is the executable and CMD provides its default arguments. CMD alone can be fully replaced by docker run args; ENTRYPOINT needs --entrypoint to override.

🧅 Images & Layers
How Docker images are built from layers, the union filesystem, copy-on-write, and how to build minimal images.
Union Filesystem & Layers
A Docker image is a stack of read-only layers. When a container starts, Docker adds a thin writable layer on top (copy-on-write). Each RUN, COPY, ADD instruction in a Dockerfile creates a new layer.
Container Layer (writable) — your running changes
Layer 4: COPY . . (app source code)
Layer 3: RUN npm install (node_modules)
Layer 2: COPY package.json .
Layer 1: FROM node:20-alpine (base)
Layers are shared — 100 containers from same image = 100 writable layers + 1 shared read-only stack
Minimizing Image Size
DOCKERFILE — SIZE OPTIMIZATION
# ❌ BAD — creates 3 layers, apt cache stays in image
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*  # too late! previous layer still has cache

# ✅ GOOD — single layer, cache cleaned in same RUN
RUN apt-get update && \
    apt-get install -y --no-install-recommends curl && \
    rm -rf /var/lib/apt/lists/*   # same layer = actually removes it

# ✅ Use specific tags (not latest)
FROM node:20.11.0-alpine3.19    # reproducible builds

# ✅ Use .dockerignore (like .gitignore for builds)
# .dockerignore:
# node_modules
# .git
# *.log
# .env
# coverage/
# dist/  (if using multi-stage)

# ✅ Distroless images — no shell, package manager, or OS tools
# ~10MB for Go apps vs ~200MB with Ubuntu
FROM gcr.io/distroless/nodejs20-debian12
🎯 INTERVIEW QUESTIONS
Q: Why does chaining RUN commands with && matter?
Each RUN instruction creates a new layer. If you run apt-get update in one layer and rm -rf /var/lib/apt/lists/* in another layer, the cache files still exist in the first layer (layers are immutable snapshots). The final image contains ALL layers, so the cache is still there, just hidden. By chaining with &&, everything runs in one layer — the cleanup actually removes the files from that layer's snapshot, reducing final image size.
Q: What is .dockerignore and why is it important?
.dockerignore excludes files from the build context — the directory sent to the Docker daemon when running docker build. Without it: node_modules (hundreds of MB), .git directory, logs, .env files, test coverage all get sent unnecessarily. This slows builds (context upload) and may accidentally COPY sensitive files (.env with credentials) into the image. Always add node_modules, .git, *.log, .env, and build artifacts to .dockerignore.

SHELL
# Create custom bridge network
docker network create --driver bridge myapp-net

# Run containers on same network — they can reach each other by NAME
docker run -d --name db --network myapp-net postgres:15
docker run -d --name api --network myapp-net -p 3000:3000 myapi

# Inside 'api' container, can reach db by hostname:
# psql -h db -U postgres  ← 'db' resolves via Docker DNS

# Port mapping: host_port:container_port
docker run -p 8080:80 nginx        # host:8080 → container:80
docker run -p 127.0.0.1:8080:80 nginx  # bind to localhost only (safer)

# List and inspect networks
docker network ls
docker network inspect myapp-net
🎯 INTERVIEW QUESTIONS
Q: How do two containers on the same Docker network communicate?
Containers on a user-defined bridge network can reach each other by container name — Docker has a built-in DNS server that resolves container names to their IP addresses. So if container "api" and "db" are on the same network, "api" can connect to postgres://db:5432 without knowing the IP. The default bridge network doesn't have this DNS feature — use custom networks. This is why docker-compose automatically creates a shared network for all services.
Q: When would you use --network host?
host networking removes network isolation — the container uses the host's network directly with no NAT. Use when: (1) performance is critical and NAT overhead is measurable (networking-intensive apps), (2) the app needs to bind to specific host interfaces, (3) debugging network issues. Downsides: security risk (container can access all host network interfaces), port conflicts (can't run two containers that both bind port 80), not portable across platforms (doesn't work on Docker Desktop Mac/Windows).

SHELL
# Named volume — Docker manages location
docker volume create pgdata
docker run -d \
  -v pgdata:/var/lib/postgresql/data \    # named_volume:container_path
  --name postgres \
  postgres:15

# Bind mount — host path explicitly specified
docker run -d \
  -v /home/hany/myapp:/app \             # host_path:container_path
  -v /home/hany/myapp:/app:ro \          # :ro = read-only
  node:20

# Modern --mount syntax (explicit, preferred)
docker run -d \
  --mount type=volume,source=pgdata,target=/var/lib/postgresql/data \
  postgres:15

docker run -d \
  --mount type=bind,source="$(pwd)",target=/app \
  node:20

# Volume management
docker volume ls
docker volume inspect pgdata
docker volume prune     # remove unused volumes
🎯 INTERVIEW QUESTIONS
Q: What happens to data in a container when it's removed?
Any data written to the container's writable layer (not a volume) is lost permanently when the container is removed with docker rm. This is why stateful services (databases, file uploads) must use volumes. Volumes exist independently of containers — they persist after container removal and can be shared between multiple containers. Named volumes are the recommended way to persist data; they're managed by Docker and easier to backup and migrate.
Q: Named volume vs bind mount — when to use each?
Named volumes: Docker manages the storage location (typically /var/lib/docker/volumes/). Easier to backup, migrate, use with Docker Swarm/K8s. Better performance (Docker optimizes I/O). Use for: production databases, persistent app state. Bind mounts: map a specific host directory. Good for: development (mount source code for live reload), reading config files from known host paths, when you need to access data from both host and container. In production, prefer named volumes for portability.

🎼 Docker Compose
Define and run multi-container applications. The essential tool for local development and simple deployments.
Complete docker-compose.yml Example
YAML — docker-compose.yml
version: '3.9'

services:
  api:
    build:
      context: .
      target: development      # use dev stage of multi-stage build
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/mydb
      - REDIS_URL=redis://cache:6379
    env_file:
      - .env.local               # load from file
    volumes:
      - .:/app                   # bind mount for hot reload
      - /app/node_modules        # anonymous volume prevents overwrite
    depends_on:
      db:
        condition: service_healthy  # wait for health check
      cache:
        condition: service_started
    restart: unless-stopped

  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_DB: mydb
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user -d mydb"]
      interval: 5s
      timeout: 5s
      retries: 5

  cache:
    image: redis:7-alpine
    command: redis-server --maxmemory 256mb --maxmemory-policy allkeys-lru

volumes:
  pgdata:                        # named volume declaration

networks:
  default:
    name: myapp-network
SHELL — COMPOSE COMMANDS
docker compose up -d              # start all services detached
docker compose up --build         # rebuild images then start
docker compose down               # stop and remove containers
docker compose down -v            # also remove volumes!
docker compose logs -f api        # follow api logs
docker compose exec api bash      # shell into api container
docker compose ps                 # list service status
docker compose scale api=3        # run 3 api instances
docker compose restart api        # restart one service
🎯 INTERVIEW QUESTIONS
Q: What is depends_on and what are its limitations?
depends_on controls start ORDER — the dependent service starts after its dependencies. But it doesn't wait for dependencies to be READY (just started). A database container may start but take 10 seconds to be ready to accept connections. Solution: use depends_on with condition: service_healthy and add a healthcheck to the dependency. Or use a wait-for-it script in your entrypoint. This is a very common interview trap.
Q: What's the difference between Docker Compose and Kubernetes?
Docker Compose: for defining multi-container apps on a SINGLE host. Simple YAML, great for development, no orchestration. Not designed for production at scale. Kubernetes: container ORCHESTRATOR for multiple hosts/nodes. Handles: auto-scaling, self-healing, rolling deployments, service discovery, load balancing across nodes, secrets management, multi-tenancy. Much more complex. Use Compose for local dev and simple single-server deployments; Kubernetes for production at scale.

🔒 Docker Security
Non-root containers, minimal images, secret management, and image scanning. Security is always asked in senior interviews.
Security Best Practices
DOCKERFILE — SECURITY
# 1. Use specific non-root user
FROM node:20-alpine
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
WORKDIR /app
COPY --chown=appuser:appgroup . .
USER appuser          # NEVER run as root!

# 2. Use specific image tags (not :latest)
FROM node:20.11.0-alpine3.19    # pinned = reproducible + auditable

# 3. Minimal base images
FROM gcr.io/distroless/nodejs20  # no shell = smaller attack surface

# 4. Don't store secrets in images!
# ❌ NEVER:
ENV DB_PASSWORD=mysecret         # visible in docker inspect, image layers!
COPY .env /app/.env              # .env baked into image

# ✅ Inject at runtime:
# docker run -e DB_PASSWORD=$SECRET myapp
# Or use Docker secrets / K8s secrets

# 5. Read-only filesystem
# docker run --read-only --tmpfs /tmp myapp
Image Scanning & Runtime Security
Tools: Trivy (free, fast vulnerability scanner), Snyk, Docker Scout, Anchore. Scan in CI/CD pipeline before pushing to registry. Set severity thresholds — fail build on CRITICAL/HIGH CVEs.
⚠️ Running as root: If an attacker escapes the container running as root, they have root on the host. Always set USER in Dockerfile. Use --security-opt=no-new-privileges to prevent privilege escalation. Use AppArmor/seccomp profiles to limit syscalls.
🎯 INTERVIEW QUESTIONS
Q: Why should containers not run as root?
If a container runs as root and an attacker exploits a vulnerability in the app, they have root privileges inside the container. If they then exploit a container escape vulnerability (Docker daemon bug, kernel exploit), they get root on the host — game over. Running as non-root limits blast radius: attacker gets limited user privileges inside the container. Also prevents accidental overwrites of system files. Set USER in Dockerfile and ensure files are owned by that user (COPY --chown).
Q: How should you handle secrets in Docker?
Never bake secrets into images — they're visible in docker inspect, image history, and any registry they're pushed to. Options: (1) Runtime environment variables via -e or env_file (simple, but visible in process list), (2) Docker secrets (Swarm feature) — secrets mounted as tmpfs files in /run/secrets/, not visible as env vars, (3) Kubernetes Secrets (base64-encoded, mountable as files or env vars), (4) External secret managers: HashiCorp Vault, AWS Secrets Manager, Azure Key Vault — inject at container startup. For production: use external secret manager + K8s secrets.

🏗️ Kubernetes Architecture
Control plane vs worker nodes, what each component does, and how they work together. The most fundamental K8s interview topic.
The Architecture
CONTROL PLANE (Master)
API Server — front door, REST API, auth
etcd — key-value store, cluster state
Scheduler — assign pods to nodes
Controller Manager — watches & reconciles
Cloud Controller — cloud provider integration
WORKER NODES
kubelet — agent, manages pods on node
kube-proxy — network rules, Service routing
Container Runtime — containerd / CRI-O
Pods — your app containers
What Each Component Does
ComponentRoleKey fact
kube-apiserverAll K8s operations go through it. Validates, authenticates, stores to etcd.Single source of truth API. Horizontally scalable.
etcdDistributed KV store. Holds ALL cluster state — pods, configs, secrets.Most critical component. Losing etcd = losing cluster. Always backup!
kube-schedulerWatches for unscheduled pods, picks best node based on resources, affinity, taints.Doesn't start the pod — just assigns it to a node.
kube-controller-managerRuns control loops: node controller, replication controller, endpoints controller.Reconciliation loop — constantly checks desired vs actual state.
kubeletNode agent. Talks to API server, manages pod lifecycle, runs health checks.Only component that talks directly to container runtime.
kube-proxyImplements Services by maintaining iptables/IPVS rules for pod-to-pod routing.Doesn't proxy traffic — just configures routing rules.
What Happens When You kubectl apply
1. kubectl sends YAML to API Server → validates, authenticates, stores in etcd
2. Controller Manager notices new Deployment → creates ReplicaSet → creates Pod objects
3. Scheduler notices unscheduled Pod → evaluates nodes → assigns to best node (writes to etcd)
4. kubelet on target node notices pod assigned to it → instructs container runtime to pull image and start container
5. kubelet reports status back to API server → pod shows Running
🎯 INTERVIEW QUESTIONS
Q: What is etcd and why is it so important?
etcd is a distributed, consistent key-value store that serves as Kubernetes' "database" — it stores the entire cluster state: all pod specs, service definitions, secrets, configmaps, RBAC rules, node state. Every kubectl get, every scheduler decision, every controller loop reads from etcd. If etcd goes down, the control plane stops — you can't create, update, or delete anything (though existing workloads keep running). This is why etcd is always run with multiple replicas (3 or 5 for production) and backed up regularly.
Q: What is the reconciliation loop in Kubernetes?
The reconciliation loop is the heart of Kubernetes' self-healing. Controllers continuously watch the desired state (what you defined in YAML) and compare it with the actual state (what's running). When they differ, the controller acts to reconcile them. Example: you define a Deployment with 3 replicas. A node crashes, killing 1 pod. The ReplicaSet controller notices actual=2 vs desired=3 and creates a new pod. This loop runs constantly — it's why K8s is "self-healing."

🚀 Pods & Deployments
The fundamental workload units. Pod lifecycle, ReplicaSets, rolling updates, rollbacks, and resource management.
Pod — The Smallest Unit
A Pod is one or more containers that share network namespace and storage. Containers in a pod communicate via localhost. Pods are ephemeral — when they die, they don't come back (use Deployments for that).
YAML — DEPLOYMENT
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-deployment
  labels:
    app: api
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api           # must match pod template labels
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # max extra pods during update
      maxUnavailable: 0    # 0 = zero-downtime deployment
  template:
    metadata:
      labels:
        app: api
    spec:
      containers:
        - name: api
          image: myrepo/api:1.2.0
          ports:
            - containerPort: 3000
          resources:
            requests:           # guaranteed resources
              memory: "128Mi"
              cpu: "100m"      # 100 millicores = 0.1 CPU
            limits:             # maximum allowed
              memory: "256Mi"
              cpu: "500m"
          env:
            - name: NODE_ENV
              value: production
Requests vs Limits
AspectRequestsLimits
PurposeGuaranteed minimum resourcesMaximum resources allowed
SchedulingUsed by scheduler to find a nodeNot used for scheduling
CPU over limit—Throttled (not killed)
Memory over limit—OOMKilled (container killed!)
Best practiceSet equal to typical usageSet 2x requests as starting point
⚠️ Always set resource requests and limits! Without requests, the scheduler can't place pods properly — nodes get overloaded. Without limits, one buggy container can consume all node memory and take down other pods (OOM kills).

🎯 INTERVIEW QUESTIONS
Q: What's the difference between a Pod, ReplicaSet, and Deployment?
Pod: single instance of running containers. If it crashes, it's gone. ReplicaSet: ensures N copies of a pod are always running. If a pod dies, it creates a new one. But doesn't handle rolling updates. Deployment: manages ReplicaSets. When you update a deployment, it creates a new ReplicaSet and gradually shifts traffic by scaling one up and the other down. Stores rollout history for rollbacks. In practice, you always use Deployments — never create ReplicaSets or Pods directly.
Q: What is OOMKilled and how do you prevent it?
OOMKilled (Out Of Memory Killed) happens when a container exceeds its memory limit. The Linux kernel's OOM killer terminates the process. In K8s, the container is restarted according to the restartPolicy. Prevention: (1) Set appropriate memory limits based on profiling, (2) Fix memory leaks in the application, (3) If the app legitimately needs more memory, raise the limit, (4) Use Vertical Pod Autoscaler to auto-tune limits. Check: kubectl describe pod <name> — look for "OOMKilled" in Last State.

🎯 INTERVIEW QUESTIONS
Q: What is an Ingress and why use it instead of LoadBalancer services?
An Ingress is an L7 HTTP router. It routes traffic to different services based on URL path or hostname. Instead of creating one cloud LoadBalancer per service (expensive — each LB costs ~$15-50/month on AWS), you create ONE LoadBalancer that feeds an Ingress controller (nginx, Traefik, HAProxy) which routes internally to many services. Benefits: cost (one LB), TLS termination in one place, host-based routing (api.myapp.com vs www.myapp.com), path-based routing (/api → api-service).
Q: How does a Service discover which pods to route to?
Services use label selectors — they route to any pod matching the selector labels. When pods are created/deleted, the Endpoints object (automatically managed by the Endpoints controller) is updated with current pod IPs. kube-proxy watches the Endpoints API and updates iptables/IPVS rules on each node to route Service ClusterIP to actual pod IPs. This is how K8s handles rolling updates — new pods are added to Endpoints before old ones are removed, enabling zero-downtime updates.

⚠️ Secrets are NOT encrypted by default! They're base64-encoded (trivially decodable). For true security: enable Envelope Encryption (EncryptionConfig) so secrets are encrypted in etcd, use external secret managers (Vault, AWS Secrets Manager) with operators like External Secrets Operator, and restrict RBAC access to secrets.
🎯 INTERVIEW QUESTIONS
Q: Are Kubernetes Secrets actually secure?
By default, no — they're only base64-encoded, not encrypted. Anyone with kubectl get secret access can decode them instantly. For proper security: (1) Enable EncryptionConfiguration to encrypt secrets at rest in etcd, (2) Use RBAC to restrict who can read secrets (principle of least privilege), (3) Use External Secrets Operator with AWS Secrets Manager, GCP Secret Manager, or HashiCorp Vault for enterprise-grade secrets management, (4) Never commit secrets to Git — use Sealed Secrets or SOPS for GitOps workflows.
Q: Environment variables vs mounted volumes for config — which is better?
Environment variables: simpler, but the app must restart to pick up changes. Also visible in process list (ps aux), logged in some tools, inherited by child processes. Mounted config files (ConfigMap/Secret as volumes): can be updated without pod restart (K8s syncs changes to mounted files automatically), not exposed in process list, can be structured as full config files. For secrets: mounted volumes are generally preferred as they're less exposed. For non-sensitive config that changes rarely: env vars are fine. For dynamic config: mounted volumes.

📈 Scaling & HPA
HorizontalPodAutoscaler, VPA, cluster autoscaler, and resource requests — the holy trinity of K8s scaling.
Three Levels of Autoscaling
AutoscalerWhat it scalesBased on
HPA (Horizontal Pod Autoscaler)Number of pod replicasCPU/memory usage, custom metrics
VPA (Vertical Pod Autoscaler)CPU/memory requests per podHistorical resource usage
Cluster AutoscalerNumber of nodes in clusterPending pods that can't be scheduled
YAML — HPA
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-deployment
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70  # scale when CPU > 70%
    - type: Resource
      resource:
        name: memory
        target:
          type: AverageValue
          averageValue: 200Mi
    - type: Pods           # custom metric example
      pods:
        metric:
          name: requests_per_second
        target:
          type: AverageValue
          averageValue: "100"
HPA requires metrics-server! HPA reads CPU/memory metrics from the metrics-server. Without it, HPA can't function. Install with: kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
🎯 INTERVIEW QUESTIONS
Q: How does HPA decide when to scale?
HPA calculates the desired replica count using: desiredReplicas = ceil(currentReplicas × (currentMetricValue / desiredMetricValue)). Example: 3 replicas, CPU at 90%, target 70% → ceil(3 × 90/70) = ceil(3.86) = 4 replicas. It checks every 15 seconds (default). Scale-up: immediate on threshold breach. Scale-down: waits 5 minutes (stabilization window) to prevent flapping. HPA requires CPU/memory requests set on the pod — without requests, HPA can't calculate utilization.
Q: What's the difference between HPA and VPA? Can you use both?
HPA scales OUT (more pods). VPA scales UP (bigger pods — adjusts requests/limits). They solve different problems: HPA handles traffic bursts (more instances), VPA optimizes resource allocation for each pod. Using both simultaneously on CPU/memory can cause conflicts — HPA scales out based on CPU, VPA changes requests which affects HPA's calculations. Best practice: use HPA for most apps, VPA for services with variable resource needs. VPA can run in recommendation-only mode (no auto-apply) which is useful for right-sizing.

❤️ Health Probes
Liveness, Readiness, and Startup probes — the key to zero-downtime deployments and self-healing applications.
Three Probe Types
ProbePurposeFailure action
LivenessIs the container alive? (not deadlocked)Restart the container
ReadinessIs the container ready to receive traffic?Remove from Service endpoints (no restart)
StartupHas the container finished starting?Restart (disables liveness until passed)
YAML — ALL THREE PROBES
containers:
  - name: api
    image: myapi:1.0

    # Startup: give slow-starting apps time (e.g., Spring Boot)
    startupProbe:
      httpGet:
        path: /health
        port: 3000
      failureThreshold: 30    # 30 × 10s = 5 min to start
      periodSeconds: 10
      # Liveness disabled until startup succeeds

    # Liveness: detect deadlocks, restart if unhealthy
    livenessProbe:
      httpGet:
        path: /health/live
        port: 3000
      initialDelaySeconds: 10
      periodSeconds: 15
      failureThreshold: 3     # restart after 3 failures
      timeoutSeconds: 5

    # Readiness: only send traffic when app is truly ready
    readinessProbe:
      httpGet:
        path: /health/ready     # checks DB connection, cache, etc.
        port: 3000
      initialDelaySeconds: 5
      periodSeconds: 5
      failureThreshold: 2     # remove from LB after 2 failures
💡 ANALOGY
Think of a doctor and a nurse. The Startup probe is the nurse checking if a new patient has arrived and is ready to be examined. The Liveness probe is the doctor checking if the patient is still alive (heart beating). If not → restart (defibrillator). The Readiness probe is checking if the patient is healthy enough to see visitors (receive traffic). If sick → no visitors (removed from load balancer), but not killed.
🎯 INTERVIEW QUESTIONS
Q: What's the difference between liveness and readiness probes?
Liveness: "Is this container alive and not stuck?" Failure = restart container. Use for detecting deadlocks, infinite loops, corrupted state. Don't make it too strict — false positives cause restart loops. Readiness: "Is this container ready to serve requests?" Failure = remove from Service endpoints (no traffic sent, but NOT restarted). Use for: startup initialization, dependency checks (DB connected, cache warmed), graceful handling of overload. Key: a pod can be alive but not ready — it's running but momentarily unable to serve.
Q: How do probes enable zero-downtime deployments?
During a rolling update: (1) New pod starts, (2) Startup probe passes — liveness/readiness begin, (3) Readiness probe passes — pod added to Service endpoints, receives traffic, (4) Old pod removed from endpoints ONLY AFTER new pod is ready, (5) Old pod gracefully shuts down (SIGTERM, then drain in-flight requests). With maxUnavailable=0 and a proper readiness probe, there's always at least 3 ready pods throughout. Without readiness probes, K8s might send traffic to a pod that's starting up but not ready — causing errors.

🎯 INTERVIEW QUESTIONS
Q: What's the difference between Deployment and StatefulSet?
Deployment: pods are identical and interchangeable. Pods get random names (api-7d8b9-xk2p1). Pods can be rescheduled to any node. Good for stateless apps. StatefulSet: pods have stable, persistent identities — ordered names (mysql-0, mysql-1, mysql-2). Each pod gets its own PersistentVolumeClaim that follows it even when rescheduled. Pods start/stop in order. Stable network identity via headless Service. Use for: databases, Kafka, Zookeeper, Elasticsearch — anything needing stable storage and identity per instance.

# ServiceAccount — identity for your app's pods
apiVersion: v1

# Role — what can be done (in this namespace)
apiVersion: rbac.authorization.k8s.io/v1

  name: pod-reader

---

apiVersion: rbac.authorization.k8s.io/v1

  name: read-pods-binding
  namespace: production

  - kind: ServiceAccount
    name: api-sa
    namespace: production

  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
🎯 INTERVIEW QUESTIONS
Q: What is a ServiceAccount and why do pods need one?
A ServiceAccount is the identity a pod uses to authenticate to the Kubernetes API. When a pod calls the API (to list pods, read ConfigMaps, etc.), it presents the ServiceAccount token. Without specifying one, pods use the default ServiceAccount. Best practices: create dedicated ServiceAccounts per application with minimal permissions (principle of least privilege). Disable auto-mounting the ServiceAccount token if the pod doesn't need API access: automountServiceAccountToken: false.
Q: What is a NetworkPolicy and how does it work?
NetworkPolicy controls traffic between pods (L3/L4 rules). By default, all pods can communicate with all other pods. NetworkPolicy lets you: allow only specific pods to reach your database, block all ingress except from your frontend, restrict egress to only specific external IPs. Policies are additive — multiple policies combine. Important: NetworkPolicy requires a CNI plugin that supports it (Calico, Cilium, WeaveNet — NOT the default kubenet). Without a supporting CNI, NetworkPolicy is accepted but ignored.
