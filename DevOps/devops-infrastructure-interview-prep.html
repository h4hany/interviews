🐳 Docker
Containers, images, Dockerfile, volumes, networks.
Containers and images
A container is an isolated process with its own filesystem (from an image), namespace, and cgroups. An image is built from layers defined in a Dockerfile; each instruction adds a layer. Same base layers are shared across images. Use multi-stage builds: build in one stage, copy only the binary to a minimal final image to keep size small and avoid build tools in production.
# Multi-stage: build then copy
FROM golang:1.21 AS builder
WORKDIR /app
COPY . . && go build -o server .
FROM alpine:latest
COPY --from=builder /app/server /server
CMD ["/server"]
Volumes and networks
Volumes persist data outside the container lifecycle; use for DB data, config. Networks let containers reach each other by name (e.g. app → db). Docker Compose defines multi-container apps with services, volumes, and networks in one file for local dev.
Q: Why use multi-stage Docker builds?
The build stage can use a full SDK (compiler, tools); the final stage only needs the binary and runtime. This keeps the production image small, reduces attack surface, and speeds deploy. Without it, the image would include build tools and source.

☸️ Kubernetes
Pods, deployments, services, ingress.
Pods and Deployments
A Pod is the smallest deployable unit; it runs one or more containers that share network and storage. A Deployment manages a set of Pods: declares replicas, image, and update strategy (rolling update, recreate). It creates a ReplicaSet; you scale and roll out by changing the Deployment. Liveness and readiness probes control restarts and traffic.
Services and Ingress
Service gives Pods a stable DNS name and IP: ClusterIP (internal), NodePort (node port), LoadBalancer (cloud LB). Ingress routes HTTP/HTTPS to Services (path, host) and can handle TLS. ConfigMaps and Secrets mount config and secrets into Pods without baking them into the image.
Q: What is the difference between a Deployment and a Service?
Deployment is the workload: it ensures N replicas of your Pods are running and handles rolling updates. Service is the network: it exposes those Pods under a stable name and load-balances traffic to them. You typically have one Deployment and one Service per app.

🔄 CI/CD
Pipelines and deployment strategies.
CI and CD
CI (Continuous Integration): On every commit (or PR), run lint, unit tests, build, and produce an artifact (image, package). Fail fast so broken code is not merged. CD (Continuous Deployment/Delivery): Automatically deploy the artifact to staging and optionally production. Stages: build → test → push image → deploy. Use pipelines as code (YAML in repo) and store secrets in a vault or pipeline secrets, not in code.
Deployment strategies
Blue-green: Two identical envs; switch traffic at once. Fast rollback by switching back. Canary: Roll out to a small % of traffic, then increase. Rolling: Replace instances gradually. Choose by risk and rollback needs; canary reduces blast radius.
Q: Blue-green vs canary — when to use which?
Blue-green: full cutover at once; simple, fast rollback (flip back). Use when you want quick, all-or-nothing releases. Canary: gradual traffic shift; if metrics degrade, only a fraction of users are affected. Use when you want to validate in production with limited risk.

📊 Observability
Logs, metrics, tracing, alerting.
Logs, metrics, tracing
Logs: Centralize (e.g. ELK, Loki); use structured format (JSON) with consistent fields (level, trace_id, message). Metrics: Counters (total requests), gauges (current connections), histograms (latency percentiles). Prometheus scrapes; Grafana dashboards. Tracing: Propagate trace_id across services (e.g. Jaeger, Zipkin) to see request flow and latency per service.
Alerting and golden signals
Alert on SLO breaches and error budget burn rate; avoid alert fatigue. Golden signals: Latency (p99), Traffic (QPS), Errors (rate, 5xx), Saturation (CPU, memory, queue depth). These four cover most service health; add business metrics as needed.
Q: What are the golden signals and why use them?
Latency, traffic, errors, saturation. They give a standard way to judge service health: is it slow (latency), overloaded (traffic/saturation), or broken (errors)? Focusing on these avoids missing critical issues and keeps dashboards and alerts consistent across teams.

📋 DevOps cheatsheet
Full quick reference for interviews.
Docker
Image (layers), container (isolated process). Multi-stage builds for small images. Volumes (persist), networks (container-to-container). Compose for local multi-container.
Kubernetes
Pod (smallest unit), Deployment (replicas, rolling update), Service (ClusterIP, NodePort, LoadBalancer), Ingress (HTTP routing, TLS). ConfigMaps, Secrets.
CI/CD and observability
CI: build, test, artifact. CD: deploy (blue-green, canary, rolling). Observability: logs (structured), metrics (Prometheus), tracing (trace_id), golden signals (latency, traffic, errors, saturation).
