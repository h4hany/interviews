// SOLID PRINCIPLES
S — Single Responsibility Principle
A class should have only
one reason to change.
The most fundamental principle. Every class, module, or function should do exactly one thing — and do it well.
The Problem Without SRP
TYPESCRIPT — VIOLATION
// ❌ BAD — UserService does TOO MUCH
class UserService {
  // Responsibility 1: Business logic
  createUser(data: UserDto): User { ... }
  validateEmail(email: string): boolean { ... }

  // Responsibility 2: Persistence (should be repository)
  saveToDatabase(user: User): void { ... }
  getUserById(id: number): User { ... }

  // Responsibility 3: Notifications (should be separate)
  sendWelcomeEmail(user: User): void { ... }
  sendSmsVerification(phone: string): void { ... }

  // Responsibility 4: Logging (cross-cutting concern)
  logUserAction(action: string): void { ... }
}
// Problem: changing the email provider forces touching UserService!
// Testing requires mocking 4 different concerns!
TYPESCRIPT — SOLID
// ✅ GOOD — each class has one job
class UserService {
  constructor(
    private repo: UserRepository,
    private notifier: UserNotifier
  ) {}

  async createUser(data: UserDto): Promise<User> {
    const user = new User(data); // business logic only
    await this.repo.save(user);
    await this.notifier.sendWelcome(user);
    return user;
  }
}

class UserRepository {       // Only: persistence
  save(user: User) { ... }
  findById(id: number) { ... }
}

class UserNotifier {          // Only: notifications
  sendWelcome(user: User) { ... }
}

// Now: change email provider → only touch UserNotifier
// Testing each class is isolated and simple
💡 ANALOGY
A chef should cook. A waiter should serve. A cashier should handle billing. If your chef also does billing and drives the delivery van, every time the billing system changes you have to retrain your chef. SRP says: keep roles separated so changes in one area don't ripple through unrelated code.
🎯 INTERVIEW QUESTIONS
Q: What is the Single Responsibility Principle?
SRP states that a class should have only one reason to change — meaning it should have only one job or responsibility. If a class handles both database persistence and email sending, changes to the email provider force you to modify the same class that handles DB logic. This creates tight coupling, harder testing, and unpredictable side effects. SRP leads to smaller, focused classes that are easier to test, reuse, and change independently.
Q: How do you identify SRP violations?
Signs of SRP violation: (1) The class name contains "And" or "Manager" covering multiple concerns, (2) The class has methods from completely different domains (save to DB AND send email), (3) The class has many private helper methods for different concerns, (4) When you describe what the class does you need to use "and" repeatedly, (5) Changes to unrelated features require modifying this class. If the class is hard to unit test because it requires mocking many different dependencies, it likely violates SRP.

// SOLID PRINCIPLES
O — Open/Closed Principle
Open for extension,
closed for modification.
Add new behavior by adding new code — not by changing existing, tested code.
The Problem Without OCP
TYPESCRIPT
// ❌ BAD — every new payment method = modify existing class
class PaymentProcessor {
  process(type: string, amount: number) {
    if (type === 'credit') {
      // credit card logic
    } else if (type === 'paypal') {
      // paypal logic
    } else if (type === 'crypto') {  // ← modifying existing code!
      // crypto logic
    }
    // Every new method = edit this file = risk breaking existing methods
  }
}

// ✅ GOOD — extend by adding new class, never modify existing
interface PaymentMethod {
  process(amount: number): Promise<void>;
}

class CreditCardPayment implements PaymentMethod {
  async process(amount: number) { /* credit logic */ }
}

class PayPalPayment implements PaymentMethod {
  async process(amount: number) { /* paypal logic */ }
}

// Adding crypto? Just add a new class — ZERO existing code changes!
class CryptoPayment implements PaymentMethod {
  async process(amount: number) { /* crypto logic */ }
}

class PaymentProcessor {
  process(method: PaymentMethod, amount: number) {
    return method.process(amount); // works for any method!
  }
}
💡 ANALOGY
A power strip is closed for modification (you can't change its wiring) but open for extension (you can plug in any device). You don't need to rewire the strip every time you want to add a new appliance. Similarly, your core classes should be "wired" once and new behavior added by plugging in new implementations.
🎯 INTERVIEW QUESTIONS
Q: What is the Open/Closed Principle?
OCP states that software entities should be open for extension but closed for modification. When requirements change or new features are needed, you should add new code rather than modifying existing, working code. This reduces the risk of introducing bugs in code that already works. OCP is typically achieved through polymorphism, interfaces, and abstract classes — new behavior is added by implementing a new class that satisfies an existing interface.
Q: How does OCP relate to the Strategy pattern?
They're closely related. The Strategy pattern is a practical implementation of OCP. Instead of if/else chains that grow with every new algorithm variant, you define an interface and implement each variant as a separate strategy class. The context class (PaymentProcessor) depends on the interface, making it closed for modification. Adding a new strategy (CryptoPayment) extends behavior without touching existing code. OCP is the principle; Strategy is one design pattern that achieves it.

// SOLID PRINCIPLES
L — Liskov Substitution Principle
Subtypes must be substitutable
for their base types.
If S is a subtype of T, objects of type T can be replaced with objects of type S without breaking the program.
The Classic Violation — Square extends Rectangle
TYPESCRIPT
// ❌ FAMOUS LSP VIOLATION
class Rectangle {
  setWidth(w: number)  { this.width = w; }
  setHeight(h: number) { this.height = h; }
  area() { return this.width * this.height; }
}

class Square extends Rectangle {
  setWidth(w: number)  { this.width = w; this.height = w; }  // BOTH!
  setHeight(h: number) { this.height = h; this.width = h; }  // BOTH!
}

// Code that works with Rectangle breaks with Square:
function doubleWidth(rect: Rectangle) {
  rect.setWidth(10);
  rect.setHeight(5);
  console.log(rect.area()); // Rectangle: 50 ✅  Square: 25 ❌ !
}
// Square CANNOT be substituted for Rectangle — LSP violated!
// A square IS-A rectangle mathematically but NOT in OOP behavior.

// ✅ FIX: Don't force inheritance. Use separate classes or interface:
interface Shape { area(): number; }
class Rectangle implements Shape { ... }
class Square implements Shape { ... }  // no inheritance!
LSP Rules
A subclass must:

✅ Not strengthen preconditions — if base accepts any string, subclass cannot require non-empty string
✅ Not weaken postconditions — if base returns positive numbers, subclass cannot return negatives
✅ Not throw new exceptions — if base doesn't throw, subclass cannot throw
✅ Not break invariants — if base guarantees balance ≥ 0, subclass must too

Rule of thumb: If you have an if instanceof check to handle a subclass differently, LSP is violated.
🎯 INTERVIEW QUESTIONS
Q: What is LSP and give a real-world violation example?
LSP says subclasses must be usable wherever their parent class is used, without breaking behavior. Classic violation: Square extends Rectangle. Mathematically a square IS a rectangle, but behaviorally it isn't — setting width on a rectangle doesn't change height, but must for a square. Code written against Rectangle breaks when given a Square. Real-world: a ReadOnlyList extending List and throwing UnsupportedOperationException on add() — LSP violated. Fix: use interfaces that accurately represent capabilities instead of forcing inheritance.

// SOLID PRINCIPLES
I — Interface Segregation Principle
No client should be forced to depend
on methods it does not use.
Prefer many small, specific interfaces over one large, general-purpose interface.
Fat Interface Problem
TYPESCRIPT
// ❌ FAT INTERFACE — forces all implementors to implement everything
interface Worker {
  work(): void;
  eat(): void;
  sleep(): void;
}

class HumanWorker implements Worker {
  work()  { console.log("working"); }
  eat()   { console.log("eating"); }
  sleep() { console.log("sleeping"); }
}

class RobotWorker implements Worker {
  work()  { console.log("working"); }
  eat()   { throw new Error("Robots don't eat!"); }  // ← forced garbage!
  sleep() { throw new Error("Robots don't sleep!"); } // ← LSP violation too!
}

// ✅ SEGREGATED — clients only depend on what they need
interface Workable  { work(): void; }
interface Feedable  { eat(): void; }
interface Sleepable { sleep(): void; }

class HumanWorker implements Workable, Feedable, Sleepable { ... }
class RobotWorker implements Workable { ... }  // ← only what it needs!

// Functions depend only on what they use:
function feedWorker(w: Feedable) { w.eat(); }    // humans only
function assignWork(w: Workable) { w.work(); }   // humans AND robots
🎯 INTERVIEW QUESTIONS
Q: What is Interface Segregation and how does it differ from SRP?
ISP focuses on INTERFACES — don't force implementors to depend on methods they don't use. Create smaller, more specific interfaces. SRP focuses on CLASSES — a class should have one reason to change. They're complementary: ISP prevents bloated interfaces that force unneeded implementations; SRP prevents classes from having too many responsibilities. ISP violation sign: implementing a method with throw new UnsupportedOperationException() — you're being forced to implement something you don't need.

// SOLID PRINCIPLES
D — Dependency Inversion Principle
Depend on abstractions,
not on concretions.
High-level modules should not depend on low-level modules. Both should depend on abstractions (interfaces).
The Problem & Solution
TYPESCRIPT
// ❌ BAD — high-level depends on low-level concrete class
class MySQLDatabase {
  query(sql: string) { /* MySQL specific */ }
}

class UserService {
  private db = new MySQLDatabase();  // ← TIGHTLY COUPLED!
  // Switching to PostgreSQL = rewrite UserService!
  // Testing = must have MySQL running!
}

// ✅ GOOD — both depend on abstraction
interface Database {            // ← the abstraction
  query(sql: string): any[];
}

class MySQLDatabase implements Database {
  query(sql: string) { /* MySQL */ }
}

class PostgresDatabase implements Database {
  query(sql: string) { /* Postgres */ }
}

class InMemoryDatabase implements Database { // for testing!
  query(sql: string) { return []; }
}

class UserService {
  constructor(private db: Database) {}  // ← depends on interface
  // Swap DB? Just inject different implementation. Zero code change.
  // Testing? Inject InMemoryDatabase. No real DB needed.
}
💡 ANALOGY
A lamp doesn't depend on a specific power plant. It depends on the electrical socket standard (abstraction). The power plant (coal, solar, nuclear) implements that standard. You can switch from coal to solar without changing your lamp. DIP says your high-level code (lamp) should depend on the interface (socket), not the implementation (power plant).
DIP vs Dependency Injection: DIP is a PRINCIPLE about depending on abstractions. Dependency Injection (DI) is a TECHNIQUE/PATTERN to achieve it — passing dependencies through constructors or setters rather than creating them inside. DI frameworks (Spring, NestJS, .NET DI) automate the wiring.
🎯 INTERVIEW QUESTIONS
Q: What is DIP and how does it enable testing?
DIP says high-level modules (business logic) should not depend on low-level modules (DB, email sender, HTTP client). Both should depend on interfaces. This enables testing by substituting real implementations with test doubles (mocks, stubs, fakes). Without DIP: testing UserService requires a real database. With DIP: inject InMemoryDatabase in tests — fast, isolated, no infrastructure needed. This is the foundation of testable design.
Q: What's the relationship between all 5 SOLID principles?
They reinforce each other: SRP keeps classes focused so they have fewer dependencies. OCP (extend, don't modify) relies on the abstractions that DIP establishes. LSP ensures substitutability so OCP actually works — you can swap implementations safely. ISP creates precise interfaces that DIP depends on — if interfaces are too fat, high-level code depends on things it doesn't need. Together they guide toward loosely coupled, highly cohesive, testable code.

// CREATIONAL PATTERNS
1️⃣ Singleton Pattern
Ensure a class has exactly one instance and provide global access to it.
Implementation
TYPESCRIPT
// Thread-safe Singleton (TypeScript/JS — single thread, simpler)
class ConfigManager {
  private static instance: ConfigManager | null = null;
  private config: Record<string, any> = {};

  private constructor() {          // ← private! prevents new ConfigManager()
    this.config = loadFromEnv();
  }

  static getInstance(): ConfigManager {
    if (!ConfigManager.instance) {
      ConfigManager.instance = new ConfigManager();
    }
    return ConfigManager.instance;
  }

  get(key: string) { return this.config[key]; }
}

// Usage
const config1 = ConfigManager.getInstance();
const config2 = ConfigManager.getInstance();
console.log(config1 === config2); // true — same instance!
When to Use & When to Avoid
✅ Good uses: Configuration manager, logger, connection pool, thread pool, caches that should be shared app-wide, registry of objects.
⚠️ Singleton is often considered an anti-pattern because: Hidden global state makes code hard to reason about. Tight coupling — dependencies not visible in constructors. Hard to test — can't inject a test double, state persists between tests. Violates SRP (manages its own lifecycle + its job). Prefer using Dependency Injection to manage single instances instead.
🎯 INTERVIEW QUESTIONS
Q: What is the Singleton pattern and why is it controversial?
Singleton ensures only one instance of a class exists and provides global access. It's controversial because: (1) It's essentially a global variable — global state makes code hard to understand and debug, (2) Creates tight coupling — classes that use the singleton are hard to test in isolation (can't inject a mock), (3) Thread safety issues in multi-threaded environments, (4) Hides dependencies — not visible in function signatures or constructors, (5) State persists between test cases. Modern approach: manage single instances via DI containers instead of the Singleton pattern.

TYPESCRIPT
// Product interface
interface Notification {
  send(message: string): void;
}

// Concrete products
class EmailNotification implements Notification {
  send(msg: string) { console.log(`Email: ${msg}`); }
}
class SMSNotification implements Notification {
  send(msg: string) { console.log(`SMS: ${msg}`); }
}
class PushNotification implements Notification {
  send(msg: string) { console.log(`Push: ${msg}`); }
}

// Factory — centralizes creation logic
class NotificationFactory {
  static create(type: 'email' | 'sms' | 'push'): Notification {
    switch(type) {
      case 'email': return new EmailNotification();
      case 'sms':   return new SMSNotification();
      case 'push':  return new PushNotification();
    }
  }
}

// Client code doesn't know or care which class was created
const notif = NotificationFactory.create('email');
notif.send("Hello!"); // works for any notification type
🎯 INTERVIEW QUESTIONS
Q: What problem does the Factory Method pattern solve?
Factory Method decouples object creation from usage. Without it, client code creates objects directly (new EmailNotification()) — tightly coupled. Changing the concrete class (or its constructor) requires changing all client code. With Factory, client code uses an interface (Notification) and the factory handles creation. Benefits: (1) Swap implementations without touching client code, (2) Centralize complex creation logic, (3) Easy to mock for testing (inject a factory that returns test doubles), (4) Follows OCP — add new notification types by adding new classes, not modifying factory heavily.
Q: Factory Method vs Abstract Factory vs Builder?
Factory Method: one factory method creates one product type. Simple, one level of abstraction. Abstract Factory: creates FAMILIES of related products (e.g., WindowsButton + WindowsCheckbox vs MacButton + MacCheckbox). One factory, many related products. Builder: constructs complex objects step by step, allowing different configurations. Used when an object has many optional parts (QueryBuilder, AlertDialog with optional title, message, buttons). Use Factory Method for simple creation, Abstract Factory for product families, Builder for complex configuration.

// CREATIONAL PATTERNS
🏗️ Abstract Factory Pattern
Provide an interface for creating families of related objects without specifying concrete classes.
Implementation — UI Toolkit Example
TYPESCRIPT
// Abstract products
interface Button   { render(): void; }
interface Checkbox { check(): void; }

// Abstract factory — interface for creating product family
interface UIFactory {
  createButton(): Button;
  createCheckbox(): Checkbox;
}

// Windows family
class WindowsButton   { render() { console.log("Windows button"); } }
class WindowsCheckbox { check()  { console.log("Windows checkbox"); } }
class WindowsUIFactory implements UIFactory {
  createButton()   { return new WindowsButton(); }
  createCheckbox() { return new WindowsCheckbox(); }
}

// Mac family
class MacButton   { render() { console.log("Mac button"); } }
class MacCheckbox { check()  { console.log("Mac checkbox"); } }
class MacUIFactory implements UIFactory {
  createButton()   { return new MacButton(); }
  createCheckbox() { return new MacCheckbox(); }
}

// Client — works with any factory, always gets consistent family
class Application {
  constructor(private factory: UIFactory) {}
  build() {
    const btn = this.factory.createButton();   // always compatible pair
    const chk = this.factory.createCheckbox();
    btn.render(); chk.check();
  }
}
🎯 INTERVIEW QUESTIONS
Q: When would you use Abstract Factory vs Factory Method?
Use Abstract Factory when you have families of related products that must be used together. UI toolkit (Windows button + Windows checkbox must match), database access layer (connection + command + transaction must be for the same DB), cross-platform widgets. Use Factory Method when you only create one type of product and need flexibility in which concrete type. Abstract Factory is essentially a set of Factory Methods working together to ensure product family consistency.

// CREATIONAL PATTERNS
🧱 Builder Pattern
Construct complex objects step by step. Eliminates telescoping constructors and makes optional configuration clean.
The Telescoping Constructor Problem
TYPESCRIPT
// ❌ TELESCOPING CONSTRUCTOR — unreadable!
new HttpRequest(url, 'POST', headers, body, timeout, retries, cache, auth);
// What's the 6th parameter?? You have to count.

// ✅ BUILDER — readable, fluent, optional steps
class HttpRequestBuilder {
  private request: Partial<HttpRequest> = { method: 'GET', timeout: 5000 };

  url(url: string)               { this.request.url = url; return this; }
  method(m: string)              { this.request.method = m; return this; }
  header(k: string, v: string)  { this.request.headers[k] = v; return this; }
  body(b: any)                   { this.request.body = b; return this; }
  timeout(ms: number)           { this.request.timeout = ms; return this; }
  withAuth(token: string)       {
    this.request.headers['Authorization'] = `Bearer ${token}`;
    return this;
  }

  build(): HttpRequest {
    if (!this.request.url) throw new Error("URL is required");
    return new HttpRequest(this.request);
  }
}

// Usage — self-documenting!
const req = new HttpRequestBuilder()
  .url('https://api.example.com/users')
  .method('POST')
  .header('Content-Type', 'application/json')
  .body({ name: 'Hany' })
  .timeout(3000)
  .withAuth(token)
  .build();
🎯 INTERVIEW QUESTIONS
Q: What problem does the Builder pattern solve?
Builder solves the "telescoping constructor" problem — constructors with many parameters (especially optional ones) become unreadable and error-prone (easy to mix up argument order). Builder provides a fluent API where each step is named and optional, making code self-documenting. Also useful when: object construction involves multiple steps that must happen in order, you want to validate the object before finalizing, or you need to create different representations of the same complex object (same builder, different directors).

🧬 Prototype Pattern
Create new objects by copying (cloning) existing ones. Avoids expensive initialization when you need many similar objects.
Implementation
TYPESCRIPT
interface Cloneable<T> { clone(): T; }

class GameCharacter implements Cloneable<GameCharacter> {
  constructor(
    public name: string,
    public health: number,
    public skills: string[],
    public equipment: { weapon: string; armor: string }
  ) {}

  clone(): GameCharacter {
    // Deep clone — copy nested objects too!
    return new GameCharacter(
      this.name,
      this.health,
      [...this.skills],                   // deep copy array
      { ...this.equipment }              // deep copy object
    );
  }
}

// Instead of expensive re-initialization, clone a template:
const warriorTemplate = new GameCharacter(
  "Warrior", 100, ["slash", "block"], { weapon: "sword", armor: "plate" }
);

// Create 1000 warriors instantly — clone, don't rebuild
const warriors = Array.from({ length: 1000 }, () => warriorTemplate.clone());
🎯 INTERVIEW QUESTIONS
Q: What is the difference between deep copy and shallow copy?
Shallow copy: creates a new object but copies only the top-level properties — nested objects are still shared references. Modifying a nested object in the copy changes the original. Deep copy: recursively copies all nested objects — the copy is completely independent. For Prototype pattern, deep copy is almost always what you want. In JavaScript: shallow = Object.assign({}, obj) or spread {...obj}. Deep = structuredClone(obj) (modern), JSON.parse(JSON.stringify(obj)) (loses functions/dates), or manual recursive cloning.

Convert the interface of a class into another interface clients expect. Make incompatible interfaces work together.

TYPESCRIPT
// Existing code expects this interface
interface Logger {
  log(message: string): void;
  error(message: string): void;
}

// Third-party library with incompatible interface
class ThirdPartyLogger {
  writeInfo(msg: string, meta: object) { ... }
  writeError(msg: string, err: Error) { ... }
}

// Adapter — bridges the gap without touching either class
class LoggerAdapter implements Logger {
  constructor(private thirdParty: ThirdPartyLogger) {}

  log(message: string) {
    this.thirdParty.writeInfo(message, { timestamp: new Date() });
  }

  error(message: string) {
    this.thirdParty.writeError(message, new Error(message));
  }
}

// Client code unchanged — just inject the adapter
const logger: Logger = new LoggerAdapter(new ThirdPartyLogger());
logger.log("Application started");
🎯 INTERVIEW QUESTIONS
Q: What is the Adapter pattern and when do you use it?
Adapter allows objects with incompatible interfaces to collaborate. Use when: integrating a third-party library with a different API, migrating legacy code gradually (adapter wraps old code, new code uses new interface), working with multiple implementations of similar functionality that have different interfaces. Real examples: adapting a payment provider API to your payment interface, wrapping a REST API client to match your repository interface, adapting old callback-based code to return Promises.

// STRUCTURAL PATTERNS
🎁 Decorator Pattern
Attach new behaviors to objects by wrapping them. An alternative to subclassing for extending functionality.
Implementation
TYPESCRIPT
interface Coffee {
  cost(): number;
  description(): string;
}

// Concrete component
class SimpleCoffee implements Coffee {
  cost() { return 1.00; }
  description() { return "Simple coffee"; }
}

// Base decorator — same interface, wraps another Coffee
abstract class CoffeeDecorator implements Coffee {
  constructor(protected coffee: Coffee) {}
  cost() { return this.coffee.cost(); }
  description() { return this.coffee.description(); }
}

// Concrete decorators — add behavior by wrapping
class MilkDecorator extends CoffeeDecorator {
  cost()        { return super.cost() + 0.25; }
  description() { return super.description() + ", milk"; }
}
class SugarDecorator extends CoffeeDecorator {
  cost()        { return super.cost() + 0.10; }
  description() { return super.description() + ", sugar"; }
}

// Compose at runtime — mix and match decorators!
let myCoffee: Coffee = new SimpleCoffee();
myCoffee = new MilkDecorator(myCoffee);
myCoffee = new SugarDecorator(myCoffee);
myCoffee = new MilkDecorator(myCoffee);  // double milk!

console.log(myCoffee.description()); // "Simple coffee, milk, sugar, milk"
console.log(myCoffee.cost());        // 1.60
💡 ANALOGY
Think of clothing layers. You start with a base outfit. You add a jacket (decorator) for warmth. Add a rain jacket over it for waterproofing. Each layer adds behavior while the person underneath stays unchanged. You can mix and match layers without changing the person or creating a separate subclass for every combination (RaincoatPerson, JacketPerson, RaincoatAndJacketPerson...).
Real-world examples: Node.js streams (pipe chains), Java I/O (BufferedReader wraps FileReader), Python decorators (@cache, @property), middleware stacks in Express/Koa, React higher-order components.
🎯 INTERVIEW QUESTIONS
Q: Decorator vs inheritance — when do you choose Decorator?
Inheritance creates a fixed hierarchy at compile time — you can't add behavior dynamically. Decorator allows dynamic composition at runtime. Problem: if you have 3 features (A, B, C) and need every combination, inheritance requires 7 subclasses (A, B, C, AB, AC, BC, ABC). With decorators: 3 decorator classes, compose any combination at runtime. Use Decorator when: you need to add behavior without modifying existing classes (OCP), behaviors should be composable in any combination, you need to add/remove behaviors at runtime.

🪟 Facade Pattern
Provide a simplified interface to a complex subsystem. Hide complexity behind a simple, clean API.
Implementation
TYPESCRIPT
// Complex subsystem — many classes with complex interactions
class VideoDecoder  { decode(file: string) { ... } }
class AudioMixer    { mix(audio: any) { ... } }
class BitrateReader  { read(file: string) { ... } }
class CodecFactory   { extract(video: any) { ... } }

// ❌ Without facade — client must know all the details
const decoder = new VideoDecoder();
const codec = new CodecFactory().extract(decoder.decode(file));
const bitrate = new BitrateReader().read(file);
const audio = new AudioMixer().mix(codec, bitrate);
// Client must know AND correctly sequence 4 subsystems

// ✅ WITH FACADE — simple interface, complexity hidden
class VideoConverterFacade {
  private decoder = new VideoDecoder();
  private mixer   = new AudioMixer();
  private reader  = new BitrateReader();
  private factory = new CodecFactory();

  convert(filename: string, format: string): string {
    // Orchestrates all subsystems internally
    const video   = this.decoder.decode(filename);
    const codec   = this.factory.extract(video);
    const bitrate = this.reader.read(filename);
    return this.mixer.mix(codec, bitrate, format);
  }
}

// Client — one method, done.
const converter = new VideoConverterFacade();
const mp4 = converter.convert("movie.avi", "mp4");
🎯 INTERVIEW QUESTIONS
Q: What is the Facade pattern and give a real-world example?
Facade provides a simplified interface to a complex subsystem. Real examples: (1) A startup() method on a Computer that internally starts CPU, memory, hard drive in the right order — client just calls computer.startup(), (2) A payment API facade that internally handles authentication, fraud detection, currency conversion, bank communication, (3) An ORM like ActiveRecord — hides complex SQL behind simple model.save(), (4) The new keyword in many frameworks that wires together many objects internally. The facade doesn't prevent access to subsystems — it just provides a simpler entry point.

// STRUCTURAL PATTERNS
🛡️ Proxy Pattern
Control access to an object. Add lazy loading, caching, logging, or access control without changing the real object.
Types & Implementation
TYPESCRIPT
interface DatabaseService {
  query(sql: string): any[];
}

// Real object — expensive
class RealDatabaseService implements DatabaseService {
  query(sql: string) {
    console.log("Hitting database...");
    return [/* expensive DB call */];
  }
}

// Caching Proxy — transparent to client
class CachingProxy implements DatabaseService {
  private cache = new Map<string, any[]>();
  constructor(private db: DatabaseService) {}

  query(sql: string) {
    if (this.cache.has(sql)) {
      console.log("Cache hit!");
      return this.cache.get(sql)!;
    }
    const result = this.db.query(sql);
    this.cache.set(sql, result);
    return result;
  }
}

// Access Control Proxy
class AuthProxy implements DatabaseService {
  constructor(private db: DatabaseService, private user: User) {}

  query(sql: string) {
    if (!this.user.hasPermission('READ'))
      throw new Error("Access denied");
    return this.db.query(sql);
  }
}

// Client uses proxy transparently — same interface!
const db: DatabaseService = new CachingProxy(new RealDatabaseService());
Proxy types: Virtual Proxy (lazy initialization — create expensive object only when needed), Caching Proxy (store results, avoid repeated work), Protection Proxy (access control), Remote Proxy (represent object in another address space), Logging Proxy (log all calls transparently).
🎯 INTERVIEW QUESTIONS
Q: What's the difference between Proxy, Decorator, and Adapter?
All three wrap an object, but for different purposes. Proxy: controls ACCESS to the real object (lazy init, caching, auth, logging) — has the same interface, client may not know it's a proxy. Decorator: ADDS new behavior to an object — stackable, composable, client knows it's decorated. Adapter: TRANSLATES one interface to another — bridges incompatible interfaces, changes the API. Memory aid: Proxy = control, Decorator = enhance, Adapter = translate.

🌳 Composite Pattern
Compose objects into tree structures. Treat individual objects and compositions uniformly.
Implementation — File System Example
TYPESCRIPT
// Common interface for both files and folders
interface FileSystemItem {
  getName(): string;
  getSize(): number;
  print(indent?: string): void;
}

// Leaf — no children
class File implements FileSystemItem {
  constructor(private name: string, private size: number) {}
  getName() { return this.name; }
  getSize() { return this.size; }
  print(indent = "") { console.log(`${indent}📄 ${this.name} (${this.size}KB)`); }
}

// Composite — has children, delegates to them
class Folder implements FileSystemItem {
  private children: FileSystemItem[] = [];

  constructor(private name: string) {}
  getName() { return this.name; }
  add(item: FileSystemItem) { this.children.push(item); }

  getSize() {                 // delegates to all children
    return this.children.reduce((sum, child) => sum + child.getSize(), 0);
  }
  print(indent = "") {
    console.log(`${indent}📁 ${this.name}/`);
    this.children.forEach(c => c.print(indent + "  "));
  }
}

// Build tree — treat files and folders identically
const root = new Folder("root");
const src  = new Folder("src");
src.add(new File("main.ts", 12));
src.add(new File("utils.ts", 5));
root.add(src);
root.add(new File("README.md", 2));

root.print();          // recursively prints tree
root.getSize();        // 19 — recursively sums all files
🎯 INTERVIEW QUESTIONS
Q: What is the Composite pattern and what problem does it solve?
Composite lets you treat individual objects and groups of objects uniformly through a common interface. It solves the problem of tree structures where you'd otherwise have to distinguish between leaf nodes and containers with if isinstance checks. Real examples: file systems (files and folders), UI component trees (widgets containing other widgets), organizational charts (employees and managers), menus (items and submenus), HTML DOM. Key insight: both leaf and composite implement the same interface, so recursive operations work uniformly.

👁️ Observer Pattern
Define a one-to-many dependency. When one object changes state, all its dependents are notified automatically.
Implementation
Subject
subscribe/notify
notifies →
Observer A
update()
Observer B
update()
Observer C
update()
TYPESCRIPT
interface Observer<T> { update(data: T): void; }

class EventEmitter<T> {
  private observers: Observer<T>[] = [];

  subscribe(obs: Observer<T>)   { this.observers.push(obs); }
  unsubscribe(obs: Observer<T>) {
    this.observers = this.observers.filter(o => o !== obs);
  }
  notify(data: T) { this.observers.forEach(o => o.update(data)); }
}

// Subject
class StockPrice extends EventEmitter<number> {
  private price = 0;

  setPrice(price: number) {
    this.price = price;
    this.notify(price);  // automatically notify all observers
  }
}

// Observers
class PriceAlert implements Observer<number> {
  update(price: number) {
    if (price > 100) console.log(`🚨 Price alert: $${price}`);
  }
}

class PriceLogger implements Observer<number> {
  update(price: number) { console.log(`📝 Logged: $${price}`); }
}

const stock = new StockPrice();
stock.subscribe(new PriceAlert());
stock.subscribe(new PriceLogger());
stock.setPrice(105); // both observers notified!
Observer is everywhere: DOM events (addEventListener), RxJS/Observables, React hooks (useEffect watching state), Redux (store subscriptions), Node.js EventEmitter, WebSocket event handlers, database triggers, Kafka consumers.
🎯 INTERVIEW QUESTIONS
Q: What is the Observer pattern and where is it used in practice?
Observer defines a one-to-many dependency — when the subject's state changes, all registered observers are automatically notified and updated. Practical uses: event systems (DOM events), reactive programming (RxJS), MVC (view observes model changes), publish-subscribe messaging (Kafka/RabbitMQ at a conceptual level), real-time notifications, change tracking. Key benefit: loose coupling — the subject doesn't know about specific observers. New observers can be added without modifying the subject (OCP).
Q: What is the difference between Observer and Pub/Sub?
Observer: subject knows about observers (direct reference). Observers subscribe directly to subject. Simpler, tighter coupling. In-process. Pub/Sub: publishers and subscribers don't know each other — there's a broker/message bus in between. More decoupled. Often distributed (Kafka, RabbitMQ). Observers are synchronous by default; Pub/Sub is typically async. Observer is a design pattern; Pub/Sub is a messaging architecture. Observer is like direct subscriptions; Pub/Sub is like a newspaper — publisher doesn't know its readers.

♟️ Strategy Pattern
Define a family of algorithms, encapsulate each, and make them interchangeable. Select the algorithm at runtime.
Implementation
TYPESCRIPT
// Strategy interface
interface SortStrategy {
  sort(data: number[]): number[];
}

// Concrete strategies
class BubbleSort implements SortStrategy {
  sort(data: number[]) { /* O(n²) for small arrays */ return [...data].sort(); }
}
class QuickSort implements SortStrategy {
  sort(data: number[]) { /* O(n log n) average */ return [...data].sort(); }
}
class MergeSort implements SortStrategy {
  sort(data: number[]) { /* O(n log n) stable */ return [...data].sort(); }
}

// Context — uses a strategy, can swap it
class Sorter {
  constructor(private strategy: SortStrategy) {}

  setStrategy(s: SortStrategy) { this.strategy = s; }

  sort(data: number[]) {
    // Select strategy dynamically based on data size
    if (data.length < 10) this.setStrategy(new BubbleSort());
    return this.strategy.sort(data);
  }
}

// Real example: payment strategies
class ShoppingCart {
  constructor(private paymentStrategy: PaymentMethod) {}

  checkout(amount: number) {
    this.paymentStrategy.process(amount);
  }
}

// Switch strategy at runtime:
const cart = new ShoppingCart(new CreditCardPayment());
cart.checkout(100);
// User switches to PayPal:
cart.paymentStrategy = new PayPalPayment();
cart.checkout(100);
🎯 INTERVIEW QUESTIONS
Q: What is the Strategy pattern and how does it relate to OCP?
Strategy encapsulates interchangeable algorithms behind a common interface, allowing them to be swapped at runtime. It implements OCP perfectly — adding a new algorithm means adding a new strategy class, not modifying the context. Use when: you have multiple variations of an algorithm (sorting, compression, pricing), you need to switch behavior at runtime based on user input/config/data, you want to eliminate large if/else or switch statements that select algorithm variants. Examples: payment methods, authentication strategies, compression algorithms, rendering engines.

📟 Command Pattern
Encapsulate a request as an object. Enables undo/redo, queuing, logging, and transactional behavior.
Implementation with Undo/Redo
TYPESCRIPT
interface Command {
  execute(): void;
  undo(): void;
}

// Receiver — the object that actually does the work
class TextEditor {
  private content = "";
  insertText(text: string) { this.content += text; }
  deleteText(len: number)  { this.content = this.content.slice(0, -len); }
  getText() { return this.content; }
}

// Concrete commands
class InsertCommand implements Command {
  constructor(private editor: TextEditor, private text: string) {}
  execute() { this.editor.insertText(this.text); }
  undo()    { this.editor.deleteText(this.text.length); }
}

// Invoker — tracks history, supports undo/redo
class CommandHistory {
  private history: Command[] = [];
  private pointer = -1;

  execute(cmd: Command) {
    this.history.splice(this.pointer + 1); // clear redo stack
    cmd.execute();
    this.history.push(cmd);
    this.pointer++;
  }

  undo() {
    if (this.pointer >= 0) {
      this.history[this.pointer--].undo();
    }
  }

  redo() {
    if (this.pointer < this.history.length - 1) {
      this.history[++this.pointer].execute();
    }
  }
}
🎯 INTERVIEW QUESTIONS
Q: What are the main use cases for the Command pattern?
Command encapsulates a request as an object, enabling: (1) Undo/Redo — store history of commands, each knows how to reverse itself, (2) Task queuing — commands can be serialized, scheduled, sent over a network, (3) Logging/auditing — log every command that changes state, (4) Macro recording — record a sequence of commands and replay them, (5) Transactional behavior — execute multiple commands, rollback all if any fails. Real examples: text editor undo, database transactions, event sourcing, task schedulers, GUI button handlers.

📋 Template Method Pattern
Define the skeleton of an algorithm in the base class, deferring some steps to subclasses.
Implementation
TYPESCRIPT
// Abstract class defines the template
abstract class DataExporter {
  // TEMPLATE METHOD — defines the algorithm skeleton
  export(data: any[]) {
    const processed = this.processData(data);   // step 1
    const formatted = this.formatData(processed); // step 2 (varies)
    this.writeOutput(formatted);                  // step 3 (varies)
    this.notifyComplete();                        // step 4 (hook - optional)
  }

  // Concrete step — same for all exporters
  private processData(data: any[]) {
    return data.filter(item => item.active);    // common logic
  }

  // Abstract steps — MUST be overridden
  protected abstract formatData(data: any[]): string;
  protected abstract writeOutput(output: string): void;

  // Hook — CAN be overridden (optional step)
  protected notifyComplete() { /* default: do nothing */ }
}

// Concrete implementations only override what differs
class CsvExporter extends DataExporter {
  protected formatData(data: any[]) {
    return data.map(row => Object.values(row).join(',')).join('\n');
  }
  protected writeOutput(output: string) { fs.writeFileSync('data.csv', output); }
}

class JsonExporter extends DataExporter {
  protected formatData(data: any[]) { return JSON.stringify(data, null, 2); }
  protected writeOutput(output: string) { fs.writeFileSync('data.json', output); }
  protected notifyComplete() { console.log("JSON export done!"); } // hook!
}
🎯 INTERVIEW QUESTIONS
Q: Template Method vs Strategy — what's the difference?
Both deal with varying algorithms, but differently. Template Method: uses INHERITANCE — the algorithm skeleton is in the base class, subclasses override specific steps. Variation happens at compile time via class hierarchy. Strategy: uses COMPOSITION — the algorithm is encapsulated in a separate strategy object, injected at runtime. "Prefer composition over inheritance" — Strategy is more flexible. Template Method is simpler when you just need a few variants of the same process. Strategy allows runtime switching; Template Method doesn't.

🔄 Iterator Pattern
Provide a way to sequentially access elements of a collection without exposing its underlying structure.
Implementation
TYPESCRIPT
// Custom iterator for a binary tree (in-order traversal)
class TreeNode<T> {
  constructor(
    public value: T,
    public left?: TreeNode<T>,
    public right?: TreeNode<T>
  ) {}
}

// Using JavaScript generator (built-in iterator protocol)
class BinaryTree<T> {
  constructor(private root: TreeNode<T>) {}

  *[Symbol.iterator](): Iterator<T> {   // implements iterable protocol
    yield* this.inOrder(this.root);
  }

  private *inOrder(node?: TreeNode<T>): Generator<T> {
    if (!node) return;
    yield* this.inOrder(node.left);
    yield node.value;
    yield* this.inOrder(node.right);
  }
}

// Client uses for...of — doesn't know about tree internals
const tree = new BinaryTree(/* ... */);
for (const value of tree) {
  console.log(value); // in-order traversal without knowing it's a tree
}
// Also works: [...tree], Array.from(tree), tree.forEach()
🎯 INTERVIEW QUESTIONS
Q: What is the Iterator pattern and where does JavaScript use it natively?
Iterator provides a standardized way to traverse a collection (array, tree, graph, set) without knowing its internal structure. JavaScript has the Iterator Protocol built-in: any object with a [Symbol.iterator] method is iterable. Arrays, Strings, Maps, Sets, Generators are all iterables. This is why for...of, spread (...), destructuring, Array.from(), and Promise.all() work on all these types — they all speak the same iterator protocol. Implementing [Symbol.iterator] on your custom class makes it work with all these built-in features automatically.
