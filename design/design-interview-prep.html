🔧 SOLID Principles
SRP, OCP, LSP, ISP, DIP.
The five principles
SRP (Single Responsibility): A class should have one reason to change; one responsibility. OCP (Open/Closed): Open for extension (new behavior via new code), closed for modification (avoid changing existing code). Use abstractions and composition. LSP (Liskov Substitution): Subtypes must be substitutable for their base type without breaking callers. ISP (Interface Segregation): Prefer many specific interfaces over one fat interface so clients do not depend on methods they do not use. DIP (Dependency Inversion): Depend on abstractions (interfaces) not concretions; high-level modules should not depend on low-level details.
Q: What does Open for extension, closed for modification mean?
You add new behavior by adding new code (e.g. new class, new implementation of an interface), not by editing existing classes. Achieve it with polymorphism, strategy-style injection, and stable interfaces so existing callers keep working.

🏭 Creational Patterns
Singleton, Factory, Abstract Factory, Builder, Prototype.
Patterns
Singleton: Ensure one instance globally (or per scope); control creation and access. Factory Method: Delegate object creation to a method (or subclass); encapsulate which concrete type is created. Abstract Factory: Create families of related objects (e.g. UI widgets for one theme) without coupling to concrete classes. Builder: Step-by-step construction; same build process can produce different representations; fluent API. Prototype: Copy an existing instance instead of creating from scratch; useful when creation is expensive or when you need a clone.
Q: When would you use Builder instead of a constructor with many params?
When you have many optional parameters or when construction is complex and you want a clear, readable API. Builder avoids telescoping constructors and makes the intent of each step explicit. Good for config objects, DTOs, and immutable objects with many fields.

🧱 Structural Patterns
Adapter, Decorator, Proxy, Facade, Composite.
Key structural patterns
Adapter: Wrap an existing class so it matches another interface the client expects; integrate legacy or third-party code. Decorator: Wrap an object to add behavior dynamically; same interface, additional responsibility (e.g. logging, caching). Proxy: Stand-in that controls access: lazy init, access control, logging, remote proxy. Facade: Single interface to a set of interfaces in a subsystem; simplify usage. Composite: Build tree structures; treat leaf and composite nodes uniformly so clients can work with the tree without branching.
Q: Adapter vs Decorator — what is the difference?
Adapter changes the interface of an object so it can be used where a different interface is expected; the goal is compatibility. Decorator keeps the same interface and adds behavior (e.g. logging, caching); the goal is to extend functionality transparently.

🔄 Behavioral Patterns
Observer, Strategy, Command, State.
Key behavioral patterns
Observer: One-to-many dependency; when the subject changes, observers are notified. Used in event systems, reactive UIs. Strategy: Encapsulate algorithms; make them interchangeable so the client can switch behavior (e.g. sort strategy, payment method). Command: Encapsulate a request as an object; supports undo, queueing, logging. State: Object behavior changes with internal state; avoid large switch/if on state. Iterator, Template Method (skeleton in base class, steps overridden), and Chain of Responsibility are also common.
Q: When do you use Strategy pattern?
When you have multiple algorithms or behaviors that are interchangeable and you want to avoid conditionals (e.g. different sort strategies, payment methods, compression algorithms). The client depends on an abstraction and receives the concrete strategy via injection or config.

📐 Object-Oriented Design
Classes, relationships, design process.
Design process
(1) Clarify requirements: Scope, constraints, who uses the system. (2) Identify main entities: Nouns often become classes (e.g. Vehicle, ParkingSpot, Elevator). (3) Relationships: Has-a (composition, aggregation), is-a (inheritance). (4) Key methods: What each entity does; public API. (5) Design for change: Apply SOLID; use patterns where they fit (e.g. State for elevator, Strategy for pricing). Consider edge cases and concurrency (e.g. two cars for one spot).
Q: How do you approach designing a parking lot?
Clarify: types of vehicles, spot types, pricing, entry/exit flow. Entities: ParkingLot, Level, ParkingSpot, Vehicle (or Car, Bike), Ticket. Relationships: Lot has Levels, Level has Spots, Spot can hold Vehicle, Ticket links Vehicle to Spot and time. Methods: findAvailableSpot(), park(), unpark(), calculateFee(). Consider thread safety if multiple entries at once.

📋 Design & OOD cheatsheet
Full quick reference for interviews.
SOLID
SRP (one reason to change), OCP (extend not modify), LSP (substitutable subtypes), ISP (small interfaces), DIP (depend on abstractions).
Patterns
Creational: Singleton, Factory, Builder. Structural: Adapter, Decorator, Proxy, Facade, Composite. Behavioral: Observer, Strategy, Command, State.
OOD process
Requirements → entities → relationships (has-a, is-a) → key methods → SOLID and patterns. Use cases: parking lot, elevator, deck of cards, etc.
