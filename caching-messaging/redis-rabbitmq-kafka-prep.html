// REDIS
⚡ What is Redis?
Remote Dictionary Server — the world's most popular in-memory data store. Used by Twitter, GitHub, Snapchat, Craigslist and millions of others.
Why is Redis so Fast?
1. In-memory: All data lives in RAM. RAM access = ~100ns. Disk access = ~100ms. That's 1000x faster.

2. Single-threaded event loop: No thread context switching. No mutex locks. One command at a time, processed atomically.

3. Simple data structures: Optimized C implementations of strings, lists, sets, sorted sets. No query parsing overhead.

4. Non-blocking I/O: Uses epoll/kqueue to handle thousands of connections with one thread.
💡 ANALOGY
Redis is like a super-fast whiteboard on your desk. Your database is like a filing cabinet in a different building. For frequently-needed info (who's logged in, what's trending), you write it on the whiteboard. Instant access. When the office closes (restart), the whiteboard is cleared (unless you configured persistence).
When to Use Redis
Session Storage Caching DB Queries Rate Limiting Leaderboards Distributed Locks Pub/Sub Job Queues Real-time Analytics
⚠️ Don't use Redis as a primary database unless you can afford data loss or have persistence configured. Redis is best as a complementary layer alongside a durable DB (Postgres, MySQL, MongoDB).
🎯 TOP INTERVIEW QUESTIONS
Q: Why is Redis single-threaded but still fast?
Redis is single-threaded for command processing, which eliminates race conditions and context switching overhead. It's fast because: (1) All operations happen in RAM (not disk), (2) Operations are O(1) or O(log n) due to optimized data structures, (3) Network I/O uses epoll (handles thousands of connections with one thread), (4) No transaction overhead. Note: Redis 6+ uses multiple I/O threads for networking, but command processing remains single-threaded.
Q: What happens to Redis data when the server restarts?
By default, Redis is in-memory only — all data is lost on restart. Options: (1) RDB (Redis Database) — periodic snapshots to disk (configurable interval, e.g., every 5 min or every 1000 writes). Data loss up to last snapshot interval. (2) AOF (Append Only File) — logs every write command. More durable, larger files. Rewrite/compact periodically. (3) Both combined — RDB for fast restarts, AOF for durability.

🗂️ Redis Data Structures
Redis is more than a cache — it's a data structure server. Each type is purpose-built for specific use cases.
The 5 Core Types
Type Commands Use Case Complexity
String GET, SET, INCR, APPEND Cache, counters, sessions O(1)
Hash HGET, HSET, HGETALL User objects, config O(1) per field
List LPUSH, RPOP, LRANGE Message queues, activity feeds O(1) push/pop
Set SADD, SMEMBERS, SINTER Tags, unique visitors, follows O(1) add/check
Sorted Set (ZSet) ZADD, ZRANGE, ZRANK Leaderboards, priority queues O(log n)
Practical Examples
REDIS CLI — REAL PATTERNS 
# String — simple cache with TTL
SET user:123:name "Hany" EX 3600
GET user:123:name

# Hash — store user object (more memory-efficient than JSON string)
HSET user:123 name "Hany" email "h@ex.com" age 28
HGET user:123 name
HGETALL user:123

# Sorted Set — real-time leaderboard
ZADD game:scores 1500 "alice"
ZADD game:scores 2300 "bob"
ZADD game:scores 1800 "hany"
ZRANGE game:scores 0 -1 WITHSCORES REV  # top players
ZRANK game:scores "hany"               # rank of "hany"

# INCR — atomic counter (rate limiting, view counts)
INCR page:views:home
INCRBY user:123:points 50

# Set — track unique users
SADD daily:visitors "user:1" "user:2" "user:3"
SCARD daily:visitors  # count

# List — simple job queue (BRPOP blocks until item available)
LPUSH job:queue '{"type":"email","to":"h@x.com"}'
BRPOP job:queue 0  # worker pops with blocking wait
Advanced Types
Type What it does Use Case
Bitmap Bit-level operations on strings Daily active users, feature flags per user
HyperLogLog Approximate unique count (0.81% error) Count unique visitors, distinct search queries
Stream Append-only log with consumer groups Event sourcing, lightweight Kafka alternative
Geo Store/query geospatial data Find nearby restaurants, drivers
HyperLogLog magic: Uses only 12KB of memory to count up to 2^64 unique items with ~1% error. A regular set counting 100M unique items would use GBs. Use PFADD/PFCOUNT for massive scale cardinality estimation.
🎯 TOP INTERVIEW QUESTIONS
Q: How would you implement a real-time leaderboard with Redis?
Sorted Set (ZADD/ZRANGE): ZADD leaderboard {score} {user_id}. Get top 10: ZRANGE leaderboard 0 9 WITHSCORES REV. Get user rank: ZRANK leaderboard user_id. Increment score: ZINCRBY leaderboard 100 user_id. This is O(log N) per update and O(log N + K) per range query. Perfect for gaming leaderboards, content upvotes, trending topics. Example: Reddit's hot posts use a sorted set with a score calculated from votes + time decay.
Q: What is the difference between Redis Hash and storing JSON as a String?
Hash: HSET user:1 name "Hany" age 28 — stores fields separately, can update one field (HSET user:1 age 29) without loading the whole object. More memory efficient for small objects. String (JSON): SET user:1 '{"name":"Hany","age":28}' — must parse/serialize full JSON on every read/write, atomic updates require read-modify-write cycle. Use Hash when you frequently access/update individual fields. Use String (JSON) when you almost always need the full object.

🔧 Redis Patterns
The production patterns every backend developer must know.
Distributed Lock (Redlock)
NODE.JS — DISTRIBUTED LOCK 
// SET NX EX = Set if Not eXists, with EXpiry
// This is ATOMIC — no race condition possible

async function acquireLock(lockKey, ttlMs) {
  const token = generateUUID();
  const result = await redis.set(lockKey, token, {
    NX: true,  // Only set if not exists
    PX: ttlMs  // Auto-expire (prevents deadlock)
  });
  return result === 'OK' ? token : null;
}

async function releaseLock(lockKey, token) {
  // Lua script ensures atomic check-and-delete
  // Don't delete if we don't own the lock!
  const script = `
    if redis.call('get', KEYS[1]) == ARGV[1] then
      return redis.call('del', KEYS[1])
    else return 0 end`;
  await redis.eval(script, [lockKey], [token]);
}
Rate Limiting
REDIS — SLIDING WINDOW RATE LIMIT 
-- Lua script (atomic) — sliding window rate limiter
-- Key: user:{id}:ratelimit | Limit: 100 requests/minute
local key = KEYS[1]
local now = tonumber(ARGV[1])
local window = 60000  -- 1 minute in ms
local limit = 100

redis.call('ZREMRANGEBYSCORE', key, 0, now - window)
local count = redis.call('ZCARD', key)
if count < limit then
  redis.call('ZADD', key, now, now)
  redis.call('EXPIRE', key, 60)
  return 1  -- allowed
else
  return 0  -- rate limited
end
Pub/Sub
NODE.JS — REDIS PUB/SUB 
// Publisher — notify all subscribers of a new message
await publisher.publish('chat:room:42', JSON.stringify({
  user: 'Hany', text: 'Hello!', ts: Date.now()
}));

// Subscriber — receive messages in real-time
await subscriber.subscribe('chat:room:42', (msg) => {
  const data = JSON.parse(msg);
  io.to('room42').emit('message', data); // forward to WebSocket
});

// Pattern subscribe — match multiple channels
await subscriber.psubscribe('chat:room:*', (msg, channel) => { ... });
⚠️ Redis Pub/Sub is fire-and-forget. If a subscriber is offline when a message is published, the message is LOST. For durable messaging, use Redis Streams (XADD/XREAD) which stores messages and allows replay, or use RabbitMQ/Kafka.
🎯 TOP INTERVIEW QUESTIONS
Q: How do you implement a distributed lock with Redis?
Use SET key value NX PX ttl (atomic: set only if not exists, with expiry). NX ensures only one process gets the lock. PX ensures automatic release if the holder crashes. Always use a unique token as the value — when releasing, use a Lua script to atomically check that you own the lock before deleting it. This prevents releasing someone else's lock. For multi-node Redis, use the Redlock algorithm (acquire lock on majority of N Redis nodes).
Q: What is Redis pipelining and when should you use it?
Pipelining sends multiple commands in one network round trip without waiting for each response. Instead of 100 round trips (each ~1ms = 100ms total), you send all 100 commands at once and receive all responses together (~1ms total). Use when: batch operations (bulk insert, bulk fetch), when commands don't depend on each other's results. Don't use when commands depend on previous results (use Lua scripts instead). Node.js: redis.pipeline().set().get().exec()

Making Redis durable and scalable — RDB, AOF, Sentinel, and Redis Cluster.
Persistence Options
Option How Data Loss Risk Performance
No persistence All in memory only All data on crash Fastest
RDB (Snapshot) Fork process, dump to disk every N min Up to last snapshot interval Good
AOF (Append-Only) Log every write command Up to last fsync (1s with everysec) Slightly slower
RDB + AOF Both Minimal (AOF recovery) Combined overhead
High Availability: Sentinel vs Cluster
Aspect Redis Sentinel Redis Cluster
Purpose Automatic failover (HA) Horizontal scaling + HA
Data All data on primary (replicated) Data sharded across 16384 slots
Scale Vertically (bigger machine) Horizontally (more nodes)
Failover Automatic primary promotion Automatic (per shard)
Use when Need HA, data fits on one machine Data too large for one machine
🎯 TOP INTERVIEW QUESTIONS
Q: What is Redis Cluster and how does it shard data?
Redis Cluster splits the keyspace into 16384 hash slots. Each node handles a range of slots. Key's slot = CRC16(key) % 16384. Minimum 3 primary nodes (each with a replica = 6 nodes total). Client libraries are cluster-aware: they compute the slot, send to the right node. Multi-key operations require all keys on the same slot (use hash tags: {user:1}:name and {user:1}:email share slot because of {user:1}). Cluster handles resharding when adding/removing nodes.

// RABBITMQ
🐰 What is RabbitMQ?
A traditional message broker implementing AMQP. Routes messages from producers to consumers via a flexible exchange/queue system.
Core Concepts
Producer: Publishes messages to an Exchange
Exchange: Receives messages and routes them to queues based on routing rules
Queue: Buffer that stores messages until consumed
Consumer: Reads messages from queues
Binding: Link between Exchange and Queue with a routing key
💡 ANALOGY
RabbitMQ is like a post office. Producer = letter writer. Exchange = sorting department (decides which mailbox/queue to put the letter in). Queue = mailbox. Consumer = person who picks up mail. The routing key is like the address on the envelope — the sorting department uses it to decide where it goes.
RabbitMQ vs Redis Queues
Feature RabbitMQ Redis List Queue
Message routing Complex (exchanges, bindings) Simple (key = queue name)
Acknowledgments Built-in ACK/NACK Manual (LMOVE + confirm)
Dead letter queues Native support Manual implementation
Message persistence Durable queues + persistent messages Via Redis persistence (RDB/AOF)
Complex routing Topic, fanout, headers exchanges Only key-based
Management UI Built-in web UI RedisInsight (separate)

🔀 Exchanges & Routing
The heart of RabbitMQ — how messages are routed to the right queues.
Exchange Types
Exchange Routing Rule Use Case
Direct Route to queue with exact routing key match Task distribution, work queues
Topic Route by pattern (* = one word, # = many words) Selective pub/sub, log routing
Fanout Route to ALL bound queues (ignores routing key) Broadcast, notifications to all services
Headers Route based on message header attributes Complex routing without key string
Topic Exchange Pattern Example
RABBITMQ — TOPIC ROUTING
# Pattern: {service}.{action}.{region}
# Routing key examples:
  user.created.us-east
  order.shipped.eu-west
  payment.failed.us-east

# Queue bindings:
  audit.queue    → "#"                 (ALL messages)
  us.queue       → "*.*.us-*"          (all US events)
  payment.queue  → "payment.#"         (all payment events)
  critical.queue → "*.failed.*"        (all failures)

# "payment.failed.us-east" goes to:
# → audit.queue (# matches all)
# → us.queue (*.*.us-* matches)
# → payment.queue (payment.# matches)
# → critical.queue (*.failed.* matches)
🎯 TOP INTERVIEW QUESTIONS
Q: What's the difference between Direct, Topic, and Fanout exchanges?
Direct: routes message to queue(s) where binding key = routing key (exact match). Use for point-to-point or load distribution between workers. Topic: routes by pattern matching (* for one word, # for zero or more words). Use for pub/sub where consumers subscribe to subsets of messages. Fanout: copies message to ALL bound queues, ignores routing key entirely. Use for broadcasting events that all services need (e.g., "user signed up" needs to notify email service, analytics service, and recommendation service simultaneously).

// RABBITMQ
🛡️ Patterns & Reliability
ACK/NACK, dead letter queues, persistence — making RabbitMQ bulletproof.
Acknowledgments — The Key to Reliability
NODE.JS — ACK/NACK PATTERN 
// Consumer with manual acknowledgment
channel.consume('task.queue', async (msg) => {
  if (!msg) return;

  try {
    const task = JSON.parse(msg.content.toString());
    await processTask(task);
    channel.ack(msg); // ✅ Success — remove from queue
  } catch (err) {
    const retries = msg.properties.headers['x-retries'] || 0;
    if (retries < 3) {
      // Requeue with retry count
      channel.nack(msg, false, false); // → goes to dead letter queue
    } else {
      channel.nack(msg, false, false); // max retries reached → DLQ
    }
  }
}, { noAck: false }); // manual ack mode!
noAck: false (manual ack) is critical for at-least-once delivery. If consumer crashes before ACKing, RabbitMQ automatically redelivers the message to another consumer. With noAck: true (auto-ack), message is deleted as soon as delivered — if processing crashes, message is LOST.
Dead Letter Queue (DLQ)
Messages go to DLQ when: rejected with nack(..., false), TTL expires, queue is full.

Set up DLQ as exchange + queue. Configure main queue with x-dead-letter-exchange argument. Messages in DLQ can be inspected, retried manually, or processed by a separate error handler.
🎯 TOP INTERVIEW QUESTIONS
Q: How do you ensure a message is not lost in RabbitMQ?
Three layers of durability: (1) Durable queue — queue survives broker restart (durable: true when declaring); (2) Persistent message — message written to disk (deliveryMode: 2 in properties); (3) Consumer ACK — message only removed from queue after explicit channel.ack(). All three are needed for zero message loss. Also consider publisher confirms (confirmSelect) — broker acknowledges that it has persisted the message, so producer knows it's safe.

// KAFKA
📡 What is Kafka?
A distributed event streaming platform. Not a message queue — a distributed commit log that retains events and enables replay.
Kafka's Key Insight
Traditional message queues delete messages after consumption. Kafka retains messages for a configurable period (default 7 days). This enables:

✅ Multiple independent consumers reading the same events
✅ Replay — reprocess historical events
✅ Event sourcing — the event log IS your source of truth
✅ Massive throughput — millions of events/sec via sequential disk I/O
💡 ANALOGY
RabbitMQ is like a to-do list — items are done and checked off. Kafka is like a bank statement / transaction log. Every transaction is permanently recorded. Multiple people (services) can read the same statement. You can replay from any date. The log is the truth.
🎯 TOP INTERVIEW QUESTIONS
Q: Why is Kafka so fast despite writing to disk?
Kafka achieves high throughput via: (1) Sequential disk writes — always appending to end of log file (sequential I/O = ~500MB/s vs random I/O = ~100MB/s); (2) OS page cache — Kafka relies heavily on OS file system cache (data lives in RAM cache most of the time); (3) Zero-copy transfer — sendfile() syscall sends data from disk to network without copying through userspace; (4) Batching — messages accumulated and sent in batches; (5) Partitions — parallel I/O across multiple disks/nodes. Result: millions of messages/sec on commodity hardware.

// KAFKA
🏗️ Kafka Architecture
Topics, partitions, brokers, replication — the internals that make Kafka scale.
Core Components
Topic: Named category for messages (like a DB table). E.g., "user-events", "orders".

Partition: A topic is split into N partitions. Each partition = an ordered, immutable, append-only log. Partitions enable parallelism and scaling.

Offset: Each message in a partition has a sequential ID (offset: 0, 1, 2...). Consumers track their offset to know where they are.

Broker: A Kafka server. Cluster has multiple brokers. Each partition has one leader broker (handles all reads/writes) and N-1 replica followers.

Replication Factor: How many copies of each partition. RF=3 means 1 leader + 2 followers. Tolerates RF-1 = 2 broker failures.
Partitioning Strategy
KAFKA — PRODUCER PARTITION ASSIGNMENT
// With key: partition = hash(key) % numPartitions
// Same key ALWAYS goes to same partition → ordering guaranteed

producer.send({
  topic: 'user-events',
  key: 'user:12345',   // all events for user 12345 → same partition
  value: JSON.stringify({ action: 'login', ts: Date.now() })
});

// Without key: round-robin across partitions
// Higher throughput, but no ordering guarantee

// Topic config: 'user-events' with 8 partitions, RF=3
// = 8 parallel consumer workers maximum
// = data on 3 brokers (survives 2 broker failures)
🎯 TOP INTERVIEW QUESTIONS
Q: How does Kafka handle replication and leader election?
Each partition has 1 leader and N replicas. Producers/consumers always talk to the leader. Replicas pull from leader and stay in sync (ISR — In-Sync Replicas list). If leader fails: Kafka controller (or KRaft in modern versions) detects it via heartbeat timeout, elects new leader from ISR list, updates metadata. Unclean leader election (promote non-ISR) risks data loss but improves availability. acks=all means producer waits for all ISR replicas to confirm — strongest durability guarantee.
Q: How many partitions should a topic have?
Partitions = max parallelism. More partitions = more throughput but more overhead. Rule of thumb: start with target throughput ÷ single-partition throughput (single partition does ~10-100MB/s). Also cap at max consumers you'll ever want (consumer group can't have more active consumers than partitions). Example: 1GB/s throughput, 50MB/s per partition = 20 partitions. Tip: overpartition slightly (2-3x expected need) since adding partitions later is easy but rebalancing is disruptive.

// KAFKA
👥 Consumers & Groups
Consumer groups enable parallel processing and independent consumption of the same topic by multiple services.
Consumer Groups
A consumer group = multiple consumers sharing the work of consuming a topic.

Key rule: Each partition is consumed by exactly one consumer within a group. If you have 8 partitions and 4 consumers → each consumer handles 2 partitions. If 8 consumers → 1 partition each. If 9 consumers → 1 is idle.

Multiple groups: Kafka tracks each group's offset independently. Group A (analytics) and Group B (notifications) can consume the same "user-events" topic completely independently at their own pace.
💡 ANALOGY
A YouTube video (Kafka topic) can be watched by multiple viewers (consumer groups) independently. Pause/resume doesn't affect other viewers. Within a group of friends watching together on one screen (consumer group), only one of them can watch each segment — they split the work.
Commit Strategies
Strategy How Risk
Auto-commit (enable.auto.commit=true) Commits offset every N ms automatically Can commit before processing complete → message loss on crash
Manual commit (sync) commitSync() after processing At-least-once. Slower (blocks).
Manual commit (async) commitAsync() after processing At-least-once. May commit out of order.
Transactional Commit offset + DB write in one transaction Exactly-once. Complex setup.
🎯 TOP INTERVIEW QUESTIONS
Q: What is consumer lag and how do you handle it?
Consumer lag = difference between latest offset in a partition and the consumer's committed offset. High lag means consumer is falling behind producers. Solutions: (1) Scale consumers (add more consumers to group, up to partition count); (2) Increase consumer parallelism (multiple threads per consumer); (3) Optimize processing logic (batch DB writes, async processing); (4) Scale partitions (more partitions = more parallel consumers); (5) Alert on lag > threshold. Monitor with kafka-consumer-groups.sh or Kafka monitoring tools (Confluent Control Center, Burrow, Grafana).

Outbox Pattern — Reliable Event Publishing
SQL — TRANSACTIONAL OUTBOX
-- Problem: Writing to DB and publishing to Kafka is two operations
-- If Kafka publish fails after DB write, event is lost!

-- Solution: Write event to DB outbox table IN SAME TRANSACTION
BEGIN;
  INSERT INTO orders (id, user_id, total) VALUES (1, 42, 99.99);
  INSERT INTO outbox (aggregate_id, event_type, payload, published)
    VALUES (1, 'ORDER_CREATED', '{"id":1,"total":99.99}', false);
COMMIT;

-- Separate relay process reads unpublished outbox events
-- and publishes to Kafka, then marks as published
-- Debezium CDC can automate this by watching DB changes
🎯 TOP INTERVIEW QUESTIONS
Q: What is event sourcing and how does Kafka support it?
Event sourcing stores state changes as a sequence of events rather than current state. E.g., instead of "account balance = $100", store: [deposited $200, withdrew $50, withdrew $50]. Current state is computed by replaying events. Kafka is perfect: immutable event log with long retention = natural event store. Replaying from offset 0 rebuilds any state. Benefits: full audit trail, debugging, time travel (what was state at 2pm?). Used by banks, e-commerce (order state machine), gaming (player action log).
